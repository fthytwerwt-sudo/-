## 6. mechanism_landability_summary（机制落地性总览）

- 当前最大问题：项目已经把“机制名、闸门、卡片字段、禁止项”写得很完整，但中间缺一层统一的 `Mechanism Inference Layer（机制推理层）`，导致很多机制只能防止误写，不能稳定指导 AI 根据现场信号主动选动作。
- 机制总数：14
- 已有较完整推理链的机制数：4
- 有局部推理链但缺统一函数的机制数：5
- 缺推理函数的机制数：10
- 最影响落地的缺口：`editing_decision_pack（剪辑决策包）` 缺“重要点识别 -> 画面动作选择 -> 不伤证据链验证”的推理函数；其次是 `content_route_card（内容路由卡）` 与 `quality_lock_card（质量锁卡）` 缺统一信号分类和状态推测。

当前机制状态粗分：

| layer | mechanisms |
| --- | --- |
| rule_name_only（只有规则名） | 少量产品化 / 工作包触发仍偏方向词，不适合作为当前修补重点 |
| gate_only（只有闸门） | route_decision 部分字段、Completion Relay Gate、quality_lock_card 部分规则 |
| process_chain（有流程链） | OPC 7 层闭环、文案生产流程、execution_supply_pack family、DeepSeek 深度配合流程 |
| signal_inference（有信号推理） | review_loop 指标体系、content_validation V1 声音 / 动作基线、DeepSeek readiness 三态 |
| action_selection（有动作选择） | review_loop 数据倒推修改方向、large_task_gate lane / parallel 降级、DeepSeek fallback / blocked |
| validation_loop（有验证闭环） | review_loop 24h / 72h / 7d、Completion Relay remaining_work_check、DeepSeek readiness_check |
| feedback_update（有反馈更新） | review_loop 单变量更新最强；其他机制多停在日志回流或状态守护 |

## 11. editing_inference_gap（中段剪辑推理缺口）

| signal | inferred_state | expected_action | validation | current_status | missing_part |
| --- | --- | --- | --- | --- | --- |
| 画面中有关键按钮、关键结果、关键数字，但原画面字号小 | 观众可能看不清关键证据 | `zoom_in` 或 `crop_focus` | 放大后 OCR / 人眼可读，且仍知道所在页面 | missing | 缺“字号 / 证据重要度 -> 放大”的判断函数 |

## 17. mechanism_inference_layer_blueprint（机制推理层蓝图）

`Mechanism Inference Layer（机制推理层）` = 把机制从“规则 / 闸门 / 卡片名”升级为“AI 能根据现场信号判断下一步动作”的中间层。

### 17.1 `trigger_router（触发路由器）`

作用：看见什么输入 / 状态 / 缺口 / 冲突，就触发哪个机制。

建议输入：

- 用户任务类型
- 当前项目阶段
- 素材是否足够
- 文案是否已定稿
- 是否涉及剪辑 / reference / DeepSeek / 复盘 / 状态文件
- 是否有 missing fields / rule conflict / after_read_gap

建议输出：

- `content_route_card_required`
- `quality_lock_card_required`
- `editing_decision_pack_required`
- `review_variable_card_required`
- `deepseek_supply_required`
- `completion_relay_gate_required`

### 17.2 `signal_taxonomy（信号分类）`

作用：把素材信号、文案信号、平台反馈、执行日志、用户反馈分成可判断类别。

建议分类：

| signal class | examples |
| --- | --- |
| `material_signal（素材信号）` | 录屏清晰度、关键画面、OCR 文字、前后差、遮挡、卡顿 |
| `copy_signal（文案信号）` | 内容类型、验证目标、口播功能、结果差、观众任务 |
| `execution_signal（执行信号）` | 技术验证、供料来源、missing_context、blocked_if、remaining_work |
| `quality_signal（质量信号）` | AI 感、声音、节奏、证据强度、画面可读性、卡片密度、人感 |
| `platform_signal（平台信号）` | 播放、留存、完播、收藏、涨粉、私信、咨询 |
| `user_feedback_signal（用户反馈信号）` | 用户审片、ChatGPT 复审、内容不舒服、像 demo、质量不够 |

### 17.3 `state_inference（状态推测）`

作用：根据多个信号推测当前状态。

建议状态：

- `evidence_insufficient（证据不足）`
- `viewer_may_miss_key_point（观众可能漏掉重点）`
- `middle_structure_shortboard（中段结构短板）`
- `voice_or_human_feel_shortboard（声音或人感短板）`
- `route_conflict_with_reference（当前路由和 reference 冲突）`
- `data_missing_waiting（数据不足等待）`
- `local_context_enough（本地上下文足够）`
- `after_read_gap_exists（读完仍有缺口）`
- `must_block（必须阻断）`

### 17.4 `action_policy（动作选择）`

作用：根据状态选择动作。

建议动作：

- `zoom_in`
- `crop_focus`
- `highlight_box`
- `insert_card`
- `split_compare`
- `do_not_touch`
- `rework_copy`
- `rework_route`
- `rework_quality`
- `wait_for_data`
- `mark_missing_fields`
- `create_supply_request`
- `fallback_local_only`
- `blocked`

### 17.5 `validation_rule（验证规则）`

作用：动作后如何判断是否有效。

建议规则：

- 剪辑动作后：关键点是否更清楚、证据链是否未被裁断、画面是否仍可理解。
- 路由动作后：中段主体是否仍由真实素材承载，API 图 / PPT 是否没有替代证据。
- 质量动作后：是否没有一票否决项，是否没有把技术通过写成内容通过。
- 复盘动作后：是否只改一个变量，是否知道改完看哪个指标。
- 供料动作后：是否未读取 secret，是否清楚 `deepseek_passed / fallback / blocked`。

### 17.6 `feedback_update（反馈更新）`

作用：结果如何更新下一轮判断，避免每次都从零调。

建议输出：

- `mechanism_feedback_entry（机制反馈条目）`
- `signal_to_state_update（信号到状态映射更新）`
- `action_effectiveness_note（动作有效性记录）`
- `next_round_default_check（下一轮默认检查）`
- `do_not_repeat_mistake（禁止重复错误）`
