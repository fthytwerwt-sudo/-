# Docker 工作台与录制减负规则

## 1. 文件定位
本文件只负责 3 件事：
- 定义 Docker 最小工作台的角色
- 定义中间层工作流顺序
- 定义录制减负的默认做法

它不负责：
- 当前待发对象切换
- Perplexity / MCP / 外部联网搜索
- 最终成片自动化承诺

## 2. Docker 工作台的正确角色
当前 Docker 工作台不是：
- 替代判断
- 替代用户录制
- 全自动成片系统

当前 Docker 工作台只负责：
- 把 `01_场景简报.md` 压成 `02_工作包正文.md`
- 把工作包正文继续压成 `03_Prompt包.md`
- 把 prompt 包继续压成 `04_录制计划.md`
- 把录制计划继续压成 `05_回审清单.md`
- 在需要时补一份结果差对比骨架和最小执行日志骨架

## 3. 当前最小工作流顺序
当前最小工作流固定为：
1. `01_场景简报.md`
2. `02_工作包正文.md`
3. `03_Prompt包.md`
4. `04_录制计划.md`
5. `05_回审清单.md`

补充输出可包括：
- `06_结果差对比.md`
- `工作台执行日志.md`

## 4. 录制减负默认规则
当前录制减负的核心不是“AI 全自动替你录”，而是：

**AI 先把最重的结构整理完，人只录最值的 4 段核心素材。**

当前默认只要求 4 段核心录制素材：
1. 原始糊话输入
2. 主 prompt 执行过程
3. 修结果 prompt 补强动作
4. 前后差值与总结收束

AI 先承担：
- 工作包正文结构整理
- 三层 prompt 包整理
- 结尾总结点整理
- `Prompt 引用尾卡` 骨架整理
- 结果差对比骨架整理

## 5. Compose 最小要求
- 至少提供一个 `workbench` 服务
- 使用 bind mount 把仓库目录挂进去
- 可额外提供一个本地持久状态 volume
- 当前不接外部 API
- 当前不接 Perplexity

## 6. 当前状态边界
- 本规则只保证“中间层工作台 scaffold”成立
- 不保证“样片最终一定产出”
- 不保证“最终成片已经 technical_validation 全通过”
- 若 Docker 环境不可运行，只能写成 `technical_validation = partial_pass`

## 7. 一句话规则
**当前 Docker 工作台只是把“场景简报 -> 工作包正文 -> Prompt 包 -> 录制计划 -> 回审清单”串起来的本地最小工作台；它的价值在于先帮用户减轻结构整理负担，而不是冒充全自动成片系统。**
