# 项目状态动作总控器与机制推理层

## 1. 文件定位

本文件是《视频工厂》的最高状态动作路由机制。

它不替代 `10_OPC一人公司闭环与多AI协作机制.md`，不替代 `08_当前正式事实.md`，不替代 `codex_source/01_execution_rules.md`。

它负责每轮先判断：

- 当前项目状态
- 输入信号
- 事实源优先级
- 触发机制
- 下一动作
- 完成标准
- 反馈更新

`Project State Action Router（项目状态动作总控器）` 的目标是：先让 AI 判断“现在是什么状态、该进哪条机制路、下一步该做什么”，再进入文案、剪辑、复盘、供料或执行。

`Mechanism Inference Layer（机制推理层）` 的目标是：把机制从“规则 / 闸门 / 卡片名”升级为可执行的 `signal -> state_inference -> action_policy -> validation_rule -> feedback_update`。

## 3. fact_source_arbitration_table（事实源裁决表）

| conflict | decision | reason | required_followup |
| --- | --- | --- | --- |
| `GPT Project static package vs GitHub main` | `GitHub main wins` | GPT Project 上传包只是静态协作包，不是实时事实库 | 若静态包落后，触发 `gpt_project_sync_needed` |
| `User current explicit instruction vs repo old fact` | 当前用户指令可指导本轮，但必须同步到 repo 才能成为下一聊天事实 | 用户本轮指令优先于旧口径，但不自动改写仓库事实 | 本轮如果形成新默认行为，必须落库、日志、提交 |
| `DeepSeek supply vs original repo files` | `original repo files win` | DeepSeek 只读供料，不拍板事实 | Codex 必须复核原文件并标冲突 |
| `Perplexity reference vs repo formal facts` | `repo formal facts win` | Perplexity 是外部研究层，只做参考 | 可沉淀为 reference，不得直接升级成已确认 |
| `technical_validation vs content_validation` | `content_validation cannot be upgraded by technical_validation` | 技术跑通不等于内容通过 | 必须保留内容复审 / 灰度状态 |
| `target_state_plan vs current_formal_fact` | `current_formal_fact wins` | 目标态计划不是当前事实 | 目标态只能作为 next target / plan |
| `latest.md vs older dated logs` | `latest.md wins, then verify direct source files` | latest 是当前摘要，但仍需回查源文件 | 若冲突，按源文件和当前正式事实修正 latest |
| `summary.json vs chat memory` | `summary.json / repo files win` | 聊天记忆不是本地事实源 | 需要本地复核路径与状态 |

## 5. mechanism_inference_template（机制推理统一模板）

```text
mechanism_inference:
  mechanism_name:
  input_signal:
  observed_evidence:
  current_project_state:
  fact_source_used:
  inferred_state:
  confidence:
  selected_action:
  action_reason:
  validation_rule:
  feedback_update:
  blocked_if:
  not_allowed:
```

使用要求：

1. 每个机制先写 `input_signal（输入信号）`，不得直接跳到动作。
2. `observed_evidence（现场证据）` 必须来自本轮用户输入、仓库文件、复盘数据、路径索引或执行日志。
3. `fact_source_used（使用事实源）` 必须说明以谁为准。
4. `confidence（置信度）` 只能写 `high / medium / low`，低置信度不得继续高风险动作。
5. `validation_rule（验证规则）` 必须能在执行后被检查。
6. `feedback_update（反馈更新）` 必须说明是否更新 latest、dated log、路径索引、机制文件或 missing fields。

## 5A. core_inference_functions（三大机制推理函数）

本节把 `editing_decision_pack（剪辑决策包）`、`content_route_card（内容路由卡）`、`quality_lock_card（质量锁卡）` 从“卡片 / 字段 / 动作名”升级为先推理、再生成卡片或动作的函数。

统一结构：

```text
input_signal（输入信号）
-> observed_evidence（现场证据）
-> state_inference（状态判断）
-> action_policy（动作策略）
-> validation_rule（验证规则）
-> blocked_if（阻断条件）
-> feedback_update（反馈回写）
```

硬规则：

1. 三个函数是机制判断，不是固定 SOP。
2. 三个函数不得推进 `content_validation / send_ready / publish_status / voice_validation / final_voice_validated / visual_master_locked`。
3. 三个函数必须先解释“为什么这样判断”，再输出卡片、动作或 blocked。
4. 证据不足时必须 blocked 或待补，不得用技术通过覆盖内容问题。

### 5A.1 editing_inference_function（剪辑推理函数）

```text
editing_inference_function:
  purpose: 判断用户录制素材 / 录屏 / 卡片 / reference 在时间线中应该如何剪辑。
  input_signal:
    - 画面信息密度过低
    - 关键操作发生但观众看不清
    - 页面变化前后差异出现
    - 结果差出现
    - 录屏中出现遮挡 / 模糊 / 黑屏 / 卡顿
    - reference 说明需要保留某个证据窗口
    - 用户反馈中段不顺 / 看不清 / 像硬拼接
  observed_evidence:
    - 时间码
    - 页面名称
    - 按钮 / 输入框 / 结果区域
    - 前后对比
    - 观众需要看到的证据点
    - reference 关键效果目标
  state_inference:
    - evidence_window_clear
    - evidence_window_unclear
    - zoom_needed
    - zoom_not_allowed
    - hold_frame_needed
    - insert_card_needed
    - keep_original_needed
    - side_by_side_needed
    - blocked_material_unclear
  action_policy:
    - zoom_in
    - crop_focus
    - keep_full_frame
    - freeze_frame
    - insert_explanation_card
    - side_by_side_compare
    - no_extra_edit
    - blocked
  validation_rule:
    - 是否让观众更快看清证据
    - 是否没有切断上下文
    - 是否没有让装饰抢走真实录屏证据
    - 是否没有用卡片替代真实证据
    - 是否符合 reference contract 的 effect_targets
  blocked_if:
    - 素材不可读
    - 时间码不清
    - 证据点不可见
    - 放大会切断必要上下文
    - 必须靠猜测判断用户操作
    - reference 关键效果无法比较
  feedback_update:
    - 写入 editing_decision_pack
    - 写入 review_pack / log
    - 如发现素材不足，回写补录或 blocked
```

补充硬规则：

- `editing_decision_pack（剪辑决策包）` 必须由本函数推理后生成；不能只列 `zoom_in / insert_card / freeze_frame` 动作名。
- 中段主体仍优先由 `用户录制素材` 承担，卡片只能辅助解释。
- 放大、裁切、定格、插卡都必须保护真实证据，不得抢证据位。

### 5A.2 content_route_inference_function（内容路由推理函数）

```text
content_route_inference_function:
  purpose: 从内容目标、素材证据、结果差、平台风险推断承载结构。
  input_signal:
    - 内容类型
    - 本轮 validation_goal
    - 强证据类型
    - 用户录制素材是否能撑住中段
    - 结果差是否可视化
    - 平台风险
    - reference 质量点
    - 是否需要 Prompt 引用尾卡
    - 是否出现产品化沉淀信号
  observed_evidence:
    - 真实录屏
    - 前后对比
    - 步骤截图
    - 结果截图
    - 平台数据
    - 用户提供素材说明
    - Codex 素材细节报告
  state_inference:
    - user_recording_as_middle_carrier
    - api_human_once_enough
    - api_human_twice_needed
    - ppt_summary_needed
    - prompt_tail_card_optional
    - prompt_tail_card_not_needed
    - work_package_not_needed_yet
    - productization_signal_present
    - blocked_evidence_missing
  action_policy:
    - 生成 content_route_card
    - 决定 API 生成人物 1 次 / 2 次 / 不确定待判断
    - 决定 PPT / 信息卡 / Prompt 尾卡是否使用
    - 决定是否只做内容验证，不沉淀工作包
    - 决定是否需要素材审计或补录
  validation_rule:
    - 中段主体是否由真实素材承担
    - 观众看完能拿走什么
    - 结果差是否被证据支撑
    - 是否没有强套旧 reference 流程
    - 是否没有先定人物次数再硬塞文案
    - 是否没有把 Prompt 尾卡写成主叙事
  blocked_if:
    - validation_goal 缺失
    - core_evidence 缺失
    - middle_carrier 无法判断
    - 素材不足以支撑文案
    - 平台风险硬阻断
    - reference 与当前项目事实冲突且无法裁决
  feedback_update:
    - 写入 content_route_card
    - 需要时写入 visual_asset_requirement_pack
    - 若值得沉淀，再触发三层 prompt 包 / 工作包
```

补充硬规则：

- `content_route_card（内容路由卡）` 是本函数的输出，不是先验模板。
- 不能先锁人物次数、PPT 数量、尾卡数量，再把文案塞进去。
- 没有 `validation_goal / core_evidence / flow_flex_reason`，不得进入视频执行。

### 5A.3 quality_issue_classifier（质量短板分类器）

```text
quality_issue_classifier:
  purpose: 当技术通过但用户感觉不对 / 怪 / 不顺 / demo 感时，先判断短板层，不直接重做。
  input_signal:
    - 用户反馈不对 / 怪 / 不顺 / demo 感
    - 技术通过但内容不舒服
    - 声音像 AI 播报 / 播音腔 / 夹子音
    - 剪辑硬拼接 / 跳切 / 上下文断裂
    - 证据不清 / 结果差看不出来
    - 卡片太密 / PPT 感过重
    - 主持壳像 NPC / 抢中段主体
    - reference 偏离
    - 灰度数据不足却要求判断下一轮变量
  observed_evidence:
    - 用户反馈原话
    - 当前 review_pack
    - summary.json
    - 时间码 / 截图 / 抽帧
    - 声音候选 / 节奏参考
    - visual_route_map
    - reference contract deviation check
  issue_categories:
    - direction_issue
    - structure_issue
    - evidence_issue
    - editing_issue
    - voice_issue
    - visual_issue
    - card_density_issue
    - host_role_issue
    - reference_deviation_issue
    - state_conflict_issue
    - platform_risk_issue
    - data_insufficient_issue
  state_inference:
    - highest_priority_issue_identified
    - multiple_issue_observed_but_one_primary
    - evidence_missing
    - human_review_required
    - blocked_need_user_input
  action_policy:
    - 只修一个最高优先级问题
    - 先补证据再改文案
    - 先改剪辑推理再重新组装
    - 先做声音复审，不直接换全片音轨
    - 先做 reference deviation check
    - blocked_need_user_input
  validation_rule:
    - 是否定位到唯一最高优先级短板
    - 是否区分 technical_validation 和 content_validation
    - 是否没有把用户感觉硬写成事实
```

- `quality_lock_card（质量锁卡）` 必须调用本分类器做短板判断；它锁质量底线，不证明内容通过。
- 当用户说“不顺 / 怪 / demo 感”时，先分类，再决定只改一个最高优先级问题。
- 没有足够证据时，不得把猜测写成事实。

## 6. completion_state_inference（完成状态判断）

```text
completed:
  conditions:
    - all required_output_inventory items done
    - validation passed
    - latest updated if state changed
    - dated log created if execution changed project state
    - no forbidden status promotion
    - no remaining must-fix items

partial_completed:
  conditions:
    - some items done
    - unresolved required item exists
    - cannot claim completed

blocked:
  conditions:
    - missing key file
    - missing user input
    - unsafe status change required
    - API/secret required but forbidden
    - evidence insufficient

continue:
  conditions:
    - no blocked item
    - remaining required item exists
    - Codex must keep executing
```

`Completion Relay Gate（补全接力闸门）` 负责防止 Codex 做一半就停；`completion_state_inference（完成状态判断）` 负责判定当前是 `completed / partial_completed / blocked / continue`。

如果 `required_output_inventory（必须交付清单）` 仍有必做项未完成，只能写 `continue` 或 `partial_completed`，不得写 `completed`。

## 7. feedback_update_rule（反馈更新机制）

```text
If an action succeeds:
  update latest / dated log / relevant mechanism note if it changes next default behavior.

If an action fails:
  write failure reason, failed signal, blocked reason, next safe action.

If user feedback says 不对 / 怪 / 不顺:
  classify into direction / structure / evidence / editing / voice / quality / route / state conflict.

If data is missing:
  do not infer next variable; mark missing fields and wait.

If Codex only partially completes:
  update completion relay failure and require next repair.

If reference / sample / target effect is provided:
  trigger reference_contract_needed, create Reference-to-Execution Contract, and do not enter Codex execution until reference_anchor / effect_targets / function_fields / deviation_check / done_when are filled.
```

落库规则：

- 改变下一轮默认判断：必须更新 `codex_log/latest.md` 和对应机制入口。
- 生成 GPT Project 静态包：必须更新 `codex_log/current_local_artifact_paths.md`。
- 发现状态冲突：必须写冲突来源、裁决依据、修正位置。
- 发现缺字段：必须写入对应 missing fields 文件或本轮日志，不得用空白掩盖。
- 用户反馈属于主观感受但无对象时：标记 `blocked_need_user_input`，请求具体对象或片段。
