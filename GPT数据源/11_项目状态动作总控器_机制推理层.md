# 项目状态动作总控器与机制推理层

## 1. 文件定位

本文件是《视频工厂》的最高状态动作路由机制。

它不替代 `10_OPC一人公司闭环与多AI协作机制.md`，不替代 `08_当前正式事实.md`，不替代 `codex_source/01_execution_rules.md`。

它负责每轮先判断：

- 当前项目状态
- 输入信号
- 事实源优先级
- 触发机制
- 下一动作
- 完成标准
- 反馈更新

`Project State Action Router（项目状态动作总控器）` 的目标是：先让 AI 判断“现在是什么状态、该进哪条机制路、下一步该做什么”，再进入文案、剪辑、复盘、供料或执行。

`Mechanism Inference Layer（机制推理层）` 的目标是：把机制从“规则 / 闸门 / 卡片名”升级为可执行的 `signal -> state_inference -> action_policy -> validation_rule -> feedback_update`。

| `material_audit_needed（素材审计）` | 用户给原始素材、录屏、图片、音频或素材说明 | 审计素材用途、证据强度、缺口和风险，形成素材判断 | 当前主线锚点、文案路由规则、路径索引、素材相关说明 | 不得直接生成视频，不得修改素材文件 | 素材可用 / 待补 / blocked 边界清楚 | 把素材存在写成素材适配 |
| `content_route_needed（内容路由）` | 内容类型、主体承载、API 人物次数、PPT / 尾卡是否出现不清楚 | 建立 `content_route_card`，先判断承载结构 | 文案路由规则、AI 知识类视频价值规则、OPC 机制、当前正式事实 | 不得强套旧 reference 流程 | 路由卡解释“为什么这样承载、为什么不照搬旧流程” | 只列镜头结构，不说明判断依据 |
| `editing_inference_needed（剪辑推理）` | Codex 不知道哪里该放大、裁切、插卡、定格或不动 | 建立或调用 `editing_inference_function` 后再剪辑 | 文案路由规则、DeepSeek supply schema、执行规则、素材描述 | 不得凭感觉乱动剪辑，不得在素材不清时继续 | 每个动作有信号、状态、动作、验证、反馈 | 只有 `zoom_in` 等动作名，没有判断函数 |
| `quality_review_needed（质量复审）` | 用户说“不对 / 怪 / 不顺”，或技术通过但内容不舒服 | 用 `quality_issue_classifier` 判断短板层 | 质量规则、OPC 机制、当前样片 / 复盘入口、latest | 不得直接改素材或升级状态 | 问题被归类为方向 / 结构 / 证据 / 剪辑 / 声音 / 质量 / 路由 / 状态冲突 | 只写主观感受，不定位机制 |
| `voice_review_needed（声音复审）` | 用户反馈声音不舒服，或 voice 状态待确认 | 只做声音复审与候选判断，不推进最终声音状态 | 当前正式事实、TTS reference、latest、声音相关记录 | 不得写 `voice_validation = passed`，不得调用 TTS / voice cloning API | 声音问题、候选、待用户确认项清楚 | 把候选音色写成最终声音通过 |
| `gpt_project_sync_needed（GPT Project 静态包同步）` | GPT Project 静态包落后于 GitHub main，或新增入口机制 | 重新生成静态上传包，更新路径索引和 manifest | `GPT数据源/`、latest、dated log、路径索引、短路由说明 | 不得写成用户已上传 UI，不得让静态包高于 GitHub main | 包存在、manifest 存在、路径索引更新、边界写清 | 只有目录，没有 manifest 或路径索引 |
| `blocked_need_user_input（阻断，需用户输入）` | 缺关键文件、缺用户素材 / 数据 / 判断、需要 secret / API / 状态推进 | 停止并输出精确 blocked 项 | 触发该状态的直接证据文件 | 不得猜、不读 secret、不调用 API、不推进动态状态 | 用户需补什么、为什么缺、补完后走哪条机制清楚 | 用推测继续执行 |

## 3. fact_source_arbitration_table（事实源裁决表）

| conflict | decision | reason | required_followup |
| --- | --- | --- | --- |
| `GPT Project static package vs GitHub main` | `GitHub main wins` | GPT Project 上传包只是静态协作包，不是实时事实库 | 若静态包落后，触发 `gpt_project_sync_needed` |
| `User current explicit instruction vs repo old fact` | 当前用户指令可指导本轮，但必须同步到 repo 才能成为下一聊天事实 | 用户本轮指令优先于旧口径，但不自动改写仓库事实 | 本轮如果形成新默认行为，必须落库、日志、提交 |
| `DeepSeek supply vs original repo files` | `original repo files win` | DeepSeek 只读供料，不拍板事实 | Codex 必须复核原文件并标冲突 |
| `Perplexity reference vs repo formal facts` | `repo formal facts win` | Perplexity 是外部研究层，只做参考 | 可沉淀为 reference，不得直接升级成已确认 |
| `technical_validation vs content_validation` | `content_validation cannot be upgraded by technical_validation` | 技术跑通不等于内容通过 | 必须保留内容复审 / 灰度状态 |
| `target_state_plan vs current_formal_fact` | `current_formal_fact wins` | 目标态计划不是当前事实 | 目标态只能作为 next target / plan |
| `latest.md vs older dated logs` | `latest.md wins, then verify direct source files` | latest 是当前摘要，但仍需回查源文件 | 若冲突，按源文件和当前正式事实修正 latest |
| `summary.json vs chat memory` | `summary.json / repo files win` | 聊天记忆不是本地事实源 | 需要本地复核路径与状态 |

## 4. trigger_routing_table（触发路由表）

| input_signal | inferred_state | trigger_mechanism | selected_action | forbidden_action | done_when | blocked_if |
| --- | --- | --- | --- | --- | --- | --- |
| 用户给截图 / 平台数据 | `gray_test_data_intake` | `review_loop / data intake` | 按视频、时间窗、数据类型录入字段和缺失项 | 不得直接判断内容通过或下一轮变量 | 数据字段、证据路径、缺失字段更新 | 缺 video_id、时间窗、数据类型 |
| 用户给最终文案 | `content_route_needed` | `content_route_card + execution_supply_pack family` | 先判断承载结构、证据、API 人物次数、PPT / 尾卡 | 不得绕过路由直接执行视频 | 路由卡与执行供料需求清楚 | 文案来源、目标或边界不清 |
| 用户给素材 | `material_audit_needed` | `material_audit / evidence detail report` | 审计素材可用性、证据强度、缺口、blocked 条件 | 不得直接改素材或生成视频 | 素材状态和下一动作清楚 | 素材不可读或权限不明 |
| Codex 执行完 | `mechanism_repair_needed` 或对应执行状态复核 | `technical_validation + content_validation boundary check + completion relay review` | 检查 required_output_inventory、禁止状态、日志、Git、同步包 | 不得只写 completed | 验证通过且剩余工作检查完成 | 任一必交付项未完成 |
| 用户说“不对 / 怪 / 不顺” | `quality_review_needed` | `quality_issue_classifier` | 归类为方向 / 结构 / 证据 / 剪辑 / 声音 / 质量 / 路由 / 状态冲突 | 不得马上随意重做 | 短板层与下一步动作清楚 | 缺可复核对象或用户反馈无法定位 |
| 发现旧口径残留 | `mechanism_repair_needed` | `old_source_residue_repair` | 降权旧口径，修入口或事实源裁决 | 不得让旧口径覆盖 main | 新入口不再误导 | 无法判断旧口径影响范围 |
| 发现灰度数据缺失 | `gray_test_waiting_data` | `missing_fields explicit + wait for data` | 写明缺哪些字段，等待或请求数据 | 不得推断短板层或下一轮唯一变量 | missing fields 更新 | 用户要求必须结论但数据不足 |
| 发现声音不适 | `voice_review_needed` | `voice_quality_review` | 只做声音问题归因和候选复审 | 不得写声音通过，不得调用 TTS API | 声音短板和待确认项清楚 | 需要真实试听但无用户反馈 |
| 发现剪辑不知道怎么做 | `editing_inference_needed` | `editing_inference_function` | 先补剪辑推理函数，再执行剪辑动作 | 不得凭动作名乱剪 | 每个剪辑动作有信号、状态、验证 | 缺素材描述或看不清证据 |
| GPT Project 静态包落后 | `gpt_project_sync_needed` | `static_package_regeneration` | 重新生成上传包、manifest、路径索引 | 不得写成 UI 已上传 | 包和 manifest 存在且路径索引更新 | 无法生成包或关键文件缺失 |
| DeepSeek fallback / missing context | `mechanism_repair_needed` 或 `blocked_need_user_input` | `supply_need_inference or local-only route` | 判断 local-only 是否足够；不足则 blocked | 不得写 DeepSeek 真实参与 | provenance 清楚，原文件已复核 | 需要 API / secret 才能继续 |

## 5. mechanism_inference_template（机制推理统一模板）

```text
mechanism_inference:
  mechanism_name:
  input_signal:
  observed_evidence:
  current_project_state:
  fact_source_used:
  inferred_state:
  confidence:
  selected_action:
  action_reason:
  validation_rule:
  feedback_update:
  blocked_if:
  not_allowed:
```

使用要求：

1. 每个机制先写 `input_signal（输入信号）`，不得直接跳到动作。
2. `observed_evidence（现场证据）` 必须来自本轮用户输入、仓库文件、复盘数据、路径索引或执行日志。
3. `fact_source_used（使用事实源）` 必须说明以谁为准。
4. `confidence（置信度）` 只能写 `high / medium / low`，低置信度不得继续高风险动作。
5. `validation_rule（验证规则）` 必须能在执行后被检查。
6. `feedback_update（反馈更新）` 必须说明是否更新 latest、dated log、路径索引、机制文件或 missing fields。

## 6. completion_state_inference（完成状态判断）

```text
completed:
  conditions:
    - all required_output_inventory items done
    - validation passed
    - latest updated if state changed
    - dated log created if execution changed project state
    - no forbidden status promotion
    - no remaining must-fix items

partial_completed:
  conditions:
    - some items done
    - unresolved required item exists
    - cannot claim completed

blocked:
  conditions:
    - missing key file
    - missing user input
    - unsafe status change required
    - API/secret required but forbidden
    - evidence insufficient

continue:
  conditions:
    - no blocked item
    - remaining required item exists
    - Codex must keep executing
```

`Completion Relay Gate（补全接力闸门）` 负责防止 Codex 做一半就停；`completion_state_inference（完成状态判断）` 负责判定当前是 `completed / partial_completed / blocked / continue`。

如果 `required_output_inventory（必须交付清单）` 仍有必做项未完成，只能写 `continue` 或 `partial_completed`，不得写 `completed`。

## 7. feedback_update_rule（反馈更新机制）

```text
If an action succeeds:
  update latest / dated log / relevant mechanism note if it changes next default behavior.

If an action fails:
  write failure reason, failed signal, blocked reason, next safe action.

If user feedback says 不对 / 怪 / 不顺:
  classify into direction / structure / evidence / editing / voice / quality / route / state conflict.

If data is missing:
  do not infer next variable; mark missing fields and wait.

If Codex only partially completes:
  update completion relay failure and require next repair.
```

落库规则：

- 改变下一轮默认判断：必须更新 `codex_log/latest.md` 和对应机制入口。
- 生成 GPT Project 静态包：必须更新 `codex_log/current_local_artifact_paths.md`。
- 发现状态冲突：必须写冲突来源、裁决依据、修正位置。
- 发现缺字段：必须写入对应 missing fields 文件或本轮日志，不得用空白掩盖。
- 用户反馈属于主观感受但无对象时：标记 `blocked_need_user_input`，请求具体对象或片段。

## 8. seed_state（当前项目初始状态）

```text
current_project_state:
  - v3.1 gray_test_active
  - content_validation = gray_testing_not_final_passed
  - send_ready = false
  - voice_validation = pending_user_chatgpt_review
  - visual_master_locked = false
  - gray_test_data_missing = true
  - current_publish_target_old_branch_residue = true
  - completion_relay_gate_written_but_long_term_unverified = true
  - mechanism_inference_layer_blueprint_written_but_not_landed = this task will land it
```

边界：

- 本 seed state 只用于下一轮状态判断起点，不推进视频动态状态。
- `content_validation` 仍是 `gray_testing_not_final_passed`，不等于内容最终通过。
- `send_ready` 仍是 `false`。
- `voice_validation` 仍是 `pending_user_chatgpt_review`。
- `visual_master_locked` 仍是 `false`。
- 灰度数据缺失时，不得判断 6000 播放门槛、短板层或下一轮唯一变量。

## 9. 一句话规则

**每轮先跑 `Project State Action Router（项目状态动作总控器）`：看信号，裁事实，判状态，选机制，定动作，验结果，再把反馈写回下一轮默认判断。**
