# 项目状态动作总控器与机制推理层

## 0A. formal_operation current route

`已确认` 当前项目状态从 `gray_test` 当前口径迁移为 `formal_operation_active（正式运营中）`。

当前默认数据链路：

`formal_operation_active -> operation_data_intake -> operation_review -> operation_next_variable_decision`

旧 `gray_test_*` 状态仅作为 `legacy alias（历史兼容别名）` 保留，不再作为当前默认路由。

## 1. 文件定位

本文件是《视频工厂》的最高状态动作路由机制。

它不替代 `10_OPC一人公司闭环与多AI协作机制.md`，不替代 `08_当前正式事实.md`，不替代 `codex_source/01_execution_rules.md`。

它负责每轮先判断：

- 当前项目状态
- 输入信号
- 事实源优先级
- 触发机制
- 下一动作
- 完成标准
- 反馈更新

`Project State Action Router（项目状态动作总控器）` 的目标是：先让 AI 判断“现在是什么状态、该进哪条机制路、下一步该做什么”，再进入文案、剪辑、复盘、供料或执行。

`Mechanism Inference Layer（机制推理层）` 的目标是：把机制从“规则 / 闸门 / 卡片名”升级为可执行的 `signal -> state_inference -> action_policy -> validation_rule -> feedback_update`。

## 3A. mandatory_commit_push_gate（强制提交推送闸门）

```text
mandatory_commit_push_gate:
  inferred_state:
    - mandatory_commit_push_required
    - git_sync_incomplete
    - remote_head_verification_required
    - unrelated_dirty_files_isolation_required
    - secret_scan_required
  trigger:
    - any_minimal_task_with_repo_file_changes
    - project_file_change
    - mechanism_repair
    - code_change
    - script_change
    - fixture_or_test_change
    - log_change
    - GPT_Project_sync_package_change
    - review_pack_schema_or_rule_change
  selected_action:
    - explicitly_stage_relevant_files
    - run_secret_scan_on_staged_diff
    - create_commit
    - push_to_current_reading_branch_or_locked_target_branch
    - verify_remote_head_or_remote_commit_readback
    - report_git_sync_status
  completed_requires:
    - relevant_files_staged_explicitly
    - commit_created
    - push_succeeded
    - remote_head_verified
    - no_unrelated_dirty_files_committed
    - no_secret_committed
  completed_forbidden_if:
    - local_changes_uncommitted
    - commit_created_but_not_pushed
    - pushed_to_wrong_branch
    - remote_head_not_verified
    - unrelated_dirty_files_mixed_into_commit
    - secret_scan_failed
    - only_local_output_exists
  allowed_non_push_cases:
    - truly_read_only_task_with_no_file_changes
    - blocked_before_any_file_change
    - user_explicitly_says_do_not_commit_or_push_this_round
  non_push_status:
    - no_file_change_completed_readonly
    - blocked
    - partial_completed
  blocked_if:
    - current_branch_unclear
    - push_failed
    - remote_head_not_verified
    - unrelated_dirty_files_cannot_be_isolated
    - secret_scan_failed
    - only_git_add_dot_would_work
```

凡本轮产生仓库文件改动，`completed` 必须同时满足：

```text
completed:
  required_if_repo_files_changed:
    - relevant_files_committed = true
    - pushed_to_current_reading_branch = true
    - remote_head_verified = true
    - unrelated_dirty_files_not_committed = true
    - secret_scan_passed = true
```

## 5. mechanism_inference_template（机制推理统一模板）

```text
mechanism_inference:
  mechanism_name:
  input_signal:
  observed_evidence:
  current_project_state:
  fact_source_used:
  inferred_state:
  confidence:
  selected_action:
  action_reason:
  validation_rule:
  feedback_update:
  blocked_if:
  not_allowed:
```

使用要求：

1. 每个机制先写 `input_signal（输入信号）`，不得直接跳到动作。
2. `observed_evidence（现场证据）` 必须来自本轮用户输入、仓库文件、复盘数据、路径索引或执行日志。
3. `fact_source_used（使用事实源）` 必须说明以谁为准。
4. `confidence（置信度）` 只能写 `high / medium / low`，低置信度不得继续高风险动作。
5. `validation_rule（验证规则）` 必须能在执行后被检查。
6. `feedback_update（反馈更新）` 必须说明是否更新 latest、dated log、路径索引、机制文件或 missing fields。

## 5A. core_inference_functions（三大机制推理函数）

本节把 `editing_decision_pack（剪辑决策包）`、`content_route_card（内容路由卡）`、`quality_lock_card（质量锁卡）` 从“卡片 / 字段 / 动作名”升级为先推理、再生成卡片或动作的函数。

统一结构：

```text
input_signal（输入信号）
-> observed_evidence（现场证据）
-> state_inference（状态判断）
-> action_policy（动作策略）
-> validation_rule（验证规则）
-> blocked_if（阻断条件）
-> feedback_update（反馈回写）
```

硬规则：

1. 三个函数是机制判断，不是固定 SOP。
2. 三个函数不得推进 `content_validation / send_ready / publish_status / voice_validation / final_voice_validated / visual_master_locked`。
3. 三个函数必须先解释“为什么这样判断”，再输出卡片、动作或 blocked。
4. 证据不足时必须 blocked 或待补，不得用技术通过覆盖内容问题。

### 5A.0 script_anchor_extraction_function（文案锚点提取函数）

`script_anchor_extraction_function（文案锚点提取函数）` 是 `content_route_inference_function（内容路由推理函数）` 的纵向补全函数。它在 `content_route_card V2（内容路由卡 V2）` 之后、视频执行 / TTS / 时间线装配之前运行，把最终文案从段落级路由补成句子级执行锚点。

```text
script_anchor_extraction_function（文案锚点提取函数）:
  purpose（用途）:
    从最终文案里自动提取句子级执行锚点，让视频执行前不再只依赖段落级素材结构。
  input_signal（输入信号）:
    - final_script（最终文案）
    - content_route_card_v2（内容路由卡 V2）
    - material_detail_report（素材细节报告）
    - platform_risk_note（平台风险提示）
  observed_evidence（现场证据）:
    - narration_line（口播句子）
    - claim_type（主张类型）
    - evidence_type（证据类型）
    - material_id（素材编号）
    - source_timecode（源素材时间码）
    - risk_phrase（风险表达）
  state_inference（状态判断）:
    - sentence_has_direct_evidence
    - sentence_has_user_experience_claim
    - sentence_is_boundary_statement
    - sentence_is_reversal_point
    - sentence_is_summary_point
    - sentence_needs_card_support
    - sentence_must_not_use_visual_claim
    - paragraph_level_mapping_insufficient
  action_policy（动作策略）:
    - output_script_function_map
    - output_evidence_anchor_map
    - output_visual_anchor_map
    - output_tts_prosody_anchor_map
    - output_card_anchor_map
    - output_forbidden_visual_map
    - output_script_to_timeline_map
    - blocked_if_sentence_level_mapping_missing
  validation_rule（验证规则）:
    - 每 1-2 句必须有 line_group_id
    - 每个 line_group_id 必须绑定素材或明确 no_direct_evidence
    - 每个 line_group_id 必须有 allowed_visuals 和 forbidden_visuals
    - 每个 evidence_sentence 必须能追溯素材或标记为用户经验陈述
    - 每个 boundary_statement 不得配误导性证明画面
  blocked_if（阻断条件）:
    - final_script missing
    - material_detail_report missing
    - script_to_timeline_map missing before video execution
    - paragraph_level_mapping_only
  feedback_update（反馈回写）:
    - 写入 content_route_card V2
    - 写入 video execution preflight
    - 写入 review pack
    - 写入 dated log
```

本函数输出字段：

```text
script_anchor_map（文案锚点表）:
  script_function_map
  evidence_anchor_map
  visual_anchor_map
  tts_prosody_anchor_map
  card_anchor_map
  forbidden_visual_map
  script_to_timeline_map
```

执行前阻断：

- 缺 `script_to_timeline_map（文案到时间线映射表）`，不得进入视频执行。
- 缺 `tts_prosody_anchor_map（TTS 韵律锚点表）`，不得生成 TTS。
- 开头路线为 `meme_gif_opening_hook（梗图 GIF 开场钩子）`、`high_emotion_hook（高情绪开头）` 或抖音抓眼路线时，缺 `opening_visual_hook_spec（开头视觉钩子规格）`，不得生成开头。
- 只有 `material_01 / material_02 / material_03` 段落级分配时，必须写 `paragraph_level_mapping_insufficient（段落级映射不足）`，不得进入成片装配。

失败路由：

- `card_ratio_mismatch`：actual ratio 不是 `horizontal_16_9` 或继续按竖屏默认执行时，必须 `blocked` 并改为横屏 16:9；不得把竖屏卡片写成当前默认。

- `card_reference_deviation`：只有局部颜色像、整体不符合 approved reference 时，必须重跑 `Reference-to-Execution Contract` 并输出 `deviation_check`。

完成判断：

- 该闸门通过只代表卡片视觉预发布质量过线或失败路由清楚，不代表 `content_validation（内容验证）`、`send_ready（可发送状态）` 或 `visual_master_locked（视觉母版锁定）`。
- 若本轮只是机制修补，完成标准是相关机制文件、fixture、latest、dated log、验证和 Git 同步完成；不生成视频。
- 任何卡片视觉失败、证据被遮挡、locked copy 被改写或 HyperFrames fallback 未授权时，都不得写 `completed`。

## 6. completion_state_inference（完成状态判断）

```text
completed:
  conditions:
    - all required_output_inventory items done
    - validation passed
    - latest updated if state changed
    - dated log created if execution changed project state
    - no forbidden status promotion
    - no remaining must-fix items
    - if task is formal-operation video delivery, result is publish_candidate_ready_for_human_review, not technical_preview_done
    - no fallback / internal_diagnostic_only / local-only output / partial result is being used as completed deliverable

partial_completed:
  conditions:
    - some items done
    - unresolved required item exists
    - cannot claim completed
    - user explicitly accepted staged delivery
    - cannot replace blocked when current task target is complete delivery

blocked:
  conditions:
    - missing key file
    - missing user input
    - unsafe status change required
    - API/secret required but forbidden
    - evidence insufficient
    - baseline cannot be met without fallback or degradation

continue:
  conditions:
    - no blocked item
    - remaining required item exists
    - Codex must keep executing
```

`Completion Relay Gate（补全接力闸门）` 负责防止 Codex 做一半就停；`completion_state_inference（完成状态判断）` 负责判定当前是 `completed / partial_completed / blocked / continue`。

如果 `required_output_inventory（必须交付清单）` 仍有必做项未完成，Codex 必须继续执行；若命中阻断条件，写 `blocked`。`partial_completed` 只允许用于用户明确接受的分阶段任务，完整交付任务不得用它替代 `blocked`，不得写 `completed`。

当 `judgment_card（判断卡）` 或 `summary_card（总结卡）` 被选中时，`required_output_inventory（必须交付清单）` 必须额外包含：

1. `codex_judgment_permission_matrix_read（判断权限表已读取）`
2. `hyperframes_card_motion_plan（HyperFrames 卡片动效方案）`
3. `hyperframes_runtime_status（HyperFrames 运行时状态）`
4. `hyperframes_visual_quality_check（HyperFrames 视觉质量检查）`
5. `subtitle_card_overlap_check（字幕卡片重叠检查）`
6. `card_text_semantic_match（卡片文字语义匹配）`

若 HyperFrames 必需但 runtime 缺失，且用户未授权静态卡片降级，`completion_state_inference（完成状态判断）` 必须写：

```text
completed = false
status = blocked
```

## 7. feedback_update_rule（反馈更新机制）

```text
If an action succeeds:
  update latest / dated log / relevant mechanism note if it changes next default behavior.

If an action fails:
  write failure reason, failed signal, blocked reason, next safe action.

If user feedback says 不对 / 怪 / 不顺:
  classify into direction / structure / evidence / editing / voice / quality / route / state conflict.

If data is missing:
  do not infer next variable; mark missing fields and wait.

  update completion relay failure and require next repair.
```

落库规则：
