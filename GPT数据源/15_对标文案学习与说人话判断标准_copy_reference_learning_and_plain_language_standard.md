# 对标文案学习与说人话判断标准

## 0. 文件定位

本文件是《视频工厂》的 `copy_reference_learning_standard（对标文案学习标准）`。

它负责把用户本轮提供的 5 篇外部对标账号文案，转成后续 ChatGPT / Codex 可读取、可判断、可回审的长期文案质量标准。

本文件解决 6 个问题：

1. 什么叫 `plain_language_standard（说人话标准）`。
2. 什么叫 `visual_sense_standard（画面感标准）`。
3. 什么时候该细写，什么时候可以一笔带过。
4. 如何学习对标账号的文案发动机，而不是照抄句子。
5. 如何把对标文案标准接入 AI 知识类视频的最终落稿、文案回审和素材证据要求。
6. 如何把外部参考迁移到《视频工厂》自己的账号身份，而不是把用户账号改造成对标账号。

它不负责：

- 写新视频文案。
- 生成下一条正式视频执行 prompt。
- 替代 `material_detail_report（素材细节报告）`。
- 替代 `data_goal_anchor（数据目标锚点）`。
- 替代 `content_validation（内容验证）`。
- 推进 `send_ready（可发送状态）`、`publish_status（发布状态）`、`voice_validation（声音验证）`、`final_voice_validated（最终声音验证）` 或 `visual_master_locked（视觉母版锁定）`。

- 禁止把本文件写成“用户最终风格已确定”。

- 禁止把“说人话 / 有画面感”替代真实素材证据。
- 禁止把本文件写成内容验证通过、发布成功、声音通过或视觉母版锁定。

## 3. 总判断：学文案发动机，不学句子

这 5 篇文案最值得学的是底层发动机：

```text
普通人眼前的具体麻烦
-> 旧方法为什么累、慢、乱、贵或容易被坑
-> AI / Agent 接手一段具体工作
-> 用 demo 把动作和结果差演出来
-> 在关键点细写，让观众看见
-> 在背景和过渡处轻写，避免堆信息
-> 保留边界，不把工具吹成万能
-> 上升到工作方式变化，而不是停在功能清单
```

后续《视频工厂》文案要学：

- 开头的判断速度。
- 案例的可视化能力。
- 中段 demo 的功能分配。
- 细节在关键处爆发的能力。
- 边界提醒的低压表达。
- 结尾把工具能力翻译成工作方式变化的能力。

不要学：

- 固定口头禅。
- 对标账号的人设、团队背书和发布会现场资源。
- 强工具种草口吻。
- 第三方产品案例。
- 对标账号的句子顺序和原句。

## 4. 对标文案共同 DNA

### 4.1 开头：先拍判断，不先介绍工具

对标文案通常不会先说“某产品发布了某功能”，而是先抛一个普通人能马上理解的判断：

- 说话就能剪辑。
- 免费工具里藏着专家模式。
- 这期会改变办公方式。
- 你以为 AI 短剧很难，但流程正在被压成工厂。
- 发布会很多，但真正有用的是这些能力。

后续《视频工厂》不能开头就念工具名、模型名、功能列表。开头应该先回答：

- 这件事跟观众有什么关系？
- 旧方法卡在哪里？
- 这次 AI / Codex 具体替人接了哪一步？
- 结果差能不能一眼看懂？

### 4.2 痛点：把技术能力翻译成普通人的麻烦

对标文案的痛点不是抽象词，而是具体麻烦：

- 剪辑软件难学，重复动作费时间。
- 买房、装修、论文这种复杂决策，普通人不会拆问题，也容易被专业信息差坑。
- 工作群、文档、评论、日历里散落的小事太多，人会漏。
- AI 短剧不是缺模型，而是缺批量保持角色、场景、镜头和剧情连贯的流程。
- 模型发布太多，普通人真正关心的是它能不能帮自己完成任务。

后续写《视频工厂》时，不要裸写“提升效率 / 优化体验 / 生成内容”。必须翻译成：

- 省掉哪一步。
- 少查哪些信息。
- 少踩哪个坑。
- 让哪个混乱问题变成能判断的表 / 报告 / 初稿 / 清单。
- 让观众看到“以前我怎么被卡住，现在 AI 接手了哪段”。

### 4.3 案例：用具体动作证明价值

对标文案会把能力拆成连续 demo：

- 剪口播、调音量、翻译字幕、找效果、包装素材、生成旁白、补镜头。
- 买房贷款计算、装修报价查坑、文献堆整理成研究框架。
- 每日巡逻、自动录 bug、个人决策分身、文档排版、建应用、建多维表格系统、用 CLI 让外部 Agent 操作飞书。
- 长剧本重构、选角选景、批量镜头、提示词微调、连续性检查、全集合并。
- 视频模型实拍编辑、代码应用实测、YouTube 联动、客服数据分析、报告和 PPT 自动生成。

这些 demo 不是功能罗列，而是每个案例都证明一个明确问题：

- 批量重复动作能不能交给 AI？
- 复杂决策能不能让 AI 先查、算、拆？
- 工作场景里的小事能不能让 Agent 自动巡逻？
- 影像生产能不能从手工拆镜压成流程？
- 新模型是不是只会聊天，还是能真的干活？

### 4.4 结果差：旧方法 vs 新方法

高价值文案必须让观众看见变化：

- 从几小时采访手动剪脏话，到十几秒完成初剪。
- 从问中介 / 普通 AI 得泛泛回答，到搜索、计算、看板一次给出结构。
- 从手动盯群录 bug，到每小时自动抓取并通知开发。
- 从手写几十个镜头提示词，到批量生成镜头再人工检查衔接。
- 从发布会概念，到真实应用能不能下载 SVG、能不能读 YouTube、能不能生成报告。

《视频工厂》后续文案必须默认写出结果差：

```text
以前：混乱、手动、靠感觉、容易漏、结果不可用
现在：AI / Codex 接手一段流程，输出可检查、可修改、可复盘的结果
```

### 4.5 边界：不纯吹，必须保留限制

对标文案的可信度来自边界提醒：

- 剪映 Agent 适合初剪，不等于能做电影级剪辑。
- 专家模式能辅助复杂决策，但不是替人最终拍板。
- 飞书 Agent 很强，但权限和外部群安全要注意，也还没做到处处可 @。
- AI 短剧工厂能大幅提效，但角色一致性和片段衔接仍要检查。
- 新模型有强项，也有抽象推理和世界知识弱一档的时候。

后续《视频工厂》文案必须保留：

- 这次 AI 做到了什么。
- 哪些地方仍需用户判断。
- 哪些地方必须人审、复查、补素材或 blocked。
- 不得把初筛写成商业验证，不得把技术通过写成内容通过。

### 4.6 上升：从工具功能上升到工作方式变化

对标文案结尾通常不是简单“这个功能很好”，而是上升到工作方式：

- 创作者不用卡在不会软件。
- 普通人手里多了一个会做事的专家。
- Agent 成为办公软件里的一等公民。
- 一人剧组时代正在接近。
- 个人或小团队可以做过去组织才能做的事。

《视频工厂》后续适合上升到：

- 一人公司如何把 AI 从聊天框变成执行层。
- 普通人如何把混乱任务压成可执行结构。
- AI 不替你拍板，但能先把问题整理到你能判断。
- Codex 的价值不是“聪明”，而是把散乱素材、步骤和结果变成可复盘证据。

## 5. 说人话判断标准

### 5.1 合格线

一段文案至少要满足：

- 技术能力已经翻译成普通人遇到的麻烦。
- 观众能听懂这件事为什么和自己有关。
- 每个关键收益都能落到具体动作或结果。
- 抽象词后面有例子，不裸跑。
- AI / Agent / Codex 出现时，是接手某段工作，不是被当成神秘名词。

### 5.2 优秀线

优秀文案要做到：

- 听起来像一个人正在给朋友讲自己的发现。
- 先让观众进入处境，再让工具出现。
- 专业细节变成坑点、成本、风险或结果差。
- demo 之间有明确功能分工，不是堆功能清单。
- 边界提醒自然出现，不打断节奏。
- 结尾能让观众知道“这改变的是哪种工作方式”。

### 5.3 一票否决

出现以下任一情况，文案回审应判为不通过：

- 先讲工具名、模型名、产品名，没有先讲观众问题。
- 只有“提高效率 / 优化流程 / 生成结果”，没有具体动作。
- 只有功能清单，没有场景和结果差。
- 只有字段、结构、机制名，读起来像项目报告。
- 把对标账号句子照搬进用户稿。
- 把外部产品能力写成用户自己已经验证的项目事实。
- 没有素材支撑，却写成真实发生。

### 5.4 禁止空泛词裸跑

以下词可以出现，但不能裸跑：

- `提升效率`
- `优化体验`
- `智能生成`
- `调用模型`
- `完成操作`
- `自动处理`
- `一站式`
- `闭环`
- `赋能`

如果要用，必须接上具体解释：

```text
它帮我省掉了哪一步？
它处理了哪个文件 / 页面 / 表格 / 素材？
它让原来哪个判断变清楚？
它输出了什么能被复核的东西？
```

### 5.5 技术词如何翻译成人话

| 技术词 | 人话翻译方向 |
| --- | --- |
| `Agent（智能体）` | 不只是回答问题，而是能接手一串动作。 |
| `多模态（multimodal）` | 它能同时看文字、图片、视频、音频，不用你把材料拆碎再喂。 |
| `精准搜索（precision search）` | 它不是随便搜一堆网页，而是先找真正能影响判断的信息。 |
| `工程文件（editable project file）` | 生成后还能继续改，不满意不用从头重来。 |
| `工作流（workflow）` | 把原来散在群聊、表格、日历、文档里的动作串起来自动跑。 |
| `CLI（命令行工具）` | 让外部 Agent 也能直接操作某个系统，而不是只能在聊天框里建议。 |
| `一致性（consistency）` | 同一个角色、场景或判断不要每一镜都变样。 |
| `模型能力（model capability）` | 它到底能不能帮你完成一个真实任务。 |

## 6. 画面感判断标准

### 6.1 观众脑子里必须看到什么

合格文案要让观众在脑子里看到：

- 人正在做什么。
- 工具界面发生了什么变化。
- 旧方法麻烦在哪里。
- 新方法具体接手了哪一步。
- 前后结果差在哪里。
- 哪个失败点、风险点或卡点被发现。

如果观众只能听到“AI 很强”，但脑子里没有画面，这段不合格。

### 6.2 工具动作如何写

工具动作优先写成：

- 导入素材。
- 识别字幕。
- 把音量拉齐。
- 查政策和利率。
- 计算不同方案。
- 把群里的 bug 自动录到表格。
- 生成角色图和场景图。
- 对同一提示词做模型对比。
- 读取客服聊天数据，生成图表、报告和 PPT。

动作句必须回答：

```text
谁把什么材料交给 AI？
AI 做了哪一步？
输出长什么样？
用户为什么能更快判断？
```

### 6.3 结果变化如何写

结果变化要写出“从 A 到 B”：

- 从素材散乱到可编辑工程。
- 从泛泛回答到结构化看板。
- 从人工盯群到自动记录和通知。
- 从逐镜头手写提示词到批量生成再检查。
- 从发布会大词到实测结果。

### 6.4 坑点和风险如何写

坑点要具体：

- 被复杂软件门槛卡住。
- 被中介、装修报价、专业术语和信息差坑住。
- 工作消息散在各处导致漏回复、漏待办、漏 bug。
- 角色和场景生成很多，但一致性可能掉。
- 模型很快，但抽象推理不一定更强。

风险提醒要短，但不能没有。它负责让观众相信你不是在硬吹。

### 6.5 没有画面支撑时如何降级

如果没有真实画面支撑：

- 只能写成经验判断、边界卡或待验证观点。
- 不得写成素材已经证明。
- 不得进入 `publish_candidate（可发布候选片）` 级口径。
- 视频执行前必须回到 `material_evidence_contract（素材证据契约）` 和 `line_group_evidence_gate（句组证据闸门）`。

## 7. 重点分配规则：什么时候细写，什么时候一笔带过

### 7.1 必须细写的位置

以下位置必须细写：

- 第一个让观众相信的 demo。
- 旧方法具体麻烦在哪里。
- AI / Agent 接手的关键动作。
- 结果差出现的地方。
- 工具价值真正成立的地方。
- 观众可能被坑、会误解或会质疑的地方。
- 能证明《视频工厂》真实项目经验的画面、字段、文件、表格、报告和结论。

### 7.2 可以轻写的位置

以下位置可以一笔带过：

- 背景常识。
- 工具名单。
- 不影响理解的模型名、参数名、版本名。
- 过渡情绪。
- 低价值铺垫。
- 不需要观众记住的周边功能。
- 已经被上一个 demo 证明过的同类能力。

### 7.3 不能细写错位置

不要把篇幅浪费在：

- 大量产品名堆叠。
- 发布会信息搬运。
- 工具功能说明。
- 背景定义。
- 对标账号的口头禅。
- 没有素材支撑的宏大判断。

细写必须服务观众理解，不是为了显得资料多。

### 7.4 细写不是堆字，而是让观众看见关键动作

细写的目标是让观众能复述：

```text
以前这一步怎么麻烦
AI 接了哪一步
输出变成什么
我接下来该怎么判断
```

如果细写之后观众仍只记得“AI 很厉害”，那不是细写，是堆信息。

## 8. 节奏判断标准

### 8.1 开头节奏

开头要快：

- 先给判断、结果或反常识。
- 不要先铺 20 秒背景。
- 不要先报产品功能表。
- CTA 可以低压，但不能压过判断。

### 8.2 中段案例节奏

中段要密，但不能乱：

- 每个 demo 服务一个明确问题。
- demo 之间要有功能分工。
- 关键 demo 细写，辅助 demo 轻写。
- 同类功能不要重复讲同一套逻辑。

### 8.3 连续 demo 节奏

连续 demo 应该像阶梯：

```text
基础重复动作
-> 复杂决策
-> 多工具串联
-> 批量生产 / 工作方式变化
```

不要像清单：

```text
功能 A、功能 B、功能 C、功能 D……
```

### 8.4 边界提醒节奏

边界提醒要短，但必须有：

- 现在能做到什么。
- 现在做不到什么。
- 哪些地方仍要人审。
- 哪些结果不能过度解释。

### 8.5 结尾上升节奏

结尾不强卖。优先收成：

- 这不是替你拍板，而是帮你少走一轮弯路。
- 这不是工具炫技，而是工作方式变化。
- 这不是最终验证成功，而是下一轮值得继续观察的能力。

## 9. 五篇对标文案逐篇高保真学习卡

### 9.1 文案 1：剪映 Agent / 剪辑生产力

**主题**：从剪辑软件升级为“会听指令、会理解素材、能调用工具”的视频创作 Agent。

**开头机制**：

- 先用“说话就能剪辑”的反常识判断抓住观众。
- 马上说明剪映不再只是剪辑软件，而是 AI 生成 + 剪辑 + AI 操作的新物种。
- 开头不是讲版本更新，而是讲创作门槛被压到一个对话框。

**普通人痛点**：

- 剪辑原本是高级技术，软件复杂、酷炫效果难学。
- 剪口播、调音量、改错字、加字幕这些不难，但极耗时间。
- 短视频创作者可能素材有意思，但不会包装。
- 有些人没素材，只有想法，也需要先跑 demo 或准备灵感。

**核心 demo**：

- 批量删除脏话口播。
- 调整全片音量一致。
- 英文演讲转写、翻译、双语字幕排版。
- 按模糊指令找效果并应用。
- 给普通素材自动做包装、音效、动画、大字。
- 无台词视频自动识别画面、生成旁白。
- 多段无逻辑素材剪成 vlog。
- 在剪映内生成补充镜头、片尾 logo、AI 转场、改词翻唱、文字成篇。

**细写位置**：

- “重复性动作”必须细写，因为它让观众立刻知道省在哪里。
- “长采访 / 批量切片”必须细写，因为它把十几秒 demo 放大成真实工作量。
- “可编辑工程文件”要写清，因为它解释了为什么这不是普通视频生成。
- “无素材 / 无台词 / 模糊指令”要细写，因为它展示了 Agent 理解素材和语义的能力。

**一笔带过位置**：

- 模型体系、模板数量、所有子功能名单。
- 只用于证明“还有更多能力”的周边功能。
- 不影响观众判断的效果名。

**结果差**：

- 从会软件才能剪，到一句话交代初剪任务。
- 从手动找功能、找模板，到让 AI 搜工具并应用。
- 从跨软件找素材，到在同一个工程里生成、编辑、剪辑。

**边界提醒**：

- 它适合初剪和批量化，不等于电影级精剪。
- 生成效果有味道，但不一定精致。
- 当前仍是第一版，不能写成终极形态已实现。

**上升判断**：

- 创作过程正在从“会不会软件”变成“会不会要求和调度”。
- 更多人的才华可以不再卡在软件门槛。

**用户账号可迁移点**：

- 把“工具功能”讲成“我原来卡在哪一步，现在 AI 接手哪一步”。
- 写清可编辑、可复核、可手动接管的结果。
- 适合迁移到 Codex 自动整理素材、生成表格、跑检查、输出报告这类内容。

**用户账号不应照搬点**：

- 不照搬剪映产品种草语气。
- 不照搬外部 demo 和第三方素材。
- 不把“软件万能”写成《视频工厂》的结论。

### 9.2 文案 2：豆包专家模式 / 复杂问题决策

**主题**：把复杂决策从“不会拆问题 / 被信息差坑”变成“AI 先查、先算、先整理到能判断”。

**开头机制**：

- 先指出好模型越来越贵，再反转到免费工具里藏着专家模式。
- 观众关心的不是模型榜单，而是更强模式到底能为普通人做什么。

**普通人痛点**：

- 买房、装修、保险、教育、医美、转行、行业研究都门槛高。
- 问专业人士很慢且不一定客观。
- 问普通 AI 速度快，但容易泛泛续写。
- 用户不会拆问题、不会写提示词，就把 AI 当搜索框。

**核心 demo**：

- 深圳首套房：首付、公积金、商业贷、利率、年限、总利息、方案选择。
- 装修报价：材料规格、核心工艺、遗漏项、消防空调、二次加钱风险。
- 文献和研究方向：把多格式材料整理成理论、共识、缺口、创新点、数据收集和分析方案。

**细写位置**：

- 买房案例要细写“政策、利率、计算、看板”，因为它证明 Agent 不是泛聊。
- 装修案例要细写“单层玻璃、石膏板、金属工艺、本地市场价、强制支出”，因为坑点具体才可信。
- 学术案例要细写“文献格式混乱、研究现状、理论、缺口和方法”，因为它证明复杂材料整理能力。

**一笔带过位置**：

- 模型榜单名次和同级模型对比。
- 复杂生活领域名单。
- “专家模式”技术包装。

**结果差**：

- 从问中介三小时还不一定靠谱，到几分钟给出结构化看板。
- 从看报价凭感觉，到发现低价背后的工艺缩水、规格模糊和后续加钱。
- 从文献越看越乱，到研究路径重新打开。

**边界提醒**：

- 专家模式不是替用户最终拍板。
- 它适合先把复杂问题整理到可判断状态。
- 数据、政策和本地价格必须有来源，不能凭模型想象。

**上升判断**：

- 抛开术语，本质是让普通人手里多一个会做事的专家。

**用户账号可迁移点**：

- 很适合迁移到“Codex 不是给建议，而是先把一团材料整理成能决策的证据包”。
- 适合写复杂工作场景：报价、选品、合同、素材、报告、项目计划。
- 重点写“少踩什么坑”，不是“AI 很聪明”。

**用户账号不应照搬点**：

- 不写成豆包专家模式产品介绍。
- 不把法律、金融、医疗等高风险判断写成 AI 已替人决策。
- 不把免费 / 榜单 / 模型强度作为用户账号主卖点。

### 9.3 文案 3：飞书 Agent 办公方式

**主题**：Agent 进入办公软件后，不只是聊天，而是巡逻、记录、建表、写文档、建应用、调工具的新同事。

**开头机制**：

- 先抛“可能全面开启 Agent 办公方式”的判断。
- 再承诺会给玩法、内部工作流、提示词和 skill。
- 开头不是介绍飞书更新，而是直接把观众拉进“办公方式会变”的大判断。

**普通人痛点**：

- 工作信息散在群聊、文件、评论、日历里，人会漏。
- 周报、会议、文档、PPT、bug 录入、表格系统都是琐碎但耗时。
- 很多人不会开发应用，也不会设计复杂表格系统。
- 外部 Agent 如果不能操作办公系统，只能停在聊天框建议。

**核心 demo**：

- 每日巡逻：从群聊、文件、评论里找真实待办，排优先级，还能先做掉一部分。
- 周报：按本周工作生成给领导看的周报和信息图。
- bug 工作流：用户群报 bug，自动填入多维表格，再通知开发群。
- 个人分身：从真实消息、评论、批注提取决策风格和个人说明卡。
- 专业版文档：语音乱讲点子，整理成策划案；在文档评论里继续修改；生成 PPT。
- skill 市场 / 本地导入：让助手持续学习和安装适用技能。
- 妙搭：非开发同学做邀请系统、资料网页和应用。
- 多维表格 Agent：从需求出系统设计、建表、仪表盘、自动化提醒。
- 飞书 CLI：让外部 Agent 下载发票、读会议纪要、拆修改点、约会、发文档、回邮件。

**细写位置**：

- 每日巡逻要细写，因为它把“清单”升级成“自动发现 + 先做一半”。
- bug 工作流要细写，因为它展示多渠道信息、表格和通知的自动串联。
- 多维表格 Agent 要细写，因为它把业务逻辑设计难点说清。
- CLI 要细写，因为它解释了外部 Agent 如何从聊天框进入真实系统操作。

**一笔带过位置**：

- 四大工具名单只需快速建立地图。
- 计费细节、福利、官方名称可以轻写。
- skill 市场中的周边示例可作为辅助，不抢主线。

**结果差**：

- 从人盯群、手填表、写周报，到 Agent 自动巡逻和执行。
- 从不会开发 / 不会建表，到自然语言生成系统。
- 从会议纪要变聊天记录，到外部 Agent 拆任务并逐个处理。

**边界提醒**：

- 权限和外部群安全必须提醒。
- 目前还不能像普通同事一样在所有场景随地 @。
- Agent 干活需要算力和费用，不是零成本魔法。

**上升判断**：

- Agent 正在成为办公系统里的一等公民。
- 个人或小团队可能做出过去需要组织才能完成的事。

**用户账号可迁移点**：

- 适合迁移到“一人公司 AI 协作系统”的账号身份。
- 写 Codex 时可以强调它不是聊天助手，而是执行层和 Integrator。
- 适合展示真实工作流：读取材料、记录问题、生成表格、同步日志、验证、Git 收尾。

**用户账号不应照搬点**：

- 不照搬飞书团队背书、福利和产品发布节奏。
- 不把企业协作权限当作《视频工厂》的默认条件。
- 不把工具名堆满，忽略用户自己的真实工作流。

### 9.4 文案 4：AI 短剧工厂

**主题**：把短剧生产从手工拆分的多环节流程，压成从剧本到镜头、角色、场景、成篇的一套 Agent 工作流。

**开头机制**：

- 先拿“AI 短剧很火”和“你以为很难”建立观众兴趣。
- 再给出强结果：长剧本变成多集短剧成篇。
- 开头承认夸张感，转而用后续 demo 证明。

**普通人痛点**：

- AI 短剧看起来门槛高。
- 长剧本要拆成短剧结构、角色、场景、镜头、提示词、连续性，工作量巨大。
- 角色一致性、画风统一、镜头连贯、剧情衔接都费心。

**核心 demo**：

- 输入长篇剧本或想法。
- 自动重构成短剧基础框架。
- 确认项目名、画风、比例等配置。
- 按每集几十秒到一分钟生成大纲。
- 生成角色形象和场景。
- 对角色照片重新描述、重新生成。
- 挑选具体集数，看到整集镜头和视频提示词。
- 微调提示词，批量生成片段。
- 检查片段之间的人物和剧情衔接。
- 合并全集，看特效、表演、剪接和一致性。

**细写位置**：

- 长剧本如何变成短剧框架必须细写，因为这是核心生产力变化。
- 角色和场景批量生成必须细写，因为一致性是 AI 影像生产痛点。
- 镜头提示词和片段衔接必须细写，因为它说明仍需要人工检查和微调。
- 全集效果要细写到表演、镜头和眼神问题，避免纯吹。

**一笔带过位置**：

- 具体玄幻剧情内容。
- 不影响判断的角色名和台词。
- 周边风格类型名单。

**结果差**：

- 从逐镜头写提示词、逐张抽图、手动保持风格，到 Agent 自动拆剧、出角色、出场景、出镜头。
- 从创作者被工作量劝退，到先跑出可看的批量成篇。

**边界提醒**：

- 自动生成不一定符合想法，需要重新描述和抽卡。
- 人物复杂时片段衔接可能出问题。
- 仍要检查每个片段，不是完全无人监管。

**上升判断**：

- 影像生产进入一人剧组 / 小团队提效阶段。
- 专业人士也会因为批量流程被重新分工。

**用户账号可迁移点**：

- 适合迁移“复杂流程被 AI 压成可检查流程”的表达。
- 可以用于讲《视频工厂》如何把素材、文案、TTS、字幕、卡片、验证拆成可复跑链路。
- 重点不是“直接成片”，而是“每一步可检查、可返工、可阻断”。

**用户账号不应照搬点**：

- 不把“爆款手到擒来”迁移成用户承诺。
- 不把娱乐短剧的强种草口吻迁移到 AI 知识类视频。
- 不跳过素材证据和人审，直接把批量生成写成完成。

### 9.5 文案 5：Google 更新 / 多模型实测对比

**主题**：从一大波发布更新里筛出真正有用的能力，用实测对比讲清每个模型的正确打开方式和边界。

**开头机制**：

- 先说更新很多，但只挑真正有用的东西。
- 直接上效果，不先讲发布会背景。
- 先用同一任务对比不同模型，再给出正确使用场景。

**普通人痛点**：

- 发布会信息太多，普通人不知道哪个对自己有用。
- 新模型名、功能名很多，但真正关心的是能不能完成任务。
- 同一类模型不能只看宣传，要看实测强项和短板。

**核心 demo**：

- 视频全模态模型：同提示词现场演唱会视频对比。
- 正确打开方式：用实拍视频加特效、加元素、换镜头、做跟踪。
- Flash 模型：用 SVG 图标应用、换装应用、YouTube 教程网站测试编码和多模态能力。
- 抽象推理对比：用音乐问题观察世界知识和抽象总结能力差异。
- Antigravity：读取本地客服聊天数据，生成分析、图表、报告和培训 PPT。
- 快速提及地图、YouTube、搜索、Spark 和无人驾驶体验，只作为信息补充。

**细写位置**：

- 同任务模型对比要细写，因为它让观众知道“不能这么比”和“正确打开方式”。
- SVG / 换装 / YouTube 网站要细写，因为它们是能不能干活的实测。
- 抽象推理短板要细写，因为它防止把“快”误写成“更聪明”。
- Antigravity 本地数据案例要细写，因为它说明 Agent 任务产物链。

**一笔带过位置**：

- 发布会全量更新表。
- 国内暂时用不上的功能。
- 现场活动和邀请背景。
- 只作为趋势补充的小功能。

**结果差**：

- 从发布会概念到真实任务产物。
- 从“某模型强”到“它强在哪类任务、弱在哪类任务”。
- 从聊天机器人到桌面 Agent 管理多个任务。

**边界提醒**：

- 不能直接拿不同定位模型硬比。
- 快不等于智慧更高。
- 有些功能国内暂时用不上。
- 工具强项要放回正确任务场景。

**上升判断**：

- 这是 Agent 时代，关键是怎么用 Agent 干活。
- 后续内容应继续围绕实用教程和真实任务，而不是只追发布名词。

**用户账号可迁移点**：

- 适合迁移“发布信息过滤 + 实测对比 + 正确打开方式”的结构。
- 适合《视频工厂》做工具链、模型链、Codex / ChatGPT / DeepSeek / Perplexity 分工讲解。
- 重点写真实任务产物，不写发布会复读。

**用户账号不应照搬点**：

- 不照搬大型发布会、团队现场和海外体验背书。
- 不把未经本地实测的能力写成用户已验证。
- 不用模型名堆砌替代观众能看到的任务结果。

## 10. 迁移到《视频工厂》的写作规则

### 10.1 用户账号更适合的身份

《视频工厂》更适合迁移成：

- `AI 生产力实战者`
- `一人公司系统搭建者`
- `真实项目踩坑复盘者`
- `多 AI 协作总控者`
- `Codex 执行层验证者`

不适合迁移成：

- 发布会测评号。
- 纯工具种草号。
- 团队资源展示号。
- 大而全 AI 新闻号。
- 只讲功能、不讲真实项目结果的教程号。

### 10.2 适合迁移的表达机制

后续最终文案可迁移：

- 先给判断，不先报工具。
- 从普通人真实麻烦切入。
- 用真实录屏、表格、报告、文件、日志和结果差做 demo。
- 把 AI / Codex 写成接手一段工作，而不是万能答案机。
- 在关键动作处细写，在背景过渡处轻写。
- 保留限制、人审和 blocked 边界。
- 结尾上升到工作方式变化。

### 10.3 不适合迁移的表达机制

后续最终文案不应迁移：

- 固定口头禅。
- 对标账号产品测评人设。
- 对外团队背书。
- 外部品牌案例。
- 强福利、强种草、强关注压力。
- 未经用户真实素材验证的功能承诺。

### 10.4 Codex / AI 生产力选题的应用方式

写 Codex / AI 生产力时，默认优先回答：

1. 这个任务原来哪里混乱？
2. 用户以前要手动做哪几步？

## 12. 与现有文案流程的关系

- `Perplexity（外部参考检索 / 研究工具）` 仍只做 `reference_pack（参考包）` 和 `raw_feeling_draft（原感初稿）`。
- 用户素材仍是最终文案硬依据。
- `Codex（执行代理）` 素材检查仍必须汇报真实细节、时间码、页面、按钮、动作、结果和证据强度。
- `ChatGPT（最终落稿与复审入口）` 仍是最终文案出口。
- `copy_granularity_mixture_rule（文案颗粒度配比规则）` 继续负责句子密度与素材颗粒度。
- `reference_copy_voice_mechanism（对标文案话语机制）` 继续负责反报告腔和普通人问题救场。
- 本文件新增的是长期 `plain_language_standard（说人话标准）`、`visual_sense_standard（画面感标准）`、`focus_allocation_rule（重点分配规则）` 和 5 篇逐篇学习卡。
- 本文件不替代素材证据、不替代数据目标、不替代内容验证。

## 13. 一句话规则

**后续《视频工厂》写文案，不是学对标账号怎么说同一句话，而是学它们怎么把普通人的真实麻烦、AI 接手的具体动作、结果差、边界和工作方式变化讲到小白也有画面；缺真实素材证据时只能降级、请求补素材或 blocked，不得用“说人话”掩盖证据不足。**
