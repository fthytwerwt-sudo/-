# 选题与文案规则

## 0A. 正式运营约束

当前阶段为 `formal_operation_active`。选题与文案修改必须先读取 `codex_log/current_operation_target.md`、`review_loop/operation_records_index.md` 和 `codex_log/current_data_goal_anchor.md`；数据不足时不得把正式运营样本写成方向成立或内容通过。

若选题 / 文案任务继续指向做视频、产视频、发片候选、运营内容或下一条视频，交付基线必须切换到 `publish_candidate（可发布候选片）` 或 `blocked（阻断）`。文案准备包、内容路由卡、时间线映射、TTS 韵律锚点、剪辑 / 装配决策包只属于执行前必备条件，不能被写成视频交付完成；技术预览、无声预览、横屏技术包或只交 JSON / Markdown 包只能作为内部诊断。

## 0B. locked_copy_contract（锁定文案契约）

ChatGPT / 用户是最终落稿和文案锁定入口。选题、标题、开头句、最终口播稿和核心判断一旦进入视频执行，必须写入 `locked_copy_contract（锁定文案契约）`。

`locked_copy_contract` 至少包含：

- `locked_topic（锁定选题）`
- `locked_title（锁定标题）`
- `locked_final_script（锁定最终口播稿）`
- `locked_opening_line（锁定开头句）`
- `allowed_copy_changes（允许改动）`
- `forbidden_copy_changes（禁止改动）`
- `copy_change_request_required_if_needed（如需改文案必须请求变更）`

Codex 只能做标点、换行、字幕分句和 TTS 停顿等非语义执行调整，不得擅自改写标题、选题、核心判断、人味表达或视觉标题卡标题。若 Codex 判断文案太长、不适合剪辑、画面无法对应或 TTS 不适配，必须输出 `copy_change_request（文案修改请求）`，等待 ChatGPT / 用户确认；不得自行压缩、重写或替换文案。

## 1. 文件定位
本文件只解决 3 件事：
- 当前什么选题值得进入主流程
- 文案怎样先压成适配当前正式主线的版本
- 当前用户偏好的最关键稳定结论是什么

它不负责：
- 当前待发对象切换
- Docker / compose 技术细节
- 最终成片执行命令

## 2. 当前文案前置层唯一适配要求
一条内容要进入当前主流程，至少要同时满足：
1. 能安排 `API 生成真人段`
2. 能提供 `用户录制素材`
3. 能用 `少量 PPT` 收住关键词或总结
4. 不需要把 `真人实拍出镜` 当默认前提
5. 不需要把 `pure PPT` 拉回主体叙事
6. 能提供 `真实 AI 使用经验`、`工作提效证据`、`前后变化` 中的至少一项强证明点

## 3. 当前阶段优先主打的内容类型

### 3.1 `AI 职场任务现场拆解`
当前优先。

更适合的原因：
- 更能展示“把一句糊话压成可执行结构”的真实能力
- 更容易放进具体任务、具体动作、具体结果
- 更容易让 `用户录制素材` 提供中段强证据
- 更容易讲清楚哪里节省了重复劳动、哪里出现了真实结果差

### 3.2 `AI 问题点破`
当前优先。

更适合的原因：
- 更容易从真实误判切入
- 更容易让开头判断句成立
- 更容易把“问题为什么会发生”压成一个可执行动作

## 4. 当前高优先级选题来源
- 高频具体职场场景
- 真实失败体验
- 有清楚前后差值的题
- 能真实录到中段证据的题
- 能自然拆成 `真实动作 + 结果差 + 最小动作`，并在必要时再沉淀工作包的题

当前更适合优先考虑：
- 模糊需求压稿
- 汇报 / 周报 / 项目方案
- 会议纪要变待办
- 材料整理与结构重写
- 为什么你用了 AI 最后还是自己重写

## 5. 当前不优先的题
- `AI 日常工作流分享`
- 纯工具新闻
- 纯模型更新播报
- 纯抽象方法论
- 只能靠 `pure PPT` 承载的题
- 必须默认 `真人实拍出镜` 才成立的题

## 6. 当前稳定文案偏好结论
以下是从旧 `30-36` 文案知识库吸收出的最关键稳定结论：

### 6.1 开头偏好
- 用户更偏好：`结论先拍桌`
- 次优先：`结果先交代`
- 可用补充：`对比拉开型`
- 暂不默认：长叙事、重悬念式开头

### 6.2 判断句偏好
- 用户偏好明显有立场的判断句
- 更像“先定性 -> 再比较 -> 再给选择”
- 不喜欢书面报告腔

### 6.3 过渡偏好
- 更稳的是：
  - `问题推进型`
  - `举例切换型`
  - `编号展开型`
- 不追求漂亮辞藻，更追求顺

### 6.4 收尾偏好
- 更偏好：
  - 一句话收束观点
  - 给一个低门槛动作
- 不默认强卖课、强关注、强会员导向

### 6.5 当前一句风格理解
**用户偏好 `有判断、有人味、有动作` 的表达。**

## 6A. 后续默认文案生产流程

以后《视频工厂》AI 知识类视频文案默认按以下流程推进：

1. `Perplexity（外部参考检索 / 研究工具）` 只负责输出 `reference pack（参考包）` 和 `raw feeling draft（原感初稿）`。
   - 作用：提供外部参考、表达方向、案例、梗感和原始感觉。
   - 边界：不是最终稿，不得直接进入视频、字幕或声音执行。
2. 用户根据参考方向录制素材。
   - 素材可以是录屏、网页操作、结果对比、过程证据、关键截图。
   - 素材是最终文案的硬依据。
3. `Codex（执行代理）` 做素材技术检查与细节汇报。
   - `Codex（执行代理）` 不负责最终文案定稿。
   - `Codex（执行代理）` 必须把素材里可写进最终稿的细节报告给 `ChatGPT（最终落稿与复审入口）`。
   - 汇报不能只写“文件存在、时长正常、技术通过”，必须写清真实素材里有什么、在哪一秒、发生了什么、能证明什么。
4. `ChatGPT（最终落稿与复审入口）` 基于 `Perplexity（外部参考检索 / 研究工具）` 参考包、用户素材和 `Codex（执行代理）` 细节报告写最终落稿。
   - 最终稿唯一默认出口是 `ChatGPT（最终落稿与复审入口）`。
   - 可以重排、压缩、改写、加口语、加微反转、加真实细节。
   - 必须贴真实素材，不能写素材里没有的动作、页面和画面。
5. `Codex（执行代理）` 再按 `ChatGPT（最终落稿与复审入口）` 最终稿执行视频、字幕、音频、卡片和审片包。
   - 执行后回到 `ChatGPT（最终落稿与复审入口）` 复审。

硬规则：
- `Perplexity（外部参考检索 / 研究工具）` 初稿不能被当成最终稿。
- `Codex（执行代理）` 素材检查不能被当成 `Codex（执行代理）` 写最终文案。
- `ChatGPT（最终落稿与复审入口）` 不能在缺少素材细节时直接写最终落稿。

## 6B. 当前最终文案风格锚点

当前最终风格锚点为：
- 用户确认的“用字更自然版长稿”
- 20260427 B 版“停顿梗感”试听方向

稳定风格要求：
1. 生活观察起手。
2. 微反转，不做极限反转。
3. 说话带一点梗，但不要刻意装幽默。
4. 自然口语，像真人能直接说出口。
5. 有轻吐槽，但不攻击用户。
6. 保留真实细节，不写空泛概念。
7. 避免 AI 感硬词。
8. 不写课程腔。
9. 不写广告腔。
10. 不写鸡血短视频腔。

明确禁用或慎用：
- `下一步从哪打`
- `开门钥匙`
- `把路铲出来`
- `一团雾`
- `强势赋能`
- `精准击穿`
- `底层逻辑`
- `高效闭环`
- `抓手`
- `撬动`
- `价值闭环`
- `强势破局`

推荐保留的表达方向：
- `陪资料加班`
- `第一行没开`
- `看着像方案，用起来像空气`
- `永远不会错，也永远没用`
- `普通问法给废话，Prompt（提示词）调对给初稿`
- `别再跟空白页硬刚`
- `先帮你把顺序理出来`
- `下一步从哪开始`

## 6C. copy_granularity_mixture_rule（文案颗粒度配比规则）

`已确认` 文案表层要顺口、自然、有冲击力，但关键价值点必须落到素材颗粒度：精彩和重要的位置讲细节，普通过渡位置顺口推进。

默认配比是写作检查比例，不是硬时长比例：

- `30% 顺口推进`：负责开头痛点、观众心理、轻吐槽、段落转场、一句话判断和结尾低压收束。
- `50% 素材颗粒度`：负责证明能力、展示 AI 操作、展示结果、解释为什么适合用户。
- `20% 判断、边界和下一步`：负责说明素材能证明什么、不能证明什么，以及下一步该核什么。

可以顺口的位置：

- 开头痛点。
- 观众心理。
- 轻吐槽。
- 段落转场。
- 一句话判断。
- 结尾低压收束。

必须加素材颗粒度的位置：

- 出现“AI / Codex 帮我做了什么”。
- 出现“自动操作电脑”。
- 出现“选品流程”。
- 出现“生成表格 / 云盘文件 / 回到聊天框结论”。
- 出现“为什么适合我”。
- 出现“风险 / 下一步复查项”。
- 出现“结果变化”。

素材颗粒度必须优先写真实画面里能看到或能从素材报告证明的元素：

- 真实页面。
- 真实动作。
- 真实输入。
- 真实点击。
- 真实字段。
- 真实表格。
- 真实文件路径 / 云盘文件。
- 真实聊天框回传结论。
- 真实结果变化。

弱写法：

```text
Codex 帮我自动筛选商品。
```

强写法：

```text
你看这里，Codex 不是在聊天框里随便给建议，它是在我的电脑上直接操作。Codex 在屏幕上打字，先在搜索框里输入品类词，再翻商品卡，一个个看客单价、佣金、销量信号、店铺分、商品分和退货风险。
```

结果弱写法：

```text
最后生成了表格。
```

结果强写法：

```text
最后它把这些商品整理成一张云盘表格。商品名、客单价、佣金、销量信号、内容可拍性、主要风险、适不适合我、下一步要核什么，都分列写清楚；再回到聊天框给出当前最适合复查的商品、理由、风险和下一步。
```

验收问题：

- 这句话有没有真实动作？
- 有没有真实页面？
- 有没有真实字段？
- 有没有结果变化？
- 有没有边界？
- 如果没有素材支撑，是不是应该降级成用户经验陈述或删掉？

硬边界：

- 不允许整篇都顺口飘过去，用“AI 帮我处理好了”遮盖素材不足。
- 不允许整篇都堆操作细节，让开头、过渡和收尾失去人味。
- 没有素材支撑的细节不得写成事实；只能降级成用户经验陈述、卡片解释、待补素材或删除。
- 本规则不授权 `Codex（执行代理）` 改写锁定最终文案；需要语义改动时仍必须输出 `copy_change_request（文案修改请求）`。

## 6D. reference_copy_voice_mechanism（对标文案话语机制）

### 6D.1 一句话定义

以后最终文案默认继承对标稿的“普通人复杂问题救场”话语机制：先站在用户正在被复杂问题坑的位置说话，再用真实案例、坑点细节和结果变化引出 AI / Codex 的价值；文案要像真人讲给朋友听，不像功能说明，也不像项目报告。

### 6D.2 机制核心

`reference_copy_voice_mechanism（对标文案话语机制）` 是在 `copy_granularity_mixture_rule（文案颗粒度配比规则）` 之上补“怎么说人话”的机制，不替换已有颗粒度机制。

核心要求：

1. 不是先讲工具，而是先讲普通人正在遇到的复杂问题。
2. 不是先展示功能，而是先展示旧方法为什么累、慢、容易踩坑。
3. 不是直接说 AI 很强，而是让 AI 在案例里救场。
4. 不是堆字段，而是把字段翻译成坑点。
5. 不是输出功能总结，而是输出观众能理解的结果变化。
6. 不是强卖，而是低压收束。
7. 结构不固定，机制固定。

### 6D.3 默认话语公式

可变结构公式：

```text
普通人正在被什么复杂问题卡住
-> 旧方法为什么麻烦 / 不靠谱 / 容易被坑
-> AI / Codex 不是来炫技，而是来接手一段具体工作
-> 具体案例展开
-> 坑点细节爆发
-> 工具做了哪些动作
-> 结果变得更清楚
-> 一句能力定义
-> 低压收尾：它不替你拍板，但能先帮你少走一轮弯路
```

硬边界：

- 这是默认机制，不是固定段落顺序。
- 如果素材更适合先给结果，可以先结果预览。
- 如果素材更适合先现场录屏，可以使用 `screen_first_opening（录屏现场先行开头）`。
- 如果素材情绪强，可以先用问题 / 梗图 / 反应卡。
- 如果证据弱，不得硬套大叙事。

### 6D.4 说人话规则

推荐使用的推进词：

```text
你有没有发现
你以为
但其实
举个例子
你看这里
最关键的是
这才是重点
说人话就是
以前你可能会
换成现在
看出区别了吗
```

禁止把稿子写成：

```text
项目报告腔
功能说明书腔
平台介绍腔
课程腔
广告腔
过度专业名词串
```

### 6D.5 坑点细节规则

文案里的细节优先写“坑点细节”，不是单纯字段清单。

弱写法：

```text
Codex 记录了客单价、佣金、销量信号、店铺分、商品分。
```

强写法：

```text
有的商品佣金看起来高，但客单价太低，拍一条视频的时间成本都未必回得来；有的商品销量信号不错，但 SKU 太复杂，观众看完根本不知道该买哪一款；还有的商品图很好看，但店铺分和退货风险一低，后面可能全是售后坑。
```

### 6D.6 能力定义规则

每条内容中段或结尾需要有一句“能力重新定义”，把工具能力翻译成人话。

示例：

```text
普通 AI 是给你建议；Codex 这类执行型 AI，是直接替你跑一段流程。但它最值钱的地方不是替你拍板，而是先把混乱商品整理成你能决策的证据表。
```

### 6D.7 低压收尾规则

收尾不能强卖，也不能鸡血。优先收成：

```text
它不是自动赚钱。
它不是替你赌商品。
它只是先把混乱问题整理到你能判断的状态。
```

### 6D.8 不写死结构规则

`reference_copy_voice_mechanism（对标文案话语机制）` 锁的是话语质量机制，不锁死结构。

后续每条内容仍要根据以下因素动态调整：

- 用户素材强弱。
- `reference pack（参考包）`。
- 证据出现顺序。
- 开头路线判断。
- 素材中最强画面。
- 平台风险。
- 观众理解路径。
- 本轮数据目标。
- 是否需要先结果预览。
- 是否需要先现场画面。
- 是否需要多案例。
- 是否只适合单案例深挖。

不得强制每条内容都有三个案例，不得强制每条内容先讲宏观趋势，不得照搬 reference 原句、案例、品牌表达或第三方资产。

### 6D.9 前台口播转换规则：后台标签不得出现在最终稿

`reference_copy_voice_mechanism（对标文案话语机制）` 里的检查字段只用于后台验收，不是最终稿结构。最终口播必须把这些字段转换成自然口播链条，像一个人正在讲一件普通人会遇到的麻烦事，而不是把机制名、句子类型或验收项念出来。

最终文案不得出现以下标签式写法：

```text
用户痛点句：
旧方法坑点句：
AI 救场句：
坑点细节句：
能力定义句：
低压收尾句：
```

最终文案不能写成机制解释、项目报告、路由说明或字段说明。对标稿真正值得继承的是“说话方式”：先把人带进具体处境，再让工具自然出现，把专业判断变成观众听得懂的坑、动作和结果变化；不是继承表格化结构。

弱写法：

```text
用户痛点句：你不知道哪个商品值得测。
旧方法坑点句：手动翻精选联盟很累。
AI 救场句：Codex 帮我初筛商品。
能力定义句：Codex 是执行型 AI。
```

强写法：

```text
你以为做带货最难的是拍视频，其实第一步就卡在：你根本不知道哪个商品值得测。你刷半天精选联盟，看到的全是商品卡。这个佣金高，那个销量好，但真要拍，你还是不知道先拍哪个。所以这次我没有继续手动翻，而是让 Codex 先替我跑一轮初筛。
```

硬边界：

```text
如果最终稿读起来像“机制字段串联”，即使字段都齐，也判为不通过。
不能写成机制字段串联。
```

## 6E. 对标文案学习标准入口

## 7A. 开头路线判断边界

`opening_route_decision（开头路由判断）` 必须先回答：

1. 开头时长是 `<= 3s` 还是 `> 3s`？
2. 话题情绪强度是 low / medium / high？
3. 争议程度是 low / medium / high？
4. 开头证据强度是否足以直接用录屏先行？
5. 本条是否强依赖品牌一致性和轻陪伴进入？
6. 核心问题能否一句话钉住？

边界：

- `element_doll_opening（元素娃娃开头）` 保留，但不得写成所有内容唯一默认开头。
- `meme_gif_opening_hook（梗图 GIF 开场钩子）` 只在需要抓眼、起情绪、抛问题时优先，不得写成新默认死流程。
- 开头 GIF / 梗图只负责停留和问题，不负责证明。
- 强证据仍来自 `用户录制素材`、前后对比、步骤截图、结果截图或平台数据。

**当前选题和文案，不是先找一个“看起来能讲”的题，而是先找一个能同时接住“开头路线判断 + 用户录制素材中段证据 + 少量 PPT 结尾收束”，并且能录出 `真实 AI 使用经验`、`工作提效证据`、`结果差` 与 `最小动作` 的题；开头固定机制、不固定素材，若后续出现明确复用价值，再沉淀为工作包。**
