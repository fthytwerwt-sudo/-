# 目标态计划

## 1. 文件定位
本文件只写目标态计划。

硬规则：
- 本文件不是当前正式事实清单
- 本文件里的内容不得回写成 `已确认`
- 目标态只有在取得执行证据并更新 `08_当前正式事实.md` 后，才能升级成当前正式事实

## 2. 当前目标态

### 2.1 路由加速目标态
- `待验证` 让当前 10 份执行包长期稳定承担 80% 路由判断
- `待验证` 让 GPT 与 Codex 不再默认深读旧 10-37 散文件

### 2.2 Docker 工作台目标态
- `待验证` 让 Docker 最小工作台稳定串起：
  - `01_场景简报.md`
  - `02_工作包正文.md`
  - `03_Prompt包.md`
  - `04_录制计划.md`
  - `05_回审清单.md`
- `待验证` 让这套工作台除了 `docker compose config` 通过之外，还能在后续轮次完成更多容器内长流程验证
- `待验证` 让 GPT 与 Codex 命中 Docker / 工作台任务时，默认先完成最小可逆修正，而不是先把排障责任丢给用户

### 2.3 录制减负目标态
- `待验证` 让每条视频默认只要求 `4 段核心录制素材`
- `待验证` 让 AI 先把工作包正文、三层 prompt 包、结果差证明点与 `Prompt 引用尾卡` 骨架整理出来
- `待验证` 让用户录制负担从“录很多碎片”收束成“只录最值的 4 段证据素材”

### 2.4 vNext 样板目标态
- `待验证` 把首条 `样板_营销方案初稿_vnext` 从模板可用，推进到真实可复用样板
- `待验证` 让 `Minecraft-inspired 原创体素方块风` 外壳、总结卡与 `Prompt 引用尾卡` 的可读性继续收敛
- `待验证` 让声音路线从“建议参数”推进到真实试配结果

### 2.4A `content_validation V1` 升级目标态
- `待验证` 让 `可爱女生向导音` 从“目标感觉 + 回审标准已写入”，推进到多轮样片稳定过线
- `待验证` 让 `可爱元素主持娃娃` 从“内部动作基线 V1 已写入”，推进到稳定可复用的主持壳
- `待验证` 让声音基线 V1 与动作基线 V1 的联合过线规则，在多轮复审中稳定成立
- `待验证` 让“阿里优先验证”先完成工程层验证轮次，但仍不偷写成最终供应商已定
- `待验证` 基于用户语音样本推进 10-15 秒最小声音试配，用样本约束 `平静 / 轻陪伴 / 低压 / 游戏向导感` 的听感方向
- `待验证` 形成 `声音参考样本 -> TTS 试配 -> 人工听感复审 -> 再决定是否替换全片音轨` 的闭环
- `待验证` 对已生成的声音复刻 trial 做用户 / ChatGPT 听感复审，先判断是否明显比上一轮 `Serena` 系统音色更接近样本，再决定是否进入下一轮最小微调。
- `待验证` 在最小声音试配通过前，不把样本参考、阿里优先验证或任一候选音色写成最终声音方案

### 2.4B C 线元素娃娃能力目标态
- `待验证` C 线若要进入 `round2`，前置条件不是继续调动作，而是先补“视频级角色资产能力”
- `待验证` 当前最值路线是：先补一套可被当前已接模型真正吃进去的主持娃娃资产，再做最小能力验证
- `待验证` 该最小能力验证只回答两个问题：
  - `detect` 能不能过
  - 生成出来后，口型 / 头部 / 动作是否达到主持壳最低可用线
- `待验证` 若新资产仍过不了 `detect`，或虽然能过 `detect` 但出来仍明显是“图片动起来 / gif 感”，则停止当前 C 线模型路线，不进入 `round2`

### 2.5 阶段与回审目标态
- `待验证` 让每轮内容推进至少明确：
  - `technical_validation`
  - `content_validation`
  - `remaining_blockers`
- `待验证` 让“结果差对比 + 回审清单 + dated log”形成固定复盘闭环
- `待验证` 让主线升级、样板升级与商业化判断都按阶段推进，而不是混写成一次性完成

### 2.6 商业化目标态
- `待验证` 当前默认变现单元继续从“单 prompt”升级成“一个场景工作包（多个 prompt）”
- `待验证` 让视频更稳定地承担“证明壳 / 入口壳”，工作包更稳定地承担“产品单元”
- `待验证` `CPS` 可以作为副线路径继续观察
- `待验证` 但 `CPS` 目前仍只能写成建议判断，不得写成市场已成立

## 4. 目标态升级条件
任一目标态若要升级成当前正式事实，至少要同时满足：
1. 有真实文件证据或执行证据
2. 不再依赖聊天补解释
3. 已在根目录 10 份执行包里完成更新
4. 已同步回仓库镜像与主读取分支
5. 至少经过多轮样片复审，而不是单轮判断
6. 没有一票否决项且核心维度稳定达合格
7. 不再依赖“旧样片通过”或聊天补解释去证明新标准成立
