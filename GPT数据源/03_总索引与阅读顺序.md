## 0A. 当前运营入口

## 0B. 正式运营交付基线

当前正式运营阶段的视频交付默认只接受 `publish_candidate（可发布候选片）` 或 `blocked（阻断）`。`technical_preview（技术预览）`、`preflight package（执行前补全包）`、`silent preview（无声预览）`、无音轨视频、横屏技术包、只交 JSON / Markdown / route card 均为 `internal_diagnostic_only（内部诊断产物）` 或 `historical_internal_diagnostic_only（历史内部诊断产物）`，不能作为用户交付物。

当前默认出片比例为 `horizontal_16_9（横屏 16:9）`，默认交付分辨率为 `1920x1080`。旧 `vertical_9_16（竖屏 9:16）` 或 `1080x1920` 只按历史样片、历史提示卡、历史视觉参考或用户另行指定竖屏任务理解，不再作为正式运营默认交付比例。

## 0C. 用户反馈分工与禁止降级完成

当前正式运营阶段新增两条默认入口规则：

1. 用户只负责 `goal_correction（目标修正）`、`page_aesthetic_reference（页面 / 美观 / 观感对标）` 和 `result_quality_feedback（结果是否合格反馈）`；用户不负责替 GPT / Codex 诊断内部执行原因。
2. Codex 不得降级完成正式运营任务。fallback、技术预览、局部结果、内部诊断、本地未同步产物、无声视频、比例错误视频、只读报告或 route card 不能写 `completed`；做不到仓库写明的目标必须 `blocked`，降级方案只能在 blocked 后等待用户明确授权。

命中用户反馈“不合格 / 不对 / 不顺 / 不美观 / 不是我要的 / 文案画面对不上 / 标题被改 / 比例错 / 声音不行 / 字幕不对”时，先读 `11_项目状态动作总控器_机制推理层.md` 的 `self_repair_audit_required（自修审计必需）` 与 Codex 执行规则里的 `no_degrade_completion_gate（禁止降级完成闸门）`。

## 0D. 锁定文案契约与已发布不回炉边界

视频执行前必须有 `locked_copy_contract（锁定文案契约）`：`locked_topic / locked_title / locked_final_script / locked_opening_line / allowed_copy_changes / forbidden_copy_changes / copy_change_request_required_if_needed`。ChatGPT / 用户是最终落稿和文案锁定入口；Codex 只能执行锁定文案，不能自行改标题、选题、开头句、核心判断或人味表达。

如 Codex 判断文案无法执行，必须输出 `copy_change_request（文案修改请求）` 或 blocked，不能为了剪辑方便自行改稿。视频执行必须生成 `line_level_script_visual_alignment_gate（逐句文案画面对齐闸门）` 级别的 `script_to_timeline_map`，通常每 1-2 句一个 `line_group`；只有段落级映射不得进入成片生成。

导出前必须完成 `subtitle_card_overlap_check（字幕卡片重叠检查）`；high severity overlap 未修复不得完成。用户明确说“视频已经发了 / 已发布”时，触发 `post_publish_no_rework_boundary（已发布视频不默认回炉边界）`：当前视频进入数据回流和复盘，只修反馈记录、仓库机制或下一轮执行规则，不默认回炉该视频。

`11_项目状态动作总控器_机制推理层.md` 是 GPT Project / ChatGPT 侧最高机制入口。每轮先用它判断当前项目状态、事实源裁决、触发机制、下一动作、完成标准和反馈更新，再进入文案、剪辑、复盘、供料或执行。

`11_项目状态动作总控器_机制推理层.md` 也承载当前三大机制推理函数：

- `editing_inference_function（剪辑推理函数）`
- `content_route_inference_function（内容路由推理函数）`
- `quality_issue_classifier（质量短板分类器）`

这三个函数统一使用 `input_signal -> observed_evidence -> state_inference -> action_policy -> validation_rule -> blocked_if -> feedback_update`，分别约束 `editing_decision_pack（剪辑决策包）`、`content_route_card（内容路由卡）` 和 `quality_lock_card（质量锁卡）`，不推进动态状态。

`content_route_card V2（内容路由卡 V2）` 的标准输出模板写在 `05_文案路由规则.md`，用于每条真实视频执行前统一判断 `opening_route_decision（开头路由判断）`、`middle_carrier_decision（中段承载判断）`、`focusee_middle_editing_decision（FocuSee 中段剪辑判断）`、`card_placement_decision（卡片位置判断）`、`core_evidence（核心证据）`、API 生成人物、少量 PPT、Prompt 尾卡和 `blocked_if（阻断条件）`。

`script_anchor_extraction_function（文案锚点提取函数）` 的标准输出模板也写在 `05_文案路由规则.md`，用于每条真实视频执行前把最终文案补成句子级 `script_to_timeline_map（文案到时间线映射表）`、`tts_prosody_anchor_map（TTS 韵律锚点表）`、`opening_visual_hook_spec（开头视觉钩子规格，条件触发）` 和 `forbidden_visual_map（禁用画面表）`。缺这些前置字段时，不得进入视频执行、TTS 生成或高情绪开头生成。

上述 route card、timeline map、TTS 韵律锚点、剪辑 / 装配决策包和数据目标对齐检查都是 `preflight_required_inputs（执行前必备输入）`，不是正式运营视频交付结果。若不能交 `publish_candidate_ready_for_human_review`，必须写 `blocked_publish_candidate_unavailable`。
