# 24｜质量基线与抖音90分标准

## 1. 文件定位
本文件用于统一当前中文文件集的质量判断口径。

它负责：
- demo 的身份
- “抖音90分标准”的内部含义
- 当前主线下必须过线什么
- 哪些情况一票否决
- 视觉信息密度和 demo 感优先怎么查

它不负责：
- 单条执行日志
- 仓库分支状态
- 临时命令

## 2. 当前 demo 的身份
当前 demo 只代表：
- 链路能跑
- 是运行锚点
- 是最小闭环验证件

当前 demo 不代表：
- 质量已经过线
- 可以当质量样片
- 可以当长期基线

## 3. “抖音90分标准”的当前定义
当前这里说的“抖音90分标准”是项目内部质量简称，不是平台官方评分。

它指的是：
- 接近知识类 / AI 类短视频可发布测试线的质量水位
- 更接近创作者经验口径，而不是平台分数

## 4. 当前主线下的质量基线
当前默认主线固定为：
- `API生成真人`
- `用户录制素材`
- `少量PPT`
- `云端剪辑`

因此当前质量判断必须先看：
- `API生成真人` 是否真的承担了进入或判断
- `用户录制素材` 是否真的承担了中段主体推进
- `少量PPT` 是否只做整理与总结

## 5. 必须过线项
- 开头 3 秒有效
- `API生成真人` 段值，不空出场
- `用户录制素材` 真的承担主体推进
- 中段有过程感或证据感
- `少量PPT` 清楚但不过课件
- 前后变化或关键判断能被看懂
- 结尾有收束
- 用户看完后能拿走一个判断、动作、步骤或自检点
- 全片至少有一段真实证据，不只是抽象讲解
- `segment` 路由没有错位：判断归判断，证据归证据，整理归整理
- 没有因为先定固定时长而把节奏压坏

## 6. 一票否决项
- 主要承载仍像纯 demo 演示
- 旧版数字人口播路线明显出戏
- `API生成真人` 段强行撑时长，但不提供判断价值
- `用户录制素材` 太碎，整条看不出主线
- `少量PPT` 过多，重新滑回课件感
- 看完仍不知道表达什么
- 看完说不出下一步怎么做或怎么判断
- 全条只有观点，没有真实证据
- 全条只有共鸣，没有动作出口
- 人物段越权，抢走本该给录屏或总结卡承担的内容
- 录屏段失职，没有真正承担证据节点
- 总结卡越权，抢走主叙事
- 因先定固定时长而导致节奏失真
- 因 `segment` 路由错误而形成 demo 感 / 说明书感 / 课件感

## 7. 加分项
- `API生成真人段` 只在关键位置出现，进入和收束都干净
- `用户录制素材` 真有过程证据，不只是装饰切片
- `少量PPT` 能承担结构整理，但不抢主体
- 节奏不是匀速播报，有明显推进感
- 整条像真实会发的短视频，而不是系统拼装
- 看完不仅觉得“说得对”，还知道“我现在先做什么”

## 8. 视觉信息密度补充规则

### 8.1 当前优先查什么
当用户觉得“空 / 平 / 像 demo / 像课件 / 拼得碎”时，优先先查：
1. `API生成真人` 是否越权或失职
2. `用户录制素材` 是否真的承担主体推进
3. `少量PPT` 是否越权抢主叙事
4. 是否因为先定固定时长，逼得 segment 承载错位
5. 是否因为 `segment` 路由错误，才显得像 demo / 说明书 / 课件

### 8.2 当前白名单做法
- 关键位置才放 `API生成真人`
- 中段尽量让 `用户录制素材` 证明过程
- 讲到哪一步，画面推进到哪一步
- 关键词、结果句、数字句做单点显影
- 在关键结论处给处理时间

### 8.3 当前黑名单做法
- 为了“有脸”而过度堆判断段
- 素材乱切，假装有节奏
- 把旁白全文搬上 PPT
- 一页同时强化多个重点
- 靠花活掩盖结构问题

## 9. 当前阶段建议：时长与目标边界
以下只属于阶段性建议，不是正式事实。

### 9.1 当前更适合的主版与短钩子版
- 当前不再先写死统一固定时长
- 当前主版应先看“问题 → 动作 → 结果”是否完整，再反推时长
- `45–55` 秒仍可作为常见主版软区间参考，但不是当前项目的固定硬规则
- `25–30` 秒仍可作为短钩子软区间参考，但不是必须遵守的默认时长
- 证据更重、动作更完整的内容，允许比原先 `45–55` 秒更长

### 9.2 当前更适合的质量理解
- 若目标更偏 `爆款型`，可以允许前 3 秒钩子更强，但不能完全丢掉中段强证据
- 若目标更偏 `承接型`，则具体任务、前后差值、最小行动和中段强证据更重要

### 9.3 当前必须避免的偷换
- 不能因为短钩子版更短，就把中段证据整段删空
- 不能因为想追求“更容易爆”，就把整条内容重新做成只剩观点和情绪
- 不能把阶段性时长建议写成平台或市场已验证的正式标准

## 10. generation / 云端剪辑边界
当前资料口径必须写清：
- `云端剪辑` 是正式方向
- 不是“已经稳定跑通”的事实
- `local preview / local mp4` 可以是辅助产物
- 不能继续被写成默认正式交付口径

## 11. 当前质量判断顺序
1. 先承认旧 demo 不具备质量参考价值
2. 先看文案与 block 规划是否成立
3. 再看 block 是否继续拆到了 `segment`
4. 再看 `API生成真人 / 用户录制素材 / 少量PPT` 的职责分配
5. 再看有没有真实证据、最小行动和收束
6. 再看视觉信息密度和节奏有没有拖后腿
7. 最后再用“90分标准”判断是否接近可发布测试水位

## 12. 一句话规则
**当前不是拿旧 demo 定质量标准，也不是回到整条PPT旧路线或旧版数字人口播默认路线，而是按 `API生成真人 + 用户录制素材 + 少量PPT + 云端剪辑` 去压质量；若人物越权、录屏失职、总结卡越权，或因为先定固定时长导致 `segment` 路由错位，就不能写成真正过线。**
