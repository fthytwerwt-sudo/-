# 文案路由规则

## 1. 文件定位
本文件只负责 4 件事：
- 判断这轮文案属于哪一类
- 定义当前 `内容表达文案` 的默认交付包
- 定义 block / segment 承载分配
- 定义 Docker / 工作台 / 录制减负任务的默认修正动作

## 2. 文案一级分类
当前先分 4 类：
1. `内容表达文案`
2. `分发包装文案`
3. `关系建立文案`
4. `变现承接文案`

当前默认主流程只优先服务：
- `内容表达文案`

## 3. 当前默认交付包
只要这轮输出的是 `内容表达文案`，默认必须一起交付：
1. 完整脚本
2. `block -> 承载分配`
3. `4 段核心录制素材`
4. 录制动作说明
5. `三层 prompt 包`
6. 结果差证明点
7. `PPT / 总结卡文案`
8. `Prompt 引用尾卡`
9. 录制验收清单

缺任一项，只能算部分交付。

## 4. 三层 prompt 包

### 第 1 层：主 `prompt`
- 先把一句糊话压成更像可交付物的初稿

### 第 2 层：修结果 `prompt`
- 把“看起来对但不能交”的版本继续补成更稳的版本

### 第 3 层：展示 / 尾卡 `prompt`
- 服务总结卡与 `Prompt 引用尾卡`

## 5. `4 段核心录制素材`
当前录制减负默认只要求 4 段：
1. 原始糊话输入
2. 主 `prompt` 执行过程
3. 修结果 `prompt` 补强动作
4. 前后差值与总结收束

## 6. 当前更适合的结构建议
- 默认主结构：`问题 -> 动作 -> 结果`
- 当前更适合的开头：
  - `紧迫任务现场型`
  - `错误示范型`
- 当前更适合的中段强证据：
  - `用户真实操作录屏`
- 当前更适合的结尾收束：
  - `少量 PPT / 总结卡`
  - 总结之后允许补一张 `Prompt 引用尾卡`

## 7. block 与 segment 默认承载

| block | 默认承载 |
| --- | --- |
| 开头进入 | `API 生成真人` |
| 问题点破 | `API 生成真人` 或 `API 生成真人 + 用户录制素材` |
| 过程演示 | `用户录制素材` |
| 关键判断 | `API 生成真人` |
| 结构整理 | `少量 PPT` |
| 结尾收束 | `API 生成真人` 或 `少量 PPT` |
| `Prompt 引用尾卡` | `少量 PPT / 尾卡面板` |

当前明确禁止：
- 不允许先定人物次数，再硬塞文案
- 不允许让 `API 生成真人` 去替代中段本该由录屏承担的主体证据
- 不允许让 `少量 PPT` 去替代中段主体推进
- 不允许让 `Prompt 引用尾卡` 去承担主叙事

## 8. 当前交付闸门
一条 `内容表达文案` 在进入执行前，至少要同时回答：
1. block 是否已经继续拆成 `segment`
2. 中段是否真的由 `用户录制素材` 承担主体推进
3. `三层 prompt 包` 是否已经交齐
4. `4 段核心录制素材` 是否已经写清
5. `Prompt 引用尾卡` 是否只承担产品单元引用
6. 是否没有把 `云端剪辑` 写成已跑通事实

## 9. Docker / 工作台默认修正动作
若任务命中：
- Docker
- 工作台
- compose
- 录制减负
- 工作包生成
- prompt 生成
- 结果差对比

默认先检查：
- `compose.yaml`
- `docker/*`
- `scripts/*`
- `工作台模板/*`
- volume / mount / 路径漂移

默认动作：
1. 先确认当前工作台目标是不是中间层 scaffold
2. 先做最小可逆修正
3. 先修路径、挂载、volume 命名、模板缺口、脚本参数问题
4. 再做 `docker compose config` 或最小脚本生成验证

必须明确：
- 默认先尝试最小修正，不先把问题丢给用户
- 但不能把“定义了默认修正动作”偷写成“链路已稳定跑通”

## 10. 一句话规则
**当前文案路由不是“先写一版再找东西塞进去”，而是先判断文案类型，再拆 block，再拆 `segment`；默认交付已经升级为 `4 段核心录制素材 + 三层 prompt 包 + Prompt 引用尾卡`，并在命中 Docker / 工作台任务时先做最小可逆修正。**
