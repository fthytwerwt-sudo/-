# 文案路由规则

## 1. 文件定位
本文件不是统一文案模板，而是当前正式资料包的文案路由规则。

它负责：
- 先判断这轮文案属于哪一类
- 再判断这轮内容如何分配给 `API 生成真人`、`用户录制素材`、`少量 PPT`
- 再决定是否能进入样片执行

它不负责：
- 商业化长文规则全文
- 正式事实清单
- lane / parallel 执行细则

## 2. 文案一级分类
当前文案先分 4 类：
1. `内容表达文案`
2. `分发包装文案`
3. `关系建立文案`
4. `变现承接文案`

当前默认优先级：
1. `内容表达文案`
2. `分发包装文案`
3. `关系建立文案`
4. `变现承接文案`

## 3. 当前默认主流程只优先服务什么
当前根目录正式主线，默认只优先服务：
- `内容表达文案`

原因很简单：
- 当前要先把主线口径收稳
- 先把默认承载写清
- 不让分发、关系或变现文案反过来抢走主线

### 当前默认文案交付包协议
只要这轮输出的是 `内容表达文案`，默认交付就不算“只给一版脚本”。

以后默认必须一起交付：
1. 完整脚本
2. `block -> 承载分配`
3. 录制素材清单
4. 录制动作说明
5. 豆包输入 `prompt`
6. 结果差证明点
7. `PPT / 总结卡文案`
8. 录制验收清单

缺任一项，只能算部分交付，不能直接冒充“文案已完成”。

## 4. 当前阶段更适合的结构建议
以下属于当前阶段建议，不是长期唯一模板。

### 4.1 当前主结构
当前更适合优先采用：

- `问题 → 动作 → 结果`

它更适合当前主线的原因是：
- 开头更容易用一个真实问题进入
- 中段更容易把动作交给 `用户录制素材`
- 结尾更容易把结果和最小行动收住

### 4.2 当前更适合的开头
当前更适合优先测试：
- `紧迫任务现场型`
- `错误示范型`

其中：
- `紧迫任务现场型` 更适合抓前 3 秒和真实任务感
- `错误示范型` 更适合点破误判和提升共鸣

### 4.3 当前更适合的中段证据
当前更适合把中段强证据交给：

- `用户真实操作录屏`

必须明确：
- 录屏不是装饰切片
- 录屏要承担主体推进和过程证明
- 不能让人物段去替代录屏本该承担的证据节点

### 4.4 当前更适合的结尾动作
当前更适合优先测试：
- 评论区关键词承接
- 私信承接
- 资料领取型承接
- 最小行动 / 自检句收束

## 5. 当前内容表达文案的两种主路由

### 5.1 `爆款型文案`
当前阶段更适合用于：
- 抢前段留存
- 测试强钩子
- 提升完播

更适合的写法是：
- 开头更猛
- 情绪钩子更强
- 中段只保留最强的 1 到 2 个证据节点
- 结尾给一句明确收束或简单承接动作

### 5.2 `承接型文案`
当前阶段更适合用于：
- 让用户看懂具体场景
- 承接资料包、私信、咨询或服务
- 沉淀收藏和长尾转化

更适合的写法是：
- 场景更具体
- 中段证据更完整
- 前后差值更清楚
- 结尾给最小行动或明确承接句

必须明确：
- `爆款型文案` 和 `承接型文案` 不是同一个目标
- 更容易出爆，不等于更容易变现
- 更容易承接，不等于更容易冲前段留存

## 6. block 与 segment 承载路由
`内容表达文案` 进入主流程后，必须继续拆 block，再拆到 `segment`。

### 常见 block
- 开头进入
- 问题点破
- 过程演示
- 关键判断
- 结构整理
- 结尾收束

### 默认承载映射
| block | 默认承载 |
|---|---|
| 开头进入 | `API 生成真人` |
| 问题点破 | `API 生成真人` 或 `API 生成真人 + 用户录制素材` |
| 过程演示 | `用户录制素材` |
| 关键判断 | `API 生成真人` |
| 结构整理 | `少量 PPT` |
| 结尾收束 | `API 生成真人` 或 `少量 PPT` |

### segment 功能 → 承载
- 判断节点 → `API生成真人`
- 证据节点 → `用户录制素材`
- 整理节点 → `少量PPT / 总结卡`
- 转场节点 → 看它更像判断还是整理，不单独乱加人物

### 当前必须明确禁止
- 不允许先定人物次数，再硬塞文案
- 不允许先定结尾一定 3 秒，再倒推总结卡要写什么
- 不允许因为一段字多，就默认给人物硬撑
- 不允许因为没有判断，就给人物强行补出场
- 不允许让 `API 生成真人` 去替代中段本该由录屏承担的主体证据
- 不允许让 `少量 PPT` 去替代本该由录屏或人物承担的主体推进

## 7. 当前阶段的时长与配比建议
以下只属于阶段性建议，不是正式事实：

- 不再先写死统一固定时长
- 主版时长应先按文案完整度、证据强度与动作出口反推
- 短钩子版可以更短，但不能靠压短时长删空关键证据
- `爆款型文案 : 承接型文案` 当前更适合先按 `7:3` 测试

补充理解：
- `45–55` 秒仍可作为当前常见主版软区间参考，但不是固定硬规则
- `25–30` 秒仍可作为短钩子软区间参考，但不是默认必须遵守的死时长
- 若一条内容的证据更重、动作更完整，允许比原先 `45–55` 秒更长
- 若一条内容只承担强钩子验证，允许明显短于主版时长

### 项目默认经验值
下面这组只属于当前项目默认经验值，不是行业硬标准：

- 人物段时长 ≈ 口播字数 ÷ 4.5–5.5 + `0.2–0.4` 秒缓冲
- 总结卡时长 ≈ 屏幕总字数 ÷ 5.5–6.5 + `0.4–0.8` 秒缓冲
- 单模块最短不要低于 `1.4` 秒
- 人物单段若超过 `22–26` 字，默认先怀疑过长，优先拆成“人物判断 + 录屏 / 卡片”
- 录屏段不按字数，而按“至少看清一个完整动作或明确变化”判断时长

### 当前一句核心判断
**先判这一小段在干什么，再判这段字有多少、观众来不来得及看。功能决定承载，字数决定时长。**

## 8. `API 生成真人段` 的出现次数
当前只能这样写：
- `API 生成真人段 1 次`
- `API 生成真人段 2 次`

当前不能这样写：
- “只写 1 次，却不说明是 API 生成真人段”
- “只写 2 次，却不说明是 API 生成真人段”

规则：
- `API 生成真人段` 出现 1 次还是 2 次，是 block 路由结果
- 不是固定模板
- 也不是默认一定要更多“像人”的镜头

## 9. 一个最小示例
题目：**为什么你明明用了 AI 写汇报，最后还是得自己重写一遍？**

一个 50 秒最小分发示例：
- 开头判断：`API生成真人`
  - 点破“问题不在写，而在没先压清判断口径”
- 中段证据 1：`用户录制素材`
  - 展示目标、边界、验收被整理进执行清单
- 中段证据 2：`用户录制素材`
  - 展示 AI 输出和人工修回之间的差值
- 中后段整理：`少量PPT / 总结卡`
  - 只显影 3 个关键词，不抢主叙事
- 若中后段出现额外判断拐点：
  - 才允许再补 1 段 `API生成真人`

这条示例要说明的不是模板，而是：
- 不是每段都用人物
- 不是每段都用总结卡
- 而是按“判断 / 证据 / 整理”分发

## 10. `真人实拍出镜` 的位置
若某条内容必须使用 `真人实拍出镜`，必须同时满足：
1. 显式写出 `真人实拍出镜`
2. 显式标注为非默认分支
3. 不得把它回写成当前唯一正式主线

## 11. 进入样片执行前的 9 条检查
1. 这轮文案是哪一类？
2. 当前是否仍属于 `内容表达文案` 主流程？
3. 当前更像 `爆款型文案` 还是 `承接型文案`？
4. block 是否已经继续拆成 `segment`？
5. 开头是否明确交给 `API 生成真人`？
6. 中段是否真的有 `用户录制素材` 承担主体推进？
7. `少量 PPT` 是否只承担结构整理与收束？
8. 是否没有把 `云端剪辑` 写成“已跑通事实”？
9. 默认文案交付包是否已经把录制素材、录制动作、豆包 `prompt`、结果差与验收清单一起交齐？

## 12. 默认改稿顺序
当前默认按下面顺序改：
1. 先改开头判断句
2. 再改中段证据层
3. 再改 block 承载分配
4. 再把 block 继续拆成 `segment`
5. 再反推人物、录屏、总结卡时长
6. 最后才改语气和节奏

## 13. 一句话规则
**当前文案路由不是“先写一版再找东西塞进去”，而是先判断文案类型，再拆 block，再拆 `segment`；当前阶段更适合用“紧迫任务现场型 / 错误示范型”开头，把“问题 → 动作 → 结果”压清，再把中段强证据交给 `用户录制素材`，并把录制素材包与豆包 `prompt` 包一起交付。**
