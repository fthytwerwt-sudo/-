# 文案路由规则

## 1. 文件定位
本文件只负责 4 件事：
- 判断这轮文案属于哪一类
- 定义当前 `内容表达文案` 的默认交付包
- 定义 block / segment 承载分配
- 定义 Docker / 工作台 / 录制减负任务的默认修正动作

## 2. 文案一级分类
当前先分 4 类：
1. `内容表达文案`
2. `分发包装文案`
3. `关系建立文案`
4. `变现承接文案`

当前默认主流程只优先服务：
- `内容表达文案`

## 3. 当前默认交付包
只要这轮输出的是 `内容表达文案`，默认必须一起交付：
1. 完整脚本
2. `block -> 承载分配`
3. `4 段核心录制素材`
4. 录制动作说明
5. `三层 prompt 包`
6. 结果差证明点
7. `PPT / 总结卡文案`
8. `Prompt 引用尾卡`
9. 录制验收清单

缺任一项，只能算部分交付。

## 3A. 默认文案生产路由

以后《视频工厂》AI 知识类视频默认按四方分工推进：

1. `Perplexity（外部参考检索 / 研究工具）` 输出 `reference pack（参考包）` 和 `raw feeling draft（原感初稿）`。
   - 只提供外部参考、表达方向、案例、梗和原感。
   - 不负责最终定稿，不能直接进入执行。
2. 用户录制真实素材。
   - 包括录屏、网页操作、结果对比、过程证据、关键截图。
   - 素材决定最终稿能写到多具体。
3. `Codex（执行代理）` 做素材技术检查与细节证据报告。
   - 目标不是写最终文案，而是把素材细节交给 `ChatGPT（最终落稿与复审入口）`。
   - 必须给出能写稿的细节，不能只报“素材存在 / 技术通过”。
4. `ChatGPT（最终落稿与复审入口）` 基于参考包、用户素材和 `Codex（执行代理）` 细节报告写最终落稿。
   - 默认最终稿出口是 `ChatGPT（最终落稿与复审入口）`。
   - 必须保证文案贴真实素材，不能写素材里没有的页面、按钮、动作和结果。
5. `Codex（执行代理）` 按最终稿执行视频、字幕、声音、卡片和审片包。
   - 执行完成后回到 `ChatGPT（最终落稿与复审入口）` 复审。

## 3B. `Codex（执行代理）` 素材细节汇报标准

`Codex（执行代理）` 检查素材时，必须按“给 `ChatGPT（最终落稿与复审入口）` 写最终稿”的标准汇报。

至少包含：

1. 素材清单
   - 文件路径、文件名、时长、分辨率、是否能打开、是否有音频。
   - 是否有明显卡顿、黑屏、模糊、遮挡。
2. 时间码细节
   - 每个关键动作从几秒开始，到几秒结束。
   - 每个画面段落对应什么内容。
   - 哪些时间段可做反面展示，哪些时间段可做正面展示，哪些时间段适合配口播解释。
3. 画面可见信息
   - 页面名称、按钮名称、输入框名称、标签名称。
   - 弹窗、提示、报错。
   - 生成结果中的可读文字。
   - 页面变化前后差异。
4. 操作步骤细节
   - 用户进入哪个网站、工具或页面。
   - 点击了哪个入口。
   - 输入了什么内容。
   - 哪个按钮触发了生成。
   - 生成后出现了什么结果。
   - 如果有 `API（应用程序接口）`、后台或控制台类页面，必须写清入口位置和动作。
   - 颗粒度示例：进入阿里百炼大模型服务平台，找到 `API Key（接口密钥）` 入口，点击新建 `Key（密钥）`，复制生成的 `Key（密钥）` 到项目配置里。
   - 上面只是颗粒度示例，不能编造素材里没有的操作。
5. 证据判断
   - 这段素材能证明什么。
   - 不能证明什么。
   - 哪些画面足够支撑文案。
   - 哪些文案点素材没有支撑，必须删掉或改写。
   - 哪些地方需要补录。
6. 给 `ChatGPT（最终落稿与复审入口）` 的写稿建议
   - 哪些细节应该写进口播。
   - 哪些细节只适合字幕或卡片。
   - 哪些地方适合微反转。
   - 哪些地方适合轻吐槽。
   - 哪些地方不适合写得太满。

## 4. 三层 prompt 包

### 第 1 层：主 `prompt`
- 先把一句糊话压成更像可交付物的初稿

### 第 2 层：修结果 `prompt`
- 把“看起来对但不能交”的版本继续补成更稳的版本

### 第 3 层：展示 / 尾卡 `prompt`
- 服务总结卡与 `Prompt 引用尾卡`

## 5. `4 段核心录制素材`
当前录制减负默认只要求 4 段：
1. 原始糊话输入
2. 主 `prompt` 执行过程
3. 修结果 `prompt` 补强动作
4. 前后差值与总结收束

## 6. 当前更适合的结构建议
- 默认主结构：`问题 -> 动作 -> 结果`
- 当前更适合的开头：
  - `紧迫任务现场型`
  - `错误示范型`
- 当前更适合的中段强证据：
  - `用户真实操作录屏`
- 当前更适合的结尾收束：
  - `少量 PPT / 总结卡`
  - 总结之后允许补一张 `Prompt 引用尾卡`

## 7. block 与 segment 默认承载

| block | 默认承载 |
| --- | --- |
| 开头进入 | `API 生成真人` |
| 问题点破 | `API 生成真人` 或 `API 生成真人 + 用户录制素材` |
| 过程演示 | `用户录制素材` |
| 关键判断 | `API 生成真人` |
| 结构整理 | `少量 PPT` |
| 结尾收束 | `API 生成真人` 或 `少量 PPT` |
| `Prompt 引用尾卡` | `少量 PPT / 尾卡面板` |

当前明确禁止：
- 不允许先定人物次数，再硬塞文案
- 不允许让 `API 生成真人` 去替代中段本该由录屏承担的主体证据
- 不允许让 `少量 PPT` 去替代中段主体推进
- 不允许让 `Prompt 引用尾卡` 去承担主叙事

## 8. 当前交付闸门
一条 `内容表达文案` 在进入执行前，至少要同时回答：
1. block 是否已经继续拆成 `segment`
2. 中段是否真的由 `用户录制素材` 承担主体推进
3. `三层 prompt 包` 是否已经交齐
4. `4 段核心录制素材` 是否已经写清
5. `Prompt 引用尾卡` 是否只承担产品单元引用
6. 是否没有把 `云端剪辑` 写成已跑通事实

## 8A. 方案 B 三类骚萌卡结构机制与节奏预算

### 8A.1 用户确认口径

`已确认` 用户本轮确认的新结构不是“多加几张好看的卡”，而是把 `骚萌卡` 写成文案功能触发机制：

- `已确认` 正面也需要一张骚萌卡，因为正面之后文案也会有反转。
- `已确认` 有反转的阶段，就可以触发一张骚萌卡。
- `已确认` 元素娃娃 slogan 后面还有一个骚萌卡位置。
- `已确认` 三张骚萌卡必须风格统一，不能变成不同角色。
- `已确认` 三张骚萌卡都必须搞笑一点，不能只可爱、不能只说明、不能变成普通信息卡。
- `已确认` 每张骚萌卡主要表现什么、承载什么，必须在文案路由阶段先写清。
- `已确认` 当前不是为了每个位置都加好看的东西，而是要保证视频可观看性。
- `已确认` 骚萌卡必须和文案、视频节奏搭配。

### 8A.2 总机制

`已确认` 骚萌卡的定义：

- 骚萌卡不是装饰图。
- 骚萌卡不是固定贴片。
- 骚萌卡是“文案功能触发的搞笑人格分身”。
- 三张骚萌卡必须统一风格，但承担不同叙事职责。
- 三张骚萌卡都必须有搞笑感。
- 骚萌卡必须和文案反转 / 问题钩子 / 视频停顿配合，不能只因为画面好看就插。
- 如果使用三张骚萌卡，必须从后半段回收时长，优先压缩或删除 `回场主持壳（API 生成真人回场段）` 和 `judgment_card（判断总结卡）`。

默认结构顺序：

1. 元素娃娃 slogan（开头定性）
2. `problem_hook_sassy_card（问题钩子骚萌卡）`
3. 反面展示 / 反面录屏
4. `negative_reversal_sassy_card（反面反转骚萌卡）`
5. 正面展示 / 正面录屏
6. `positive_reversal_sassy_card（正面反转骚萌卡）`
7. 结果差提示卡
8. `Prompt 引用尾卡`

### 8A.3 `problem_hook_sassy_card（问题钩子骚萌卡）`

位置：
- 元素娃娃 slogan 后，反面展示前。

核心职责：
- 点出问题。
- 把观众从“听介绍”拉到“这说的不就是我吗”。
- 用骚萌搞笑的方式把问题讲出来。

它承载：
- 当前视频的真实问题。
- 用户痛点。
- 一句轻吐槽。
- 进入反面展示的理由。

它不承载：
- 不讲完整方法。
- 不解释正反流程。
- 不展示结果差。
- 不做总结。
- 不做产品介绍。

情绪：
- 发现问题。
- 眯眼吐槽。
- 轻微嫌弃但可爱。
- 必须搞笑，不能只是卖萌。

表情动作：
- 挑眉。
- 侧眼。
- 捂嘴偷笑。
- 手指轻点。
- 像发现大家都在犯同一个小毛病。

文案例子：
- “我发现一个问题：很多人用 AI，不是不会问，是第一句就把它问懵了。”
- “你不是不会用 AI，你是开头就把任务丢成一团。”
- “很多人用 AI，第一步就把路问歪了。”

建议时长：
- `1.6s-2.2s`

失败线：
- 变成普通解释卡。
- 文案太正经。
- 不搞笑。
- 像课程标题页。
- 和元素娃娃 slogan 重复。

### 8A.4 `negative_reversal_sassy_card（反面反转骚萌卡）`

位置：
- 反面录屏里，旧做法结果刚露馅之后。

核心职责：
- 替观众吐槽翻车结果。
- 把“看起来有东西，实际不能用”的落差变成笑点。

它承载：
- 反面结果的反转。
- 空泛 / 泛泛而谈 / 看起来对但交不了的落差。
- 观众的无语感。
- 第一处强笑点。

它不承载：
- 不讲原因。
- 不讲方法。
- 不提前讲正面答案。
- 不做产品介绍。

情绪：
- 贱萌吐槽。
- 得瑟拆台。
- 欠揍可爱。
- 必须搞笑，不能只是可爱。

表情动作：
- 挑眉。
- 小坏笑。
- 捂嘴偷笑。
- 手一摊。
- 像在说“你看，我就知道会这样”。

文案例子：
- “方案很满，用起来空。”
- “看着像方案，实际像空气。”
- “字挺多，但没一处能直接用。”
- “这不是初稿，这是礼貌废话。”

建议时长：
- `1.2s-1.6s`

失败线：
- 反面结果还没露出来就插。
- 只卖萌不吐槽。
- 把观众骂了。
- 破坏录屏证据链。
- 时长太长导致反面展示断掉。

### 8A.5 `positive_reversal_sassy_card（正面反转骚萌卡）`

位置：
- 正面录屏里，新做法结果明显变好之后。

核心职责：
- 放大“这回真的能用了”的正向反转。
- 把正面结果的可用度提升变成第二个轻笑点。

它承载：
- 正面结果突然变清楚。
- 工作包 / prompt 调整后的差值。
- “终于像能交的初稿了”的爽点。
- 第二处轻笑点。

它不承载：
- 不继续吐槽旧做法。
- 不重复解释方法。
- 不抢结果差提示卡的总结职责。

情绪：
- 小得意。
- 小惊喜。
- 轻微炫耀。
- 像在说“这才对嘛”。
- 必须搞笑，不能变成普通鼓励卡。

表情动作：
- wink（眨眼）。
- 点头。
- 小骄傲。
- 小手势。
- 歪头得瑟。
- 轻微小扭动。

文案例子：
- “诶，这回像能交的初稿了。”
- “这才像回事嘛。”
- “你看，顺序一对，结果就不飘了。”
- “终于不是空气方案了。”

建议时长：
- `1.2s-1.6s`

失败线：
- 正面结果没有明显反转却硬插。
- 还在继续吐槽旧做法，导致情绪重复。
- 变成炫技。
- 抢走正面录屏的证据主体。

### 8A.6 统一风格锁定

所有骚萌卡必须保持同一视觉风格：

- 原创 Q 版 AI 向导 / 可爱元素主持娃娃。
- 高质量 3D / GIF 感。
- 9:16 竖屏整页 reaction card（整页反应卡）。
- 橙黄爆炸 / 放射线 / 表情包式背景。
- 大字 punchline（大字笑点）清楚可读。
- 表情骚萌、得瑟、戏精、欠揍可爱。
- 必须搞笑。
- 同一个角色体系，不能每张像换了角色。
- 不能写实人物。
- 不能普通贴纸。
- 不能低幼卖萌。
- 不能性感擦边。
- 不能出现胸口 AI 标识。
- 不能使用已有 IP / meme 角色 / 平台 UI。
- 不能回到本地程序绘图感。

统一情绪边界：

可以：
- 挑眉。
- wink（眨眼）。
- 捂嘴偷笑。
- 歪头。
- 侧眼。
- 吐槽脸。
- 小得意。
- 小坏笑。
- 轻微扭动。

不要：
- 大哭。
- 惊恐。
- 抱头崩溃。
- X 眼。
- 旋涡眼。
- 尖叫嘴。
- 撒娇。
- 擦边。
- 幼态。
- 软色情。
- 低幼装可爱。

### 8A.7 触发规则

`已确认` 骚萌卡不是按段落固定出现，而是按文案功能触发。

触发条件：

1. 问题钩子触发：
   - 当元素娃娃 slogan 后，需要用一句人话点出用户痛点时，触发 `problem_hook_sassy_card（问题钩子骚萌卡）`。
2. 反面反转触发：
   - 当旧做法结果暴露出“空泛 / 不能用 / 看着对但交不了”时，触发 `negative_reversal_sassy_card（反面反转骚萌卡）`。
3. 正面反转触发：
   - 当新做法结果暴露出“突然清楚 / 能交 / 前后差明显”时，触发 `positive_reversal_sassy_card（正面反转骚萌卡）`。

禁止触发：
- 没有文案钩子。
- 没有反转。
- 没有停顿。
- 只是为了画面热闹。
- 会打断录屏证据链。
- 会让视频超过节奏预算。
- 会让骚萌卡和真实录屏争夺主体。

### 8A.8 节奏预算原则

当前 round34 结构预算锚点：

- `已确认` 当前 full 约 `56 秒`，精确记录为 `55.92s`。
- `已确认` 当前中段约 `28.5 秒`，精确记录为 `28.52s`。
- `已确认` 三张骚萌卡预计总时长约 `4.0s-5.4s`。

三张骚萌卡建议预算：

| card | 建议时长 |
| --- | --- |
| `problem_hook_sassy_card（问题钩子骚萌卡）` | `1.6s-2.2s` |
| `negative_reversal_sassy_card（反面反转骚萌卡）` | `1.2s-1.6s` |
| `positive_reversal_sassy_card（正面反转骚萌卡）` | `1.2s-1.6s` |
| 合计 | `4.0s-5.4s` |

如果启用三张骚萌卡，必须从后半段回收至少 `4-6 秒`。

优先压缩 / 删除：
- `shot07_clean_host（回场主持壳）`
- `shot08_judgment_card（判断总结卡）`

不压缩：
- `shot03_negative_recording（反面真实录屏）`
- `shot05_positive_recording（正面真实录屏）`
- `shot09_prompt_tail（Prompt 引用尾卡）` 的必要展示

节奏收束规则：
- `shot06_result_diff_card（结果差提示卡）` 负责收住判断，不再继续搞笑。
- 如果保留三张骚萌卡，`回场主持壳（API 生成真人回场段）` 默认删除或极短化。
- `judgment_card（判断总结卡）` 默认合并进 `结果差提示卡（结果差卡）`。
- `Prompt引用尾卡（Prompt 引用尾卡）` 保留，但不得重复讲结论。

### 8A.9 最终一句规则

骚萌卡的风格必须统一，且每张都要搞笑；但每张卡承载的功能不同：

`problem_hook_sassy_card（问题钩子骚萌卡）` 负责“把问题讲得好笑”；
`negative_reversal_sassy_card（反面反转骚萌卡）` 负责“把翻车讲得好笑”；
`positive_reversal_sassy_card（正面反转骚萌卡）` 负责“把变好讲得好笑”；

`结果差提示卡（结果差卡）` 负责收住判断，`Prompt引用尾卡（Prompt 引用尾卡）` 负责承接工作包。

## 9. Docker / 工作台默认修正动作
若任务命中：
- Docker
- 工作台
- compose
- 录制减负
- 工作包生成
- prompt 生成
- 结果差对比

默认先检查：
- `compose.yaml`
- `docker/*`
- `scripts/*`
- `工作台模板/*`
- volume / mount / 路径漂移

默认动作：
1. 先确认当前工作台目标是不是中间层 scaffold
2. 先做最小可逆修正
3. 先修路径、挂载、volume 命名、模板缺口、脚本参数问题

必须明确：

## 10. 一句话规则
**当前文案路由不是“先写一版再找东西塞进去”，而是先判断文案类型，再拆 block，再拆 `segment`；默认交付已经升级为 `4 段核心录制素材 + 三层 prompt 包 + Prompt 引用尾卡`，并在命中 Docker / 工作台任务时先做最小可逆修正。**
