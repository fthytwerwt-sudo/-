# 文案路由规则

## 0A. 正式运营路由约束

当前阶段为 `formal_operation_active`。任何正式文案改写前，必须先确认 `operation_review` 是否完成、`current_data_goal_anchor` 是否 ready；当前 V003 仍是 `partial_data_recorded`，不得直接生成下一条正式视频执行 prompt。

## 0A-1. 正式运营视频交付边界

若文案路由继续进入做视频、产视频、发片候选、运营内容、下一条视频或发布候选，默认交付结果必须是 `publish_candidate_ready_for_human_review（可发布候选片，待人工复审）` 或 `blocked_publish_candidate_unavailable（可发布候选片不可交付阻断）`。

当前默认出片比例为 `horizontal_16_9（横屏 16:9）`，默认交付分辨率为 `1920x1080`；适配屏幕录制、文档、表格、指标分层和客资评分类内容。旧 `vertical_9_16（竖屏 9:16）`、`1080x1920` 不再作为正式运营默认交付比例。

用户提供录屏 / 桌面素材进入正式视频执行时，必须优先应用 `source_native_no_mask_visual_rule（源素材比例 + 无遮挡视觉规则）`。该规则优先于旧固定横屏交付口径：除非用户本轮明确指定横屏 16:9 / 1920x1080，Codex 不得为了统一比例强行补灰边、白边、黑边、letterbox、pillarbox 或 padding bands。

`content_route_card（内容路由卡）`、`script_to_timeline_map（文案到时间线映射表）`、`tts_prosody_anchor_map（TTS 韵律锚点表）`、`editing_decision_pack（剪辑决策包）`、`assembly_decision_pack（装配决策包）`、`data_goal_alignment_check（数据目标对齐检查）` 只属于执行前必备条件，不是用户最终视频交付物。`technical_preview（技术预览）`、无声预览、横屏技术包或只交 JSON / Markdown route card，不能写成文案 / 视频执行完成。

## 0A-1b. source_native_no_mask_visual_rule（源素材比例 + 无遮挡视觉规则）

`已确认` 用户提供素材时，默认执行规则是“用户给什么素材，就优先使用什么素材本身”。除非用户本轮明确授权，Codex 不得为了隐私、美观、比例、字幕、卡片或平台风险主动遮挡、洗白、模糊或重塑用户素材。

硬规则：

1. 默认不遮挡用户提供素材；默认不使用 `privacy whiteout / full-frame mask / gray dim overlay / black block / final-stage whiteout`。
2. 默认不强制 `16:9`，默认不强制 `1920x1080` 横屏；优先使用 `source_native_ratio（源素材比例）` 或 `user_provided_material_ratio（用户提供素材比例）`。
3. 多素材比例不一致时，优先保持核心素材可读；允许等比 cover 与最小裁切，但不得用灰边 / 白边 / 黑边 / padding bands 填充。
4. 字幕和卡片不得遮挡中段核心证据；需要卡片时，优先使用独立卡片段、前后转场卡或 sidecar 字幕，不盖住真实商品卡、表格、聊天框结论。
5. 商品名、商品价格、佣金、月销、商品分、店铺分、表格字段、云盘文件名、浏览器边栏和页面结构，默认不自动遮挡；只在用户明确授权时做局部 blur / 小块 mask，并记录授权来源。
6. 如素材里出现真实 `secret / API key / token / 身份证号 / 银行卡 / 手机号 / 私人地址 / 明确不可公开隐私`，不得擅自遮挡后继续，必须 blocked 并请求用户授权处理。
7. 如果遮挡后商品卡、表格或聊天框结论不可读，不能交片；宁可 blocked，也不得把遮挡后的不可读画面写成可发布候选。

## 0A-2. locked copy and line-level execution gate

正式文案进入视频执行前，必须建立 `locked_copy_contract（锁定文案契约）`。它必须包含 `locked_topic / locked_title / locked_final_script / locked_opening_line / allowed_copy_changes / forbidden_copy_changes / copy_change_request_required_if_needed`。

Codex 执行视频时不得重新定稿。允许做素材映射、字幕分句、TTS 停顿、剪辑节奏、卡片位置、证据窗口和导出参数；不允许改标题、选题、开头句、核心判断、人味表达、语义或视觉标题卡标题。若需要改文案，必须输出 `copy_change_request（文案修改请求）`，等待 ChatGPT / 用户确认。

`script_to_timeline_map（文案到时间线映射表）` 必须做到逐句 `line_group` 级别，通常每 1-2 句一组。每组必须包含 `line_group_id / narration_text / source_timecode / expected_visual / allowed_visuals / forbidden_visuals / subtitle_text / card_text_if_any / evidence_strength / alignment_status / blocked_if_visual_mismatch`。只有段落级映射、画面无法直接支撑口播、或出现“文案说 A，画面显示 B”时，必须修复、请求改稿或 blocked。

字幕和卡片生成后必须跑 `subtitle_card_overlap_check（字幕卡片重叠检查）`。标题卡、解释卡、总结卡、口播字幕、画面 OCR 和关键证据区域出现 high severity overlap 时，必须修复；修不了不得导出或写完成。

## 0A-3. material_evidence_gate 默认素材证据闸门

以后凡是文案继续进入 `video_sample_or_assembly（视频样片 / 装配）`、`publish_candidate_delivery（发布候选交付）`、`locked_copy_video_execution（锁定文案视频执行）` 或 `video_repair_execution（视频修复执行）`，不得只凭“主题相关”把文案硬配到素材。

默认链路必须变成：

```text
material_detail_report
-> material_evidence_contract
-> line_group_evidence_gate
-> auto_storyboard_preflight_report
-> auto_edit_allowed check
-> render only if passed
```

`material_evidence_contract（素材证据契约）` 必须从素材细节报告提取每段素材能证明什么、不能证明什么、最佳用途、禁止用途、平台风险、隐私风险和公开安全性。`background_context_only（背景上下文）` 不能当 direct evidence（直接证据）；`privacy_risk = high` 的素材不得默认入片；`cannot_support` 明确禁止的文案点不得绑定该素材。

`line_group_evidence_gate（句组证据闸门）` 必须逐个 `line_group（句组）` 输出 `claim_type / required_evidence_type / selected_visual_type / material_evidence_refs / evidence_match_status / mismatch_risk / blocked_if`。硬规则：

- `data_sentence（数据句）` 必须绑定真实数据来源、素材直接证据或 `data_result_card（数据成果卡）`。
- `judgment_sentence（判断句）` 如果素材不能直接证明，必须用 HyperFrames 判断卡承接，不得硬配普通录屏。
- `action_sentence（动作句）` 必须用明确动作画面、动作卡或成果卡，不得硬配主题相关录屏。
- `boundary_sentence（边界句）` 优先用边界卡。
- `process_sentence（过程句）` 才适合用真实录屏素材。
- `proxy_match（代理匹配）` 只能用于过渡句 / 背景句，不能当证明。

视频生成前必须生成 `auto_storyboard_preflight_report（自动分镜预检报告）`。若 `blocked_no_evidence_count / high_mismatch_risk_count / privacy_high_selected_count / data_sentence_without_source_count / judgment_sentence_hard_mapped_to_recording_count / action_sentence_without_card_or_direct_visual_count / selected_material_in_cannot_support_count / card_required_unresolved_count` 任一大于 0，`auto_edit_allowed = false`，必须 `blocked`，不得生成 `full.mp4`。

## 0A-4. prompt_delta 与视频执行组件判断

文案进入视频执行链路后，GPT prompt 只代表本轮 `prompt_delta（增量目标）`，不代表完整流程源。Codex 必须先跑 `process_boot_gate（流程启动闸门）`，输出 `process_boot_report（流程启动报告）`，再决定能否进入素材映射、TTS、字幕、卡片、时间线或审片包执行。

prompt 没写某个组件，不代表可静默跳过。以下组件必须进入 `component_decision_table（组件判断表）`：

```text
component_decision_table:
  judgment_card:
    needed:
    reason:
    not_needed_reason:
  summary_card:
    needed:
    reason:
    not_needed_reason:
  result_diff_card:
    needed:
    reason:
    not_needed_reason:
  prompt_tail_card:
    needed:
    reason:
    not_needed_reason:
  tts_prosody_anchor_map:
    needed:
    reason:
    not_needed_reason:
  script_to_timeline_map:
    needed:
    reason:
    not_needed_reason:
  data_goal_alignment_check:
    needed:
    reason:
    not_needed_reason:
```

硬规则：

- 判断卡、总结卡、结果差卡、Prompt 尾卡是否出现，由 `content_route_card V2`、`card_placement_decision`、素材证据、数据目标和观众理解路径共同判断，不由 prompt 点名决定。
- 卡片需要就执行；不需要必须写 `not_needed_reason（不需要理由）`。
- `script_to_timeline_map` 与 `tts_prosody_anchor_map` 是视频执行默认前置项；不能因为 prompt 没写就省略。
- `data_goal_alignment_check` 命中正式运营视频执行时默认必需；缺失时不得写视频执行完成。
- 若组件判断需要改 locked 文案语义，Codex 只能输出 `copy_change_request（文案修改请求）` 或 blocked。

## 0A-5. publish_candidate_preflight_suite 文案到执行预检接入

`content_route_card V2`、`script_anchor_extraction_function`、`card_placement_decision`、`tts_prosody_anchor_map` 和 `script_to_timeline_map` 后续必须进入 `publish_candidate_preflight_suite（发片候选预检套件）`。它们不再只是“字段存在”，而是导出前要被检查、能阻断、能回写、能进入 review_pack 的执行证据。

接入关系：

```text
script_anchor_extraction_function
  -> script_to_timeline_map
  -> line_level_alignment_preflight
  -> line_visual_tolerance_preflight
  -> near_equivalent_material_substitution_preflight

tts_prosody_anchor_map
  -> tts_route_and_prosody_preflight

tts_route_report
  -> publish_candidate_voice_gate
  -> b_voice_feel_minimax_preflight

card_placement_decision
  -> card_decision_preflight

forbidden_visual_map / forbidden_copy_changes / forbidden_status_promotion
  -> forbidden_action_preflight

visual_evidence_check / readability_check / subtitle_card_overlap_check
  -> visual_evidence_readability_preflight

locked_copy_contract
  -> locked_copy_diff_preflight

publish_candidate_required_inventory / review_pack / user_publish_candidate_standard
  -> publish_candidate_user_standard_preflight
  -> completion_truth_preflight
```

硬规则：

- `script_to_timeline_map` 必须能被脚本检查到每个 `line_group` 的实际观察画面、证据强度、错位原因和修复动作；只有 `expected_visual` 不够。
- `line_visual_tolerance_rule（文案画面一致性容差规则）`：只允许最多约 `5%` 的局部非核心、极其相近素材替代；连续近似替代最多 1 个句组；核心证据错位为 0；全程漂移、主题相近冒充证据、需要观众猜测或缺用户素材时必须 blocked。
- `near_equivalent_material_substitution_report（近似素材替代报告）`：后续每条候选片必须列出 `total_line_group_count / exact_match_count / near_equivalent_count / near_equivalent_ratio / consecutive_near_equivalent_max / core_evidence_mismatch_count / whole_video_drift_detected / substitutions / final_decision`。`极其相近素材` 不等于“主题相近”，不得为了完成出片硬剪。
- `tts_prosody_anchor_map` 必须记录目标 TTS 路线和实际 TTS 路线的对照；不能把“已生成 TTS 文件”写成声音路线通过。
- `tts_route_report` 是正片候选必填报告；后续默认 TTS route 必须是 MiniMax `speech-2.8-hd / MiniMax/speech-2.8-hd`。B 方案升级为正式听感标准 `formal_voice_feel_reference`，但不能继续作为正片候选默认 provider。
- `b_voice_feel_minimax_formal_voice_rule（B 方案听感 + MiniMax 正式语音规则）`：B 听感必须体现轻陪伴、低压、自然中文口语、B 停顿梗感、轻微停顿玩笑节奏、游戏向导感，并明确不是播音腔、销售腔、客服腔或幼稚可爱音；正式生成路线必须是 MiniMax。
- `b_voice_identity_lock（B 声音身份锁）`：B 方案不能再只靠 `voice_feel_tags` 或 `b_voice_feel_reflected = true` 放行；后续正片候选必须有 `expected_b_minimax_voice_id`、`locked_voice_setting` 和 `human_voice_review_status = user_confirmed`，且实际生成报告中的 `actual_voice_id` 必须完全等于 `expected_b_minimax_voice_id`。
- `female-tianmei` 不得继续作为 B 方案默认音色，除非用户明确试听并确认；情绪、停顿、上扬和语速优化只能发生在同一 `voice_id` 内，不得换音色。
- 缺 `tts_route_report`、实际 provider / model 非 MiniMax、使用阿里 Qwen TTS / Serena / macOS say / local fallback 的候选，只能 `blocked_publish_candidate_unavailable` 或 `internal_diagnostic_only`，不得写 `publish_candidate_ready_for_human_review = true`。
- `card_placement_decision` 必须包含判断卡、总结卡、结果差卡、边界卡、Prompt 尾卡的需要 / 不需要理由；prompt 未提不是不需要理由。
- `publish_candidate_user_standard_rule（候选可发布用户标准）`：候选片必须达到用户打开后原则上可直接发、再进入人工复审的标准；微小节奏 / 非核心转场 / 不影响理解的字幕节奏 / 小审美瑕疵可记录为 minor flaw，但 locked copy 改写、标题变化、声音路线错误、fallback、无声、全程漂移、核心证据错位、字幕/卡片遮挡核心证据、明显黑灰边/白屏/黑块/整屏遮挡、技术预览或内部诊断冒充候选都必须 blocked。
- `publish_candidate_ready_for_human_review（可发布候选片，待人工复审）` 不等于 `send_ready = true`；`send_ready` 必须等待用户或 ChatGPT 最终确认。
- 预检报告必须进入后续 `review_pack_required_preflight_reports`，缺任一报告不得写视频执行完成。

## 1. 文件定位
本文件只负责 4 件事：
- 判断这轮文案属于哪一类
- 定义当前 `内容表达文案` 的默认交付包
- 定义 block / segment 承载分配
- 定义 Docker / 工作台 / 录制减负任务的默认修正动作

## 2. 文案一级分类
当前先分 4 类：
1. `内容表达文案`
2. `分发包装文案`
3. `关系建立文案`
4. `变现承接文案`

当前默认主流程只优先服务：
- `内容表达文案`

## 3. 当前默认交付包
只要这轮输出的是 `内容表达文案`，默认必须一起交付：
1. 完整脚本
2. `block -> 承载分配`
3. `4 段核心录制素材`
4. 录制动作说明
5. 结果差证明点
6. `PPT / 总结卡文案`
7. 发布前风险检查要点

若这条内容已经出现明确可复用流程、强结果差、用户需求信号或产品化承接空间，再补：
8. `三层 prompt 包`
9. `Prompt 引用尾卡`
10. 录制验收清单
11. `工作包正文 / 回审清单`

缺任一基础项，只能算部分交付；未补第 `8-11` 项，不等于交付不完整，只代表当前仍是内容验证优先。

## 3A. 默认文案生产路由

以后《视频工厂》AI 知识类视频默认按四方分工推进：

1. `Perplexity（外部参考检索 / 研究工具）` 输出 `reference pack（参考包）` 和 `raw feeling draft（原感初稿）`。
   - 只提供外部参考、表达方向、案例、梗和原感。
   - 不负责最终定稿，不能直接进入执行。
2. 用户录制真实素材。
   - 包括录屏、网页操作、结果对比、过程证据、关键截图。
   - 素材决定最终稿能写到多具体。
3. `Codex（执行代理）` 做素材技术检查与细节证据报告。
   - 目标不是写最终文案，而是把素材细节交给 `ChatGPT（最终落稿与复审入口）`。
   - 必须给出能写稿的细节，不能只报“素材存在 / 技术通过”。
4. `ChatGPT（最终落稿与复审入口）` 基于参考包、用户素材和 `Codex（执行代理）` 细节报告写最终落稿。
   - 默认最终稿出口是 `ChatGPT（最终落稿与复审入口）`。
   - 必须保证文案贴真实素材，不能写素材里没有的页面、按钮、动作和结果。
5. `Codex（执行代理）` 按最终稿执行视频、字幕、声音、卡片和审片包。
   - 正式运营视频交付必须产出 `publish_candidate_ready_for_human_review` 或 `blocked_publish_candidate_unavailable`。
   - 执行完成后回到 `ChatGPT（最终落稿与复审入口）` 复审；`publish_candidate` 不等于 `send_ready`。

## 3A-1. 数据目标驱动文案修改闸门

当任务命中文案修改、下一条视频、根据数据改、播放低 / 收藏低 / 客资弱、复盘后重写、数据飞轮、目标驱动、单主变量或内容结构反馈时，必须先读：

- `13_目标驱动数据飞轮与文案执行闭环_goal_driven_data_flywheel_and_copy_execution_loop.md`
- `14_数据目标执行总线_data_goal_execution_bus.md`
- `codex_log/current_data_goal_anchor.md（当前数据目标锚点）`
- 当前 `video_goal_card（视频目标卡）`
- 当前 `post_publish_review_card（发布后复盘卡）`
- 当前 `data_flywheel_memory（数据飞轮记忆）`
- 当前 `content_structure_feedback_card（内容结构反馈卡）`

正式文案修改前置字段：

```text
data_goal_copy_revision_gate:
  required_before_copy_revision:
    - current_north_star_goal
    - current_stage_goal
    - threshold_config_v1
    - video_goal_card
    - post_publish_review_card
    - data_flywheel_memory
    - content_structure_feedback_card
    - main_bottleneck
    - primary_variable
    - supporting_variables
    - forbidden_variables
    - success_metric
    - failure_metric
```

输出必须包含：

- `data_diagnosis（数据诊断）`
- `threshold_check（阈值检查）`
- `main_bottleneck（主短板）`
- `variable_plan（变量计划）`
- `copy_revision_strategy（文案修改策略）`
- `revised_script（修改后文案）`
- `next_video_execution_prompt（下一条视频执行 prompt）`
- `post_publish_validation_metric（发布后验证指标）`
- `data_goal_anchor_used（使用的数据目标锚点）`
- `main_bottleneck_supported（服务的主短板）`
- `primary_variable_supported（服务的主变量）`
- `forbidden_variables_avoided（避开的禁止变量）`

硬规则：

- 没有 `threshold_config_v1（阈值配置 V1）`，不得做数据驱动判断。
- 没有 `video_goal_card（视频目标卡）`，不得进入正式文案修改。
- 没有 `post_publish_review_card（发布后复盘卡）`，不得声称“根据数据修改文案”。
- 没有 `main_bottleneck（主短板）`，不得重写正式文案。
- 没有 `primary_variable（主验证变量）`，不得生成 Codex 执行 prompt。
- 数据不足时只能输出假设版文案，并标注缺失数据。
- `single_primary_variable_rule（单主变量规则）` 允许 1 个主验证变量 + 1-2 个协同变量；超过 3 个变量且未标 `major_revision（大改版）` 不得进入执行。
- 没有 `next_video_execution_prompt（下一条视频执行 prompt）`，Codex 不得进入视频执行。
- 没有 `data_goal_anchor_used（使用的数据目标锚点）`，不得进入内容路由、剪辑、编排或装配。
- 没有 `current_data_goal_anchor_source（当前数据目标锚点来源）`，不得把 `content_route_card V2（内容路由卡 V2）`、`script_to_timeline_map（文案到时间线映射表）`、`editing_decision_pack（剪辑决策包）` 或 `assembly_decision_pack（装配决策包）` 写成可执行完成态。
- 内容路由不是只判断画面承载，而是先判断这条内容如何服务本轮数据目标。

## 3A-2. copy_iteration_decision_system（文案迭代决策系统）

当任务要求记录每期文案、根据数据判断文案该改哪一层、给 ChatGPT 输出下一版文案修改简报，必须先进入 `copy_iteration_decision_system（文案迭代决策系统）`，不得直接改稿或生成正式下一条视频执行 prompt。

当前可运行入口：

- `scripts/文案迭代决策系统_copy_iteration_decision_system.py`
- `review_loop/copy_iteration/copy_registry.json`
- `review_loop/copy_iteration/latest_copy_iteration_report.json`
- `review_loop/copy_iteration/latest_copy_iteration_report.md`
- `review_loop/copy_iteration/V003/V003_next_copy_revision_brief.md`

文案修改前必须先判断本轮问题层级：

- `opening_packaging（开头包装）`
- `bridge_3_8s（3-8 秒承接）`
- `middle_structure（中段结构）`
- `evidence_expression（证据表达）`
- `tone_and_language（语气与人话程度）`
- `topic_angle（选题角度）`
- `target_audience（目标人群）`

硬规则：

- V003 当前仍是 `interim_36h_snapshot / partial_data_recorded` 时，`formal_copy_revision_allowed = false`。
- 只允许 `low_confidence_prepare（低置信度准备）` 开头 0-3 秒和 3-8 秒承接。
- 不得把 `opening_packaging` 问题直接升级成 `topic_angle` 失败或 `target_audience` 错误。
- 不得全文重写，不得改目标用户、核心选题方向或承接 / offer。
- `raw copy（原始文案）` 必须保真保存；疑似错字只能写入 `copy_quality_notes / suspected_typos`，不得覆盖 raw 原文。
- ChatGPT 读取 `V003_next_copy_revision_brief.md` 后负责最终判断、汇报和改稿；Codex 只负责记录、结构化、生成报告和验证。

## 3B. `Codex（执行代理）` 素材细节汇报标准

`Codex（执行代理）` 检查素材时，必须按“给 `ChatGPT（最终落稿与复审入口）` 写最终稿”的标准汇报。

至少包含：

1. 素材清单
   - 文件路径、文件名、时长、分辨率、是否能打开、是否有音频。
   - 是否有明显卡顿、黑屏、模糊、遮挡。
2. 时间码细节
   - 每个关键动作从几秒开始，到几秒结束。
   - 每个画面段落对应什么内容。
   - 哪些时间段可做反面展示，哪些时间段可做正面展示，哪些时间段适合配口播解释。
3. 画面可见信息
   - 页面名称、按钮名称、输入框名称、标签名称。
   - 弹窗、提示、报错。
   - 生成结果中的可读文字。
   - 页面变化前后差异。
4. 操作步骤细节
   - 用户进入哪个网站、工具或页面。
   - 点击了哪个入口。
   - 输入了什么内容。
   - 哪个按钮触发了生成。
   - 生成后出现了什么结果。
   - 如果有 `API（应用程序接口）`、后台或控制台类页面，必须写清入口位置和动作。
   - 颗粒度示例：进入阿里百炼大模型服务平台，找到 `API Key（接口密钥）` 入口，点击新建 `Key（密钥）`，复制生成的 `Key（密钥）` 到项目配置里。
   - 上面只是颗粒度示例，不能编造素材里没有的操作。
5. 证据判断
   - 这段素材能证明什么。
   - 不能证明什么。
   - 哪些画面足够支撑文案。
   - 哪些文案点素材没有支撑，必须删掉或改写。
   - 哪些地方需要补录。
6. 给 `ChatGPT（最终落稿与复审入口）` 的写稿建议
   - 哪些细节应该写进口播。
   - 哪些细节只适合字幕或卡片。
   - 哪些地方适合微反转。
   - 哪些地方适合轻吐槽。
   - 哪些地方不适合写得太满。

## 3B-1. copy_granularity_mixture_gate（文案颗粒度配比闸门）

ChatGPT 最终文案进入视频执行前，必须标注每个关键段落或 `line_group（句组）` 属于哪类密度：

- `smooth_line（顺口推进句）`
- `material_granularity_line（素材颗粒度句）`
- `judgment_boundary_line（判断 / 边界句）`
- `result_transition_line（结果变化句）`

`smooth_line（顺口推进句）`：

- 可以不绑定具体时间码。
- 负责痛点、过渡、人味、轻吐槽和低压收束。
- 不得承载事实证明；不能用顺口表达替代素材证据。

`material_granularity_line（素材颗粒度句）`：

- 必须绑定素材时间码，或明确标为 `no_direct_visual_but_user_experience_statement（无直接画面但属于用户经验陈述）`。
- 必须说明页面、动作、字段、结果中至少两个。
- 示例包括：`Codex 在屏幕上打字`、点击确认、搜索商品、翻商品卡、记录商品名、价格、佣金、销量信号、店铺分、商品分、风险，写入云盘表格，最后回到聊天框给结论。

`judgment_boundary_line（判断 / 边界句）`：

- 必须说明素材能证明什么、不能证明什么。
- 不得写成商品一定能卖、商业闭环成立、内容已验证成功或 `send_ready = true`。

`result_transition_line（结果变化句）`：

- 必须说明从什么变成什么。
- 例如：从一堆商品卡 -> 候选表 -> 4 个复查商品 -> 回到聊天框结论。

`script_anchor_extraction_function（文案锚点提取函数）` 必须新增并输出以下字段：

- `granularity_type（颗粒度类型）`
- `detail_density（细节密度：low / medium / high）`
- `material_detail_required（是否必须绑定素材细节）`
- `must_include_detail_types（必须包含的细节类型）`
- `allowed_smoothness（允许顺口程度）`
- `blocked_if_key_claim_without_material_detail（关键主张缺素材细节时阻断）`

阻断条件：

- 如果文案说 Codex 自动操作电脑，但素材没有操作电脑画面，必须 blocked 或退回改写。
- 如果文案说 Codex 生成表格，但没有表格、云盘、文件或结果画面，必须 blocked 或退回改写。
- 如果文案说“最适合我的商品”，必须绑定聊天框结论、表格字段或用户规则，不得只凭感觉。
- 如果关键高光段只有顺口表达，没有素材动作细节，必须退回 ChatGPT 改稿。
- 如果整篇都是操作细节，导致开头和过渡没有人味，也必须退回改稿。

本闸门与 `locked_copy_contract（锁定文案契约）` 不冲突：它只负责执行前密度标注和证据检查，不授权 Codex 自行改写最终文案。需要改语义时，Codex 只能输出 `copy_change_request（文案修改请求）` 或 blocked。

## 3B-2. reference_copy_voice_gate（对标文案话语闸门）

ChatGPT 最终文案进入视频执行前，必须检查是否完成 `reference_copy_voice_mechanism（对标文案话语机制）`。该闸门和 `copy_granularity_mixture_gate（文案颗粒度配比闸门）` 配合使用：前者检查话语功能，后者检查句子密度和素材颗粒度。

```text
reference_copy_voice_check:
  user_problem_positioned:
  old_way_pain_clear:
  ai_as_rescue_not_feature_intro:
  concrete_case_present:
  pitfall_details_present:
  tool_actions_visible:
  result_change_clear:
  ability_definition_present:
  low_pressure_ending:
  structure_adapted_to_material:
```

若文案只是写成“工具功能介绍”，必须退回改稿。若文案只是按对标结构机械复刻，但和当前素材、证据强弱、平台风险或观众理解路径不匹配，也必须退回改稿。

句子类型继续接入已有四类：

```text
smooth_line（顺口推进句）
material_granularity_line（素材颗粒度句）
judgment_boundary_line（判断 / 边界句）
result_transition_line（结果变化句）
```

新增检查字段：

```text
copy_voice_function（话语功能）:
  - user_pain_line（用户痛点句）
  - old_way_trap_line（旧方法坑点句）
  - ai_rescue_line（AI 救场句）
  - pitfall_detail_line（坑点细节句）
  - ability_definition_line（能力定义句）
  - low_pressure_close_line（低压收尾句）
```

阻断条件：

```text
blocked_if:
  - 文案先讲工具，没先讲用户问题。
  - 文案只有字段，没有坑点。
  - 文案只有结构，没有人话。
  - 文案只说 AI 很强，没有具体案例。
  - 文案照搬 reference 结构，不按当前素材调整。
  - 文案照搬 reference 原句。
  - 文案把初筛写成商业验证。
```

本闸门不授权 `Codex（执行代理）` 自行重写锁定最终文案。命中阻断时，Codex 只能输出 `copy_change_request（文案修改请求）`、退回 ChatGPT 改稿或 blocked，不得把机制检查改写成新第四期最终稿。

### 3B-2A. anti_report_voice_gate（反报告腔话语闸门）

`copy_voice_function（话语功能）` 只允许作为后台标注和验收字段，不允许作为最终文案标题、段落名或口播标签。最终稿不能把 `user_pain_line / ai_rescue_line / ability_definition_line` 这类字段翻译成前台结构；它们只能帮助检查“有没有把话说成人话”。

最终稿进入视频执行前，必须额外检查：

```text
anti_report_voice_check:
  no_label_like_output:
  no_project_report_tone:
  no_function_intro_first:
  no_mechanism_explanation_as_narration:
  natural_spoken_chain_present:
  concrete_human_situation_first:
  pitfall_detail_embedded_in_sentence:
  ai_tool_appears_as_rescue:
  result_change_sounds_like_viewer_benefit:
```

阻断条件：

```text
blocked_if:
  - 最终稿出现“用户痛点句 / AI 救场句 / 能力定义句”等标签化输出。
  - 最终稿像在解释机制，而不是讲一个普通人遇到的具体问题。
  - 最终稿先讲 Codex 是什么，而不是先讲用户为什么卡住。
  - 最终稿只是把字段补齐，但读起来不像真人口播。
  - 最终稿只有“机制正确”，没有“观众愿意听下去”的话语链。
```

Codex 只能检查和退回，不能自行把最终文案改成新稿。命中本闸门阻断时，只能输出 `copy_change_request（文案修改请求）`、退回 ChatGPT 改稿或 blocked。

通过本闸门不代表 `content_validation（内容验证）` 通过，只代表文案话语不再明显报告腔。

## 3C. content_route_card（内容路由卡）

每条 `内容表达文案` 进入视频执行前，必须先生成 `content_route_card（内容路由卡）`。

这张卡不是新的固定模板，而是 `content_route_inference_function（内容路由推理函数）` 的输出卡。它负责把文案目标、素材证据、结果差、平台风险和观众理解路径压成可执行路由，并解释“为什么这条视频这样变”。

`content_route_inference_function（内容路由推理函数）` 必须先判断：

```text
input_signal（输入信号）
-> observed_evidence（现场证据）
-> state_inference（状态判断）
-> action_policy（动作策略）
-> validation_rule（验证规则）
-> blocked_if（阻断条件）
-> feedback_update（反馈回写）
```

如果没有完成这条推理链，不能只填一张 `content_route_card（内容路由卡）` 就进入视频执行。

内容路由卡至少回答：

| 字段 | 填写要求 |
| --- | --- |
| `content_type（内容类型）` | 例如 AI 职场任务现场拆解、AI 问题点破、发布包装复盘等。 |
| `validation_goal（本轮验证目标）` | 写清本条验证什么，不能泛写“做一条视频”。 |
| `data_goal_anchor_used（使用的数据目标锚点）` | 写清本条内容继承的当前阶段目标、主短板、主变量、禁止变量和验证指标。 |
| `main_bottleneck_supported（服务的主短板）` | 写这条内容优先修哪个数据短板，不能泛写“提升表现”。 |
| `primary_variable_supported（服务的主变量）` | 写本条内容主要验证哪一个变量，不能同时塞多个主变量。 |
| `forbidden_variables_avoided（避开的禁止变量）` | 写清本条内容不能引入哪些变量，避免发布后无法归因。 |
| `post_publish_validation_metric（发布后验证指标）` | 写发布后要看哪个指标判断本轮变量是否有信号。 |
| `opening_route_decision（开头路由判断）` | 先判断开头路线，再执行开头；不得把元素娃娃或梗图 GIF 写成所有内容唯一默认。 |
| `card_placement_decision（卡片位置判断）` | 先判断总结卡、反转卡、结果差卡、Prompt 尾卡的文案功能和证据窗口，再决定是否插入与放在哪里；不得固定旧 shot。 |
| `core_evidence（核心证据）` | 写真实录屏、前后对比、步骤截图、结果截图或平台数据。 |
| `middle_carrier（中段主体承载）` | 默认优先用户录制素材；若不是，必须说明原因。 |
| `api_human_usage（API 生成真人使用方式）` | 写 `1 次` / `2 次` / `不确定待判断`，并说明依据。 |
| `ppt_usage（少量 PPT 使用方式）` | 写关键词显影、结构整理、总结卡或不适用。 |
| `prompt_tail_card_usage（Prompt 引用尾卡是否使用）` | 写使用 / 不使用 / 待判断；不得默认强塞。 |
| `reference_inheritance（继承哪些质量点）` | 只写质量点、风格边界、展示边界，不写旧镜头流程照搬。 |
| `flow_flex_reason（为什么本条不照搬固定流程）` | 说明当前文案、素材、风险或结果差为什么需要局部变化。 |
| `platform_risk_note（平台风险提示）` | 写是否触发发布前平台风险检查，以及风险词或画面风险。 |

硬规则：

- Codex 不得先定人物次数、卡片数量、PPT 数量，再把文案塞进去。
- 内容路由卡不得写成所有视频都沿用同一结构。
- 如果当前文案目标和旧 reference 流程冲突，先保留实时路由判断，只继承质量点。
- 缺 `validation_goal`、`core_evidence` 或 `flow_flex_reason` 时，不得进入视频执行。
- 缺 `content_route_inference_function（内容路由推理函数）` 的 `state_inference / action_policy / blocked_if` 时，不得把 `content_route_card（内容路由卡）` 写成已完成。

#### `element_doll_opening（元素娃娃开头）`

适用条件：

- 开头时长通常 `<= 3s`。
- 需要统一品牌感、轻陪伴、低压进入。
- 话题不是强争议 / 强情绪开场。
- 内容偏讲解、复盘、判断、轻引导。
- 不需要用强梗图冲击先拦住观众。

职责：

- 轻进入。
- 建立“有人带你看”的感觉。
- 承担品牌识别。
- 不抢中段真实录屏证据。

禁止：

- 不得长时间连续主持。
- 不得替代中段主体。
- 不得写成所有内容唯一默认开头。

#### `meme_gif_opening_hook（梗图 GIF 开场钩子）`

适用条件：

- 开头时长 `> 3s`，或需要先让观众停一下。
- 话题有明显情绪、反差、争议或商业焦虑。
- 需要先抓眼，再抛一个强问题。
- 例如：`AI 到底能不能赚钱？`、`普通人做 AI 是机会还是被割？`、`为什么你用 AI 还是赚不到钱？`

职责：

- 抓眼。
- 起情绪。
- 抛核心问题。
- 让开头“存在”，但不负责证明。

允许继承的机制：

- 漫画速度线 / 冲击线。
- 反应姿态人物。
- 大字压屏。
- 弹出 / 抖动 / 快速推进动效。
- GIF 感 / 梗图感。

禁止：

- 不得复刻用户参考图里的具体人物、头像、字体、构图或可识别风格。
- 不得复用第三方素材。
- 不得把梗图 GIF 开头写成新唯一默认。
- 不得替代中段真实录屏证据。
- 不得承担主要证明。
- 不得变成强卖课、鸡血、夸张割韭菜风。

#### `direct_question_title_card（直接问题标题卡）`

适用条件：

- 主题一句话本身很强。
- 开头需要干净、直接、短。
- 不需要角色出场。
- 不需要强 GIF 情绪。
- 适合 `1-2s` 内钉住问题。

职责：

- 一句话把问题钉住。
- 快速进入中段证据或判断。

禁止：

- 不得变成 PPT 课件感。
- 不得铺长定义。
- 不得用大段说明代替问题钩子。

#### `screen_first_opening（录屏现场先行开头）`

适用条件：

- 真实画面本身有冲击力。
- 一开头就能看到失败、结果差或强证据。
- 适合“先给证据，再解释”。

职责：

- 用真实画面直接建立可信度。
- 把观众带进工作现场。
- 服务真实 AI 使用经验和工作提效实录。

禁止：

- 不得让画面看不清。
- 不得用装饰卡片替代真实证据。
- 不得在证据不足时假装现场感成立。

选择规则：

```text
if opening_duration > 3s and topic_emotion_level in [medium, high] and controversy_level in [medium, high]:
  prefer meme_gif_opening_hook

if opening_duration <= 3s and brand_consistency_need = high and topic_emotion_level != high:
  prefer element_doll_opening

if core_question_can_be_stated_in_one_sentence = true and no strong visual evidence needed at opening:
  consider direct_question_title_card

if evidence_start_strength = high:
  consider screen_first_opening

if meme_gif_opening_hook selected:
  must verify it only hooks and asks, not proves

if element_doll_opening selected:
  must verify it does not become the only default route

if route cannot be determined:
  output opening_route = pending_judgment
  do not proceed to video execution
```

硬规则：

- 开头固定机制，不固定素材；Codex / ChatGPT 必须先判断开头路线，再执行开头。
- `element_doll_opening（元素娃娃开头）` 保留为合法 opening route，但不再是所有内容唯一默认开头。
- `meme_gif_opening_hook（梗图 GIF 开场钩子）` 是条件优先路线，不是新默认死流程。
- 开头 GIF 只负责抓眼、起情绪、抛问题，不负责证明，不替代中段真实录屏证据。
- 用户参考图只能继承“漫画冲击线 / 反应姿态 / 大字压屏 / GIF 感 / 快速抛问题”的机制，不得复刻具体人物、头像、字体、构图或第三方可识别资产。

### 3C-2. `card_placement_decision（卡片位置判断）`

总结卡和反转卡是文案功能卡，不是固定 shot 模板。职责可以稳定，位置必须由最终文案、素材证据、结果差、转折点和观众理解路径判断。

最小字段：

```text
card_placement_decision（卡片位置判断）:
  card_type:
    - judgment_card（判断卡）
    - summary_card（总结卡）
    - reversal_card（反转卡）
    - result_diff_card（结果差卡）
    - boundary_card（边界卡）
    - prompt_tail_card（Prompt 引用尾卡）
    - no_card_needed（不需要卡片）
  copy_function:
    - hook（钩子）
    - problem_setup（问题建立）
    - negative_example（反面示例）
    - reversal_point（反转点）
    - positive_example（正面示例）
    - evidence_explanation（证据解释）
    - result_diff（结果差）
    - conclusion（结论）
    - next_action（下一步动作）
  candidate_position:
    - before_evidence（证据前）
    - after_evidence（证据后）
    - between_negative_and_positive（反面与正面之间）
    - before_result_diff（结果差前）
    - after_result_diff（结果差后）
    - final_closure（结尾收束）
    - not_applicable（不适用）
  position_reason:
  evidence_dependency:
  interrupt_risk:
  selected_position:
  not_allowed:
  fallback_action:
```

#### `judgment_card（判断卡）`

职责：

- 强化核心判断句。
- 帮观众识别本条内容的关键定性。
- 在必要时把 `core_claim / judgment / boundary_statement（核心判断 / 判断 / 边界声明）` 做成轻动效卡片。

适合出现的位置：

- 核心判断句已经由真实素材或上下文支撑之后。
- 需要把一句重要判断从口播中轻轻提出来，但不打断证据窗口。
- 结尾前需要短暂停顿，帮助观众记住本条判断。

不适合出现的位置：

- 中段真实录屏正在承担关键证据时。
- 关键判断本身还没有被素材支撑时。
- 只是为了让画面更好看。

禁止：

- 不得改写判断句语义。
- 不得替代真实录屏证据。
- 不得写成卖课式大字报。
- 不得把判断卡写成 `content_validation（内容验证）` 通过证据。

#### `boundary_card（边界卡）`

职责：

- 标记本条内容能证明什么、不能证明什么。
- 降低观众对素材证据的过度误读。
- 在争议点、策略边界或数据边界出现时，帮助区分经验、推测和已确认事实。

适合出现的位置：

- 口播出现 `不能证明 / 只代表 / 目前只是 / 还待验证` 一类边界声明时。
- 结果差或平台数据容易被误读为商业验证成立时。

禁止：

- 不得把边界声明包装成新的结论。
- 不得新增素材里没有的数据结论。
- 不得抢真实证据窗口。

#### `summary_card（总结卡）`

职责：

- 收束观点。
- 压缩结论。
- 给观众一个最小行动或最小判断。
- 帮助观众记住“这条到底讲了什么”。

适合出现的位置：

- 一段证据展示之后，需要帮观众收住结论。
- 结果差展示之后，需要把差异翻译成人话。
- 结尾之前，需要把本条观点落成一句判断。
- 结尾处，需要给一个低门槛动作。

不适合出现的位置：

- 中段真实录屏正在承担关键证据时。
- 文案还没完成证据展示时。
- 只是为了让画面更好看。
- 结尾一句话已经足够清楚时。

禁止：

- 不得替代中段真实录屏证据。
- 不得把总结卡做成密集 PPT。
- 不得每条强行固定结尾总结卡。
- 不得把总结卡写成 `content_validation（内容验证）` 通过证据。

#### `reversal_card（反转卡）`

职责：

- 承担“认知翻转”。
- 点出旧做法为什么错。
- 点出新做法为什么更有效。
- 把反面示例和正面示例之间的转折讲清楚。

适合出现的位置：

- 反面示例之后，正面示例之前。
- 旧理解被推翻的那一句。
- 结果差出现前，需要先提示观众“关键不是这里，而是那里”。
- 文案从吐槽 / 问题转向解决方案时。

不适合出现的位置：

- 文案没有明确反转点。
- 只是普通并列步骤。
- 真实录屏已经自然展示了变化，不需要再插卡解释。
- 插卡会打断 FocuSee 原始运镜和证据窗口。

禁止：

- 不得为了“有趣”强行插反转卡。
- 不得把反转卡固定为某个 shot。
- 不得把反转卡当普通信息卡。
- 不得替代真实前后对比。
- 不得让反转卡抢走中段证据。

#### `result_diff_card（结果差卡）`

职责：

- 展示普通做法 vs 改进做法的可感知差异。
- 帮观众快速理解这条内容的价值。
- 可以和总结卡相邻，但职责不同。

适合出现的位置：

- 前后对比素材之后。
- 结果生成之后。
- 需要把“看得见的差异”变成“观众听得懂的差异”时。

禁止：

- 不得新增素材里没有的数据结论。
- 不得冒充真实平台数据。
- 不得做成密集 dashboard。
- 不得替代用户录制素材。

#### `prompt_tail_card（Prompt 引用尾卡）`

职责：

- 低压承接。
- 给用户一个 prompt / 工作包 / 资料包入口。
- 只做引用，不做主结论。

适合出现的位置：

- 总结卡之后，且总结卡确实存在。
- 视频主叙事已经结束之后。
- 内容出现明确可沉淀价值时。

禁止：

- 不得承担主叙事。
- 不得做第二个主结尾。
- 不得强卖课。
- 不得每条默认强塞。

选择规则：

```text
if copy_function = reversal_point and negative_example_exists = true and positive_example_follows = true:
  consider reversal_card between_negative_and_positive

if copy_function = conclusion or next_action:
  consider summary_card near final_closure

if copy_function = core_claim or copy_function = judgment:
  consider judgment_card only after evidence_dependency is clear

if copy_function = boundary_statement:
  consider boundary_card only to reduce over-claim risk

if result_diff_is_visible and viewer_needs_translation = true:
  consider result_diff_card after_evidence

if prompt_or_work_package_handoff_needed = true and main_story_finished = true:
  consider prompt_tail_card after summary_card or after narrative closure

if evidence_window_active = true:
  avoid summary_card / reversal_card unless card clarifies evidence without interrupting it

if FocuSee recording_layer_motion_baked_in = true:
  preserve original motion; do not insert card if it breaks the evidence window

if no clear reversal point:
  do not insert reversal_card

if no clear conclusion or next action:
  do not force summary_card

if card_position_unclear:
  output card_placement = pending_judgment
  do not proceed to video execution

if card_type in [judgment_card, summary_card]:
  set hyperframes_required = true
  require hyperframes_card_motion_baseline
  require hyperframes_runtime_status
  if hyperframes_runtime_status in [missing, not_found, not_verified]:
    output blocked_if_hyperframes_unavailable
    do not use fallback static card unless user explicitly authorizes fallback
```

硬规则：

- Codex 不得先固定总结卡 / 反转卡位置，再把文案塞进去。
- `card_placement_decision（卡片位置判断）` 必须先说明 `copy_function（文案功能）`、`evidence_dependency（证据依赖）` 和 `interrupt_risk（打断风险）`。
- 判断卡 / 总结卡是否出现仍由 `card_placement_decision（卡片位置判断）` 判断；它们不是每条视频固定强插卡。
- 如果真实录屏已经承担证据，卡片不得抢真实证据位。
- 如果文案没有明确反转点，不得强行插 `reversal_card（反转卡）`。
- 如果结尾一句话能自然收住，不得强行堆 `summary_card（总结卡）`。
- 如果判断卡 / 总结卡被选中，HyperFrames 是正式运营阶段的默认卡片动效和视觉优化基线；若 runtime / plugin / script 不存在或不可用，必须 blocked 或等待用户授权降级，不能用普通静态卡片冒充 HyperFrames 已执行。
- 卡片职责可以稳定；卡片位置不得固定为旧 shot、旧模板或旧 reference 时间点。

## 3D. 文案进入执行后的全执行供料链

一条最终文案交给 `Codex（执行代理）` 后，不得直接进入“生成视频”。

必须先经过：

```text
0. data_goal_anchor（数据目标锚点）
1. content_route_card（内容路由卡）
2. script_anchor_extraction_function（文案锚点提取函数）
3. script_to_timeline_map（文案到时间线映射表）
4. tts_prosody_anchor_map（TTS 韵律锚点表）
5. visual_asset_requirement_pack（视觉素材需求包）
6. api_asset_generation_pack（API 素材生成包）
7. image_prompt_pack（图片 prompt 包）
8. asset_validation_pack（素材验收包）
9. assembly_decision_pack（装配决策包）
10. editing_decision_pack（剪辑决策包）
11. data_goal_alignment_check（数据目标对齐检查）
12. review_pack / 审片包回流
```

每层职责：

| 层级 | 职责 |
| --- | --- |
| `data_goal_anchor（数据目标锚点）` | 锁定阶段目标、主短板、主变量、禁止变量、成功 / 失败 / 发布后验证指标。 |
| `content_route_card（内容路由卡）` | 判断这条内容为什么这样承载，不能先锁人物段、卡片数或 PPT 数。 |
| `script_anchor_extraction_function（文案锚点提取函数）` | 从最终文案提取句子级功能、证据、画面、TTS、卡片和禁用画面锚点。 |
| `script_to_timeline_map（文案到时间线映射表）` | 每 1-2 句绑定素材、时间码、画面职责、字幕职责、卡片职责和禁用画面。 |
| `tts_prosody_anchor_map（TTS 韵律锚点表）` | 在 TTS 生成前定义分句、合并、停顿、重音、禁用上扬词和音高走势。 |
| `visual_asset_requirement_pack（视觉素材需求包）` | 判断每个 block / segment 需要真实录屏、截图、PPT / 总结卡、API 生成图、角色、背景、图标还是不需要额外素材。 |
| `api_asset_generation_pack（API 素材生成包）` | 判断哪些素材可由 API 生成、是否允许调用、供应商候选、数量、尺寸、风格、降级路线和阻断条件。 |
| `image_prompt_pack（图片 prompt 包）` | 为每张候选生成图写 prompt、negative prompt、构图、风格、用途和验收标准；不生成真实图片。 |
| `asset_validation_pack（素材验收包）` | 检查素材是否符合风格、是否抢真实证据位、是否 AI 感过强、是否有版权 / 官方资产复用风险。 |
| `assembly_decision_pack（装配决策包）` | 决定素材进入哪段时间线，谁是主证据，谁是辅助，TTS / 字幕 / 卡片怎么配合。 |
| `editing_decision_pack（剪辑决策包）` | 由 `editing_inference_function（剪辑推理函数）` 先判断证据窗口、上下文风险和动作必要性，再决定放大、保留原画面、插卡、框选 / 高亮、定格、左右对比或不动。 |
| `data_goal_alignment_check（数据目标对齐检查）` | 验证每段、每个剪辑动作、每个装配选择是否仍服务主变量并避开禁止变量。 |
| `review_pack（审片包）` | 回流给 ChatGPT / 用户复审，不替代最终内容判断。 |

硬规则：

- Codex 拿到最终文案后，不能直接生成视频。
- 缺 `data_goal_anchor（数据目标锚点）`，不得进入视频执行、剪辑、编排、装配或成片生成。
- 必须先判断素材需求。
- 只有段落级 `material_01 / material_02 / material_03` 分配不算可执行供料；缺 `script_to_timeline_map（文案到时间线映射表）` 不得装配时间线。
- TTS 生成前必须先完成 `tts_prosody_anchor_map（TTS 韵律锚点表）`，不得只按标点机械切句。
- 高情绪 / 抖音抓眼 / 梗图 GIF / 抽象动效开头生成前必须先完成 `opening_visual_hook_spec（开头视觉钩子规格）`。
- 没有真实录屏证据时，不能用 API 图片冒充证据。
- API 生成图只做辅助表达、视觉壳、信息卡、氛围、角色、背景、图标等，不替代核心证据。
- 图片生成失败时必须有降级路线：少量 PPT、信息卡、真实截图、不使用额外图片或 `blocked（阻断）`。
- 不得把图片好看写成 `content_validation（内容验证）` 通过。
- 中文可读文字默认由后期卡片层处理，不默认交给图片模型生成。
- 不得复用官方 Minecraft 资产、logo、字体、texture、model、sound。
- `data_goal_alignment_check（数据目标对齐检查）` 缺失时，不得写执行完成。

`editing_decision_pack（剪辑决策包）` 必须额外包含：

```text
editing_decision_pack_data_goal_fields:
  current_data_goal_anchor_source:
  data_goal_anchor_used:
  line_group_goal:
  primary_variable_support:
  evidence_role_for_metric:
  forbidden_visuals_by_goal:
  edit_action_reason_against_data_goal:
  post_publish_validation_metric:
```

`assembly_decision_pack（装配决策包）` 必须额外包含：

```text
assembly_decision_pack_data_goal_fields:
  current_data_goal_anchor_source:
  data_goal_anchor_used:
  segment_goal:
  carrier_reason:
  metric_supported:
  variable_preserved:
  forbidden_variable_avoided:
  post_publish_validation_metric:
```

`forbidden_*` 字段映射规则：

```text
source_of_truth:
  data_goal_anchor.forbidden_variables

content_route_card V2:
  forbidden_variables_avoided

script_to_timeline_map:
  forbidden_variables_avoided

editing_decision_pack:
  forbidden_visuals_by_goal

assembly_decision_pack:
  forbidden_variable_avoided

data_goal_alignment_check:
  current_data_goal_anchor_source
  no_forbidden_variable_introduced
```

硬规则：

- 字段名可以按对象职责不同而不同，但必须全部回指同一组 `data_goal_anchor.forbidden_variables（数据目标锚点里的禁止变量）`。
- `script_to_timeline_map（文案到时间线映射表）` 每个 `line_group_id` 必须同时绑定 `main_bottleneck_supported`、`primary_variable_supported`、`forbidden_variables_avoided` 和 `post_publish_validation_metric`。
- `content_route_card V2`、`script_to_timeline_map`、`editing_decision_pack`、`assembly_decision_pack` 和 `data_goal_alignment_check` 必须回指同一个 `codex_log/current_data_goal_anchor.md（当前数据目标锚点）`。
- 当前锚点状态为 `draft / waiting_data` 时，只能输出假设版或 blocked，不得进入正式数据驱动视频执行。
- 若只能说明“画面好看 / 剪辑顺”，但不能说明避开了哪个禁止变量，不得进入装配或剪辑完成态。
- 若 `forbidden_visuals_by_goal` 与 `forbidden_variable_avoided` 无法追溯到同一组上游 `forbidden_variables`，必须 blocked。

## 3D-1. editing_inference_function 与 editing_decision_pack 的关系

`editing_decision_pack（剪辑决策包）` 是剪辑推理后的执行输出，不是独立动作清单。

最小链路：

```text
editing_inference_function（剪辑推理函数）
-> state_inference（状态判断）
-> action_policy（动作策略）
-> editing_decision_pack（剪辑决策包）
-> review_pack / log（审片回流 / 日志）
```

必须先判断：

- 当前 `data_goal_anchor（数据目标锚点）` 是否存在。
- 观众需要看清的真实证据点是什么。
- 当前 `line_id / line_group_id（句子编号 / 句组编号）` 在证明什么。
- 当前 1-2 句服务哪个 `main_bottleneck（主短板）`、`primary_variable（主验证变量）` 和 `post_publish_validation_metric（发布后验证指标）`。
- 当前 1-2 句必须出现什么画面、不能出现什么画面、是否需要卡片辅助。
- 当前句子是否是反转点、证据点、边界声明或总结点。
- 放大 / 裁切会不会切断必要上下文。
- 插卡会不会抢走真实录屏证据位。
- 定格 / 左右对比是否真的能让结果差更清楚。
- 素材不可读、时间码不清、证据点不可见时是否必须 blocked。

硬规则：

- 中段主体仍优先由 `用户录制素材` 承担，卡片只能辅助。
- 剪辑动作必须写清 `edit_action_reason_against_data_goal（相对数据目标的剪辑动作理由）`。
- 装配动作必须写清 `metric_supported（支持的指标）` 和 `variable_preserved（保留的变量）`。
- 放大、裁切、定格、插卡都必须保护真实证据。
- 缺 `editing_inference_function（剪辑推理函数）` 时，不得把 `editing_decision_pack（剪辑决策包）` 写成完成。
- 缺 `script_to_timeline_map（文案到时间线映射表）` 时，不得生成 `assembly_decision_pack（装配决策包）` 或 `editing_decision_pack（剪辑决策包）`。
- 缺 `data_goal_anchor_used（使用的数据目标锚点）` 时，不得生成 `assembly_decision_pack（装配决策包）` 或 `editing_decision_pack（剪辑决策包）`。
- `content_route_card / script_to_timeline_map / tts_prosody_anchor_map / editing_decision_pack / assembly_decision_pack / data_goal_alignment_check` 不是视频交付；缺发布候选片基础线时必须 blocked，不得用这些中间层或技术预览写完成。
- 如果文案句子与素材证据冲突，必须 blocked 或回到 ChatGPT 复审，不得硬剪。

## 3D-2. FocuSee 自带运镜录屏素材的中段剪辑边界

当用户录制素材来自 `FocuSee`，且素材已经自带 `3D Motion（3D 运镜）`、自动跟随、自动 zoom 或镜头引导时，默认视为：

```text
recording_layer_motion_baked_in（录制层运镜已内置）
```

这类素材仍属于 `用户录制素材`，仍承担中段真实证据主体。变化只发生在 Codex 的中段剪辑职责：Codex 不再默认把“二次放大 / 裁切 / 重新运镜”当成可读性增强的默认动作。

默认不得执行：

- 二次 zoom。
- crop 放大。
- 重新运镜。
- 为了可读性强行插入额外放大段。

默认只做：

- 根据最终文案识别对应时间码。
- 切掉等待、重复、空白、失败操作。
- 保留 FocuSee 原始 `3D Motion（3D 运镜）` 节奏。
- 按 block / segment 衔接口播、字幕、卡片。
- 在证据窗口不清时标记 `blocked（阻断）` 或要求补录。

只有以下情况，Codex 才允许重新判断是否需要辅助放大、定格或卡片：

- FocuSee 运镜没有覆盖关键证据点。
- 关键文字仍不可读。
- 文案要求观众必须看清某个结果差，但素材未展示清楚。
- 用户明确要求二次剪辑增强。

即使允许辅助处理，也不得破坏 FocuSee 原有 `3D Motion（3D 运镜）` 节奏，不得让卡片或二次放大抢走真实录屏证据。

## 4. 三层 prompt 包

`三层 prompt 包` 当前保留，但定位已调整为：`内容辅助结构 / 可选沉淀结构`。

不是每条发布验证内容的默认硬前置；只有当这条内容需要补强结果差、形成可复用流程或继续沉淀工作包时，再补全三层。

### 第 1 层：主 `prompt`
- 先把一句糊话压成更像可交付物的初稿

### 第 2 层：修结果 `prompt`
- 把“看起来对但不能交”的版本继续补成更稳的版本

### 第 3 层：展示 / 尾卡 `prompt`
- 服务总结卡与 `Prompt 引用尾卡`

## 5. `4 段核心录制素材`
当前录制减负默认只要求 4 段：
1. 原始糊话输入
2. 主 `prompt` 执行过程
3. 修结果 `prompt` 补强动作
4. 前后差值与总结收束

## 6. 当前更适合的结构建议
- 默认主结构：`问题 -> 动作 -> 结果`
- 当前更适合的开头：
  - `紧迫任务现场型`
  - `错误示范型`
- 当前更适合的中段强证据：
  - `用户真实操作录屏`
- 当前更适合的结尾收束：
  - `少量 PPT / 总结卡`
  - 总结之后允许补一张 `Prompt 引用尾卡`

## 7. block 与 segment 默认承载

| block | 默认承载 |
| --- | --- |
| 开头进入 | `API 生成真人` |
| 问题点破 | `API 生成真人` 或 `API 生成真人 + 用户录制素材` |
| 过程演示 | `用户录制素材` |
| 关键判断 | `API 生成真人` |
| 结构整理 | `少量 PPT` |
| 结尾收束 | `API 生成真人` 或 `少量 PPT` |
| `Prompt 引用尾卡` | `少量 PPT / 尾卡面板` |

当前明确禁止：
- 不允许先定人物次数，再硬塞文案
- 不允许让 `API 生成真人` 去替代中段本该由录屏承担的主体证据
- 不允许让 `少量 PPT` 去替代中段主体推进
- 不允许让 `Prompt 引用尾卡` 去承担主叙事
- 不允许因为已有 `reference（参考）`、`locked reference（锁定参考）` 或 `visual route（视觉路由）`，就预先锁死人物段次数、卡片数量、PPT 比例和尾卡出现与否

`已确认` 本节 block / segment 承载表是默认职责地图，不是固定镜头 SOP。

必须同时默认理解：
- 如果默认承载表与当前文案目标、真实素材、结果差结构、平台风险、观众理解路径或情绪节奏冲突，应先做机制判断，再重排 block / segment。
- `reference（参考）`、`locked reference（锁定参考）`、`visual route（视觉路由）` 在文案路由里只负责质量、风格和展示边界，不直接替代当前文案的实时路由判断。

## 7A. HyperFrames 卡片动效承载边界

`已确认` HyperFrames 当前只允许作为卡片动效承载层，不改变 block / segment 的文案职责，不替代用户录制素材，不替代云端剪辑，也不新增视觉路由。

1. `data_or_result_diff_card_motion（数据卡 / 结果差卡动效）`

## 8. 当前交付闸门
一条 `内容表达文案` 在进入执行前，至少要同时回答：
1. 是否已经生成 `content_route_card（内容路由卡）`
2. block 是否已经继续拆成 `segment`
3. 中段是否真的由 `用户录制素材` 承担主体推进
4. 若这条内容需要沉淀，`三层 prompt 包` 是否已经交齐
5. `4 段核心录制素材` 是否已经写清
6. `Prompt 引用尾卡` 是否只承担可选沉淀 / 产品化承接引用
7. 是否没有把 `云端剪辑` 写成已跑通事实
8. 若使用 HyperFrames，是否只作为卡片动效承载，且没有接入中段录屏
9. 若判断卡 / 总结卡被选中，是否已经通过 `hyperframes_runtime_gate（HyperFrames 运行时闸门）` 与 `hyperframes_visual_quality_gate（HyperFrames 视觉质量闸门）`
10. 是否没有把 `reference / locked reference / visual route` 写成固定镜头流程

若任务命中：
- Docker
- 工作台
- compose
- 录制减负
- 工作包生成
- prompt 生成
- 结果差对比

- `工作台模板/*`
- volume / mount / 路径漂移

默认动作：

3. 先修路径、挂载、volume 命名、模板缺口、脚本参数问题
4. 再做 `docker compose config` 或最小脚本生成验证

必须明确：
- 默认先尝试最小修正，不先把问题丢给用户
- 但不能把“定义了默认修正动作”偷写成“链路已稳定跑通”

## 10. 一句话规则
**当前文案路由不是“先写一版再找东西塞进去”，而是先判断文案类型，再拆 block，再拆 `segment`；默认先交付 `内容表达结构包 + 4 段核心录制素材 + 结果差证明点`，只有当内容值得沉淀时，再补 `三层 prompt 包 + Prompt 引用尾卡 + 工作包正文`；HyperFrames 当前只能服务卡片动效，不替代中段用户录制素材；判断卡 / 总结卡一旦被选中，HyperFrames 是默认动效和视觉优化基线，runtime 缺失则 blocked 或等待用户授权降级。**
