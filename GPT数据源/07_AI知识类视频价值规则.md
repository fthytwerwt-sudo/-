# AI 知识类视频价值规则

## 1. 文件定位
本文件只解决一个问题：

**当前正式主线下，什么样的 AI 知识类视频才算有价值，什么样的承接单元才更像能变现。**

## 2. 当前一句话总判断
当前价值底线不是“讲了很多”，也不是“只把一个 AI 观点讲懂了”，而是：

**用户看完后能更快拿到一版更像真实交付物的专业初稿，并且能看到为什么结果更专业的证据。**

## 2A. 最终稿细节价值标准

最终稿必须尽量具体到真实操作细节，默认至少追问并呈现：

- 具体工具 / 网站
- 具体页面
- 具体按钮
- 具体输入动作
- 具体生成结果
- 具体前后对比
- 具体失败点
- 具体下一步怎么做

禁止只写：
- `配置 API（应用程序接口）`
- `调用模型`
- `生成内容`
- `优化效果`
- `提升效率`
- `改善体验`
- `完成操作`

应该尽量写成：
- 进入某个平台。
- 点左侧某个入口。
- 新建某个 `Key（密钥）`。
- 把 `Key（密钥）` 复制到某个配置项。
- 重新运行后生成了某个结果。
- 普通问法输出的是空泛建议；场景包问法输出的是可以直接改的结构。

价值判断硬规则：
- 没有真实素材细节支撑的文案点，必须删掉、降级或标记待补。
- 只有抽象概念、漂亮判断和泛泛收益，不算“有细节、能落地”。
- `ChatGPT（最终落稿与复审入口）` 写最终稿时，必须基于 `Codex（执行代理）` 的素材细节报告校准页面、按钮、动作和结果。

## 3. 当前价值承载形态
当前更适合优先承载价值的内容形态是：
- `AI 职场任务现场拆解`

原因：
- 更容易展示真实 AI 使用过程
- 更容易让观众看到哪里节省了重复劳动
- 更容易把模糊需求压成可执行任务
- 更容易展示真实工作流证据
- 更容易把前后差值讲清
- 更容易留下最小行动

## 4. 当前真正的默认价值单元
`已确认` 当前默认优先验证的，不是“单 prompt”，也不是“每条都长成工作包”。

当前默认更适合优先承载价值的单元是：
- 一条 `真实 AI 使用经验 + 工作提效实录`

它至少要让用户看见：
- 真实 AI 使用过程
- 哪里节省了重复劳动
- 前后变化或结果差
- 最小可执行动作

## 5. 视频与工作包的关系
- 视频默认仍是：`证明壳 / 入口壳`
- 当前更准确地说，视频默认是：`真实经验证明壳 / 提效证据入口壳`
- `场景化专业输出工作包` 当前只作为：`可选产品化承接单元`
- 结尾总结后的 `Prompt 引用尾卡`，默认只负责把用户从视频引到可选沉淀，不再假设每条都必须落到工作包

## 6. 强证据优先级
当前更强的证据优先级是：
1. 同一个工具在“普通输入” vs “工作包输入”下的结果差值
2. `用户录制素材`
3. 前后对比
4. 步骤验证截图
5. 结果截图

当前更弱的证据是：
1. 只有 PPT 总结
2. 只有抽象判断
3. 只有口头说明

## 6A. 什么时候才建议沉淀工作包

只有同时或部分出现以下信号时，才建议把内容继续沉淀成 `场景化专业输出工作包`：
- 高收藏或高评论
- 明确咨询或用户追问
- 可复用流程已经相对稳定
- 结果差足够强，适合做产品化承接

在这些信号不足前，默认先把视频当成内容价值验证与平台反馈验证，不把工作包写成当前唯一中心目标。

## 6A. HyperFrames 动效与真实证据边界

`已确认` HyperFrames 当前只能提高三类卡片的表达清晰度和情绪节奏，不能替代强证据。

当前允许：

- 数据卡 / 结果差卡动效服务“普通输入 vs 工作包输入”的结果差理解。
- Prompt 引用尾卡动效服务低压承接和工作包 / prompt 包入口。
- 骚萌卡动效服务问题钩子、反面反转、正面反转，放大笑点和爽点。

价值判断硬规则：

- HyperFrames 动效必须服务问题、反转、结果差。
- HyperFrames 不得补充真实素材没有证明的新结论。
- HyperFrames 不得把数据卡做成密集 dashboard，导致观众误以为有更多真实数据。
- HyperFrames 不得把 Prompt 尾卡做成第二个主结尾或强卖课 CTA。
- HyperFrames 不得把骚萌卡做成普通可爱贴纸；三张骚萌卡必须服务“这说的不就是我吗”、“旧做法看着有东西但不能用”、“这回真的能用了”。
- 中段强证据仍必须优先由 `用户录制素材`、前后对比、步骤验证截图和结果截图承担。

## 7. 爆款潜力和变现潜力必须分开判断

### 更容易出爆的内容
往往更偏：
- 情绪钩子强
- 前几秒抓人
- `紧迫任务现场型` 或 `错误示范型` 开头

### 更容易变现的内容
往往更偏：
- 具体职场场景
- 有前后差值
- 有具体任务
- 有最小行动
- 能自然接工作包、资料包、咨询或服务

必须明确：
- 更容易出爆，不等于更容易变现
- 更容易讲懂，不等于更容易承接

## 8. 商业化副线路径
`CPS` 可以作为副线路径考虑。

但当前只能写成：
- `建议判断`
- `待验证`

当前不能写成：
- 市场已成立
- 当前主商业化模型已经验证通过

## 9. 进入样片前的价值闸门
至少要回答：
1. 用户看完后能做什么？
2. 用户看完后能判断什么？
3. 强证据是什么？
4. 中段证据是否真的由 `用户录制素材` 承担？
5. 结尾总结与 `Prompt 引用尾卡` 是否各守其位？
6. 这条内容当前是只做验证，还是已经出现足够信号值得沉淀为工作包？

## 10. 一句话规则
**当前 AI 知识类视频的最低价值标准，不是“看起来内容很多”，而是“能让用户看到真实 AI 使用过程、工作提效证据、前后结果差，并知道下一步最小能怎么做”；`场景化专业输出工作包` 当前只作为可选产品化承接单元，只有高反馈、强结果差和稳定流程信号出现后才继续沉淀；`CPS` 只能先当副线路径建议，不能冒充已成立事实；HyperFrames 动效只能强化问题、反转和结果差，不得替代真实录屏证据。**

- `待验证` 当前任何真实样片是否已经按这套 V1 过线

### 11.2 `可爱女生向导音` 方向桥接

#### 11.2.1 目标感觉
- `已确认` 当前必须保留的目标感觉是：
  - 轻陪伴
  - 低压
  - 像游戏向导
  - 不是幼稚夹子音
  - 不是 AI 播报
  - 不是新闻播音

#### 11.2.2 外部资料包桥接结论
- `已确认` 单模型（直接 API 调用 + 默认音色）默认不够用于可发布状态。
- `已确认` 主要瓶颈不在音色本身，而在：
  - 韵律
  - 停顿
  - 情绪弧线
  - 句间过渡
- `部分成立` 推荐主链路当前作为方向判断写入：
  - 文案韵律重写（Prompt Engineering）
  - 豆包语音合成 2.0（语音指令 + 可爱女生音色）
  - 后期轻处理（EQ + 压缩 + 去齿音）
- `部分成立` 低门槛备选路线当前作为方向判断写入：
  - Azure TTS `zh-CN-XiaoxiaoNeural`
  - `SSML` 精控
  - 后期轻处理
- `待验证` 当前用户已有阿里 API，所以“阿里优先验证”只作为工程顺序建议写入，不得偷写成最终路线。

#### 11.2.3 平台判断桥接表
| 路线 / 平台 | 当前桥接结论 | 当前状态 |
| --- | --- | --- |
| 豆包语音合成 2.0 | 中文自然度高，可爱女生 / 轻陪伴感潜力强，自然语言指令控情绪，支持“引用上文”改善韵律；风险是指令偶不稳定、夹子音边界难控 | `部分成立` |
| `CosyVoice 2.0`（阿里云百炼） | 中文自然度高，`instruct` 模式可做自然语言控制，音色稳定性较强；风险是自然语言指令与情绪控制稳定性仍要实测 | `待验证` |
| Azure Speech `zh-CN-XiaoxiaoNeural` | 稳定性高，`SSML` 可精控 `pitch / rate / break`；风险是中文韵律容易偏正式，播音感更明显，可爱感上限弱于豆包 | `部分成立` |
| 腾讯云 TTS | 有情绪枚举参数；风险是情绪切换颗粒粗，可爱感上限偏低 | `待验证` |
| ElevenLabs / OpenAI TTS | 当前不适合作为中文短视频主链路；主要风险是中文自然度与口音问题 | `已确认` |

#### 11.2.4 工程提醒
- `已确认` 推荐主链路内不需要默认再叠 `Voice Conversion`
- `已确认` 同批次必须做音色一致性检查
- `待验证` 只要同批次音色明显漂移，就要重生成该段后再回审

#### 11.2.5 声音基线 V1
| 维度 | 合格线 | 优秀线 | 一票否决 |
| --- | --- | --- | --- |
| 自然度 | `18 秒内不出现 2 次以上明显机器切句感；句子不碎；不会一听就是 TTS 念稿` | `像“先理解了再说出来”；句尾会按提问 / 判断 / 引导自然变化；关键句能听出重心移动` | `整句断裂、每字一样重、每句同一种收法、一听就是机器朗读` |
| 向导感 | `像在旁边带你看，不是上课，不是推销，不是纠正你` | `有“陪你过一遍”的感觉；引导动作词轻轻领出来；像把复杂东西变简单` | `老师上课感、客服话术感、销售开场白感、命令感过强` |
| 可爱感 | `比普通知识讲解更轻、更软、更灵一点；不装嫩、不夹、不撒娇` | `可爱感来自轻巧、明亮、微笑感，不来自高音发嗲；像“可爱的女生向导”，不是“卖萌角色配音”` | `夹子音、撒娇音、装嫩音、假可爱` |
| 停顿与韵律 | `停顿跟语义边界一致；不乱断；不会因停顿把意思切碎` | `不是按标点机械停，而是按理解停；转折词前有准备感；重点给空间` | `逗号一律一个时长、句句像切片、转折词没有层次` |
| 情绪弧线 | `开头 / 判断 / 收尾至少能听出不同功能，不再像说明书` | `推荐弧线为“开头轻惊喜 → 中段平稳清楚 → 结尾温暖收束”` | `全程一个平面、开头就太满、结尾突然打鸡血` |
| 音色一致性 | `18 秒内不明显漂移，不像换人，不像换麦` | `即使分句生成再拼，也像同一个角色同一次说完` | `同段像换人 / 换麦 / 情绪一变音色就散` |
| 播音腔 / AI 感 | `没有 CCTV 播音腔、客服播报感、字正腔圆到不像人说话` | `发音清楚但不刻意；重音落在意思上` | `新闻播音感、政务播报感、朗诵腔、过度字正腔圆` |
| 文案贴合度 | `声音重点和文案重点一致，不会整段一个力度` | `判断句更稳、动作句更轻、结果句略留白；结构词能帮助观众听懂层级` | `判断句念成广告句、结果句一带而过、声音与字幕 / 画面抢主次` |

#### 11.2.6 声音过线规则
- 没有一票否决项，才允许继续判断“合格片”
- 核心维度至少包括：
  - 自然度
  - 向导感
  - 停顿与韵律
  - 播音腔 / AI 感
- 以上核心维度全部达合格，才允许把声音部分写成“合格”
- 当前这套声音标准只能标记为：
  - `已确认`：方向与回审维度已写入
  - `部分成立`：方向收敛但样片通过记录不足
  - `待验证`：真实效果、具体样片和最终供应商
- 当前明确禁止：
  - 直接写“声音路线已最终确定”
  - 直接写“当前样片已完成声音内容验证”

### 11.3 `可爱元素主持娃娃` 动作基线 V1

#### 11.3.1 当前写入状态
- `已确认` 当前用户锁定方向是：`Minecraft-inspired 原创体素方块风` 下的 `可爱元素主持娃娃`
- `已确认` 当前参考感觉是：
  - 更接近“可爱的红色衣服元素娃娃”
  - 更像主持娃娃
  - 不像普通游戏 NPC
  - 不像只是站着说话的僵硬数字人
- `待验证` 当前没有“动作”那份 Perplexity 外部原文
- `待验证` 所以动作部分当前只能落成：
  - 当前用户方向下的内部 `动作基线 V1`
  - 不能冒充外部已验证结论

#### 11.3.2 动作基线 V1
| 维度 | 合格线 | 优秀线 | 一票否决 |
| --- | --- | --- | --- |
| 主持感 | `前 5 秒内一眼能看出“这是带你看的主持壳”，不是普通角色占位` | `开头、判断、转折、收尾各有明确主持功能，像在带节奏而不是在自演` | `像普通游戏 NPC、像会动的摆件、看不出主持职责` |
| 可爱感 | `可爱来自轮廓、配色、轻微 smile 感，不来自装嫩和卖萌` | `像“可爱的主持娃娃”，亲近但不幼态，亮点来自轻巧和友好` | `装嫩、卖萌、幼态过强、可爱感靠夸张动作硬堆` |
| 活泼感 | `有轻微 bounce、头部反应和小手势，不是站桩` | `活泼但不乱，能随着文案功能轻轻起伏，像主持人在“点一下重点”` | `动作过僵像站桩，或动作过闹像乱跳` |
| 文案与动作对齐 | `提问、判断、转折、收尾至少能看出动作差异；口型、表情、头部、手势服务同一层意思` | `重点词、判断句、过渡句都能看出动作重心变化，像“听懂了再表达”` | `口型、表情、头部、轻微 bounce、手势各做各的；动作跟文案功能冲突` |
| 镜头与画面适配 | `镜头稳定、构图留白够，字幕和旁侧信息还有安全区` | `一眼能看出“这是主持壳 + 信息区”的画面组织，角色存在感和信息可读性都成立` | `构图挤压字幕、镜头乱晃、角色动作出框、背景过抢` |
| 不抢中段主体 | `主持壳只承担开头进入、关键判断、对比转折、结尾收束，不越权占中段` | `主持壳进得快、退得快，把主体稳稳让给用户录制素材` | `主持壳长时间连讲、替代中段真实录屏、变成整条主叙事` |

#### 11.3.3 文案功能 -> 动作映射 V1
| 文案功能 | 推荐动作 | 推荐镜头 | 推荐表情 | 动作密度建议 | 失败线 |
| --- | --- | --- | --- | --- | --- |
| 开头提问 | 轻微前倾、单手打开、轻点头 | 半身近景，人物居中或偏一侧给标题留白 | 好奇、友好、轻提醒 | `2-3 秒内 1 个主动作 + 1 个轻 bounce` | 站桩不动，或一上来就大挥手 / 大摇摆 |
| 关键判断 | 小幅点头、手掌轻切、短停顿后再落手势 | 半身稳定镜头，正面或微侧面 | 稍认真，但不凶 | `低密度`，让判断句先成立 | 像上课训人、像销售拍桌、像 idle loop |
| 对比转折 | 身体微转、手势指向左右信息区、头部有切换 | 半身或中近景，画面留出对比区 | 先提眉再收稳 | `每个转折 1 个明显切换动作` | 动作太碎，看不出“前后差” |
| 操作提示 | 手掌向侧边轻引、视线短暂带向屏幕信息区 | 半身偏侧构图，给操作区留白 | 清楚、鼓励、低压 | `只点一下动作词，不连续挥动` | 像客服演示、像老师板书、像在抢录屏主体 |
| 结尾收束 | 小幅收手、轻点头、温和邀请式手势 | 半身近景或中近景，构图稳定 | 温暖、收住、微笑感 | `低密度`，以收住结论为主 | 突然打鸡血、突然大幅卖萌、收尾比开头更吵 |

### 11.4 联合过线规则
- 声音不过线 = `content_validation（内容验证）` 不通过
- 动作不过线 = `content_validation（内容验证）` 不通过
- 没有一票否决项，且核心维度达合格，才允许写成“合格片”
- 当前声音部分核心维度至少包括：
  - 自然度
  - 向导感
  - 停顿与韵律
  - 播音腔 / AI 感
- 当前动作部分核心维度至少包括：
  - 主持感

  - `部分成立` → `已确认`
