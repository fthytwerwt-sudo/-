# AI 知识类视频价值规则

## 0A. 正式运营价值判断边界

正式运营样本的数据可以用于观察价值信号，但不能直接证明方向成立、商业成立或内容通过。V003 收藏率等早期信号只能保留为 `draft_low_confidence`，需等 72h / 7d 与人审复盘。

## 1. 文件定位
本文件只解决一个问题：

**当前正式主线下，什么样的 AI 知识类视频才算有价值，什么样的承接单元才更像能变现。**

## 2. 当前一句话总判断
当前价值底线不是“讲了很多”，也不是“只把一个 AI 观点讲懂了”，而是：

**用户看完后能更快拿到一版更像真实交付物的专业初稿，并且能看到为什么结果更专业的证据。**

## 2A. 最终稿细节价值标准

最终稿必须尽量具体到真实操作细节，默认至少追问并呈现：

- 具体工具 / 网站
- 具体页面
- 具体按钮
- 具体输入动作
- 具体生成结果
- 具体前后对比
- 具体失败点
- 具体下一步怎么做

禁止只写：
- `配置 API（应用程序接口）`
- `调用模型`
- `生成内容`
- `优化效果`
- `提升效率`
- `改善体验`
- `完成操作`

应该尽量写成：
- 进入某个平台。
- 点左侧某个入口。
- 新建某个 `Key（密钥）`。
- 把 `Key（密钥）` 复制到某个配置项。
- 重新运行后生成了某个结果。
- 普通问法输出的是空泛建议；场景包问法输出的是可以直接改的结构。

价值判断硬规则：
- 没有真实素材细节支撑的文案点，必须删掉、降级或标记待补。
- 只有抽象概念、漂亮判断和泛泛收益，不算“有细节、能落地”。
- `ChatGPT（最终落稿与复审入口）` 写最终稿时，必须基于 `Codex（执行代理）` 的素材细节报告校准页面、按钮、动作和结果。

## 2A-1. 精彩点颗粒度价值标准

一条 AI 知识 / AI 电商视频的价值，不只来自判断正确，还来自观众看见“AI 具体怎么干活”。精彩点不能一句“AI 帮我处理好了”带过，必须让观众看到素材颗粒度。

高价值细节包括：

- AI / Codex 在真实屏幕上打字。
- AI / Codex 自动点击、搜索、筛选。
- AI / Codex 翻商品卡。
- AI / Codex 读取价格、佣金、销量、评分、风险。
- AI / Codex 把乱商品整理成表。
- AI / Codex 把结果传到云盘，形成云盘表格或文件。
- AI / Codex 回到聊天框给清楚结论。
- AI / Codex 说出哪个商品更适合用户、理由、风险和下一步。

这些细节出现时，文案必须停下来讲清，不得一句“AI 帮我处理好了”带过。

低价值位置可以顺口：

- “我以前也这样”
- “看到后面眼睛都花”
- “这一步才是重点”
- “别急着说它能爆”

验收标准：

- 观众能不能在脑子里看到 AI 正在干活？
- 观众能不能看到从混乱页面到清楚表格的变化？
- 观众能不能理解这个结果为什么适合用户？
- 观众能不能知道下一步该核什么？
- 文案有没有避免把初筛写成商业验证？

## 2A-2. 对标话语价值标准

AI 知识类视频的价值，不只是“解释 AI 功能”，而是把一个普通人正在遇到的复杂问题讲明白。

高价值文案要让观众感觉：

- 这个问题我也会遇到。
- 旧方法确实麻烦。
- 普通 AI 只给建议不够。
- 执行型 AI / 专家模式 / Codex 能接手一段具体工作。
- 最后结果不是炫技，而是让我更容易判断。

高价值细节优先是“坑点细节”：

- 钱会亏在哪里。
- 时间会浪费在哪里。
- 哪个字段会骗人。
- 哪个环节会中途加钱。
- 哪个判断会导致后面返工。

对新第四期的应用示例：

```text
不要写：
Codex 帮我筛选商品。

要写：
你以为做带货最难的是拍视频，其实第一步就卡在：你根本不知道哪个商品值得测。佣金高的不一定适合你，销量好的可能太卷，内容好拍的也可能退货风险高。所以我让 Codex 先替我跑一轮初筛，把一堆商品卡整理成我能判断的复查表。
```

验收问题：

```text
viewer_can_feel_problem:
old_way_pain_visible:
pitfall_details_specific:
tool_value_shown_through_case:
result_change_visible:
ability_definition_clear:
ending_not_over_selling:
structure_not_fixed:
```

本标准只提升话语价值，不推进 `content_validation（内容验证）`、`send_ready（可发送状态）`、`publish_candidate_ready_for_human_review（可发布候选片，待人工复审）`、`voice_validation（声音验证）` 或 `visual_master_locked（视觉母版锁定）`。

### 2A-2A. 反报告腔价值标准

AI 知识类视频不能只把专业结构写完整，还必须让观众感觉“这个问题和我有关”。高价值文案不是把机制说明、字段清单和执行步骤排整齐，而是用真人叙事、具体坑点和工具救场，把复杂判断讲到观众能听懂、能代入、能知道下一步。

价值判断不能只看：

- 有没有问题。
- 有没有动作。
- 有没有结果。
- 有没有边界。

还要看：

- 像不像真人在讲。
- 是否先站在普通人处境。
- 有没有“这不就是我吗”的感觉。
- 专业细节有没有变成坑点，而不是字段。
- AI / Codex 出现时是不是救场，而不是产品介绍。

如果一条文案结构正确但像项目报告，仍然不能算高价值文案。自然口播链条必须优先于标签化结构：先让观众听懂为什么卡住，再让工具动作和结果变化顺着案例出现。

硬规则：

```text
说人话不是降低专业度，而是把专业判断翻译成普通人能感到痛、能看懂坑、能立刻判断下一步的表达。
```

## 2A-3. 素材证据契约与句组证据价值底线

`已确认` AI 知识类视频的价值底线必须先由素材证据成立，再由文案和卡片放大表达。素材只能“主题相关”不等于能证明文案。

进入视频执行前，Codex 必须把 `material_detail_report（素材细节报告）` 转成 `material_evidence_contract（素材证据契约）`，并逐个 `line_group（句组）` 运行 `line_group_evidence_gate（句组证据闸门）`：

- 过程句优先绑定真实录屏、页面、字段、按钮、模板、报告或结果页。
- 数据句必须绑定真实数据来源、素材中可见数据或数据成果卡；没有来源不得写 direct match。
- 判断句 / 边界句若不是素材直接可证明，必须由 HyperFrames 判断卡 / 边界卡承接。
- 动作句必须有动作画面、动作卡或结果卡；普通录屏背景不能硬当动作证据。
- 高隐私素材、`cannot_support` 命中的素材、背景上下文素材不得承接证明性文案。

价值验收时必须区分：

- `direct_match（直接证据匹配）`：素材本身可证明该句。
- `card_required_resolved（需卡片且已解决）`：卡片承接判断 / 边界 / 数据压缩，但不新增事实。
- `proxy_match（代理匹配）`：只允许过渡 / 背景，不算证明。
- `blocked_no_evidence（无证据阻断）`：不得进入剪辑 / 装配 / 成片生成。

## 2A-4. visual_evidence_must_remain_visible_rule（视觉证据必须可见规则）

`已确认` AI 知识类视频的强证据必须保持可见。保护隐私、追求美观、统一比例或增加卡片，都不能牺牲核心证据。

硬规则：

1. 商品卡、候选表、明细表、复查表、聊天框结论是“Codex 选品初筛”类视频的价值主体，必须尽量保留结构可读。
2. 保护隐私不能把主体页面洗白、灰化或遮成黑块；如果遮挡后证据不可读，宁可 blocked，不可交片。
3. 画面好看不等于可读，安全遮挡不等于内容可用。观众必须能看出“商品卡 -> 表格 -> 复查清单 / 结论”的证据链。
4. 卡片、字幕、局部放大只能帮助理解真实证据，不能替代真实录屏证据，不能覆盖核心表格或商品卡。
5. 发现真实 `secret / API key / token` 等高敏信息时，不能默认遮挡后继续，必须先 blocked 并取得用户授权。
6. 商品名、价格、佣金、销量、店铺分、商品分和表格字段默认属于内容证据，不得因泛化隐私策略自动洗白。

## 2A-5. publish_candidate 人感质量与内容状态边界

AI 知识类视频的 `publish_candidate（发片候选）` 必须同时看技术完整性、证据可读性、卡片 / 字幕不遮挡、TTS 韵律、人感顺滑、平台风险和审片包完整性。卡片好看、视频生成成功、TTS 文件存在、技术检查通过，都不等于 `content_validation（内容验证）` 通过。

发片候选进入人审前，必须至少具备：

- `publish_candidate_required_inventory（发片候选必交付清单）` 已逐项判断。
- `review_pack（审片包）` 能让 ChatGPT / 用户复核标题、文案、画面、声音、字幕、卡片、证据和剩余阻断。
- `visual_evidence_check（画面证据检查）` 能证明关键证据没有被遮挡、洗白、裁掉或卡片覆盖。
- `subtitle_card_overlap_check（字幕卡片重叠检查）` 无 high severity overlap。
- `tts_prosody_anchor_map（TTS 韵律锚点表）` 已约束语速、停顿、重音和句间节奏。
- `tts_route_report（TTS 路线报告）` 明确 `actual_tts_provider = minimax`，`actual_tts_model in [speech-2.8-hd, MiniMax/speech-2.8-hd]`，且 `audio_present / non_silent / no_fallback` 为真。
- `data_goal_alignment_check（数据目标对齐检查）` 已说明这条视频如何服务本轮目标、主短板、主变量和验证指标。
- `remaining_blockers（剩余阻断）` 已显式列出，不能用“看起来能发”替代。

硬规则：

- `publish_candidate_ready_for_human_review` 只表示可进入人工复审，不表示内容已通过。
- `content_validation` 只能由 ChatGPT / 用户最终复审或明确规则链路推进；技术成功不得自动推进。
- `send_ready`、`voice_validation`、`visual_master_locked`、`current_data_goal_anchor_ready` 不得由发片候选机制自动推进。
- 人感质量检查缺失时，发片候选只能 `continue` / `blocked`，不得写 `completed`。
- MiniMax `speech-2.8-hd` 是后续正片候选唯一默认 TTS 路线；阿里 B 方案只保留为 `voice_feel_reference（声音听感参考）`。
- 非 MiniMax TTS，包括阿里 Qwen TTS、Serena、macOS say、本地低质 TTS 或未授权 fallback，不得写成 `publish_candidate_ready_for_human_review = true`；只能写 `blocked_publish_candidate_unavailable` 或 `internal_diagnostic_only`。

## 2A-6. publish_candidate_preflight_suite 价值验收边界

AI 知识类视频的发布候选价值判断，必须把 `publish_candidate_preflight_suite（发片候选预检套件）` 作为进入人工复审前的结构化证据之一。套件通过只表示导出前八个闸门报告齐全且没有已声明阻断，不表示内容最终通过。

必须纳入套件的价值相关 gate：

- `line_level_alignment_preflight`：证明每个关键句组没有“口播说 A，画面显示 B”的未修复错位。
- `tts_route_and_prosody_preflight`：证明目标声音路线、实际声音路线、语速、停顿、重音和 fallback 授权边界清楚。
- `publish_candidate_voice_gate`：证明正片候选实际使用 MiniMax `speech-2.8-hd / MiniMax/speech-2.8-hd`，B 方案只作听感参考，非 MiniMax 语音不能进入人工复审候选。
- `card_decision_preflight`：证明判断卡、总结卡、结果差卡、边界卡、Prompt 尾卡服务观众理解，不抢真实证据。
- `forbidden_action_preflight`：证明没有改 locked copy、删必需卡、未授权遮挡、未授权 fallback 或状态推进。
- `visual_evidence_readability_preflight`：证明商品卡、表格、聊天框结论、平台数据或真实工作流证据没有被洗白、黑块、字幕、卡片或比例处理破坏。
- `locked_copy_diff_preflight`：证明实际字幕、TTS 和卡片没有把 ChatGPT / 用户锁定文案改成另一条内容。
- `completion_truth_preflight`：证明 `full.mp4`、技术验证或字段存在没有被误写成内容通过、可发或完成。

若视觉证据、人感质量、卡片决策、声音路线或 locked copy 差异任一预检不过，当前只能写 `blocked / continue / human_review_required`，不得写 `content_validation = passed`、`send_ready = true` 或 `publish_candidate_ready_for_human_review = true`。

## 2B. data_goal_anchor（数据目标锚点）价值闸门

AI 知识类视频价值判断必须接入 `data_goal_anchor（数据目标锚点）`。

当前实例来源必须优先读取：

- `codex_log/current_data_goal_anchor.md（当前数据目标锚点）`

该文件只代表当前这一条 / 下一条视频实际使用的锚点实例；如果它的 `anchor_instance_status（锚点实例状态）` 仍是 `waiting_data / draft`，只能做假设版价值判断或 blocked，不能写成正式数据驱动执行 ready。

一条视频不只要让用户“看懂 / 拿走东西”，还必须说明它如何服务本轮：

- `current_stage_goal（当前阶段目标）`
- `main_bottleneck（主短板）`
- `primary_variable（主验证变量）`
- `forbidden_variables（禁止变量）`
- `success_metric（成功指标）`
- `failure_metric（失败指标）`
- `post_publish_validation_metric（发布后验证指标）`

硬规则：

- 视频价值判断必须先回答“这条内容服务哪个数据目标”，不能只回答“这条好不好看 / 顺不顺 / 懂不懂”。
- 视频价值判断必须回指 `current_data_goal_anchor_source（当前数据目标锚点来源）`，不能只引用抽象规则。
- 剪辑顺、卡片好看、画面更美，只能提高人感质量，不得写成数据目标完成。
- 人感质量是发布前验收，不能替代发布后数据验证。
- 数据目标锚点不能压掉素材证据；素材不能证明的点，仍必须删改、降级或 blocked。
- 数据目标锚点不能压掉用户 / ChatGPT 最终内容判断；`send_ready（可发送状态）` 仍需人工复审。

## 3. 当前价值承载形态
当前更适合优先承载价值的内容形态是：
- `AI 职场任务现场拆解`

原因：
- 更容易展示真实 AI 使用过程
- 更容易让观众看到哪里节省了重复劳动
- 更容易把模糊需求压成可执行任务
- 更容易展示真实工作流证据
- 更容易把前后差值讲清
- 更容易留下最小行动

## 4. 当前真正的默认价值单元
`已确认` 当前默认优先验证的，不是“单 prompt”，也不是“每条都长成工作包”。

当前默认更适合优先承载价值的单元是：
- 一条 `真实 AI 使用经验 + 工作提效实录`

它至少要让用户看见：
- 真实 AI 使用过程
- 哪里节省了重复劳动
- 前后变化或结果差
- 最小可执行动作

## 5. 视频与工作包的关系
- 视频默认仍是：`证明壳 / 入口壳`
- 当前更准确地说，视频默认是：`真实经验证明壳 / 提效证据入口壳`
- `场景化专业输出工作包` 当前只作为：`可选产品化承接单元`
- 结尾总结后的 `Prompt 引用尾卡`，默认只负责把用户从视频引到可选沉淀，不再假设每条都必须落到工作包

## 6. 强证据优先级
当前更强的证据优先级是：
1. 同一个工具在“普通输入” vs “工作包输入”下的结果差值
2. `用户录制素材`
3. 前后对比
4. 步骤验证截图
5. 结果截图

当前更弱的证据是：
1. 只有 PPT 总结
2. 只有抽象判断
3. 只有口头说明

## 6A. 什么时候才建议沉淀工作包

只有同时或部分出现以下信号时，才建议把内容继续沉淀成 `场景化专业输出工作包`：
- 高收藏或高评论
- 明确咨询或用户追问
- 可复用流程已经相对稳定
- 结果差足够强，适合做产品化承接

在这些信号不足前，默认先把视频当成内容价值验证与平台反馈验证，不把工作包写成当前唯一中心目标。

## 6A. HyperFrames 动效与真实证据边界

`已确认` HyperFrames 当前只能提高三类卡片的表达清晰度和情绪节奏，不能替代强证据。

当前允许：

- 数据卡 / 结果差卡动效服务“普通输入 vs 工作包输入”的结果差理解。
- Prompt 引用尾卡动效服务低压承接和工作包 / prompt 包入口。
- 骚萌卡动效服务问题钩子、反面反转、正面反转，放大笑点和爽点。
- 判断卡动效服务核心判断句 / 关键定性句的轻量强化。
- 总结卡动效服务观点收束、最小行动和低压结束。

价值判断硬规则：

- HyperFrames 动效必须服务问题、反转、结果差。
- HyperFrames 不得补充真实素材没有证明的新结论。
- HyperFrames 不得把数据卡做成密集 dashboard，导致观众误以为有更多真实数据。
- HyperFrames 不得把 Prompt 尾卡做成第二个主结尾或强卖课 CTA。
- HyperFrames 不得把骚萌卡做成普通可爱贴纸；三张骚萌卡必须服务“这说的不就是我吗”、“旧做法看着有东西但不能用”、“这回真的能用了”。
- `judgment_card（判断卡）` 和 `summary_card（总结卡）` 被 `card_placement_decision（卡片位置判断）` 选中时，HyperFrames 是正式运营阶段默认动效与视觉优化基线；画面必须比普通静态 PPT 卡片更优美、更轻、更像发布候选片。
- `clean_soft（干净柔和）` 与 `cute_ai_guide（可爱 AI 向导）` 已锁定为判断卡 / 总结卡的最小 allowed visual skins；前者优先服务温和解释、低压收束和柔和总结，后者优先服务 AI 知识类轻陪伴引导和友好判断。
- `sharp_judgment（清晰判断）` 本轮不纳入默认锁定基线；不得作为后续判断卡 / 总结卡的默认皮肤。
- 如果 HyperFrames plugin / script / runtime entry 不存在或不可用，必须写 `blocked`、`runtime_missing` 或 `not_verified`，不得用普通静态卡片冒充 HyperFrames 已执行。
- 判断卡 / 总结卡是否出现仍由 `card_placement_decision（卡片位置判断）` 判断；不得把它们变成每条视频固定强插卡。
- HyperFrames 视觉更优美、最小皮肤通过或用户通过最小卡片预览，都不等于 `content_validation（内容验证）` 通过，不等于 `send_ready（可发送状态）`，不等于发布成功，也不等于最终视觉母版锁定。
- 中段强证据仍必须优先由 `用户录制素材`、前后对比、步骤验证截图和结果截图承担。

## 6B. AI / API 生成图片与真实证据边界

AI 图片、API 生成图、角色图、背景图、图标和氛围图只能服务辅助表达，不能替代强证据。

当前允许：

- 作为视觉壳、背景、角色、图标或信息卡辅助理解。
- 作为 `Prompt 引用尾卡`、总结卡或段落提示的视觉承载。
- 在不抢真实证据位的前提下，帮助观众更快理解结构。

当前禁止：

- 把 API 生成图当成用户真实录屏、真实截图、真实结果差或真实平台数据。
- 在中段核心证据位置用 AI 图替代 `用户录制素材`。
- 用好看的图片弥补素材无法证明的文案点。
- 把“素材视觉好看”写成 `content_validation（内容验证）` 通过。
- 复用官方 Minecraft 资产、logo、字体、texture、model、sound，或制造官方背书错觉。

如果真实证据不足，默认路线是删改文案、补录、用少量 PPT / 信息卡降级，或直接 `blocked（阻断）`；不得用 API 图冒充证据链。

## 7. 爆款潜力和变现潜力必须分开判断

### 更容易出爆的内容
往往更偏：
- 情绪钩子强
- 前几秒抓人
- `紧迫任务现场型` 或 `错误示范型` 开头

### 更容易变现的内容
往往更偏：
- 具体职场场景
- 有前后差值
- 有具体任务
- 有最小行动
- 能自然接工作包、资料包、咨询或服务

必须明确：
- 更容易出爆，不等于更容易变现
- 更容易讲懂，不等于更容易承接

## 7A. 开头钩子与强证据边界

`meme_gif_opening_hook（梗图 GIF 开场钩子）` 可以提升前几秒停留，但它只负责：

- 抓眼。
- 起情绪。
- 抛核心问题。
- 让观众愿意进入中段证据。

它不负责：

- 证明观点成立。
- 替代中段真实录屏。
- 替代前后对比、步骤截图、结果截图或平台数据。
- 推进 `content_validation（内容验证）`。

强证据仍必须来自：

- `用户录制素材`。
- 前后对比。
- 步骤截图。
- 结果截图。
- 平台数据。

验收边界：

- 梗图 GIF 开头不算强证据，只算 opening hook。
- 开头路线选择不等于内容验证通过。
- 如果中段真实证据不足，不能用更强的开头包装补证据缺口，必须删改文案、补录或 blocked。
- 用户参考图只能继承“抓眼 / 起情绪 / 快速抛问题”的机制，不得复刻具体人物、头像、字体、构图或第三方可识别资产。

## 7B. 文案功能卡与强证据边界

`data_result_card（数据成果卡）`、`summary_card（总结卡）`、`reversal_card（反转卡）`、`result_diff_card（结果差卡）` 和 `prompt_tail_card（Prompt 引用尾卡）` 都只能服务观众理解路径，不等于强证据本身。

价值边界：

- `summary_card（总结卡）` 只帮助观众收束观点、压缩结论和记住最小行动，不等于强证据。
- `reversal_card（反转卡）` 只承担认知转折，不等于结果差证明。
- `result_diff_card（结果差卡）` 可以翻译“看得见的差异”，但不得新增素材里没有的数据结论。
- `data_result_card（数据成果卡）` 只能压缩已存在的真实数据、AI 判断、下一步变量和验证指标；它优先级高于普通流程总结卡，但不能把指标名称、推测或缺失字段包装成真实数据。
- `prompt_tail_card（Prompt 引用尾卡）` 只做低压引用和承接，不承担主叙事或主结论。

强证据仍必须来自：

- `用户录制素材`。
- 前后对比。
- 步骤截图。
- 结果截图。
- 平台数据。

验收边界：

- 卡片好看、卡片动效顺，不等于 `content_validation（内容验证）` 通过。
- 卡片位置选择正确，不等于真实视频长期效果已验证。
- 如果真实证据不足，不能用总结卡、反转卡、结果差卡或 Prompt 尾卡补证据缺口，必须删改文案、补录或 blocked。
- 如果某段真实录屏正在承担关键证据，卡片不得抢走证据窗口。
- 卡片判断必须先跑 `card_budget_gate（卡片预算闸门）` 与 `cluster_merge_rule（信息簇合并规则）`；不得看到一个数据句、判断句或总结句就插一张卡。
- 超出预算时，优先保留 `data_result_card（数据成果卡）`、关键判断卡和结尾总结卡，普通 `process_summary_card（流程总结卡）` 必须合并、降级或删除。

## 8. 商业化副线路径
`CPS` 可以作为副线路径考虑。

但当前只能写成：
- `建议判断`
- `待验证`

当前不能写成：
- 市场已成立
- 当前主商业化模型已经验证通过

## 9. 进入样片前的价值闸门
至少要回答：
1. 用户看完后能做什么？
2. 用户看完后能判断什么？
3. 强证据是什么？
4. 中段证据是否真的由 `用户录制素材` 承担？
5. 结尾总结与 `Prompt 引用尾卡` 是否各守其位？
6. 这条内容当前是只做验证，还是已经出现足够信号值得沉淀为工作包？

## 9A. 数据阈值与需求信号价值闸门

AI 知识类视频的价值判断必须接入 `13_目标驱动数据飞轮与文案执行闭环_goal_driven_data_flywheel_and_copy_execution_loop.md` 中的 `threshold_config_v1（阈值配置 V1）` 和 `lead_score_model（客资评分模型）`。

判断边界：

- 播放量是前置入口，不是最终目标。
- 点赞不是需求。
- 收藏率低于 `1%` 默认视为可复用价值弱；`1%-2%` 只算可观察；高于 `2%` 才值得重点复盘其方法 / 步骤 / 模板 / 结果差价值。
- 点赞率低于 `2%` 默认视为观点或表达认可弱；`2%-5%` 是基本认同；高于 `5%` 仍需结合收藏 / 评论 / 私信判断需求。
- 有效评论 `0-2 条` 默认互动信号弱；`3-5 条` 可观察用户是否复述痛点；超过 `5 条` 才可能出现真实共鸣或需求线索。
- 私信总数不是客资质量；客资必须按 `lead_score_model` 评分。
- `lead_score >= 3` 才计入有效客资；`lead_score >= 4` 才计入高价值客资；`lead_score = 5` 才计入可转化客资。
- 30 天内少于 `3 个 3 分以上客资` 时，不得急着产品化。
- 出现 `>= 8 个 3 分以上客资` 且至少 `2 个 4 分以上客资` 时，才进入需求类型归纳。
- 出现 `1 个 5 分客资` 可以考虑轻咨询 / 工作包 / 服务雏形，但仍不得写成商业模式已经成立。

必须区分：

- `play_signal（播放信号）`
- `value_signal（价值信号）`
- `interaction_signal（互动信号）`
- `lead_signal（客资信号）`
- `commercial_validation_signal（商业验证信号）`

禁止：

- 不得把一次播放高写成方向成立。
- 不得把一次客资高写成商业模式成立。
- 不得把阈值写成行业定论；当前全部阈值都是阶段工作假设。
- 数据不足时只能写 `待验证`，不能强行输出内容方向或产品方向成立。

## 10. 一句话规则
**当前 AI 知识类视频的最低价值标准，不是“看起来内容很多”，而是“能让用户看到真实 AI 使用过程、工作提效证据、前后结果差，并知道下一步最小能怎么做”；`场景化专业输出工作包` 当前只作为可选产品化承接单元，只有高反馈、强结果差和稳定流程信号出现后才继续沉淀；`CPS` 只能先当副线路径建议，不能冒充已成立事实；HyperFrames 动效和梗图 GIF 开头只能强化问题、反转和停留，不得替代真实录屏证据。**

#### 11.2.5 声音基线 V1

| 停顿与韵律 | `停顿跟语义边界一致；不乱断；不会因停顿把意思切碎` | `不是按标点机械停，而是按理解停；转折词前有准备感；重点给空间` | `逗号一律一个时长、句句像切片、转折词没有层次` |
| 情绪弧线 | `开头 / 判断 / 收尾至少能听出不同功能，不再像说明书` | `推荐弧线为“开头轻惊喜 → 中段平稳清楚 → 结尾温暖收束”` | `全程一个平面、开头就太满、结尾突然打鸡血` |
| 音色一致性 | `18 秒内不明显漂移，不像换人，不像换麦` | `即使分句生成再拼，也像同一个角色同一次说完` | `同段像换人 / 换麦 / 情绪一变音色就散` |

| 文案贴合度 | `声音重点和文案重点一致，不会整段一个力度` | `判断句更稳、动作句更轻、结果句略留白；结构词能帮助观众听懂层级` | `判断句念成广告句、结果句一带而过、声音与字幕 / 画面抢主次` |

#### 11.2.6 声音过线规则
- 没有一票否决项，才允许继续判断“合格片”
- 核心维度至少包括：
  - 自然度
  - 向导感
  - 停顿与韵律
  - 播音腔 / AI 感
- 以上核心维度全部达合格，才允许把声音部分写成“合格”
- 当前这套声音标准只能标记为：
  - `已确认`：方向与回审维度已写入
  - `部分成立`：方向收敛但样片通过记录不足
  - `待验证`：真实效果、具体样片和最终供应商
- 当前明确禁止：
  - 直接写“声音路线已最终确定”
  - 直接写“当前样片已完成声音内容验证”

| 可爱感 | `可爱来自轮廓、配色、轻微 smile 感，不来自装嫩和卖萌` | `像“可爱的主持娃娃”，亲近但不幼态，亮点来自轻巧和友好` | `装嫩、卖萌、幼态过强、可爱感靠夸张动作硬堆` |
| 活泼感 | `有轻微 bounce、头部反应和小手势，不是站桩` | `活泼但不乱，能随着文案功能轻轻起伏，像主持人在“点一下重点”` | `动作过僵像站桩，或动作过闹像乱跳` |
| 文案与动作对齐 | `提问、判断、转折、收尾至少能看出动作差异；口型、表情、头部、手势服务同一层意思` | `重点词、判断句、过渡句都能看出动作重心变化，像“听懂了再表达”` | `口型、表情、头部、轻微 bounce、手势各做各的；动作跟文案功能冲突` |
| 镜头与画面适配 | `镜头稳定、构图留白够，字幕和旁侧信息还有安全区` | `一眼能看出“这是主持壳 + 信息区”的画面组织，角色存在感和信息可读性都成立` | `构图挤压字幕、镜头乱晃、角色动作出框、背景过抢` |
| 不抢中段主体 | `主持壳只承担开头进入、关键判断、对比转折、结尾收束，不越权占中段` | `主持壳进得快、退得快，把主体稳稳让给用户录制素材` | `主持壳长时间连讲、替代中段真实录屏、变成整条主叙事` |

#### 11.3.3 文案功能 -> 动作映射 V1
| 文案功能 | 推荐动作 | 推荐镜头 | 推荐表情 | 动作密度建议 | 失败线 |
| --- | --- | --- | --- | --- | --- |
| 开头提问 | 轻微前倾、单手打开、轻点头 | 半身近景，人物居中或偏一侧给标题留白 | 好奇、友好、轻提醒 | `2-3 秒内 1 个主动作 + 1 个轻 bounce` | 站桩不动，或一上来就大挥手 / 大摇摆 |
| 关键判断 | 小幅点头、手掌轻切、短停顿后再落手势 | 半身稳定镜头，正面或微侧面 | 稍认真，但不凶 | `低密度`，让判断句先成立 | 像上课训人、像销售拍桌、像 idle loop |
| 对比转折 | 身体微转、手势指向左右信息区、头部有切换 | 半身或中近景，画面留出对比区 | 先提眉再收稳 | `每个转折 1 个明显切换动作` | 动作太碎，看不出“前后差” |
| 操作提示 | 手掌向侧边轻引、视线短暂带向屏幕信息区 | 半身偏侧构图，给操作区留白 | 清楚、鼓励、低压 | `只点一下动作词，不连续挥动` | 像客服演示、像老师板书、像在抢录屏主体 |
| 结尾收束 | 小幅收手、轻点头、温和邀请式手势 | 半身近景或中近景，构图稳定 | 温暖、收住、微笑感 | `低密度`，以收住结论为主 | 突然打鸡血、突然大幅卖萌、收尾比开头更吵 |

### 11.4 联合过线规则
- 声音不过线 = `content_validation（内容验证）` 不通过
- 动作不过线 = `content_validation（内容验证）` 不通过
- 没有一票否决项，且核心维度达合格，才允许写成“合格片”
- 当前声音部分核心维度至少包括：
  - 自然度
  - 向导感
  - 停顿与韵律
  - 播音腔 / AI 感
- 当前动作部分核心维度至少包括：
  - 主持感
  - 文案与动作对齐
  - 不抢中段主体
- 连续多轮复审稳定成立后，相关结论才允许从：
  - `待验证` → `部分成立`
  - `部分成立` → `已确认`
- 当前明确禁止：
  - 用旧样片通过状态，去证明新 `content_validation V1` 已验证通过
  - 用某个供应商“先验证”去替代内容效果“已过线”

## 12. 一句话规则
**当前 `content_validation V1` 的作用，不是先定最终供应商，而是先把 `可爱女生向导音` 和 `可爱元素主持娃娃` 写成可直接回审的标准：声音要轻陪伴、低压、像游戏向导但不夹不播报，动作要活泼可爱但不乱、不像 NPC、也不抢中段真实录屏；没有一票否决项且核心维度达合格，才算内容过线。**

## 13. quality_lock_card（质量锁卡）

每条 AI 知识类视频进入成片执行前，必须生成 `quality_lock_card（质量锁卡）`。

这张卡的作用是锁执行前质量底线，不证明内容已经通过，也不替代用户 / ChatGPT 最终复审。技术生成成功不等于质量锁卡通过。

`quality_lock_card（质量锁卡）` 必须调用 `quality_issue_classifier（质量短板分类器）`。质量锁卡负责锁底线，分类器负责在用户觉得“不对 / 怪 / 不顺 / demo 感”或技术通过但内容不舒服时，先判断唯一最高优先级短板。

质量锁卡至少包含：

| 字段 | 填写要求 |
| --- | --- |
| `viewer_takeaway（用户看完能拿走什么）` | 写最小可理解结论、最小行动或最小判断。 |
| `proof_type（证明类型）` | 从真实录屏、前后对比、步骤截图、结果截图、平台数据中选择，可多选。 |
| `result_diff（结果差）` | 写普通做法 vs 这次做法的可感知差异。 |
| `minimum_action（最小行动）` | 写观众下一步能做什么，不写泛泛收益。 |
| `voice_quality_line（声音质量底线）` | 写声音必须满足哪些 V1 底线；不写声音已验证通过。 |
| `host_role_line（主持壳职责底线）` | 写主持壳承担进入、判断、转折、收束，不抢中段主体。 |
| `ppt_density_line（PPT 密度底线）` | 写 PPT 只做结构整理 / 总结，不替代中段证据。 |
| `reference_quality_points（继承的 reference 质量点）` | 写要继承的质量点、风格边界、展示边界。 |
| `one_vote_fail_items（一票否决项）` | 写 AI 感过强、证据不足、声音 / 动作明显不过线、平台风险硬阻断等。 |
| `not_validated_yet（仍未验证项）` | 写声音、动作、内容、平台数据或产品信号仍待验证的部分。 |

硬规则：

- 没有质量锁卡，不得进入成片执行。
- 质量锁卡只能写成执行前约束，不能写成 `content_validation（内容验证）` 已通过。
- 如果 `proof_type（证明类型）` 没有真实素材支撑，必须降级为 `待验证` 或 blocked。
- 若出现一票否决项，不能用 `technical_validation（技术验证）` 通过来覆盖。
- `reference（参考）`、`locked reference（锁定参考）` 和 `visual route（视觉路由）` 只能继承质量点，不能把旧流程写成当前固定流程。

## 13A. quality_issue_classifier（质量短板分类器）

当技术通过但用户感觉不对 / 怪 / 不顺 / demo 感时，必须先运行 `quality_issue_classifier（质量短板分类器）`，不得直接重做、直接换素材、直接换声音或推进状态。

最小结构：

```text
quality_issue_classifier:
  input_signal:
  observed_evidence:
  issue_categories:
  state_inference:
  action_policy:
  validation_rule:
  blocked_if:
  feedback_update:
```

分类范围：

- `direction_issue（方向问题）`
- `structure_issue（结构问题）`
- `evidence_issue（证据问题）`
- `editing_issue（剪辑问题）`
- `voice_issue（声音问题）`
- `voice_prosody_issue（声音韵律问题）`
- `opening_visual_hook_issue（开头视觉钩子问题）`
- `script_visual_mismatch_issue（文案画面错位问题）`
- `visual_issue（画面问题）`
- `card_density_issue（卡片密度问题）`
- `host_role_issue（主持壳职责问题）`
- `reference_deviation_issue（reference 偏离问题）`
- `state_conflict_issue（状态冲突问题）`
- `platform_risk_issue（平台风险问题）`
- `data_insufficient_issue（数据不足问题）`

硬规则：

- 必须定位唯一最高优先级短板；多个问题同时存在时，只能选一个 primary issue 作为下一轮改动变量。
- 必须区分 `technical_validation（技术验证）` 和 `content_validation（内容验证）`。
- 用户反馈是主观信号，不能直接写成事实；必须用 review_pack、summary、时间码、截图、抽帧、声音候选、visual_route_map 或 reference deviation check 佐证。
- 用户反馈“某个字突然上扬 / 节奏不自然 / 分句断裂 / 重音机器感”时，优先进入 `voice_prosody_issue（声音韵律问题）`，诊断顺序是 `prosody -> pause_timing -> sentence_segmentation -> emphasis -> pitch_contour -> voice_timbre`，不得默认换音色。
- 开头只有两行静态标题、普通标题卡或 PPT 封面感，且本条需要高情绪 / 抖音抓眼 / 梗图 GIF / 抽象动效开头时，优先进入 `opening_visual_hook_issue（开头视觉钩子问题）`，必须生成 `opening_visual_hook_spec（开头视觉钩子规格）`。
- 文案和画面局部错位、证据句没有对应画面、边界声明配误导性证明画面或只有段落级素材分配时，优先进入 `script_visual_mismatch_issue（文案画面错位问题）`，必须生成 `script_to_timeline_map（文案到时间线映射表）`。
- 缺复审对象、缺用户反馈对象、缺灰度数据或需要用户听感 / 审美判断时，必须 `blocked_need_user_input` 或 `human_review_required`。
- 若只是候选判断，不得推进 `content_validation / send_ready / publish_status / voice_validation / final_voice_validated / visual_master_locked`。

新增分类动作：

```text
quality_issue_classifier_new_categories:
  voice_prosody_issue（声音韵律问题）:
    signals:
      - sudden_pitch_rise
      - unnatural_pause
      - broken_sentence_flow
      - robotic_equal_weight
    preferred_action:
      - generate_tts_prosody_anchor_map
      - rewrite_sentence_segmentation
      - do_not_default_to_voice_change

  opening_visual_hook_issue（开头视觉钩子问题）:
    signals:
      - static_two_line_title
      - plain_title_card
      - low_douyin_visual_appeal
      - no_motion_energy
    preferred_action:
      - generate_opening_visual_hook_spec
      - block_static_title_if_high_emotion_hook_needed

  script_visual_mismatch_issue（文案画面错位问题）:
    signals:
      - sentence_visual_mismatch
      - evidence_sentence_without_matching_visual
      - boundary_statement_with_misleading_visual
      - paragraph_level_mapping_only
    preferred_action:
      - generate_script_to_timeline_map
      - block_video_execution_until_line_level_mapping_exists
```
