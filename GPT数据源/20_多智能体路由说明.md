# 20｜多智能体路由说明（给 GPT Project 用）

## 1. 文件定位

本文件放进 **GPT Project 数据源**，只负责一层短路由说明。

它回答 5 个问题：

1. 这轮更适合 `fast_lane`、`standard_lane` 还是 `audit_lane`
2. 这轮能不能并发
3. 能并发时更适合哪一种并发结构
4. ChatGPT 命中这类任务时默认该输出什么
5. 完整执行细则正文应该去哪里读

它不负责：

- 完整执行模板正文
- Codex 执行层细则全文
- worktree / git / skills 的完整操作说明
- 某一轮具体任务的执行单全文

一句话：

**本文件只负责让 GPT 知道“先选执行车道，再选并发结构”，不负责替代 Codex 执行层正文。**

---

## 2. 先判 lane，再判 parallel

默认顺序必须固定为：

1. 先判断当前任务该走哪条 `lane`
2. 再判断当前任务能不能 `parallel`

不要反过来。

因为如果当前任务连风险等级、读取深度、是否需要先审计都没判清，直接上并发只会更快放大跑偏。

---

## 3. 当前默认 3 条 lane

### 3.1 `fast_lane`

只在以下条件同时成立时才建议：

- 当前对象已固定
- 当前状态已足够新，不需要重审大量事实
- 当前唯一最高优先级目标已固定
- 本轮只改单个对象或少量局部文件
- 轻量证据已经足够支撑判断
- 不需要先重判“这是规则问题、内容问题还是实现问题”

默认动作：

- 少解释，直接执行
- 压缩读取和对齐时间
- 条件失效就立刻降级

### 3.2 `standard_lane`

这是当前项目的默认执行车道。

适合：

- 目标和边界已经基本锁定
- 但还不能当成低风险快车道
- 仍需要正常读取、正常验证、正常自检
- 本轮虽聚焦，但仍需要吸收多份资料后再落地

默认动作：

- 先读固定入口
- 再按已锁定目标执行
- 过程正常核对
- 不为了显得快而跳过验证

### 3.3 `audit_lane`

命中以下任一情况时优先建议：

- 当前对象指针未刷新或可能过期
- 当前状态说法之间存在冲突
- 本轮需要重判“该改哪一层”
- 本轮会同时牵连规则、内容、实现三层判断
- 当前慢点主要不是执行，而是判断本身

默认动作：

- 先审计
- 先分层结论
- 先判断能不能进入执行
- 不急着提速

---

## 4. `fast_lane` 不是默认总开

这条必须写给 GPT：

- `fast_lane` 只是条件成立时的提速车道
- 不是默认总开
- 不是“任务看起来小就自动走”

只要出现以下任一情况，就不要给 `fast_lane`：

- 指针可能过期
- blocker 未收口
- 当前状态还依赖大块重读
- 本轮还没收清边界
- 本轮需要先判断冲突事实

这时默认降级到：

- `standard_lane`
- 或 `audit_lane`

---

## 5. 当前默认 4 种 parallel mode

### `serial_only`

默认保守值。

适合：

- 当前任务主要在单点落地
- 多个子任务会碰同一批核心文件
- 风险主要在执行和验证，不在读取

### `read_parallel`

只读并发。

适合：

- 主要慢在读入口、读规则、读日志、读现状
- 多条读取线可以并行
- 最终仍由单点整合

### `explore_plus_integrate`

当前项目里最常用的并发结构。

等价于：

- `2 explorers + 1 integrator`

适合：

- 并行压缩“读现状 / 找规则 / 收束改法”的时间
- explorers 只读
- integrator 独占写权

### `true_multi_task_parallel`

真正多任务并发。

只在以下成立时才建议：

- 多个子任务彼此独立
- 写入范围清楚且不冲突
- 可各自验收再汇总

---

## 6. 什么时候不要并发

只要命中以下任一项，就默认不要并发：

- 多个子任务会同时改同一份核心文件
- 当前对象还没锁定
- 当前 blocker 还没锁定
- 当前慢在判断，不慢在执行
- 当前会把高风险验证绑在同一路输出上

必须明确：

- 多智能体只是并发结构之一
- 不是所有提速都靠多智能体
- 并发只提升读取、定位、收束效率
- 不承诺 runtime 一定更快

---

## 7. ChatGPT 命中时默认输出什么

以后命中这类任务时，ChatGPT 默认输出至少应包含：

1. `lane_recommendation`
2. `lane_reason`
3. `lane_invalid_if`
4. `parallel_recommendation`
5. `parallel_reason`
6. `parallel_invalid_if`
7. 最省时且不增风险的执行结构
8. 一份可直接给 Codex 的 prompt

## 9. 当前一句话规则

**先判 `lane`，再判 `parallel`；满足条件才提速，满足条件才并发，条件失效就降级；当前项目最常用的并发结构仍是 `explore_plus_integrate`，但只有在读取真的比落地更慢、且写入权能单点收束时才建议启用。**
