# v3.1 灰度测试指标体系 V1

## 1. 文件定位

本文件是《我用 AI 做 PPT 踩过的坑》v3.1 发布后灰度测试的指标体系 V1。

这套指标体系不是运营数据大表。

它是下一轮改动定位器。

它的目标不是把所有平台数据填满，而是用最少必要字段回答：下一轮最应该只改哪一个变量。

当前硬边界：

- v3.1 已发片，进入灰度测试。
- 发片不等于内容过线。
- 灰度测试不等于验证成功。
- `content_validation` 仍是 `gray_testing_not_final_passed`。
- `send_ready = false` 不等于“没有发片”；发片状态另看 `publish_status = gray_test_published`。
- Codex 只负责记录、初检、归档和下一轮任务草稿。
- 最终判断仍交给 ChatGPT / 用户。

## 2. 当前基础测试流量门槛

`7d_play_count（7 天播放量）= 6000` 是当前小样本阶段基础测试流量门槛。

必须明确：

- 6000 播放量不是最终商业目标。
- 6000 播放量不是账号方向成立。
- 6000 播放量不是内容最终通过。
- 6000 播放量只表示这条视频拿到了足够做初步判断的基础测试流量。
- 后续样本增多后，该门槛可以校准。

## 3. 后续复盘只先回答四个问题

每次复盘默认收成 4 个问题：

1. 这条有没有达到 6000 播放基础门槛？
2. 当前最短板在哪一层：流量 / 内容 / 账号 / 转化？
3. 下一轮只改哪一个变量？
4. 为什么先改它，改完看哪个指标？

如果数据不足，以上问题可以写 `待验证`，不能硬定结论。

## 4. 四层指标体系

### 4.1 流量层

回答：这条视频有没有拿到足够的基础测试流量。

核心指标：

- `24h_play_count（24 小时播放量）`
- `72h_play_count（72 小时播放量）`
- `7d_play_count（7 天播放量）`

诊断用途：

- 播放量低时，先不要直接否定内容。
- 优先排查平台分发、标题、封面、选题吸引力、开头承诺、发布时间、账号基础权重。
- 播放量是后续比例指标的母数。

### 4.2 内容层

回答：观众点进来后，内容有没有接住、撑住、值得保存。

核心指标：

- `3s_retention（3 秒留存）`
- `completion_rate（完播率）`
- `average_watch_time（平均观看时长）`
- `favorite_count（收藏数）`
- `favorite_rate（收藏率）`

诊断用途：

- 3 秒留存差，优先看开头。
- 完播差，优先看中段结构和画面推进。
- 收藏低，优先看结果差表达和方法沉淀。

### 4.3 账号增长层

回答：观众看完后，是否愿意认识这个账号、进入长期关系。

核心指标：

- `profile_visit_count（主页访问数）`
- `new_follow_count（新增关注数）`
- `play_to_follow_rate（播放转粉率）`

诊断用途：

- 内容有用但不涨粉，优先看结尾承接 / 账号价值承接。
- 要让观众知道关注后持续能得到什么。

### 4.4 私域 / 客户转化层

回答：内容是否触发真实需求、有效咨询和可跟进客户。

核心指标：

- `dm_count（私信数）`
- `effective_dm_count（有效私信数）`
- `effective_consult_count（有效咨询数）`
- `clear_need_customer_count（有明确需求的客户数）`

诊断用途：

- 私信 / 咨询低，不等于内容没价值，可能是承接弱。
- 有私信但客户不精准，优先看场景边界和服务边界。
- 播放不高但咨询质量高，不能轻易否定方向。

## 6. 小样本阶段初始建议阈值

## 7. 根据数据倒推修改方向

### 情况 A：播放量低

判断：

- 如果 7 天播放 `< 3000`，先不要直接否定内容。
- 优先排查平台分发、标题、封面、选题吸引力、开头承诺、发布时间、账号基础权重。

下一轮优先改：

- 标题或封面，二选一。
- 若不能判定，优先改标题。

### 情况 B：播放量够，但留存差

判断：

- 播放量达标，但 3 秒留存 `< 50%`，说明开头没接住。

下一轮优先改：

- 开头 3 秒。
- 包括开头第一句、第一屏画面、前 3 秒承诺。

### 情况 C：留存可以，但完播差

判断：

- 3 秒留存合格，但完播率 `< 10%-15%` 或平均观看时长低，说明中段撑不住。

下一轮优先改：

- 中段文案结构。
- 重点排查：中段拖、解释太多、结果差不明显、画面推进弱、说明书感强。

### 情况 D：完播可以，但收藏低

判断：

- 完播率可以，但收藏率 `< 0.6%`，说明观众能看完，但不觉得值得保存。

下一轮优先改：

- 结果差表达或方法沉淀。
- 优先改结果差表达。

### 情况 E：收藏可以，但涨粉低

判断：

- 收藏率合格，但新增关注少，说明内容有用，但账号长期价值没建立。

下一轮优先改：

- 结尾承接 / 账号价值承接。
- 让观众知道关注后持续能得到什么。

### 情况 F：涨粉可以，但私信 / 咨询低

判断：

- 涨粉可以，但私信 / 有效咨询低，说明观众把内容当知识看，没有意识到可以找用户解决问题。

下一轮优先改：

- 私信引导或工作包承接。
- 优先改私信引导。

### 情况 G：私信有，但客户不精准

判断：

- 私信有，但都是泛泛求教程、免费资源或非目标客户，说明场景边界和服务边界不清。

下一轮优先改：

- 场景聚焦或服务边界。
- 优先改结尾服务边界。

### 情况 H：播放不高，但咨询质量高

判断：

- 播放低，但出现高质量客户咨询，说明商业价值可能成立，不能轻易否定方向。

下一轮优先改：

- 标题或开头，扩大分发。
- 不大改核心内容。

## 8. 下一轮只改一个变量

后续每轮复盘必须收成：

```text
下一轮只改一个变量：
为什么先改它：
其余变量保持不变：
做到什么算改完：
用哪些指标验证：
```

可选变量范围：

- 标题
- 封面
- 开头 3 秒
- 第一屏画面
- 中段文案结构
- 结果差表达
- 结尾承接
- 私信引导
- 工作包承接
- 声音节奏
- 视觉节奏

硬规则：

- 不允许一次建议同时改 3 个以上变量。
- 默认只改 1 个变量。
- 特殊情况下最多 2 个，但必须说明主变量和辅助变量。
- 如果数据不足，不能硬定变量，先标记 `待验证`。

## 9. Codex 与 ChatGPT 分工

Codex 负责：

- 记录数据
- 标注缺失字段
- 做四层初检
- 归档单条记录
- 生成下一轮任务草稿

ChatGPT / 用户负责：

- 判断最终问题层
- 判断是否继续同方向
- 决定下一轮唯一改点
- 判断内容是否真正过线

Codex 不得：

- 把灰度测试写成内容通过
- 把已发片写成最终成功
- 跳过数据直接生成下一条文案
- 新建重复复盘系统替代 `review_loop/`
