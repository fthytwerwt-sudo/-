# 诊断模板

## 1. 文件定位

本模板用于把“结果现象”翻成“下一步先排查什么”的执行动作。

它不是绝对因果表，也不是最终拍板表。

当前只能把它当作：

- 优先排查表
- 结构化初检表
- 给 ChatGPT 继续判断前的收口底稿

## 2. 固定输出格式

每次诊断都按下面五项写：

- 结果现象：
- 最可能问题层：
- 次可能问题层：
- 优先排查变量：
- 当前判断状态：

状态口径固定为：

- `已确认`
- `部分成立`
- `待验证`

## 3. 常用现象到优先排查动作

v3.1 灰度测试 V1 的详细阈值、四层指标和 A-H 倒推规则统一看：

- `review_loop/07_v31灰度测试指标体系_v31_gray_test_metrics_v1.md`

当前诊断优先收成四层：

- 流量层
- 内容层
- 账号增长层
- 私域 / 客户转化层

诊断时默认先回答四个问题：

1. 这条有没有达到 6000 播放基础门槛？
2. 当前最短板在哪一层：流量 / 内容 / 账号 / 转化？
3. 下一轮只改哪一个变量？
4. 为什么先改它，改完看哪个指标？

### 情况 A：播放低

- 结果现象：
  - 播放量明显起不来，或明显低于同批次其他样本
- 最可能问题层：
  - `场景层`
- 次可能问题层：
  - `结构层`
- 优先排查变量：
  - 主场景是否命中
  - 标题与开头是否让人快速知道“值不值得点开”
- 当前判断状态：
  - 默认先写 `待验证` 或 `部分成立`

### 情况 B：前 3 秒掉得厉害

- 结果现象：
  - 前 3 秒留存明显差，开头撑不住
- 最可能问题层：
  - `结构层`
- 次可能问题层：
  - `心理机制层`
- 优先排查变量：
  - 开头钩子类型
  - 第一屏是否太像说明书或太慢进入重点
- 当前判断状态：
  - 默认先写 `部分成立`

### 情况 C：前段留住了，中段明显掉

- 结果现象：
  - 开头能留住，但中段开始明显流失
- 最可能问题层：
  - `结构层`
- 次可能问题层：
  - `画面表现层`
- 优先排查变量：
  - 中段是否进入说明书腔
  - 画面是否没有承接“变化”或信息密度过平
- 当前判断状态：
  - 默认先写 `部分成立`

### 情况 D：完播不差但点赞低

- 结果现象：
  - 大家能看完，但不愿意点赞或表达认可
- 最可能问题层：
  - `心理机制层`
- 次可能问题层：
  - `结构层`
- 优先排查变量：
  - 是否命中身份代入
  - 结果是否足够具体
  - 结尾是否给出明确落点
- 当前判断状态：
  - 默认先写 `待验证` 或 `部分成立`

### 情况 E：收藏高但转化低

- 结果现象：
  - 观众觉得有用，愿意先收着，但后续动作弱
- 最可能问题层：
  - `心理机制层`
- 次可能问题层：
  - `结构层`
- 优先排查变量：
  - 结尾动作类型
  - 结果证明是否足够让人现在就动
- 当前判断状态：
  - 默认先写 `部分成立`

### 情况 F：各项都平平

- 结果现象：
  - 播放、完播、收藏、点赞等都没有明显亮点，也没有单一爆点
- 最可能问题层：
  - `场景层`
- 次可能问题层：
  - `结构层`
- 优先排查变量：
  - 主场景是否够准
  - 主结构是否把真正的变化讲出来
- 当前判断状态：
  - 默认先写 `待验证`

## 4. 使用边界

必须明确：

- 这是一张优先排查表，不是绝对因果表
- 不能把“播放低 = 选题差”写成死规则
- 不能把“完播低 = 结构错”写成死规则
- 不能跳过单条视频记录，直接只看现象下结论

更稳妥的做法是：

1. 先回到单条视频记录看本轮唯一改动变量
2. 再用这张表找“最值得先查哪一层”
3. 最后交给 ChatGPT 做最终判断拍板

## 5. 当前一句话规则

结果现象只能帮助我们缩小排查范围，不能直接替代最终判断；Codex 先把最可能问题层压出来，最终拍板仍交给 ChatGPT。
