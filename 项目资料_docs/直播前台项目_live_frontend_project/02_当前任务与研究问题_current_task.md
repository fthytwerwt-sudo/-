# 当前任务与研究问题

## 1. 当前研究对象
当前研究对象不是“能不能做一个数字人”，而是：

- 在中国大陆 2025–2026 当前公开能力下
- 怎样把“AI 直播前台链路”做到接近抖音高分直播间标准

研究重点只放在：
- 视觉效果
- 听觉效果
- 实时回复能力
- 稳定性
- 并发扩展
- 服务器承载方式

## 2. 当前重点研究问题
当前必须研究清楚的问题包括：

### 2.1 前台 90 分验收表
把“抖音高分直播间前台标准”翻译成一份可执行验收表，要求可用于后续路线比较与 MVP 验收。

### 2.2 阿里前台路线总表
梳理阿里当前所有可用于直播前台的公开路线，至少覆盖数字人、语音、对话、RTC / 推流 / 云直播相关组合方式。

### 2.3 最小 MVP 技术链路
给出最小可验证链路，明确：
- 最小必要组件
- 输入输出关系
- 先验证什么
- 到什么表现再继续加码

### 2.4 多房间扩容判断表
研究“多房间扩容”到底该怎么理解，至少区分：
- 单房链路哪些部分能共享
- 哪些部分必须按房间独立计算
- 哪些瓶颈更可能出现在云侧

### 2.5 “1 台阿里云服务器带 5 个直播间”可行性判断
严格审查该设想是否成立，不能凭经验拍结论，必须拆成链路段判断。

### 2.6 3 档成本模型
给出保守档、平衡档、激进档三档成本模型，用于判断先按量还是进入包月。

## 3. 资料来源要求
资料来源要求如下：
- 优先使用官方公开资料、产品页、价格页、能力说明、限制说明
- 需要外部文章时，优先用可追溯来源，不把营销文案当能力证据
- 涉及价格、配额、能力限制时，必须尽量标清时间点
- 涉及大陆环境时，优先关注中国大陆可落地条件
- 对没有公开证据支撑的说法，必须标为“待验证”或“推测”

## 4. 判断要求
研究与结论必须满足以下判断要求：
- 区分“技术能跑”和“前台质量接近可继续投入”
- 区分单房可行与多房扩容可行
- 区分短期按量验证与长期包月投入
- 区分公开已确认能力与未经证实推测
- 对承载问题按链路拆开判断，不允许一句话打包下结论

## 5. 输出结构要求
后续正式研究输出必须包含以下结构：
1. 一句话总判断
2. 前台 90 分验收表
3. 阿里前台路线总表
4. 最小 MVP 技术链路
5. 多房间扩容判断表
6. “1 台阿里云服务器带 5 个直播间”可行性判断
7. 3 档成本模型
8. 最终明确结论

## 6. 最终明确结论必须回答的 4 句
最终明确结论里必须回答以下 4 个问题：
1. 现在最值得先做的 MVP 路线是什么
2. 达到什么标准后再去包月才合理
3. 一个包月到底更接近带几个直播间
4. “云上跑多个直播间、本地只监控”这个设想到底成立到哪一步
