# 当前项目执行合同与验收

## 1. 当前轮真正目标
当前轮真正目标是把“AI 直播前台验证项目”的研究边界、验收口径、主路线与交付结构压实，确保后续搜索、判断、执行都围绕同一问题展开：

- 在中国大陆 2025–2026 当前公开能力下
- 如何把“AI 直播前台链路”做到接近抖音高分直播间标准

本合同偏执行与约束，不复写账号层长期协作规则。

## 2. 当前明确不做
当前轮明确不做以下事项：
- 不做货品、SKU、库存、成交、售后相关设计
- 不做旧项目事实迁移
- 不把“能跑”当成“前台质量过关”
- 不把单房验证阶段直接包装成多房商业方案
- 不默认接受未经验证的“1 台服务器带多个直播间”结论

## 3. 当前研究范围
当前研究范围只包括：
- 前台视觉效果
- 前台听觉效果
- 实时回复能力
- 运行稳定性
- 并发扩展理解
- 云上承载方式
- 成本结构

## 4. 当前验收维度
本阶段的研究与后续执行，至少按以下维度验收：
- 问题边界是否压实
- 输出物是否完整覆盖既定结构
- 结论是否区分“已确认 / 部分成立 / 待验证 / 推测 / 通用建议”
- 是否形成最小 MVP 路线
- 是否对扩容、承载、成本给出可判断口径
- 是否明确下一步该先验证什么

## 5. 当前状态标记规则
当前项目默认只使用以下状态：
- 已确认：有明确依据或已完成验证
- 部分成立：方向成立，但仍有关键条件未闭合
- 待验证：当前缺少足够证据，必须继续查证
- 推测：基于经验或有限信息作出的暂时判断
- 通用建议：不直接等于项目结论，只作为普遍参考

禁止把中间态写成完成态。

## 6. 当前执行顺序
当前项目执行顺序默认如下：
1. 先把“抖音高分直播间前台标准”翻译成验收表
2. 再梳理阿里前台可用路线
3. 再压出最小 MVP 技术链路
4. 再判断多房间扩容口径
5. 再审查“1 台阿里云服务器带 5 个直播间”是否成立
6. 最后给出 3 档成本模型与最终明确结论

## 7. 当前失败判定
出现以下任一情况，本阶段输出可判失败或需回炉：
- 研究对象跑偏成“做整套直播业务”
- 输出缺少核心交付物之一
- 结论没有标状态
- 以未验证猜想替代承载或成本判断
- 无法回答最终必须明确的 4 个问题
- 前台质量判断仍停留在空泛描述，不能指导下一步

## 8. 当前主路线
当前主路线：
- 先做研究型收敛，形成一套可判断、可比较、可继续执行的前台验证框架
- 再基于该框架推进 MVP 验证，而不是先堆技术后再找标准

## 9. 当前必要备选
如果短期内外部资料不足以支撑完整商业级判断，必要备选是：
- 先做“单房、按量、最小可观察前台链路”判断
- 先拿到最小真实观感和延迟表现
- 再决定是否进入更重的包月与扩容方案

## 10. 什么时候切搜索型 AI
命中以下情况，默认切搜索型 AI：
- 需要最新产品能力、价格、配额、官方限制
- 需要收集阿里数字人、ASR、TTS、RTC、云直播等公开资料
- 需要对比多条公开路线
- 需要判断成本与承载是否存在公开依据

搜索型 AI 的任务是补充外部事实，不直接替代项目结论。

## 11. 什么时候切执行型 AI
命中以下情况，默认切执行型 AI：
- 当前目标明确
- 输入输出结构已锁定
- 资料包已足够支撑一轮明确产出
- 验收标准已写清
- 已到需要真实落文件、改仓库、出产物的时候

## 12. 什么时候形成“小闭环”并 push
只要命中以下任一情况，就视为形成可判断的小闭环：
- 已写完一版核心研究文档
- 已更新验收表、路线表、成本表等关键产物
- 已出现新的 blocker，需要基于真实文件或真实结果复审
- 已完成一轮可读、可评估、可继续的仓库改动

形成小闭环后默认动作：
1. 更新最新摘要
2. 必要时补完整执行日志
3. commit 当前改动
4. push 当前工作分支 / 当前 PR
5. 再交回上层复审
