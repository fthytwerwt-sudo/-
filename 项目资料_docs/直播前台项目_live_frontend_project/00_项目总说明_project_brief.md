# AI 直播前台验证项目总说明

## 1. 项目定义
本项目是一个新项目，当前只做“AI 直播前台验证项目”。

本阶段只验证直播前台质量，不验证后端交易链路，不验证货品经营，不验证完整商业闭环。

当前项目关注的前台问题只有以下几类：
- 数字人前台观感
- 声音自然度
- 回复速度与稳定性
- 是否接近抖音高分直播间前台标准

## 2. 与账号层协作协议的关系
本项目受账号层协作协议约束，但不继承旧项目的业务事实、术语体系、主线方案、历史结论或旧验收口径。

关系边界如下：
- 账号层文件负责规定协作方式、判断顺序、表达方式、状态标记与多 AI 路由
- 项目层文件只负责描述当前“AI 直播前台验证项目”的目标、范围、阶段、成功标准与输出物

## 3. 当前阶段目标
当前阶段目标不是“把整套直播业务跑起来”，而是判断现阶段公开能力下，能否把直播前台链路做到值得继续投入验证的程度。

本阶段需要形成的核心输出包括：
- 前台验收表
- 路线总表
- 最小 MVP 技术链路
- 扩容判断
- 云上承载判断
- 成本模型

## 4. 当前明确不做什么
当前阶段明确不做：
- 货品策略
- SKU 管理
- 库存体系
- 成交优化
- 售后链路
- 复杂运营后台
- 完整商业化打法
- 旧项目任何业务事实迁移

## 5. 当前技术范围
当前默认技术范围围绕以下链路展开：
- 阿里数字人
- ASR
- TTS
- 大模型对话
- RTC / 推流 / 云直播

当前阶段默认先按量验证，再决定是否包月。
当前阶段默认先单房验证，再讨论多房间扩容。

## 6. 当前项目成功标准
当前阶段成功标准不是“技术能跑”，而是“前台质量接近可继续投入”。

至少要接近以下判断：
- 前台观感不过分出戏
- 声音自然度达到可继续优化的水平
- 回复速度与稳定性不拖后腿
- 整体前台表现接近抖音高分直播间的可对标区间
- 能形成清楚的 MVP 路线、扩容判断与成本判断

## 7. 当前阶段失败判定
以下情况可直接判为当前阶段失败或不值得继续投入：
- 像低配念稿机
- 声音机械
- 回复明显拖沓
- 系统不稳定
- 观感强烈出戏
- 只能“技术跑通”，但前台质量不足以继续投入

## 8. 当前阶段输出物
当前阶段应沉淀的输出物包括：
- 可执行的前台 90 分验收表
- 阿里前台路线总表
- 最小 MVP 技术链路
- 多房间扩容判断表
- “1 台阿里云服务器带 5 个直播间”可行性判断
- 3 档成本模型
- 一版明确结论与下一步建议
