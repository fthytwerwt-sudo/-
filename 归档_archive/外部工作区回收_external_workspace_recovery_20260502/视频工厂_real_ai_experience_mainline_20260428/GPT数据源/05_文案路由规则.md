# 文案路由规则

## 1. 文件定位
本文件只负责 4 件事：
- 判断这轮文案属于哪一类
- 定义当前 `内容表达文案` 的默认交付包
- 定义 block / segment 承载分配
- 定义 Docker / 工作台 / 录制减负任务的默认修正动作

## 2. 文案一级分类
当前先分 4 类：
1. `内容表达文案`
2. `分发包装文案`
3. `关系建立文案`
4. `变现承接文案`

当前默认主流程只优先服务：
- `内容表达文案`

## 3. 当前默认交付包
只要这轮输出的是 `内容表达文案`，默认必须一起交付：
1. 完整脚本
2. `block -> 承载分配`
3. `4 段核心录制素材`
4. 录制动作说明
5. 真实 AI 使用经验 / 工作提效过程 / 真实踩坑的证据说明
6. 结果差证明点
7. `PPT / 总结卡文案`
8. 可选 `Prompt 引用尾卡` 或经验承接尾卡
9. 录制验收清单
10. 如果内容自然适合复用，再补 `三层 prompt 包 / 工作包骨架`

核心项缺任一项，只能算部分交付；但 `三层 prompt 包 / 工作包骨架` 是可选沉淀项，不再是每条内容默认必须交付的产品单元。

## 3A. 默认文案生产路由

以后《视频工厂》AI 知识类视频默认按四方分工推进：

1. `Perplexity（外部参考检索 / 研究工具）` 输出 `reference pack（参考包）` 和 `raw feeling draft（原感初稿）`。
   - 只提供外部参考、表达方向、案例、梗和原感。
   - 不负责最终定稿，不能直接进入执行。
2. 用户录制真实素材。
   - 包括录屏、网页操作、结果对比、过程证据、关键截图。
   - 素材决定最终稿能写到多具体。
3. `Codex（执行代理）` 做素材技术检查与细节证据报告。
   - 目标不是写最终文案，而是把素材细节交给 `ChatGPT（最终落稿与复审入口）`。
   - 必须给出能写稿的细节，不能只报“素材存在 / 技术通过”。
4. `ChatGPT（最终落稿与复审入口）` 基于参考包、用户素材和 `Codex（执行代理）` 细节报告写最终落稿。
   - 默认最终稿出口是 `ChatGPT（最终落稿与复审入口）`。
   - 必须保证文案贴真实素材，不能写素材里没有的页面、按钮、动作和结果。
5. `Codex（执行代理）` 按最终稿执行视频、字幕、声音、卡片和审片包。
   - 执行完成后回到 `ChatGPT（最终落稿与复审入口）` 复审。

## 3B. `Codex（执行代理）` 素材细节汇报标准

`Codex（执行代理）` 检查素材时，必须按“给 `ChatGPT（最终落稿与复审入口）` 写最终稿”的标准汇报。

至少包含：

1. 素材清单
   - 文件路径、文件名、时长、分辨率、是否能打开、是否有音频。
   - 是否有明显卡顿、黑屏、模糊、遮挡。
2. 时间码细节
   - 每个关键动作从几秒开始，到几秒结束。
   - 每个画面段落对应什么内容。
   - 哪些时间段可做反面展示，哪些时间段可做正面展示，哪些时间段适合配口播解释。
3. 画面可见信息
   - 页面名称、按钮名称、输入框名称、标签名称。
   - 弹窗、提示、报错。
   - 生成结果中的可读文字。
   - 页面变化前后差异。
4. 操作步骤细节
   - 用户进入哪个网站、工具或页面。
   - 点击了哪个入口。
   - 输入了什么内容。
   - 哪个按钮触发了生成。
   - 生成后出现了什么结果。
   - 如果有 `API（应用程序接口）`、后台或控制台类页面，必须写清入口位置和动作。
   - 颗粒度示例：进入阿里百炼大模型服务平台，找到 `API Key（接口密钥）` 入口，点击新建 `Key（密钥）`，复制生成的 `Key（密钥）` 到项目配置里。
   - 上面只是颗粒度示例，不能编造素材里没有的操作。
5. 证据判断
   - 这段素材能证明什么。
   - 不能证明什么。
   - 哪些画面足够支撑文案。
   - 哪些文案点素材没有支撑，必须删掉或改写。
   - 哪些地方需要补录。
6. 给 `ChatGPT（最终落稿与复审入口）` 的写稿建议
   - 哪些细节应该写进口播。
   - 哪些细节只适合字幕或卡片。
   - 哪些地方适合微反转。
   - 哪些地方适合轻吐槽。
   - 哪些地方不适合写得太满。

## 3C. 真实经验主线文案路由

当前内容表达文案默认先判断是否属于以下三类之一：

| 路由类型 | 核心问题 | 默认承载 | 工作包关系 |
| --- | --- | --- | --- |
| `经验记录型` | 用户这次真实用了 AI 做什么、哪里卡住、后来怎么判断 | 真实过程、页面细节、判断变化、可复盘动作 | 可沉淀为方法或经验卡，不强行变工作包 |
| `提效实录型` | 原来慢在哪里，AI 介入后快在哪里，前后差值是否可见 | 实际录屏、操作步骤、结果差、节省的返工 | 只有稳定可复用时才沉淀工作包 |
| `真实踩坑型` | 一开始为什么翻车，错误结果如何露馅，后来怎么修正 | 失败画面、轻吐槽、修正动作、结果变化 | 可以只沉淀避坑经验，不必导向完整工作包 |

写稿默认顺序：
1. 先找真实素材里已经发生的事。
2. 再判断它更像经验记录、提效实录还是真实踩坑。
3. 再安排 `API 生成真人`、用户录屏、PPT / 总结卡与尾卡。
4. 最后判断是否自然需要 `Prompt 引用尾卡`、工作包或服务承接。

硬规则：
- 视频不再只是工作包的证明壳。
- 视频首先是“真实经验记录 + 提效证明”。
- 工作包是当内容自然适合沉淀时的承接方式。
- `Prompt 引用尾卡` 可以保留，但不必每条都强行导向完整工作包。
- 结尾动作可以是保存经验、看工作包、继续下一条实录、进入咨询 / 服务，或其他与真实经验自然匹配的低压动作，不限于领取 prompt 包。

## 4. 三层 prompt 包 / 可选工作包沉淀

`三层 prompt 包` 仍然可以保留，但它的角色从“每条内容默认必须交付”降级为“可选沉淀单元”。

适合沉淀的条件：
- 这条真实经验已经形成稳定可复用方法。
- 有清楚前后差值或结果差证明点。
- 用户录制素材能证明这套方法真的被用过。
- 观众拿到后可以复用，而不是只看一个观点。

### 第 1 层：主 `prompt`
- 先把一句糊话压成更像可交付物的初稿

### 第 2 层：修结果 `prompt`
- 把“看起来对但不能交”的版本继续补成更稳的版本

### 第 3 层：展示 / 尾卡 `prompt`
- 服务总结卡与 `Prompt 引用尾卡`

## 5. `4 段核心录制素材`
当前录制减负默认只要求 4 段：
1. 真实问题 / 原始输入 / 原始工作状态
2. AI 使用过程或主 `prompt` 执行过程
3. 试错、修正、补强或判断变化
4. 前后差值与总结收束

## 6. 当前更适合的结构建议
- 默认主结构：`问题 -> 动作 -> 结果`
- 当前更适合的开头：
  - `紧迫任务现场型`
  - `错误示范型`
- 当前更适合的中段强证据：
  - `用户真实操作录屏`
- 当前更适合的结尾收束：
  - `少量 PPT / 总结卡`
  - 总结之后允许补一张 `Prompt 引用尾卡`

## 7. block 与 segment 默认承载

| block | 默认承载 |
| --- | --- |
| 开头进入 | `API 生成真人` |
| 问题点破 | `API 生成真人` 或 `API 生成真人 + 用户录制素材` |
| 过程演示 | `用户录制素材` |
| 关键判断 | `API 生成真人` |
| 结构整理 | `少量 PPT` |
| 结尾收束 | `API 生成真人` 或 `少量 PPT` |
| `Prompt 引用尾卡` | `少量 PPT / 尾卡面板` |

当前明确禁止：
- 不允许先定人物次数，再硬塞文案
- 不允许让 `API 生成真人` 去替代中段本该由录屏承担的主体证据
- 不允许让 `少量 PPT` 去替代中段主体推进
- 不允许让 `Prompt 引用尾卡` 去承担主叙事

## 8. 当前交付闸门
一条 `内容表达文案` 在进入执行前，至少要同时回答：
1. block 是否已经继续拆成 `segment`
2. 中段是否真的由 `用户录制素材` 承担主体推进
3. 这条内容属于 `经验记录型 / 提效实录型 / 真实踩坑型` 哪一种，真实素材证据是什么
4. `4 段核心录制素材` 是否已经写清
5. `Prompt 引用尾卡` 是否只承担可选沉淀、经验承接或低压动作出口
6. 是否没有把 `云端剪辑` 写成已跑通事实
7. 工作包是否真的自然需要；如果没有自然工作包，是否仍能作为真实经验视频成立

## 9. Docker / 工作台默认修正动作
若任务命中：
- Docker
- 工作台
- compose
- 录制减负
- 工作包生成
- prompt 生成
- 结果差对比

默认先检查：
- `compose.yaml`
- `docker/*`
- `scripts/*`
- `工作台模板/*`
- volume / mount / 路径漂移

默认动作：
1. 先确认当前工作台目标是不是中间层 scaffold
2. 先做最小可逆修正
3. 先修路径、挂载、volume 命名、模板缺口、脚本参数问题
4. 再做 `docker compose config` 或最小脚本生成验证

必须明确：
- 默认先尝试最小修正，不先把问题丢给用户
- 但不能把“定义了默认修正动作”偷写成“链路已稳定跑通”

## 10. 一句话规则
**当前文案路由不是“先写一版再找东西塞进去”，也不是每条都强行收成工作包，而是先判断它属于 `经验记录型 / 提效实录型 / 真实踩坑型` 哪一类，再拆 block，再拆 `segment`；默认核心交付是 `真实 AI 使用经验 + 工作提效实录 + 用户录制素材证据 + 结果差`，`三层 prompt 包 / Prompt 引用尾卡` 只在内容自然适合沉淀时启用，并在命中 Docker / 工作台任务时先做最小可逆修正。**
