# 文案路由规则

## 1. 文件定位

本文件保存稳定的文案路由和承载分配规则。

它不保存当前 round、当前样片状态或当前发布状态。

## 2. 文案类型

默认先分四类：

1. `内容表达文案`
2. `分发包装文案`
3. `关系建立文案`
4. `变现承接文案`

若用户没有特别说明，优先按 `内容表达文案` 判断。

## 3. 内容表达文案默认交付结构

内容表达文案至少要能拆出：

1. 完整脚本。
2. block 到 segment 的承载分配。
3. 录制素材清单。
4. prompt 包或工作包结构。
5. 结果差证明点。
6. 总结卡或收束文案。
7. 回审清单。

## 4. 承载分配原则

| 内容段 | 默认承载 |
| --- | --- |
| 开头进入 | `API 生成真人` 或清楚的判断段 |
| 问题点破 | 判断句、对比句、短证据 |
| 过程演示 | `用户录制素材` |
| 结构整理 | 少量 PPT / 卡片 |
| 结尾收束 | 总结句、少量 PPT、必要时人像段 |
| 工作包引用 | 尾卡或补充说明 |

## 5. 禁止路由

- 不允许让 PPT 替代主体证据。
- 不允许让人像段替代真实过程证据。
- 不允许先定视觉次数，再硬塞文案。
- 不允许把尾卡做成第二个主结尾。
- 不允许把云端剪辑方向写成已经稳定执行的事实。

## 6. Docker / 工作台类任务

若任务命中 Docker、工作台、compose、录制减负、工作包生成、prompt 生成或结果差对比：

1. 先确认是不是中间层 scaffold 或规则问题。
2. 再检查 GitHub 当前文件、脚本和日志。
3. 只做最小可逆修正。
4. 不把“有默认修正路径”写成“链路已稳定跑通”。

## 7. 一句话规则

文案路由先拆任务、证据和承载，再决定脚本和视觉；动态执行状态去 GitHub 读取。
