# GPT Project 协作包总述

## 1. 文件定位

`GPT 数据源/` 是 GPT Project 协作规则包。

`已确认` 本目录是 GPT Project 静态协作包唯一修改和上传目录，真实本地路径固定为：

`/Users/fan/Documents/视频工厂/GPT 数据源/`

以后不再创建桌面上传包，不再让用户从桌面、临时目录、PR worktree 或多个导出包里判断哪个是最新版。

它是给 ChatGPT 进入这个 Project 后使用的配合说明书，不是当前项目事实库，也不是执行状态快照。

它负责说明：
- ChatGPT 在 Project 里如何接手
- 如何使用账号层长期规则
- 如何区分协作规则和项目事实
- 每轮如何去 GitHub 读取当前事实
- 静态资料与 GitHub 冲突时如何裁决
- 如何与 Codex 分工
- 如何避免把旧静态资料当成当前事实

它不负责保存：
- 当前 round
- 当前 `latest_review_pack`
- 当前 `send_ready`
- 当前 `content_validation`
- 当前 `full_content_validation`
- 当前声音试配状态
- 当前执行产物状态

## 3. 三方分工

### GPT Project 资料源

负责协作规则、读取顺序、冲突裁决和表达边界。

唯一修改和上传目录：

`/Users/fan/Documents/视频工厂/GPT 数据源/`

无空格目录 `GPT数据源/` 是 GitHub 项目执行包 / 动态项目文件层，不作为 GPT Project 静态包默认上传来源。

### GitHub

负责保存项目事实、执行状态、日志、审片入口、声音试配证据和当前可读文件。

### Codex

负责维护 GitHub 文件、执行日志、最新状态、分支同步和证据口径。

### ChatGPT

先按本包协作规则接手，再去 GitHub 读取当前事实，最后基于当前事实和用户目标协作判断。

## 4. 冲突裁决

若 `GPT 数据源/` 静态资料与 GitHub 当前文件冲突：

1. 以 GitHub 当前文件为准。
2. 标记冲突来源。
3. 不得用本包旧内容补写当前事实。
4. 无法读取 GitHub 时，只能保守接手并标记 `待验证`。

## 6. 当前稳定内容定位

`已确认` 当前项目中心价值已从 `场景化专业输出工作包` 调整为：

`真实 AI 使用经验 + 工作提效实录`

- 用户录制素材在新口径下更核心。
- 元素娃娃 / 主持娃娃 / 骚萌卡不替代真实证据。
