# 当前事实读取规则

## 1. 文件定位

本文件不保存当前事实本体。

它只说明：每次需要当前事实时，从哪里读、如何裁决冲突、读不到时怎么办。

## 2. 默认事实源

每次需要当前事实时，优先读 GitHub 当前文件。

默认主读取分支：
- `codex/user-readable-map`

默认读取顺序：

1. `AGENTS.md`
   - 仓库顶层入口规则。
2. `codex_log/latest.md`
   - 最新执行摘要。
3. `codex_log/current_publish_target.md`
   - 当前复审 / 发布目标。
4. `codex_log/current_publish_target_light_evidence.md`
   - 当前复审轻证据。
5. `dist/latest_review_pack/summary.json`
   - 最新审片包状态总结。
6. `dist/latest_review_pack/review_manifest.md`
   - 最新审片入口。
7. 如涉及 10 份执行包，再读 GitHub 当前 `GPT数据源/`。
8. 如涉及声音、样片、具体执行或历史判断，再读对应 `codex_log/YYYYMMDD_*.md`。

## 3. 需要从 GitHub 读取的动态事实

以下内容不得由 `GPT 数据源/` 静态资料保存或推断：

- 当前 round。
- 当前 `latest_review_pack`。
- 当前 `content_validation`。
- 当前 `full_content_validation`。
- 当前 `send_ready`。
- 当前声音 trial 状态。
- 当前 TTS / voice cloning 状态。
- 当前审片入口。
- 当前执行产物状态。
- 当前 remaining blockers。

## 4. 冲突裁决

- GitHub 当前文件 > `GPT 数据源/` 静态资料。
- `GPT 数据源/` 只给协作方式，不给当前事实。
- 若 `GPT 数据源/` 与 GitHub 说法冲突，以 GitHub 为准。
- 若无法读取 GitHub，必须保守接手，不得用旧静态资料补当前事实。

## 5. 禁止误写

- 不得把历史样片写成当前最新样片。
- 不得把技术验证写成内容验证。
- 不得把用户未确认写成可发送。
- 不得把声音试配写成声音通过。
- 不得把目标态写成当前事实。
- 不得把本包旧内容写成 GitHub 已同步事实。

## 6. 读不到 GitHub 时怎么办

如果无法读取 GitHub：

1. 明确标记 `待验证`。
2. 说明缺少哪个路径或分支。
3. 不补写具体当前状态。
4. 不用静态资料里的历史描述替代当前事实。
5. 要求先恢复 GitHub 读取或让 Codex 同步当前事实文件。

## 7. 一句话规则

当前事实永远从 GitHub 当前文件读取；`GPT 数据源/` 只提供协作规则。
