## 1. 一句话结论

`已确认` 真实素材流程可以写成：

> 我给豆包一句简单需求 → 豆包给出 Trae 搭建方案和可复制 prompt → 我把 prompt 放进 Trae SOLO → Trae 自动 plan 并生成项目骨架 → 后续再接 API / 云端剪辑 / Codex 检查。

`边界` 这只能证明“从需求到可执行项目骨架的流程证据链存在”，不能证明豆包已经把工程跑通、Trae 代码已经运行成功、阿里云剪辑正式稳定、或 Codex 已证明内容过线。

## 6. 给 ChatGPT 的最终文案输入

新文案必须遵守这条真实流程：

1. `需求入口`：用户先给豆包一句简单需求：`我想用 trae 做一个短视频自动流`。
2. `豆包输出`：豆包先给出 Trae 短视频自动流方案，包含从 0 基础轻量版到无人值守全自动化版；核心链路是选题策划、脚本生成、分镜制作、视频生成 / 剪辑、配音字幕、封面标题、自动发布。
3. `豆包 prompt`：用户进一步要求 Vlog 自动流 prompt；豆包输出 `Trae Vlog 自动流 核心搭建 Prompt`，标题明确写着可直接复制粘贴到 Trae SOLO，一键生成完整架构 + 可运行脚本。
4. `Trae 执行`：Trae SOLO 画面确认 prompt 已进入 `SOLO Coder`，随后自动 plan，生成 `11 待办`，并创建 `vlog_automation_workflow` 项目骨架。
5. `API`：只能写成后续接工具能力 / API 能力的方向，不得写成已全部接通。
6. `阿里云剪辑`：只能写成用户选择的云端总装方向或后续验证方向，不得写成正式稳定链路。
7. `Codex`：Codex 负责检查执行结果、路径、文件、命令、Git 状态，并回报边界；不得写成 Codex 证明内容过线。
8. `即梦对比`：可以写成“单点素材生成 vs 可复用流程”的差异；不要贬低成绝对不可用，也不要说本链路已经全自动稳定。
