# 内容回审与循环治理规则

## 1. 文件定位

本文件用于写死当前项目在内容回审循环里的收口动作，尤其是“展示路由要不要切”与“过线后必须交付可见样片”这两条。

## 2. 回审循环先判什么

当前回审循环默认先判：

1. 当前问题更像内容不过线，还是展示路由选错
2. 当前主路由策略是否成立
3. block 路由表是否真的按职责分工

每一轮回审记录里至少要写：

- 本条视频主路由策略
- 本条 block 路由表
- 为什么这样路由
- 当前是否存在展示路由错误
- 下一轮是否继续同一路由，还是切路由

## 3. 回审循环的默认出口

当前内容回审循环的默认出口不是：

- 只在仓库里写 `quality_passed`
- 只在聊天里说“已经 90 分了”
- 只说“本地有成片”

当前默认出口必须同时满足：

1. 质量结论已明确
2. 展示路由结论已明确
3. 可见样片已明确交付给用户

## 4. 路由错误时的默认循环动作

若当前回审判断是“展示路由错误”，下一轮默认动作不是大改全部，而是：

1. 只保留当前最高优先级路由改点
2. 先决定继续同一路由，还是切路由
3. 切路由时优先按正式 fallback 规则处理

默认动作写死为：

- `human` 失败：
  - 优先切到 `hybrid`
  - 或直接切到 `ppt`
- `ppt` 失败：
  - 优先升级到 `hybrid`
  - 必要时升级到 `human_primary`
- `hybrid` 失败：
  - 优先压回单一路由

## 5. 过线后的强制交付动作

当本轮回审达到以下任一状态时：

- 项目内部 90 分水位通过
- `quality_passed`
- 通过可发布测试线

必须立刻补齐样片交付动作。

样片交付动作至少包括：

1. 给出本轮样片路径
2. 指明用户当前应优先看哪个样片文件
3. 若还需辅助判断，补充关键回审帧或回审目录

## 6. `local_only` 样片的处理

若样片属于 `.gitignore` / `local_only`：

- 必须明确说明 GitHub 上看不到
- 必须明确说明本地已生成
- 必须明确给出本地路径
- 必须明确说明该本地样片是否足以完成当前验收

不得写成：

- “GitHub 上看不到是正常的”

而不补本地交付方式。

## 7. 进入试发后的循环约束

进入试发后，展示路由可以作为正式变量继续验证。

但循环治理必须写死：

- 同一轮不允许同时改平台包装、内容本体和展示形式
- 如果本轮主要怀疑是路由错，就只先切路由
- 如果本轮主要怀疑是内容不过线，就先稳住路由，只修内容

## 8. 禁止的收口写法

明确禁止以下写法：

- “质量已过线”但不给样片路径
- “已经 90 分了”但用户看不到任何样片
- “本地有成片”但不告诉用户在哪
- “样片已生成”但不说明用户该看哪一个文件
- “看着不对”却不判断到底是内容问题还是路由问题

## 9. 当前默认样片交付物

当前项目默认样片交付物优先写死为：

1. `dist/formal_api_demo/final.mp4`
2. 如需辅助验收，再补：
   - `dist/formal_api_demo/review_frames/`

## 10. 当前一句话规则

内容回审循环必须先区分“内容不过线”还是“路由选错”；一旦判定通过 90 分 / `quality_passed` / 可发布测试线，就必须额外把可见样片交付给用户，没有样片路径的“过线”不算完整收口。
