# 录制素材与 Prompt 交付规则

## 1. 文件定位
本文件只负责 4 件事：
- 以后 GPT 给文案时，必须附带哪些录制交付
- 录制素材清单怎么写
- “录制动作说明”和“豆包输入 prompt”如何区分
- 录完后用户应回传哪些内容给 GPT 复审

它不负责：
- 单条样片的最终镜头表
- 发布文案全文
- 当前待发对象切换
- 代码实现

## 2. 以后 GPT 给文案时，必须附带哪些录制交付
只要当前输出的是 `内容表达文案`，默认必须同时交付：
1. 完整脚本
2. `block -> 承载分配`
3. 录制素材清单（默认先收成 `4 段核心录制素材`）
4. 录制动作说明
5. 三层 `prompt` 包
6. 结果差证明点
7. `PPT / 总结卡文案`
8. `Prompt 引用尾卡`
9. 录制验收清单

少其中任一项，都只能算部分交付。

## 3. 录制素材清单怎么写
录制素材清单默认至少写清：
- 要录哪一段
- 录什么页面或工具
- 这一段要证明什么
- 需要出现什么输入
- 需要出现什么结果
- 这一段录完后怎样判断“可用”

建议格式如下：
- 素材名
- 页面 / 工具
- 目标证明点
- 必出镜输入
- 必出镜结果
- 录制完成判定

## 4. “录制动作说明”和“豆包输入 prompt”如何区分
### 4.1 录制动作说明
录制动作说明写的是：
- 人要怎么操作
- 先开哪个页面
- 先点什么
- 先贴哪段内容
- 哪一步要停留
- 哪个结果一定要录到

它本质上是：
- 给拍摄与录屏动作看的执行单

### 4.2 三层 `prompt` 包
三层 `prompt` 包写的是：
- 第 1 层主 `prompt`：先把一句糊话压成更像可交付物的初稿
- 第 2 层修结果 `prompt`：把“看起来对但不能交”的版本补成更稳版本
- 第 3 层展示与尾卡 `prompt`：把判断点、总结卡与 `Prompt 引用尾卡` 压成适合视频展示的文本

它本质上是：
- 给工具看的输入内容

必须明确：
- 录制动作说明不等于三层 `prompt` 包
- 三层 `prompt` 包也不能替代录制动作说明
- `Prompt 引用尾卡` 不是再讲一遍理论，而是把产品单元引用出来

## 5. 录完后用户应回传哪些内容给 GPT 复审
录完后，用户默认应回传：
1. 原始录屏或已裁好的关键片段
2. 实际贴入豆包的 `prompt`
3. 普通输入 vs 工作包输入的结果截图或文本
4. 需要保留的结果差证明点
5. 录制中遇到的问题或偏差
6. 当前认为可用 / 不可用的素材判断
7. 录制验收清单勾选结果

若这些信息缺失较多，GPT 只能做部分复审，不能冒充完整复审。

## 6. 一句话规则
**以后文案默认不是只交一版脚本，而是默认连“4 段核心录制素材 + 录制动作单 + 三层 prompt 包 + 结果差证明点 + Prompt 引用尾卡 + 验收清单”一起交；其中录制动作说明写的是人怎么录，三层 `prompt` 包写的是工具吃什么输入。**
