# 阶段与验收闸门规则

## 1. 文件定位

本文件用于写死当前项目在阶段推进与验收收尾时，哪些条件算过线，哪些条件还不能收口。

## 2. 当前阶段定位

当前阶段必须固定理解为：

- 已完成技术阶段
- 当前主阶段是内容阶段，正往试发阶段过渡
- 当前允许真人固定背景，但不要求所有内容默认走真人
- 当前优先做“路由正确”，不是“真人优先主义”
- 展示路由规则属于内容阶段模板稳定工具，不是增长、商业化或平台分发策略

## 3. 当前验收闸门

当前项目至少区分以下 4 类闸门：

- 技术链闸门
- 内容与展示路由闸门
- 质量闸门
- 样片交付闸门

## 4. 技术链闸门

技术链闸门通过，表示：

- TTS / 图片 / 视频 / assembly 等链路已跑通
- 但这不等于内容质量已过线
- 也不等于用户已经收到可见样片

## 5. 内容与展示路由闸门

内容与展示路由闸门通过，至少表示：

- 场景已经判清
- 主价值已经判清
- 观众最需要先接住什么已经判清
- 路由单位已经压到 block，而不是整条视频固定模板
- 当前载体选择没有违背 block 职责

必须同时明确：

- 不允许为了上真人而破坏结构成立
- 不允许把 15 秒样片写成长期统一时长
- 不允许把混合承载写成固定顺序
- 若母版仍在内容阶段，应优先保证内容过线，不允许为了换载体而把内容打散

## 6. 质量闸门

质量闸门通过，至少指以下任一状态成立：

- 项目内部“90 分水位”通过
- `quality_passed`
- 通过“可发布测试线”

但必须明确：

- 质量闸门通过，不自动等于验收完成
- 若还没有把可见样片交付给用户，当前收尾仍不完整

## 7. 样片交付闸门

当质量闸门通过后，必须立刻进入样片交付闸门。

样片交付闸门通过的最低要求是：

1. 已明确给出本轮样片路径
2. 已明确用户当前应优先看哪个样片文件
3. 若样片属于 `local_only`，已明确说明：
   - 不会上传到 GitHub
   - 但本地已生成
   - 当前本地路径是什么
4. 已说明该样片是否足以完成当前验收

未满足以上 4 条时，不得把收尾写成：

- “已完成验收”
- “已正式过线”
- “用户已可验收”

## 8. 进入试发后的变量控制

进入试发后，展示路由本身可以成为变量。

但每一轮必须固定：

- 不允许同一轮同时又改平台包装、又改内容本体、又改展示形式
- 若本轮怀疑是路由错误，优先只切路由
- 若本轮怀疑是内容不过线，优先先修内容本体

## 9. 当前默认样片交付物

当前默认样片交付物优先级写死为：

1. `dist/formal_api_demo/final.mp4`
2. 若需要辅助验收，再补：
   - `dist/formal_api_demo/review_frames/`

当前不要把样片交付物写得过散。

## 10. 当前收尾最小回报字段

若本轮达到 90 分 / `quality_passed` / 可发布测试线，收尾必须额外回报：

1. 本轮样片路径
2. 是否属于 `local_only`
3. 用户当前应优先看哪一个样片文件
4. 若 GitHub 无法直接看，必须给本地可见路径

## 11. 当前一句话规则

当前阶段必须先过“内容与展示路由闸门”，再谈质量闸门；达到 90 分 / `quality_passed` / 可发布测试线，只表示质量闸门通过，只有把可见样片明确交付给用户后，当前验收收尾才完整。
