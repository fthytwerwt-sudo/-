# 质量基线与“抖音 90 分标准”规则

## 1. 文件定位

这份文件用于统一当前项目的质量判断口径。

它主要解决以下四件事：

- demo 的身份到底是什么
- 当前质量基线该怎么定
- “抖音 90 分标准”到底是什么意思
- 当前第一优先增强路线为什么先是火山引擎 TTS

这份文件不是平台官方规则汇编，也不是平台发布 API 方案说明。它只负责统一本项目内部的质量判断口径。

## 2. 当前 demo 的真实身份

当前 demo 的身份必须明确写死：

- demo 只证明链路跑通
- demo 是技术闭环底座 / 运行锚点
- demo 是最小闭环验证件
- demo 不具备质量参考价值
- 不能再继续把 demo 当成质量样片

这意味着：

- demo 可以证明当前本地视频链路成立
- demo 不能证明当前质量已经过线
- demo 不能再被用来定义质量下限

## 3. “抖音 90 分标准”的真实定义

“抖音 90 分标准”在本项目中的正式定义如下：

- 它是项目内部质量简称
- 它不是平台官方评分体系
- 它指“接近抖音知识类 / AI 类 / 无人出镜短视频可推荐门槛”的质量水位
- 它在现实中更接近创作者行业经验口径，而不是平台数字打分

以后 GPT 侧文件里如果再出现这个说法，必须显式写成“项目内部质量简称”，不得写成平台官方标准或官方打分规则。

## 4. 当前 90 分标准的判断框架

### 4A. 当前 AI 知识类内容的过线底线

当前 AI 知识类内容是否过线，不只看有没有成片，还要看：

- 看完后用户是否有收获
- 有没有一个判断
- 有没有一个动作
- 有没有一个可执行步骤
- 有没有一个自检标准
- 有没有证据支撑

明确不过线的情况包括：

- 只说问题
- 只给观点
- 只有共鸣、没有帮助
- 没有动作
- 没有证据
- 看完后仍不知道下一步做什么

必须同时明确：

- generation / assembly 成功，不等于内容已过线
- 技术链路可行，只能说明“能产出”
- 不能说明“这条内容已经值得进样片”或“已经达到内容质量线”

### 必须过线项

- 配音不再明显像系统播报
- 字幕与配音基本同步
- 开头 3 秒有效
- 内容不像说明书
- 画面不只是静态轮播
- 前后变化能被看懂
- 结尾有落点

### 一票否决项

- 明显机械配音
- 字幕和配音明显不同步
- 全程像 PPT 轮播 / demo 演示
- 开头 3 秒没有钩子
- 内容高度模板化、像批量生产
- 画面明显低质或 AI 瑕疵
- 看完整条仍不知道表达什么

### 加分项

- 配音有轻微情绪起伏
- 有真实案例或数据支撑
- 信息卡 / 字体 / 层次更接近原生短视频
- 节奏不是匀速播报
- 看起来像真人真的会发

## 5. 当前第一优先增强路线

当前第一优先增强路线已经确认：

- 当前第一优先是现成火山引擎 TTS API
- 当前先解决机械配音这一一票否决项
- 当前“走火山引擎一套”仅表示供应商统一优先
- 本轮执行上只先接 TTS，不默认同时扩 ASR、图生视频、平台 API 等

同时必须明确：

- 当前第一优先不是平台发布 API
- 当前第一优先不是自动化运营
- 当前第一优先不是图生视频
- 当前第一优先不是数字人

当前路线的核心不是“把火山能力全接一遍”，而是“先用最直接的能力把最致命的 demo 感打掉”。

## 5A. 当前纯 PPT 主线的 assembly 口径

当前纯 PPT / 信息卡主线的 assembly 口径已升级为：

- OSS 已具备
- 阿里云剪已具备
- `北京区 OSS + 云剪工程 = 唯一 assembly 主路径`
- `local assembly` 已退出 pure PPT / 信息卡主线，不再作为 fallback / 兜底 / 应急主交付
- 若缺密钥、缺云端参数或缺 provider implementation，必须如实标记为 `待注入` / `待验证`

必须同时明确：

- 这次升级只适用于纯 PPT / 信息卡主线
- 当前暂不考虑动态 PPT
- 当前不扩成复杂 motion design / 高成本视觉特效路线

## 6. 90 分 / `quality_passed` 后的样片交付硬规则

当内容达到以下任一条件时，必须向用户交付“可见样片”：

- 项目内部“90 分水位”判定通过
- `quality_passed`
- 通过“可发布测试线”

这里的“可见样片”不是只在仓库里写一句“已过线”，而是必须让用户实际看到至少一种东西：

- 成片本地路径
- 可直接打开的样片文件
- 固定回审帧集合
- 明确的验收样片目录

当前项目默认样片交付物优先写死为：

1. `dist/formal_api_demo/final.mp4`
2. 若需要辅助验收，再补：
   - `dist/formal_api_demo/review_frames/`

若样片或回审帧属于 `.gitignore` / `local_only`，必须同时显式说明：

- 这些文件不会上传到 GitHub
- 但在本地已生成
- 用户当前应看哪一个本地路径
- 它是否足以完成当前验收

明确禁止以下模糊写法：

- “质量已过线”但不给样片路径
- “已经 90 分了”但用户看不到任何样片
- “本地有成片”但不告诉用户在哪
- “GitHub 上看不到是正常的”却不补交付方式

## 7. 当前一句话规则

当前不是拿旧 demo 定质量标准；纯 PPT / 信息卡主线统一走北京区 `OSS + 云剪工程` 组装，本地 assembly 不再作为 fallback；一旦达到 90 分 / `quality_passed` / 可发布测试线，就必须把当前可见样片明确交付给用户，默认先交 `dist/formal_api_demo/final.mp4`。
