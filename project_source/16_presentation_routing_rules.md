# 展示路由规则

## 1. 文件定位

本文件用于把《视频工厂》的展示路由正式写成一套分层判断系统。

它解决的是：

- 一条视频整体该偏什么主承载形式
- 每个段落块该由什么载体承担
- 失败时默认往哪里回退
- 回审时如何判断问题是内容不过线，还是路由选错

它不是：

- 固定三段式模板
- 固定混合顺序说明
- 固定 15 秒脚本模板
- 增长策略、商业化策略或平台分发策略

## 2. 当前阶段边界

当前展示路由规则必须按以下边界理解：

- 当前主阶段是内容阶段，正往试发阶段过渡
- 当前正式默认主线已经改成：
  - 人物判断
  - 用户自己的真实录制素材
  - 少量 PPT / 图片辅助
- 当前阶段允许 pure PPT 次级支路，但它不再是默认主承载
- 当前阶段优先做“路由正确”，不是“真人优先主义”
- 当前阶段若母版仍处于内容阶段，应优先保证内容过线
- 不允许为了上真人而破坏结构成立
- 不允许因为“像真人更省事”就把 AI talking avatar / 数字人口播重新抬成默认主承载
- 展示路由规则属于内容阶段的模板稳定工具，不是增长、商业化或平台分发策略
- 进入试发后，路由本身可以成为变量
- 但不允许同一轮同时又改平台包装、又改内容本体、又改展示形式

## 3. 正式判断顺序

展示路由必须按以下顺序判断，不得跳步：

1. 先判场景
2. 再判这条视频整体主价值
3. 再判观众最需要先接住什么
4. 再判 block 级职责
5. 最后才判每个 block 用什么展示形式

明确禁止：

- 先凭个人偏好选真人或 PPT
- 因为真人“更高级”就默认真人
- 因为 PPT“更稳”就默认 PPT
- 因为当前样片是 15 秒，就把所有结构和时长写死

## 4. 路由单位与输出口径

展示路由的正式执行单位是：

- `block`
- 即段落块 / 镜头块

整条视频只负责给出主策略，不负责把所有 block 压成同一种载体。

正式输出默认至少包含以下结构：

```text
video_routing_plan
- video_scene:
- video_goal:
- primary_value:
- audience_need_first:
- video_route_strategy:
- blocks:
  - block_id:
  - block_goal:
  - block_need_first:
  - block_carrier:
  - asset_requirement:
  - why_this_carrier:
- fallback_rules:
- review_diagnosis:
```

## 5. 四层路由系统

### L1：整条视频层（Video-Level Routing）

L1 负责回答：

- 这条视频整体更像什么内容
- 当前主承载形式应该偏什么

L1 必填字段：

1. `video_scene`
2. `video_goal`
3. `primary_value`
   - `结构`
   - `证据`
   - `判断`
   - `信任`
   - `前后变化`
   - `行动`
4. `audience_need_first`
   - `先看懂`
   - `先相信`
   - `先代入`
   - `两者都要`
5. `video_route_strategy`
   - `ppt_primary`
   - `human_primary`
   - `hybrid`
   - `screen_demo_primary`
   - `case_visual_primary`
6. `route_profile`
   - `human_self_footage_light_ppt`
   - `pure_ppt_secondary`

L1 硬规则：

- L1 只定义整条视频的主策略层
- L1 不等于每个 block 都必须用同一种形式
- L1 决定的是“整体偏哪种承载”，不是“整条视频固定模板”
- 当前项目默认主线的 L1 推荐值是：
  - `video_route_strategy = hybrid`
  - `route_profile = human_self_footage_light_ppt`

### L2：段落块层（Block-Level Routing）

L2 负责回答：

- 视频里的每一段，应该由谁承担

L2 必填字段：

1. `block_id`
2. `block_goal`
   - `抓手`
   - `判断`
   - `证据`
   - `推进`
   - `对比`
   - `收束`
   - `最小行动`
3. `block_need_first`
   - `看懂`
   - `相信`
   - `代入`
   - `行动`
4. `block_carrier`
   - `ppt`
   - `human`
   - `human_with_overlay`
   - `screen_demo`
   - `case_visual`
   - `mixed`
5. `asset_requirement`
6. `why_this_carrier`

L2 硬规则：

- 路由判断的真正执行单位是 block，不是整条视频的固定套路
- 一条视频允许多个 block 用不同载体
- 开头、中段、后段、结尾可以各自承担不同职责
- 混合只表示职责分配，不表示固定顺序

### L3：回退层（Fallback Rules）

L3 负责回答：

- 如果当前展示形式不成立，默认往哪里退

L3 默认动作必须写死为：

- `human` 失败时：
  - 优先回退到 `hybrid`
  - 若观众更需要结构而不是先看人，则直接回退到 `ppt`
- `ppt` 失败时：
  - 优先升级到 `hybrid`
  - 若核心缺的是“谁在说、为什么可信”，必要时升级到 `human_primary`
- `hybrid` 失败时：
  - 优先压回单一路由
  - 默认保留最能完成当前 block 职责的那一路
- `screen_demo` 失败时：
  - 若没有真实演示过程，优先回退到 `case_visual`
  - 若连真实证据都没有，回退到 `ppt`
- `case_visual` 失败时：
  - 若无法承担证据或前后差异，优先回退到 `ppt`
  - 若观众必须看见操作过程，改走 `screen_demo`

L3 的决定顺序固定为：

1. 先保 block 职责
2. 再保观众最先需要接住的东西
3. 再看现有资产是否成立
4. 最后才考虑成本与美观

### L4：回审层（Review / Diagnosis）

L4 负责回答：

- 这条内容是不是路由选错了
- 下一轮是继续同一路由，还是切路由

L4 必查字段：

1. 本条视频主路由策略
2. 本条 block 路由表
3. 为什么这样路由
4. 当前是否存在“展示路由错误”
5. 下一轮是否继续同一路由，还是切路由

L4 必问问题：

- 这条本来该 PPT 却用了真人？
- 本来该真人却用了 PPT？
- 本来最适合混合，却被压成单一路线？
- 当前问题到底是内容不过线，还是路由选错？

## 6. 各类载体的正式口径

### A. 优先 `ppt / 信息卡` 的条件

当 block 的核心任务更偏以下内容时，优先 `ppt`：

- SOP
- 拆解
- 结构化证据
- 输入输出关系
- 对比
- 流程
- 表格
- 接手链路
- 归类 / 收束 / 归位

正式一句话：

- 观众最需要先看懂结构

### B. 优先 `human / human_with_overlay` 的条件

当 block 的核心任务更偏以下内容时，优先 `human`：

- 判断
- 立场
- 可信度
- 身份代入
- 掌控感
- 劝服
- 情绪承接
- “这是我在对你说”

正式一句话：

- 观众最需要先接住“谁在说、为什么可信”
- 这里的 `human` 当前默认指：
  - 真人表达
  - 用户自己的真实录制人物素材
  - 不是 AI talking avatar / 数字人口播默认补位

### C. 优先 `screen_demo` 的条件

当 block 的核心任务是让观众直接看懂“怎么做”时，优先 `screen_demo`：

- 实操路径
- 操作过程
- 界面步骤
- 输入到输出的现场演示

正式一句话：

- 观众最需要先看见过程，而不是只听结论

### D. 优先 `case_visual` 的条件

当 block 的核心任务是交付真实证据或前后差异时，优先 `case_visual`：

- 案例截图
- 成果对比
- 前后变化
- 局部证据

正式一句话：

- 观众最需要先看见证据或差异

### E. 优先 `mixed` 的条件

当 block 或整条视频同时需要：

- 人来建立进入感、判断感、信任感
- 证据来完成结构证明、前后变化或 SOP 演示

正式一句话：

- 既要“人”来建立进入感，又要“证据”来完成说服

混合硬规则：

- 混合不等于固定顺序
- 混合不等于硬拼接
- 混合应按 block 职责自然分配

## 7. 文案 / prompt 触发信号

以下信号只用于辅助判断，不得替代场景判断、心理机制判断和 block 职责判断。

### 更偏 `ppt` 的常见触发信号

- 步骤
- SOP
- 表格
- 字段
- 流程
- 对比
- 接手
- 结构化
- 归位
- 输入 / 输出
- 清单
- 拆解

### 更偏 `human` 的常见触发信号

- 为什么很多人
- 真正问题
- 我最想说
- 你最容易卡住
- 我更建议
- 先别急着
- 判断
- 立场
- 掌控感
- 你现在最该做什么

### 更偏 `hybrid` 的常见触发组合

- 判断 + 证据
- 代入 + 结构
- 可信度 + SOP
- 立场 + 前后变化
- 人先接住 + 证据再证明

## 8. 时长与比例规则

展示路由默认按“职责与比例”写，不按固定秒数写。

必须写死：

- 15 秒样片只是当前示例，不是长期统一时长
- 今后允许 15 秒、30 秒、45 秒、60 秒及更长知识类视频
- 每段时长必须由以下因素共同决定：
  - 总时长
  - 当前阶段
  - block 职责
  - 信息密度
  - 展示载体承载能力

可用的默认比例建议只有职责级参考，不得写成硬秒数：

- `抓手 / 进入感`：约 15% - 25%
- `判断 / 证据 / 推进`：约 50% - 70%
- `收束 / 最小行动`：约 10% - 20%

## 9. 资产闸门

### `ppt` 资产闸门

- 必须有清晰前后变化或结构证明
- 单页单主判断
- 不得长期静态轮播
- 不得长期说明书化

### `human` 资产闸门

- 必须有人物承载判断或进入感
- 画面中的人不能只是摆件
- 固定背景不得抢戏
- 若真人承担不了信息，不得强撑

### `screen_demo` 资产闸门

- 必须真的在演示
- 不得只是截图堆叠
- 必须服务于“我看懂了怎么做”

### `case_visual` 资产闸门

- 必须真的承担证据或前后差异
- 不得只是装饰素材

### `hybrid` 资产闸门

- 必须明确哪个 block 由人承担
- 必须明确哪个 block 由证据承担
- 不得重复解释同一层意思
- 不得只切来切去没有职责分工

## 10. 失败回退规则

### `human` 路由失败时

触发条件示例：

- 真人承载不了信息
- 信任感没增强，反而削弱
- 固定背景不够稳
- 观众更需要结构，不是先看人

默认动作：

- 优先回退到 `hybrid`
- 或直接回退到 `ppt`

### `ppt` 路由失败时

触发条件示例：

- 说明书感压不掉
- 进入感太弱
- 信任感不足
- 需要“谁在说”来接住

默认动作：

- 优先升级到 `hybrid`
- 必要时升级到 `human_primary`

### `hybrid` 路由失败时

触发条件示例：

- 像硬拼接
- 节奏断裂
- 成本过高
- 两边都在重复
- 每段职责不清

默认动作：

- 优先回退成单一路由
- 保留当前最能完成职责的那一条

## 11. 当前阶段建议

当前阶段的正式建议写死为：

- 下一条最值得验证的样片，是“展示路由判断样片”
- 优先验证 prompt 是否能先判断：
  - 这条是结构主导
  - 还是信任主导
  - 还是需要混合承载
- 重点不是“再出一条视频”
- 而是验证“路由系统能不能选对”

## 12. 当前一句话规则

展示路由必须先分层、再判断、再输出、再回退、再回审；路由单位是 block，不是整条视频的固定模板，15 秒只是当前示例，混合也不是固定顺序。
