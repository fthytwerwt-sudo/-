# 20｜多智能体路由说明（给 GPT Project 用）

## 1. 文件定位

本文件用于放进 **GPT Project 数据源**，作为一份短路由说明。

它只负责回答 4 个问题：

1. 什么时候应该启用 Codex 多 agent
2. 默认应该采用什么多 agent 结构
3. ChatGPT 在这类任务里该输出什么
4. 真正的执行模板正文应该去哪里读

它不负责：

- 具体多 agent prompt 全文
- Codex 的执行细则全文
- worktree / git / skills 的完整操作说明
- 项目内具体某轮任务的执行单

一句话：

**本文件只负责让 GPT 知道“什么时候该把任务路由成 Codex 多 agent 执行”，不负责替代 Codex 的执行模板库。**

---

## 2. 为什么这份文件应该放进 GPT Project

因为这份内容属于：

- 稳定判断层
- 协作路由层
- 总控层

它的作用不是教 Codex 怎么执行，而是让 ChatGPT 在项目协作时更快判断：

- 这轮是否适合交给 Codex
- 如果交给 Codex，是否值得启用多 agent
- 多 agent 应该采用哪种结构最稳

所以它适合放进 GPT Project 数据源。

---

## 3. 当前项目下，什么时候适合启用 Codex 多 agent

默认只有命中以下任一情况时，才建议启用 Codex 多 agent：

### 3.1 需要先读清现状，再决定怎么改
例如：

- 要先读 AGENTS、latest、codex_source、project_source 才能动手
- 仓库现状复杂，不能直接裸改
- 任务前需要先确认分支、日志、规则、代码和已有产物

---

### 3.2 任务能明显拆成“探索 / 收束 / 落地”
例如：

- 一个 agent 负责只读探索现状
- 一个 agent 负责只读收束最省改法
- 主 agent 负责统一落地

---

### 3.3 读取量大，但真正改动点相对集中
例如：

- 规则文件较多
- 项目文件较多
- 历史日志较多
- 但最终只需要 1 个 agent 真正改少数文件

---

### 3.4 需要提高执行效率，但又不想增加混乱
多 agent 的价值，不在于很多 agent 一起乱改文件，而在于：

- 并行压缩读取时间
- 并行压缩定位时间
- 并行压缩“找规则 / 找冲突 / 找验收”的时间

---

## 4. 当前项目下，什么时候不适合启用 Codex 多 agent

以下情况默认不建议启用：

### 4.1 就改一个很小的文件
例如：

- 一个字段
- 一个短文档
- 一个低风险单点修改

这类任务单 agent 往往更快。

---

### 4.2 当前任务还没收口
如果现在真正缺的是：

- 边界还没定
- 主目标还没定
- 验收还没定
- 该改哪一层还没定

那先不该上多 agent，而应该先让 ChatGPT 收口。

---

### 4.3 多个 agent 会同时碰同一批核心文件
只要多个 agent 会同时改：

- 同一个 project_source 文件
- 同一个 codex_source 文件
- 同一组核心代码文件

默认不适合并行写文件。

---

### 4.4 当前任务真正慢的不是执行，而是判断
如果用户真正缺的是：

- 这轮到底该不该做
- 这轮该先改哪个点
- 这轮属于哪一层问题

那应该先由 ChatGPT 判断，不要用多 agent 假装提速。

---

## 5. 当前默认推荐的多 agent 结构

当前项目默认推荐的结构，不是“多个 agent 一起改”，而是：

## **2 explorers + 1 integrator**

### 5.1 Explorer A
职责：

- 只读
- 读仓库
- 找相关文件
- 找现状
- 找冲突
- 找遗漏

不负责：

- 写文件
- 跑高风险命令
- 最终拍板

---

### 5.2 Explorer B
职责：

- 只读
- 在现状基础上收束“最省、最稳、最少返工”的改法
- 补齐验收标准
- 提出不该动的边界

不负责：

- 写文件
- 越权拍板项目结论

---

### 5.3 Integrator / Executor
职责：

- 汇总两个 explorer 的结果
- 做最终裁决
- 独占写文件权
- 独占跑验证
- 独占更新日志 / 桥接文件
- 输出最终结果

一句话：

**explorer 负责并行探路，integrator 负责单点落地。**

---

## 6. ChatGPT 在这类任务里的默认动作

当这轮命中多 agent 触发条件时，ChatGPT 默认不需要长篇解释原理，而是应直接做下面 3 步：

### 第一步
先判断这轮是不是适合交给 Codex 执行。

### 第二步
再判断这轮是否值得启用多 agent。

### 第三步
若值得启用，则直接给出一份适合 Codex 的多 agent 执行 prompt，而不是只讲原则。

---

## 7. 默认输出口径

以后命中这类任务时，ChatGPT 默认按下面顺序输出：

1. 一句话判断：这轮是否适合多 agent
2. 当前建议采用的结构：
   - 单 agent
   - 2 explorers + 1 integrator
3. 为什么是这个结构
4. 直接给可复制的 Codex prompt

---

## 8. 真正的模板正文放哪里

GPT Project 不存完整执行模板全文。

真正的多 agent 模板正文，应优先放在：

### 项目内版本
- `codex_source/10_codex_multi_agent_prompt_library.md`

### 全局通用版本
- `~/.codex/skills/` 下对应的多 agent skill

因此，GPT Project 里只需要保留本文件这样的短路由说明，不需要把完整模板库也塞进来。

---

## 9. 当前一句话规则

**当任务需要并行压缩“读现状 / 找规则 / 收束改法”的时间时，优先采用“2 explorers + 1 integrator”结构；GPT Project 只负责知道何时启用，而完整模板正文放在 Codex 执行层和全局 skill 中。**
