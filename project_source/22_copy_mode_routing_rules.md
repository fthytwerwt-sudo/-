# 文案路由规则

## 1. 文件定位
本文件不是统一文案模板，而是当前正式资料包的文案路由规则。

它负责：
- 先判断这轮文案属于哪一类
- 再判断这轮内容如何分配给 `API 生成真人`、`用户录制素材`、`少量 PPT`
- 再决定是否能进入样片执行

## 2. 文案一级分类
当前文案先分 4 类：
1. `内容表达文案`
2. `分发包装文案`
3. `关系建立文案`
4. `变现承接文案`

当前默认优先级：
1. `内容表达文案`
2. `分发包装文案`
3. `关系建立文案`
4. `变现承接文案`

## 3. 当前默认主流程只优先服务什么
当前根目录正式主线，默认只优先服务：
- `内容表达文案`

原因很简单：
- 当前要先把主线口径收稳
- 先把默认承载写清
- 不让分发、关系或变现文案反过来抢走主线

## 4. block 承载路由
`内容表达文案` 进入主流程后，必须继续拆 block。

### 常见 block
- 开头进入
- 问题点破
- 过程演示
- 关键判断
- 结构整理
- 结尾收束

### 默认承载映射
| block | 默认承载 |
|---|---|
| 开头进入 | `API 生成真人` |
| 问题点破 | `API 生成真人` 或 `API 生成真人 + 用户录制素材` |
| 过程演示 | `用户录制素材` |
| 关键判断 | `API 生成真人` |
| 结构整理 | `少量 PPT` |
| 结尾收束 | `API 生成真人` 或 `少量 PPT` |

## 5. segment 级文案路由规则
`block` 不是最后一级。

每个 block 在进入执行前，还要继续拆成更小的 `segment`，再按功能分发承载。

### segment 功能 → 承载
- 判断节点 → `API生成真人`
- 证据节点 → `用户录制素材`
- 整理节点 → `少量PPT / 总结卡`
- 转场节点 → 看它更像判断还是整理，不单独乱加人物

### 当前必须明确禁止
- 不允许先定人物次数，再硬塞文案
- 不允许先定结尾一定 3 秒，再倒推总结卡要写什么
- 不允许因为一段字多，就默认给人物硬撑
- 不允许因为没有判断，就给人物强行补出场

### 项目默认经验值
下面这组只属于当前项目默认经验值，不是行业硬标准：

- 人物段时长 ≈ 口播字数 ÷ 4.5–5.5 + `0.2–0.4` 秒缓冲
- 总结卡时长 ≈ 屏幕总字数 ÷ 5.5–6.5 + `0.4–0.8` 秒缓冲
- 单模块最短不要低于 `1.4` 秒
- 人物单段若超过 `22–26` 字，默认先怀疑过长，优先拆成“人物判断 + 录屏 / 卡片”
- 录屏段不按字数，而按“至少看清一个完整动作或明确变化”判断时长

### 当前一句核心判断
**先判这一小段在干什么，再判这段字有多少、观众来不来得及看。功能决定承载，字数决定时长。**

## 6. `API 生成真人段` 的出现次数
当前只能这样写：
- `API 生成真人段 1 次`
- `API 生成真人段 2 次`

当前不能这样写：
- “只写 1 次，却不说明是 API 生成真人段”
- “只写 2 次，却不说明是 API 生成真人段”

规则：
- `API 生成真人段` 出现 1 次还是 2 次，是 block 路由结果
- 不是固定模板
- 也不是默认一定要更多“像人”的镜头

## 7. 一个最小示例
题目：**为什么你明明用了 AI 写汇报，最后还是得自己重写一遍？**

一个 50 秒最小分发示例：
- 开头判断：`API生成真人`
  - 点破“问题不在写，而在没先压清判断口径”
- 中段证据 1：`用户录制素材`
  - 展示目标、边界、验收被整理进执行清单
- 中段证据 2：`用户录制素材`
  - 展示 AI 输出和人工修回之间的差值
- 中后段整理：`少量PPT / 总结卡`
  - 只显影 3 个关键词，不抢主叙事
- 若中后段出现额外判断拐点：
  - 才允许再补 1 段 `API生成真人`

这条示例要说明的不是模板，而是：
- 不是每段都用人物
- 不是每段都用总结卡
- 而是按“判断 / 证据 / 整理”分发

## 8. `真人实拍出镜` 的位置
若某条内容必须使用 `真人实拍出镜`，必须同时满足：
1. 显式写出 `真人实拍出镜`
2. 显式标注为非默认分支
3. 不得把它回写成当前唯一正式主线

## 9. 进入样片执行前的 7 条检查
1. 这轮文案是哪一类？
2. 当前是否仍属于 `内容表达文案` 主流程？
3. block 是否已经继续拆成 `segment`？
4. 开头是否明确交给 `API 生成真人`？
5. 中段是否真的有 `用户录制素材` 承担主体推进？
6. `少量 PPT` 是否只承担结构整理与收束？
7. 是否没有把 `云端剪辑` 写成“已跑通事实”？

## 10. 默认改稿顺序
当前默认按下面顺序改：
1. 先改开头判断句
2. 再改中段证据层
3. 再改 block 承载分配
4. 再把 block 继续拆成 `segment`
5. 再反推人物、录屏、总结卡时长
6. 最后才改语气和节奏

## 11. 一句话规则
**当前文案路由不是“先写一版再找东西塞进去”，而是先判断文案类型，再拆 block，再拆 `segment`；判断节点给 `API 生成真人`，证据节点给 `用户录制素材`，整理节点给 `少量 PPT / 总结卡`，转场节点看它更像判断还是整理，不先定人物次数再硬塞文案。**
