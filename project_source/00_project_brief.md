# 视频工厂：AI 垂类场景化视频内核

## 1. 项目定义

本项目当前正式口径是：

一个服务于用户本人个人内部使用的、Prompt 驱动、可由 Codex 重复执行的 AI 视频内核项目。

它当前不是一个面向外部客户的大而全产品，也不是一个“只差平台 API 就能成立”的发布系统。它的核心目标是把 AI 相关内容压成：

- 内容边界清楚
- 结构选择稳定
- 质量判断统一
- 工作流可复用
- 协作链路可由 ChatGPT、Perplexity、Codex 反复执行

的“场景化视频内核”。

## 2. 当前项目状态

当前阶段已经不是“技术可不可行”的阶段，而是“内容质量能不能过线、结构能不能稳定、流程能不能复用”的阶段。

当前成立的正式主线是：

- 本地视频内核主线成立
- 当前正式默认主线已切到：
  - 人物
  - 用户自己的真实录制素材
  - 少量 PPT / 图片辅助
- 正式 assembly 继续固定为“北京区 OSS + 云剪唯一主路径”
- `local assembly` 不再保留为正式默认主线的 fallback / 兜底路径
- AI talking avatar / 数字人口播由于口型、性别、动作与文案匹配风险，不再作为默认主承载
- 当前不是平台 API 前置依赖
- 当前允许云端组装后人工上传；若缺密钥、缺云端参数或缺 provider implementation，必须诚实标记待注入 / 待验证
- 当前重点是内容质量、结构稳定、可复用、可由 Codex 重复执行
- 当前不是“只差字节 / 抖音 API 就能成立”

换句话说，项目当前的关键问题已经不是“能不能生成一条视频”，而是“能不能持续生成足够像真人会发的 AI 知识类短视频”。

## 3. 固定边界

### 内容边界

- 只处理 AI 相关视频
- 当前高频核心场景：
  1. AI 项目讲解
  2. AI 方法分享
  3. AI 学习实操
  4. AI 案例拆解

### 表现边界

- 第一阶段默认主线改为“人物判断 + 自录素材证据 + 少量 PPT / 图片辅助”
- 当前允许真人口播、自录镜头、真实录屏、案例画面、少量辅助信息卡
- 当前正式默认主线统一走北京区 OSS + 云剪工程组装
- 当前不再把本地 assembly / 本地 mp4 写成正式默认主线的 fallback、兜底或应急正常交付
- 允许云端导出后人工上传；若真实云端导出尚未验证，只能写待注入 / 待验证，不能再用本地补位冒充完成
- 当前不把平台发布 API 作为前置依赖

### 当前明确不做什么

当前阶段不主动展开以下方向：

- 平台发布 API 先行
- 自动化运营
- 图生视频优先
- 数字人优先
- AI talking avatar / 数字人口播重新承担默认主承载
- 直播、售卖、获客、商业包装、账号增长
- 前端工作台优先建设
- 大而全产品化扩展

这不是说这些方向永远不做，而是当前不允许它们抢走“视频质量过线”这条主线。

## 5. 当前 demo 的真实身份

当前 demo 的定位必须单独写清楚：

- demo 只证明链路能跑通
- demo 是最小闭环验证件 / 运行锚点
- demo 是技术闭环底座，不是质量参考件
- demo 不具备质量参考价值
- demo 不能作为当前质量基线样片

以后项目脑里凡提到 demo，都必须先把它理解为“运行锚点”，而不是“质量样片”。

## 6. 当前质量问题的真实判断

当前真正的问题不是：

- 能不能生成视频
- 能不能本地跑通脚本、字幕、配音、合成

当前真正的问题是：

- 旧 demo 质量太差，不能继续充当质量参考件
- 当前 demo 的 demo 感、系统播报感、说明书感太重
- 因此不能再拿旧 demo 去反推质量下限

这意味着当前推进顺序必须调整为：

- 先承认旧 demo 不具备质量参考价值
- 再优先打掉最致命的一票否决项
- 再形成新的质量参考样片 1.0

## 7. 当前第一优先质量增强路线

当前第一优先路线已经明确：

- 当前第一优先是用户现成可用的火山引擎 TTS API
- 当前目标是先解决机械配音 / 系统播报感
- 当前“走火山引擎一套”只代表供应商统一优先
- 执行上本轮只先接 TTS

必须同时明确：

- 当前第一优先不是平台发布 API
- 当前第一优先不是自动化运营
- 当前第一优先不是图生视频
- 当前第一优先不是数字人
- 本轮不默认同时扩到 ASR、图生视频、平台 API 等其他能力

## 8. “抖音 90 分标准”的事实表述

“抖音 90 分标准”在本项目中的正式定义必须写清楚：

- 它是本项目内部质量简称，不是平台官方规则
- 它指向“接近抖音知识类 / AI 类 / 无人出镜短视频可推荐门槛”的质量水位
- 它更接近创作者 / 操盘手的行业经验口径，不是平台数字评分体系

以后在 GPT 侧文件里，凡出现“抖音 90 分标准”，都必须明确标注为“项目内部简称”，不得写成平台官方标准。

## 9. 当前 90 分标准的判断口径

当前项目内部对“90 分标准”的最低判断口径，可先压成三层：

### 定义层

- “抖音 90 分标准”是项目内部质量简称，不是平台官方规则

### 必须过线项

- 配音不再明显像系统播报
- 字幕与配音基本同步
- 开头 3 秒有效
- 内容不像说明书
- 画面不只是静态轮播
- 前后变化能被看懂
- 结尾有落点

### 一票否决项

- 明显机械配音
- 字幕和配音明显不同步
- 全程像 PPT 轮播 / demo 演示
- 开头 3 秒没有钩子
- 内容高度模板化、像批量生产
- 画面明显低质或 AI 瑕疵
- 看完整条仍不知道表达什么

完整规则以后统一以 `project_source/08_quality_baseline_and_90_score_rules.md` 为准。

## 10. 当前项目推进顺序

当前推进顺序已经明确，不再按“旧 demo 就是标准样片”的思路走。

当前正确顺序是：

1. 先承认旧 demo 不具备质量参考价值
2. 先用现成火山引擎 TTS 打掉机械配音这一条一票否决项
3. 再围绕字幕同步、开头有效、画面变化等项继续补线
4. 再拿新的样片作为质量参考件 1.0
5. 再继续定后续增强路线

这条顺序的核心不是“先堆更多能力”，而是“先把最致命的 demo 感打掉”。
