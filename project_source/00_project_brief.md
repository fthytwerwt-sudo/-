# 视频工厂：AI 垂类场景化视频内核

## 1. 项目定义

本项目当前正式口径是：

一个服务于用户本人个人内部使用的、Prompt 驱动、Codex 可执行的 AI 自媒体内部视频项目。

它当前不是一个面向外部客户的大而全平台，也不是一个只会吐固定模板的单一生成器，更不是一个以前端页面、工作台或产品化界面为当前优先级的项目。它的目标是逐步沉淀成一套：

- 固定 AI 垂类内容边界
- 支持多种视频场景模式
- 支持不同脚本结构
- 但执行骨架稳定

的“场景化视频内核”。

项目的最终价值，不在于证明技术能生成视频，也不在于先搭一个前端产品壳，而在于把 AI 相关内容压成用户本人会发、能发、值得发的短视频工作流，并把这套流程压成可被 Prompt 稳定驱动、可由 Codex 反复执行的内部视频内核。

## 2. 当前阶段

当前阶段已经从“技术可不可行”进入“内容能不能成立、质量能不能过线、流程能不能复用”的阶段。

更具体地说：

- 15 秒视频只是最小闭环验证件，不是项目终点
- 当前重点不是再证明能出片，而是把现有能力压到内容可发布
- 当前优先做：
  - 内容成立
  - 质量优化
  - 可复用工作流
  - Codex 可重复执行的视频产出链路
  - Prompt 驱动的协作闭环稳定
- 第一阶段默认优先无人出镜
- 当前最重要的不是营销包装，而是把视频项目本身压实
- 当前判断标准不是“先有页面”，而是“给定 prompt 后，能否稳定得到想要的内容与产物结果”

## 3. 固定边界

### 内容边界

- 只处理 AI 相关视频
- 当前高频核心场景：
  1. AI 项目讲解
  2. AI 方法分享
  3. AI 学习实操
  4. AI 案例拆解

### 可扩展但非第一阶段核心

- AI 工具实测
- AI 问题回答
- AI 观点拆解
- AI 日常工作流分享
- 生活片段辅助的 AI 表达型内容

### 表现边界

- 第一阶段优先建设无人出镜视频内核
- 可允许少量生活片段作为辅助素材
- 但辅助素材不得抢占主题，不得偏离 AI 主线

## 4. 当前明确不做什么

当前阶段不主动展开以下方向：

- 直播
- API 中控
- 获客
- 售卖
- 商业包装
- 账号增长
- 前端工作台优先建设
- 页面型产品化
- 大而全产品化扩展

这不是说这些方向永远不做，而是当前项目脑必须保持聚焦，不让视频内核建设被外围目标稀释。

## 5. 当前最小技术闭环事实

当前最小 demo 已经跑通，且本地真实产物存在。

已确认成功生成：

- `dist/demo/script.txt`
- `dist/demo/captions.srt`
- `dist/demo/voice.mp3`
- `dist/demo/final.mp4`

已确认的技术路线：

1. Python 主脚本解析 `cases/demo.md`
2. 生成脚本与字幕
3. 使用 macOS `say` + `afconvert` 生成配音中间音频
4. 使用项目内置 `ffmpeg-static` 输出 `voice.mp3`
5. 使用 Swift + AVFoundation 生成 3 页 9:16 静态 PPT 风格视频并合成 `final.mp4`

## 6. 环境审计结论

本地环境已知事实：

- system `ffmpeg` 不可用
- Node `v25.6.1` 可用
- Python `3.9.6` 可用
- `say` 可用
- `afconvert` 可用
- Swift `6.2.3` 可用

当前环境结论：

- 这套 demo 运行依赖更偏 macOS 原生能力
- 音频链路依赖系统 TTS 和音频转换
- `ffmpeg` 需要依赖项目内置 `ffmpeg-static`
- 当前路线适合继续做 demo 到可复用骨架的过渡，不适合现在就扩成跨平台复杂架构

## 8. 当前项目真实状态

项目当前的真实状态是：

- 已完成可行性验证
- 已具备最小出片能力
- 正在进入质量优化与工作流沉淀阶段
- 当前正式理解应是“个人内部使用的 Prompt 驱动型视频工厂 / 视频内核”
- 当前重点是 prompt、内容、执行链路与回审闭环，不是前端页面

换句话说，项目已经不是“能不能跑出一条 demo”的问题，而是：

- 这条内容是不是像短视频
- 音画是否过线
- 结构是否可复用
- 场景是否可扩展
- 能否形成给 ChatGPT、Perplexity、Codex 各自稳定接手的项目脑和执行链路

## 9. 第一阶段优先级

第一阶段的优先级如下：

1. 把 AI 视频项目本身压实
2. 把 Prompt 驱动的内容生成与 Codex 执行链路压稳
3. 让内容更像真实短视频内容，而不是项目说明书
4. 把最小 demo 提升到“可发布测试”水位
5. 建立可复用工作流和项目脑文件

## 10. 目前最值得持续优化的主线

当前最值得持续优化的主线，不是新加更多功能，而是先把“Prompt 驱动 → ChatGPT 收束 → Codex 执行 → 出片 → 回审”这条内部闭环压稳，再沿下面这条内容主线持续推进：

场景成立
→ 结构匹配
→ 心理机制接住人
→ 表现层不过度 demo 感
→ 输出稳定可复用

这意味着项目后续优化应优先围绕：

- 内容结构质量
- 开头抓力
- 中段口语感
- 结果页说服力
- 配音与字幕节奏
- PPT 信息卡表现力
- 场景模式与脚本结构的分层抽象

## 11. 当前 demo 事实锚点

当前 demo 的稳定事实锚点如下：

- 当前最关键输入文件：`cases/demo.md`
- 当前 demo 固定约束：
  - 中文
  - 15 秒
  - PPT 演示风格
  - 9:16 竖屏
  - 3 页以内
  - AI 配音
  - 简体中文字幕
- 当前默认最合理视频结构：
  - 问题 → 动作 → 结果

需要特别说明的是：

- 这组锚点只代表当前最小闭环
- 它们不应被误解为整个项目未来都只能固定成三段式生成器
- 项目正式方向仍然是“多场景模式 + 多结构选择 + 稳定执行骨架”

## 12. 当前一句话判断

当前仓库不是“一个前端产品工作台”，而是“一个已跑通的 AI 视频最小闭环 demo，正准备被压成个人内部使用、Prompt 驱动、Codex 可执行的场景化视频内核基座”。
