# 执行规则用户说明

## 1. 这份文件解决什么问题

`codex_source/01_execution_rules.md` 是原始硬规则，主要写给 Codex 看。

这份 `.user_guide` 不是去改原规则，而是把它翻成人话，帮用户理解：

- 原规则主要在管什么
- 你什么时候需要关心这份规则
- 哪些地方只是 Codex 必须遵守的硬动作
- 哪些地方会直接影响你对当前仓库推进方式的理解
- 如果你觉得“流程不对”“push 节奏不对”“不确定该不该走 PR”，应该先看哪里

如果只记一句话：

原规则不是在讨论内容好不好，而是在规定 Codex 该怎么接手、怎么推进、什么时候才能算说话有证据。

---

## 2. 原执行规则文件主要在管什么

原文件主要在管 6 件事：

### 2.1 先读什么，再动手

它规定了接到任务后不能直接凭聊天记忆开做，而要先读：

- `AGENTS.md`
- 本地 / 全局 skill
- `project_source/06_project_index.md`
- `codex_source/00_codex_readme.md`
- `codex_source/01_execution_rules.md`
- 以及和本轮任务直接相关的文件

这解决的是：

- 为什么新会话不能假装自己已经知道仓库背景
- 为什么有些任务会先读文件，再开始改

### 2.2 先判断这是哪一层的问题

原规则要求先分清：

- 项目脑层
- 执行层
- 代码层

它不允许把“结构问题”直接当成“执行问题”，也不允许把“执行流程问题”误写成“内容方向问题”。

### 2.3 什么时候可以直接做，什么时候要先停下来审计

如果任务边界清楚、文件明确、风险低，可以直接做。

但如果任务会影响：

- 项目边界
- 默认工作流
- 运行链路
- 真实产物判断

就不能硬冲，要先审计。

### 2.4 本轮能改什么，不能顺手改什么

原规则明确不允许：

- 没授权就跨层扩改
- 顺手把 demo 升级成复杂平台
- 把 `project_source/`、`codex_source/`、代码层混成一锅

### 2.5 仓库型任务怎么走 GitHub / PR 线路

只要命中仓库改动，就默认进入仓库型任务线路：

先看现状 → 开分支改 → 提 PR → 跑 checks → AI 复审 → 用户拍板

这也是为什么：

- 基线建立后不能默认直接 push `main`
- 本轮形成小闭环后，通常要 commit 并 push 当前分支

### 2.6 什么情况下不能把话说满

原规则明确禁止：

- 把未确认内容写成已确认
- 没验证就说“已完成”“已成功”“已通过”
- 没核实 remote / 认证 / 权限就假装 GitHub 流程已经走通

所以有时你会感觉它“很保守”，本质上是在防止假完成。

## 3. 用户什么时候需要看这份文件

不是每次都要看。

你通常在下面这些时刻才需要看：

- 你觉得这轮流程怪，不像你预期的推进方式
- 你不理解为什么这轮先读文档，不先改文件
- 你不理解为什么明明是补文档，还要写 `codex_log/latest.md`
- 你不理解为什么这轮不能直接 push `main`
- 你不确定这轮到底该不该走 PR
- 你怀疑这轮明明没验证够，却被说成“已经完成”

如果你只是想判断：

- 这条内容结构对不对
- 文案像不像人话
- 节奏顺不顺

那优先看的通常不是这份，而是 `project_source/` 下的内容文件。

## 4. 哪些部分主要只是给 Codex 的硬规则

这些部分你可以知道，但不需要逐条背。

| 原规则位置 | 人话解释 | 为什么主要是给 Codex 的硬规则 |
| --- | --- | --- |
| `EXEC-001` 默认读取顺序 | 进仓后先读什么，不能裸奔开做 | 这是 Codex 的接手动作要求，不是用户要记的操作手册 |
| `EXEC-002` 层级判断 | 先分清是项目脑、执行层还是代码层 | 这是 Codex 的判断纪律，防止乱改层 |
| `EXEC-003` / `EXEC-004` | 哪些任务能直接做，哪些必须先审计 | 这是 Codex 的风险控制规则 |
| `EXEC-005` 默认允许修改什么 | 本轮没授权的层，默认不顺手改 | 这是 Codex 的边界线 |
| `EXEC-006` skill 检查 | 本地 / 全局 skill 怎么查，查不到要如实说 | 这是 Codex 的执行前动作要求 |
| `EXEC-007` 缺文件 / 缺 skill / 缺 AGENTS 的处理 | 找不到关键东西时不能装作没事 | 这是 Codex 的停止线 |
| `EXEC-008` 最小必要背景包 | 进入执行动作前必须补齐哪些项目背景 | 这是 Codex 的最低理解要求 |
| `EXEC-011` 禁止假装成功 | 没验证就不能说完成 | 这是 Codex 的真实性规则 |

你不用把这些编号全背下来，但可以知道：

Codex 在这个仓库里不是“看心情推进”，而是被这些硬规则约束住的。

## 5. 哪些部分会直接影响用户理解当前仓库怎么推进

这些地方虽然写给 Codex，但会直接影响你看到的协作体验。

| 原规则位置 | 它会直接影响你看到什么 | 你最容易感受到的现象 |
| --- | --- | --- |
| `EXEC-001` 默认读取顺序 | 为什么新会话会先读一组固定入口文件 | 你会看到它先读 `AGENTS.md`、`project_source/06_project_index.md`、`codex_source/00_codex_readme.md`，再结合 `codex_log/latest.md` 接手最近状态 |
| `EXEC-006A` GitHub / PR 线路 | 为什么仓库型任务默认不直接改 `main` | 你会看到它先查分支、remote、再在功能分支推进 |
| `EXEC-006B` 执行日志规则 | 为什么明明只是补文档，也可能要刷新 `latest.md` 和补完整日志 | 你会看到仓库型任务结束前要落日志，而不是只在聊天里汇报 |
| `EXEC-006C` 小闭环后 commit / push | 为什么这轮一旦形成可复审版本，就要 commit 并 push 当前分支 | 你会看到它把结果同步到 GitHub，方便 ChatGPT 直接复审 |
| `EXEC-010` 最终汇报栏目 | 为什么最后汇报总带读取、skill、验证、未做项这些栏目 | 你会看到答复更像“真实执行汇报”，不是随手总结 |
| `EXEC-011` 验证口径 | 为什么它有时宁愿说“未验证”也不愿把话说满 | 你会看到它对“已完成”这类说法很克制 |

## 6. 如果你觉得“流程不对 / push 节奏不对 / 规则不顺 / 不知道该不该走 PR”，应该看哪几段

| 你的感受 | 先看原规则哪里 | 这几段主要回答什么 |
| --- | --- | --- |
| “为什么这轮一上来先读文件，不先改？” | `EXEC-001` 到 `EXEC-004` | 读取顺序、层级判断、能不能直接做 |
| “为什么这轮不直接 push main？” | `EXEC-006A` | 仓库型任务默认要走分支 / PR 线路 |
| “为什么只是补文档，也要写日志？” | `EXEC-006B` | 只要命中仓库型任务且有真实执行结果，就要落日志 |
| “为什么这轮已经有结果了，还要 push 当前分支？” | `EXEC-006C` | 小闭环后默认同步当前分支，方便 ChatGPT 复审 |
| “为什么你不肯直接说已完成？” | `EXEC-011` | 没验证就不能把话说满 |
| “为什么最后汇报要按固定栏目写？” | `EXEC-010` | 最终汇报必须把读取、验证、未做项讲清楚 |
| “为什么这轮你说不能继续做，要先停下汇报？” | `EXEC-007`、`EXEC-011` | 缺关键文件、缺验证、缺权限时必须停下 |

## 7. 用户如何用人话告诉 ChatGPT“我觉得执行层哪里不对”

下面这些说法，通常比“你流程怪怪的”更有用。

### 7.1 你怀疑是读取顺序不对

```text
这轮我觉得你跳过了必要的接手文件。
请先按仓库默认读取顺序自检，再继续推进。
```

### 7.2 你怀疑是分支 / PR 判断不对

```text
这轮先别讨论内容本身，我想先确认执行层。
请先判断这次是不是仓库型任务，是否应该走功能分支和 PR 线路。
```

### 7.3 你怀疑是日志节奏不对

```text
我觉得这轮已经有真实执行结果了。
请先判断是否命中了 `codex_log/` 的写日志条件，而不是只停在聊天汇报。
```

### 7.4 你怀疑是 push 节奏不对

```text
我觉得这轮还没形成可判断的小闭环，先别急着 push。
请先对照小闭环规则，判断现在适不适合同步当前分支。
```

### 7.5 你怀疑是“话说得太满”

```text
我觉得这轮验证还不够，请不要把未确认内容写成已确认。
请按“已验证 / 未验证 / 未确认”重新表述一次结果。
```

### 7.6 你只想先定位执行层问题，不想先扩讨论

```text
这轮先别扩到内容优化，我现在想讨论的是执行层。
请先告诉我问题更像：读取顺序、分支流程、日志规则、push 节奏，还是验证口径。
```

---

## 8. 最后一条提醒

这份用户说明最核心的价值，不是让你变成规则专家，而是帮你区分两件事：

- 你现在不满意的是内容本身
- 还是仓库推进方式本身

只有先把这两件事分开，ChatGPT 才更容易给出对路的下一步，而不是一边改内容、一边又误改流程。
