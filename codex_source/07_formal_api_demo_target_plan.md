# 正式版 API 视频 Demo 目标态执行计划

## 1. 文件定位

本文件是《视频工厂》正式版 API 视频 demo 的目标态执行计划。

它负责回答以下问题：

- 正式版目标质量标准先怎么锁死
- 为达到该标准，需要怎样的输入、生成、组装、复审与修正循环
- 当前仓库事实与目标态计划应如何拆开写
- 哪些前提补齐后，才允许声称正式版云端链路真实跑通
- 新会话如何接手这条正式版目标态主线

它不是：

- 当前仓库已跑通事实说明
- 现有本地 demo 的运行文档
- 代码已实现能力清单

当前仓库的已确认运行事实仍以 `codex_source/05_runtime_and_artifact_rules.md` 为准。
本文件只定义正式版目标态，不得写成“当前仓库已经成立”的事实文档。

## 2. 当前状态 vs 目标状态

### A. 当前已确认状态（仓库事实）

- 当前仓库已确认跑通的 legacy 链路仍是本地 demo：
  `cases/demo.md` → `generate_demo.py` → `say / afconvert / ffmpeg-static` → `video_builder.swift` → `dist/demo/` 四件套产物
- 旧 demo 只保留运行锚点价值，不再作为质量样片或质量基线
- “抖音 90 分标准”是项目内部质量简称，不是平台官方评分体系
- 当前正式分支的正式主线代码已改成：
  - API 生成真人
  - 用户本地录制素材
  - 少量 PPT / 图片
  - 北京区 `OSS + 云剪 cloud-only`
- 但是否能写成“已真实跑通”，仍取决于：
  - API 真人实际调用验证
  - 用户本地素材注入验证
  - 云端导出验证

### B. 本计划要推进的目标状态

- 正式版方向锁定为：
  文本需求 → API 生成素材 / 配音 → 云端组装 / 拼接 → 按质量要求验收
- 默认思路不是“先做一个低配验证版”，而是“先锁死目标标准，再把未达标项按 bug / 缺口推进修正”
- 首条样片不是终点，而是进入修正循环的入口

### C. 拆写底线

- 当前仓库事实只写“已确认存在 / 已确认跑通 / 已确认产物”
- 目标态计划只写“要达到什么标准 / 需要什么层 / 缺口如何修”
- 不得把“路径明确”写成“已跑通”
- 不得把“未达标”直接写成“路线不成立”

## 3. 执行前提 / Gate

以下前提补齐到最小执行集合前，不允许把正式版云端链路写成“已真实跑通”：

- 火山正式凭证可用
- 空间名 / 资源存储配置可用
- 关键接口可用性已补齐到最小执行集合
- 组装导出所需配置可用
- 必需接口不存在权限或白名单阻塞

在上述前提仍缺失时，本仓库只允许推进：

- 目标态计划文件
- 文件骨架规划
- 接口占位设计
- 校验框架
- 日志与交接机制

在上述前提仍缺失时，本仓库不得假装：

- 正式版 API 配音已跑通
- 正式版云端组装已跑通
- 正式版最终成片已可稳定导出

## 4. 标准先锁死

### 4.1 当前正式版目标质量标准

当前正式版目标不是“先做出一条视频”，而是“做出达到正式复审入口水位的视频样片”。

至少要达到：

- 15 秒中文知识 / 案例向视频，主题明确
- 开头 3 秒有效，有 hook
- 中段信息推进成立，不像说明书
- 结尾有落点，前后变化看得懂
- 配音不明显机械，不像系统播报
- 字幕与配音基本同步，切分利于理解
- 画面不只是 still 轮播，视觉变化服务理解
- 机器硬校验与人工复审都能给出明确结论

### 4.2 一票否决项

只要命中以下任一项，本轮样片直接判定为“未达正式复审水位”，进入修正循环：

- 明显机械配音
- 字幕与配音明显不同步
- 全程 still 太重，整体像 PPT / demo 轮播
- 开头 3 秒没有 hook
- 看完整条仍不知道表达什么
- 结果变化讲不清楚
- AI 瑕疵明显到出戏
- 发生本地正式 fallback
- 缺失关键事实锚点，导致问题不可复盘

### 4.3 可继续复审水位

“可继续复审”不等于已经完全达标，而是：

- 机器硬校验全部通过
- 没有一票否决项
- 主要问题已收束为可修正 bug / 缺口
- 当前样片足以进入人工复审，不需要再回到“先验证能力存在”

## 5. 达标链路总览

### 5.1 输入层

输入层存在的原因：

- 锁定目标
- 固定分段
- 固定每段配音 / 字幕 / 视觉意图
- 减少生成层自由发挥导致的质量漂移

它服务的质量目标：

- 主题不跑偏
- 开头、中段、结尾目标清楚
- 每段都能追溯“为什么这样生成”

### 5.2 生成层

生成层存在的原因：

- 把正式需求压成可用素材集
- 先解决配音与素材可用性
- 给后续组装提供可编排资产

它服务的质量目标：

- 配音先过线
- 视觉素材围绕段落目标生成
- 素材时长、风格、内容有约束

### 5.3 组装层

组装层存在的原因：

- 把素材变成正式样片
- 解决音画节奏、时间线、字幕压制与导出

它服务的质量目标：

- 音画同步
- 节奏成立
- 输出不是素材堆砌，而是正式成片

### 5.4 复审层

复审层存在的原因：

- 判断当前样片是否达到正式复审入口
- 形成统一的机器项与人工项双闸门

它服务的质量目标：

- 先机器拦截，再人工判断
- 未达标项能被结构化记录，而不是停留在感受层

### 5.5 修正循环层

修正循环层存在的原因：

- 把首轮不完美当成正常状态
- 但不允许因此主动降标准

它服务的质量目标：

- 所有未达标项默认归入 bug / 缺口
- 每轮只抓最高优先级 1 到 2 项修
- 持续逼近目标标准

- 轮询状态
- 下载并整理素材

### 6.3 组装脚本

- `scripts/assemble_formal_api_demo.py`

- 读取 manifest
- 调用云端组装 / 拼接能力

- `config/formal_api_demo.local.toml`

- 承载凭证
- 承载模型名、时长、比例、输出参数
- 承载存储与组装配置
- 承载轮询与校验参数

它直接服务“可复现、可调参、可追责”。

## 7. 输入输出契约

### 7.1 输入层

输入层吃：

- 自然语言正式需求
- 分段结构
- 每段配音 / 字幕 / 视觉要求
- 当前质量要求

输入层吐：

- 结构化 `video_spec`
- 每段目标
- 每段素材需求
- 全局质量约束

### 7.2 生成层

生成层吃：

- `video_spec`
- 配置文件
- 远端配置

生成层吐：

- 语音素材
- 图片 / 视频素材
- 任务状态与任务 ID
- 结构化 `manifest`

### 7.3 组装层

组装层吃：

- `manifest`
- 配置文件
- 云端组装参数

组装层吐：

- 最终成片
- `result_summary`
- 最终导出状态
- 当前样片的机器可追溯事实

### 7.4 复审层

复审层吃：

- `manifest`
- `result_summary`
- 最终成片
- 当前目标标准

复审层吐：

- 机器硬校验结果
- 人工复审结论
- 未达标项清单
- 下一轮修正优先级

### 7.5 manifest 的事实锚点定位

manifest 不是普通中间文件，而是后续修正与复审的核心事实锚点之一。

至少要承载：

- 原始输入摘要
- 每段目标
- 每段配音与字幕文本
- 每段素材来源与类型
- 每段任务 ID / 资源 ID / 状态
- 每段时长与时间线
- 当前已知问题标记
- 当前样片是否通过机器硬校验

后续修正不能只看最终视频感觉，必须能回到 manifest 判断问题出在输入、生成、组装还是复审口径。

这些都必须如实保留，但不等于路线否定。

## 9. bug / 缺口分类表

### 9.1 稳定性类

典型问题：

- 同一需求多次输出波动大
- 风格、镜头、人物不一致
- 时长不稳定

通常怎么修：

- 收紧输入结构
- 固定关键参数与段落边界
- 给每段增加正向风格锚点与负向约束
- 在 manifest 中记录每轮生成参数

### 9.2 输出污染类

典型问题：

- API 吐出不想要的元素
- 画面内容跑偏
- 多余信息、错误字幕、错误视觉进入成片

通常怎么修：

- 增加禁止元素约束
- 把每段只允许表达什么写得更死
- 在生成后增加素材筛除规则
- 在组装前做文本与视觉二次过滤

### 9.3 编排链路类

典型问题：

- 顺序不顺
- 拼接不顺
- 字幕时间轴不准
- 配音与画面不协同

通常怎么修：

- 把每段目标、时长、字幕、画面时序写进 manifest
- 组装脚本严格按时间线编排，不临时猜
- 字幕从附带文本提升为正式时间线资产
- 优先修音画不同步与节奏错位

### 9.4 质量门槛类

典型问题：

- 开头不抓人
- still 太重
- demo 感重
- 结果不清楚
- 结尾没落点
- AI 瑕疵明显

通常怎么修：

- 开头问题优先回到输入层改 hook
- still 太重优先回到生成层与组装层补动态变化
- demo 感重优先回到视觉风格与字幕编排
- 结果不清楚优先回到段落目标与结尾表达
- AI 瑕疵优先做素材替换、重生或屏蔽

## 10. 修正循环

正式流程不是“成功 / 失败”二元判断，而是固定进入修正循环：

1. 每轮样片产出后，先生成未达标项清单
2. 每个未达标项必须归类到 bug / 缺口类别
3. 每轮只抓当前最优先的 1 到 2 项问题
4. 为每个优先问题指定下一轮修正动作
5. 修正后重新生成样片并重新复审
6. 直到没有一票否决项，且样片达到可继续复审水位

首条样片的意义是“进入修正循环”，不是“宣告完成”。

## 11. 最小回归样本集

### 11.1 AI 项目讲解

用途：

- 覆盖“问题 → 动作 → 结果”主结构
- 检查 hook、结果表达、案例说服力

回归要求：

- 每次修完开头、结构、结尾类问题都必须回归这一条

### 11.2 AI 方法分享

用途：

- 覆盖“方法拆解 + 节奏推进”
- 检查字幕切分、知识表达密度、说明书感是否过重

回归要求：

- 每次修完字幕、配音、编排类问题都必须回归这一条

### 11.3 AI 学习实操

用途：

- 覆盖“步骤展示 + 画面辅助理解”
- 检查视觉素材是否真正帮助理解，而不是只做装饰

回归要求：

- 每次修完 still、素材跑偏、视觉污染类问题都必须回归这一条

为什么需要最小回归样本集：

- 当前目标不是做出一条偶然可看的片，而是形成可复用正式链路
- 没有回归样本，就会出现“这条修好了，别的场景坏掉”的假进展

## 12. 机器硬校验 vs 人工复审

### 12.1 机器硬校验

只放程序能够稳定判断的项：

- 输入文件是否完整
- 配置文件是否完整
- task_id / resource_id / output_id 是否齐全
- 素材是否齐全
- manifest / result_summary 是否完整
- 时间线是否连续
- 总时长是否落在容差内
- 是否发生本地 fallback
- 最终成片是否真实导出
- 成片、字幕、配音、manifest 是否可读取

### 12.2 人工复审

只放不能假装自动判断的项：

- 开头有没有 hook
- 画面 still 重不重
- demo 感强不强
- AI 瑕疵重不重
- 结果是否清楚
- 结尾有没有落点
- 整体是否更接近正式质量要求
- 是否值得继续复审

正式闸门必须是双闸门：

- 机器不过线，直接拦截
- 机器过线后，再进入人工复审
- 人工不过线，不降标准，进入修正循环

## 13. 验收节奏 / 验收时机 / 阶段完成标志

这份目标态计划里，“当前阶段验收”和“正式成片质量验收”不是一回事。

- 当前阶段验收，回答的是：这一阶段的骨架、Gate、接入准备、修正动作有没有做到可继续往下走。
- 正式成片质量验收，回答的是：这条样片有没有达到正式复审入口水位，是否还有一票否决项。

当前项目不是把所有验收都压到最终成片那一刻，而是按阶段做闸门式验收。前一阶段没过，下一阶段就不能假装继续。

### 13.1 骨架验收

什么时候验：

- `formal_api_demo` 输入骨架、config example、generate / assemble 骨架、manifest / result_summary schema、Gate 框架都落库后
- 且至少能完成一次不调用真实远端的 dry-run

验什么：

- 输入能解析
- dry-run 能跑
- 结构化产物能生成
- 无凭证时不会假装 success
- 没有本地 fallback 被当成正式主链

做到哪算过：

- 可以稳定产出结构化骨架结果
- blocked / missing 前提会被明确写出
- 不会把“还没接真实 API”伪装成“已经成功出片”
- 通过后，才允许进入“真实 API 接入前验收”

### 13.2 接 API 前验收

什么时候验：

- 准备把 generate / assemble 骨架接入火山真实 API 之前

验什么：

- 凭证位
- space / 存储配置位
- provider 配置位
- 组装参数位
- Gate 拦截逻辑

做到哪算过：

- 缺前提时会明确给出 blocked，而不是继续硬跑
- 前提补齐后，系统才允许进入真实 API 调用

必须明确：

- 这一步没过，禁止声称“正式链路开始跑通”

### 13.3 首轮样片验收

什么时候验：

- 第一次真实远端生成 + 云端组装完成后

验什么：

- 先跑机器硬校验
- 再做人工复审

做到哪算过：

- 没有一票否决项
- 至少达到“可继续复审水位”
- 当前问题已收束为可修正缺口，而不是路线级错误

必须明确：

- 首条样片的意义是进入修正循环，不是宣告完成

### 13.4 修正循环验收

什么时候验：

- 每轮只修 1 到 2 个最高优先级问题之后

验什么：

- 当前目标问题是否真的改善
- 是否引入新问题
- 最小回归样本集是否被打坏

做到哪算过：

- 当前轮改动值成立
- 没有把别的高频场景修坏
- 最小回归样本集通过本轮应回归的检查

必须明确：

- 不通过回归样本集，不能算这一轮修正完成

### 13.5 阶段完成标志

#### 1. 骨架阶段完成

做到哪算完：

- 输入骨架、配置骨架、generate / assemble 骨架、manifest / result_summary schema、Gate 框架都已落位
- dry-run 可跑
- 缺前提时会明确 blocked

#### 2. API 接入准备阶段完成

做到哪算完：

- 凭证位、space、存储配置位、provider 位、组装参数位都已补齐到最小执行集合
- Gate 已能正确拦截未补齐前提
- 现在才允许发起真实 API 调用

#### 3. 首轮样片阶段完成

做到哪算完：

- 已完成第一次真实远端生成与云端组装
- 机器硬校验已过
- 人工复审后没有一票否决项
- 样片已达到可继续复审水位

#### 4. 修正循环阶段单轮完成

做到哪算完：

- 本轮只抓的 1 到 2 个问题已验证改善
- 没有引入新的明显问题
- 最小回归样本集没有被修坏

### 13.6 当前用户何时验收

用户当前最该介入验收的时间点，不是骨架刚落时逐行盯文件，而是以下节点：

- 骨架 dry-run 跑通后，做一次骨架验收
- 真实 API 前提补齐后，做一次接入前验收
- 首轮样片出来后，做一次正式人工复审
- 每轮修正后，做一次改后复审

这样安排的目的，是让用户介入发生在“该拍板的时候”，而不是在骨架尚未成形时被迫替执行层兜底。
