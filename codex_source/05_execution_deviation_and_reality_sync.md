# 执行偏差与现实同步

## 1. 文件定位

本文件用于规定：当想法方案与真实执行方案发生偏差时，Codex 应如何改标、回写、同步并阻止旧方案继续污染后续执行。

它解决的是：

- 什么叫执行偏差
- 哪些偏差必须回写
- 偏差如何分级
- 发现偏差后默认怎么处理
- 回写到哪里才算形成执行事实

它不是：

- 运行命令手册
- 项目脑正文
- 单次任务的全部实施步骤

## 2. 什么叫执行偏差

只要出现“原想法方案”和“真实可执行方案”之间的有效差异，就叫执行偏差。

以下情况都属于执行偏差：

### 资源问题

- 本地机器算力、内存、磁盘、时间窗口不足
- 所需素材、账号额度、模型额度不足

### 权限问题

- 无法访问目标目录、远程仓库、外部服务、密钥或账号
- 有账号但没有对应的读写 / push / PR / 上传权限

### 环境问题

- 当前系统缺工具
- 版本不兼容
- 路径、运行时、平台能力与原设想不一致

### 依赖 / 接口问题

- 依赖包不可用
- SDK / API 不可用
- 外部接口返回与预期不一致
- 云端 / 第三方服务能力与想法方案不匹配

### 成本问题

- 方案能做，但成本明显不合适
- 当前预算 / 付费意愿 / 供应商成本与原设想冲突

### 素材 / 数据质量问题

- 输入素材不足
- 样本质量过低
- 输出素材可生成，但质量不足以支撑原方案

### 方案部分成立 / 未成立

- 某个想法只在局部成立
- 原方案的关键前提没有成立
- 想法方案理论可讲，但现实里只能落一部分

## 3. 偏差分级

### 轻微偏差

定义：

- 不改项目主线
- 不改本轮目标
- 只需要调整执行细节

默认处理：

- 原方案状态改为 `细节已调整`
- 可继续执行
- 至少写入本轮回报

### 中度偏差

定义：

- 已影响当前方案落地
- 本轮执行方案需要改写
- 但还没有推翻主线前提

默认处理：

- 原方案状态改为 `部分成立 / 本轮方案已改写`
- 必须在本轮回报中写清影响范围
- 若后续还会受影响，必须同步升级到 `codex_source/03_research_findings_bridge.md`

### 重度偏差

定义：

- 原方案关键前提失效
- 继续沿旧方案推进会误导后续执行
- 当前已无法安全把旧方案继续当成成立事实

默认处理：

- 原方案状态改为 `前提失效 / 已被偏差覆盖`
- 必须暂停旧方案
- 必须先回写，再决定是否待确认或重新拍板

## 4. 发现偏差后的默认动作

### 第一步：先标状态

发现偏差后，先给原方案改标，不允许先在脑内偷偷调整。

可用状态至少包括：

- `细节已调整`
- `部分成立`
- `本轮方案已改写`
- `已被偏差覆盖`
- `前提失效待确认`

### 第二步：再写清偏差是什么

必须写清：

- 偏差类别
- 真实表现
- 证据来自哪里
- 原方案哪一段受影响

### 第三步：再判断影响范围

至少判断三件事：

1. 只影响执行细节，还是影响本轮方案
2. 只影响本轮，还是会影响后续默认执行
3. 是否已经触发暂停旧方案的条件

### 第四步：再决定动作

只能从以下四类里选：

- `继续执行`
  - 仅适用于轻微偏差
- `改写本轮执行方案`
  - 适用于中度偏差
- `停下待确认`
  - 适用于当前无法自行安全收口
- `升级进研究桥接文件`
  - 适用于会影响后续默认执行的偏差

### 第五步：同步改原方案状态

若偏差已经改变原方案成立性，必须把旧方案从“默认成立”改成新的状态，而不是继续沿旧假设写文档或执行。

## 5. 必须回写到哪里

### 只影响本轮

写入：

- 本轮最终回报
- 本轮执行日志

### 会影响后续执行

除本轮回报外，还必须同步写入：

- `codex_source/03_research_findings_bridge.md`

此时旧结论应改为：

- `部分采用`
- `已被偏差覆盖`
- `已失效`

中的一种，而不是继续保留为默认已采用。

### 会影响长期执行前提

除以上回写外，还必须同步更新：

- `codex_source/02_current_execution_context.md`

典型情形包括：

- 当前主路径变了
- 当前阶段不做事项变了
- 某个原本被当成前提的条件已被现实推翻

## 6. 禁止行为

明确禁止：

- 明知前提失效还继续沿旧方案执行
- 把“部分成立”写成“已成立”
- 把“现实只能落一部分”写成“方案已跑通”
- 只在脑内调整，不回写仓库事实
- 只口头说明，不更新文件
- 明知偏差会影响后续执行，却只在当前回复里轻描淡写提一句

## 7. 一句话规则

只要真实执行已经证明原方案不再完整成立，Codex 就必须先回写现实、改掉旧状态，再决定继续执行、改写方案还是停下待确认。
