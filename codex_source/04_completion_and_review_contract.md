# 完成与回审契约

## 1. 文件定位

本文件用于固定 Codex 的执行后汇报格式与状态口径。

它解决的是：

- 任务完成后必须怎么汇报
- 什么叫已完成 / 部分完成 / 被阻断 / 待确认
- 哪些栏目不能省
- 建议、计划、分析和完成事实如何分开写

它不负责：

- 代替项目脑判断方向
- 代替执行前上下文
- 代替执行偏差分级规则

## 2. 适用范围

只要本轮进入了真实执行，并出现以下任一结果，就必须按本文件汇报：

- 读取了仓库文件并形成新结论
- 修改了仓库文件
- 跑了命令
- 生成了产物
- 形成了新的阻断点 / 风险点 / 待确认项

如果只有纯聊天、纯建议、纯讨论，才可以不走完整契约。

## 3. 状态定义

### `已完成`

同时满足以下条件才能写：

- 本轮要求的关键文件或动作已真实完成
- 完成标准已逐条对照
- 关键验证已真实执行，或已如实说明为何无法验证
- 没有把“建议 / 计划 / 应该可以”冒充成完成事实

### `部分完成`

适用于：

- 已完成一部分真实改动
- 但仍有剩余目标未落地
- 或者关键验证 / 同步动作未全部完成

### `被阻断`

适用于：

- 权限不足
- 关键文件缺失
- 环境异常
- 依赖 / 接口不可用
- 事实冲突已影响动作选择

被阻断时不得写成“已完成”或“基本完成”。

### `待确认`

适用于：

- 关键方向还需用户或上层重新拍板
- 当前只能保守给出结论
- 本轮不宜把假设直接升格为事实

## 4. 固定汇报栏目

最终汇报默认必须按以下栏目输出，不得自由省略：

### 1. 读取结果

必须写清：

- 实际读取了哪些文件
- 哪些文件没读到
- 缺失或未读取对本轮判断的影响

### 2. 实际改动

必须写清：

- 实际修改了哪些文件
- 每个文件改了什么
- 哪些是新建，哪些是更新

### 3. 未改动项

必须写清：

- 哪些本来可能需要改
- 这轮为什么没改
- 是因为不在范围内、避免越权，还是因为未形成事实依据

### 4. 完成标准对照

必须写清：

- 本轮完成标准有哪些
- 哪些已满足
- 哪些仍未满足

若用户本轮给了 Definition of Done，必须逐条对照，不得只写笼统总结。

### 5. 当前状态

只能从以下四类里选：

- `已完成`
- `部分完成`
- `被阻断`
- `待确认`

不得自造模糊状态词替代。

### 6. 阻断 / 风险 / 待确认

只写最关键的，必须区分：

- 已经阻断推进的
- 还没阻断但会影响下一轮的
- 当前仍需确认的

### 7. 下一步建议

必须写清：

- 现在先做什么
- 为什么是它
- 做到哪算完

若当前已被阻断，下一步建议应优先指向解阻断，而不是继续扩改。

## 5. 固定模板

```text
1. 读取结果
- 实际读取了哪些文件
- 哪些没读到
- 影响是什么

2. 实际改动
- 实际修改了哪些文件
- 每个文件改了什么
- 哪些是新建，哪些是更新

3. 未改动项
- 哪些本来可能要改，但这轮没改
- 为什么没改

4. 完成标准对照
- 本轮完成标准有哪些
- 当前已满足哪些
- 哪些还没满足

5. 当前状态
- 已完成 / 部分完成 / 被阻断 / 待确认

6. 阻断 / 风险 / 待确认
- 只写最关键的

7. 下一步建议
- 现在先做什么
- 为什么是它
- 做到哪算完
```

## 6. 禁止事项

最终汇报中明确禁止：

- 把建议写成已完成
- 把计划写成已完成
- 把分析写成已完成
- 把“已经开始做”写成“已经完成”
- 省略未改动项
- 省略完成标准对照
- 不写状态直接收尾
- 用“应该 / 大概 / 基本可以”替代验证结果

## 7. 一句话契约

Codex 的执行后汇报必须把“读了什么、改了什么、没改什么、完成到哪、卡在哪里、下一步做什么”一次写清，不能拿建议、计划或分析冒充完成事实。
