# 执行闸门与并行规则

## 1. 文件定位

这份文件解决的是：

- 什么时候才允许进入 Codex 执行
- 什么叫“顶层已收口”
- Codex 的自动补全边界是什么
- 什么情况下必须停下汇报
- 什么情况下允许多个 Codex 窗口并行
- 并行执行如何与 ChatGPT 收口衔接

这里说的“顶层已收口”，当前应理解为：

- 项目主线已对齐
- 当前版本目标已收清
- 关键边界与当前不做事项已明确
- 剩下主要是执行问题，不是继续拍板问题

必须同时明确：

- 这不是项目脑文件
- 这不是代码运行规则文件
- 这不是 Git / PR 教程

它属于 `codex_source/` 的执行层补充规则，只负责“什么时候能执行、能补到哪、什么时候能并行”。

## 2. 为什么当前要补这份文件

当前仓库原有执行规则已经覆盖了：

- 默认读取顺序
- skill 检查
- 运行与产物事实
- GitHub / PR 线路
- 日志与汇报口径

但还缺以下关键口径：

- “顶层收口后才能进入执行”的明确闸门
- “自动补全可以补到哪、不能补什么”的边界
- “多个独立 Codex 窗口并行执行”的正式规则
- “判断层与执行层如何衔接”的停止线

如果不补这份文件，执行层很容易继续替判断层补拍板，或把仍在摇摆的边界误写成已确认任务。

## 3. 执行闸门规则

### GATE-001：同时满足以下条件才允许进入 Codex

1. 顶层对齐
2. 关键边界已定
3. 当前版本目标已收口
4. 剩下主要是执行问题，不是继续判断问题

可操作化地说，至少应明确以下问题：

- 当前项目主线是什么
- 当前版本明确做什么、不做什么
- 本轮要落到哪个文件、哪个产物或哪个执行动作
- 本轮不需要由 Codex 代替用户或 ChatGPT 决定方向

### GATE-002：出现以下情况必须先停下，不得继续执行

- 顶层定义未对齐
- 主线可能被带偏
- 边界未定
- 当前版本产物形态未收清
- 执行层正在替判断层补拍板

只要继续执行会影响主线判断、边界判断或当前版本定义，就应先停下并如实汇报，而不是靠补写文档或补改文件硬推进。

### GATE-003：闸门通过后的默认动作

闸门通过后，Codex 才进入：

- 文件改动
- 结构补齐
- 已确认边界内的自动补完
- 代码或执行链路落地
- 日志记录与分支同步

也就是说，闸门解决的是“能不能进执行”，不是“进执行后一定怎么实现”。

## 4. 自动补全边界

### Codex 可以自动补全什么

在已确认边界内，Codex 可以自动补全：

- 文件结构整理
- 已确认目标下的细化
- 明确范围内的重写
- 低风险补齐
- 已确认任务边界内的执行性补完

更直白地说，Codex 可以补的是：

- 已经拍板过的东西如何更完整落地
- 已经明确范围内哪些点还没写全
- 已经确认任务边界内哪些执行细节可以顺手补齐

### Codex 不可以自动补全什么

Codex 不可以自动补全：

- 项目主线判断
- 未确认业务事实
- 是否扩到 API / 自动化运营 / 平台化
- 是否把后续增强项当成当前前置依赖
- 用户尚未拍板的关键边界

因此必须明确：

- Codex 不能替用户拍板项目主线
- Codex 不能替 ChatGPT 拍板协作模式
- Codex 不能把“后续可选增强层”偷换成“当前阻塞项”

## 5. 多 Codex 并行规则（不是 subagents）

当前仓库允许多个独立 Codex 执行窗口并行工作。

这里说的并行，必须明确为：

- 多个外部拆开的独立 Codex 工位
- 不是 Codex 内部多 agent 框架
- 不是 subagents
- 不是把拍板工作拆成并行判断

### 允许并行的前提

1. 边界明确
2. 低耦合
3. 可独立验收
4. 合并成本可控
5. ChatGPT 已先完成统一收口

### 适合并行的任务类型

例如：

- 一个窗口改 `project_source/`
- 一个窗口改 `codex_source/`
- 一个窗口整理执行单
- 一个窗口做资料库整理
- 一个窗口做 API 路线对比整理

这些任务共同的前提是：

- 各自目标清楚
- 改动范围可分离
- 合并后不会互相覆盖核心判断

### 不适合并行的情况

例如：

- 顶层未对齐
- 多任务强耦合
- 同时会改同一批核心文件
- 还在做路线判断

如果并行会让多个工位各自拍板、各自收口、各自改主线，那就不是提效，而是放大跑偏风险。

## 6. 与 ChatGPT 层的衔接规则

这一层的默认衔接方式必须写清：

- ChatGPT 负责先收口、再拆工位
- Codex 不负责各自拍板项目方向
- 多个 Codex 执行结果必须统一回到 ChatGPT 回审
- 执行层不能替项目脑层拍板

换句话说：

- ChatGPT 负责把“做什么、做到哪、哪些不做”先收清
- Codex 负责在已确认边界内落地执行
- 多工位执行后，仍要回到统一回审，而不是让执行层各自形成不同版本主线

## 7. 当前项目的防跑偏护栏

当前项目的执行护栏必须继续保持一致：

- 当前视频主线不以字节 / 抖音 API 为前置依赖
- 当前允许本地生成 + 人工上传
- 平台 API 仅属于后续可选增强层
- 当前不要自动带到自动化运营闭环 / 多账号 / 平台化软件 / 直播 API 中控主线

当前质量增强 API 的边界也必须写清：

- 当前接外部 API 的核心原因，是质量增强，不是项目成立性缺失
- 当前第一优先是用户现成可用的火山引擎 TTS API
- 不得因为“已有现成火山引擎 API”就自动扩写成“现在全套能力都接”
- 当前仍然是“先接 TTS，再看下一步”，不是一次性展开整套供应商能力

这几条只是在当前阶段保护主线，不代表未来永远不做相关增强层。

但在当前执行层里，必须防止出现以下偷换：

- 把质量增强 API 写成项目是否成立的前置
- 把平台 API 写成当前必备前置
- 把运营闭环写成当前默认目标
- 把平台化软件线写成当前自然延伸

## 8. 当前一句话规则

先由 ChatGPT / 用户完成顶层收口，再让 Codex 在已确认边界内执行；未收口不执行，可补全但不代拍板，允许多工位并行但不允许并行拍板。
