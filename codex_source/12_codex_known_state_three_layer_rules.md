# Codex 已知状态三层规则

## 1. 文件定位

本文件用于把当前项目里“什么算已知”正式分层，避免：

- 把 GPT 聊天里说过的话当成 Codex 已知
- 把 bridge 摘要当成 project_source 正文已存在
- 把任务分支已 push 当成主读取分支正式已知

## 2. 三层定义

### 第一层：GPT 已知

只存在于以下位置的内容，属于 `GPT 已知`：

- 聊天里的背景
- 外部研究结果
- GPT Project 替换包
- 还没进当前仓库正文的内容

硬规则：

- 不得把第一层写成 Codex 已知
- 不得把第一层直接写成仓库正式事实

### 第二层：Codex 条件已知

已进入以下位置，但还没进入 project_source 正文或主读取分支的内容，属于 `Codex 条件已知`：

- 当前任务分支里的 bridge
- `codex_source/*`
- `codex_log/latest.md`
- 本轮执行单

硬规则：

- 不得把第二层写成 project_source 正文已存在
- 不得把第二层写成主读取分支正式已知

### 第三层：Codex 正式已知

当前项目把“Codex 正式已知”拆成两个子状态：

#### 3A. 当前分支正式已知

满足以下条件：

- 已进入当前任务分支的 `project_source` 正文
- 已进入当前任务分支的 codex 侧读取口径

#### 3B. 主读取分支正式已知

满足以下条件：

- 已满足 3A
- 且已同步回 `codex/user-readable-map`

硬规则：

- 不得把 3A 写成 3B
- 不得把“当前分支正式已知”写成“主读取分支正式已知”

## 3. 当前执行单必须显式标层

以后给 Codex 的执行单，至少要说明当前资料属于哪一层：

- `GPT 已知`
- `Codex 条件已知`
- `当前分支正式已知`
- `主读取分支正式已知`

## 4. 一句话规则

不得把 `GPT 已知` 写成 `Codex 条件已知`，不得把 `Codex 条件已知` 写成 `当前分支正式已知`，更不得把 `当前分支正式已知` 写成 `主读取分支正式已知`。
