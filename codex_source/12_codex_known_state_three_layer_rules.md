# Codex 已知状态三层规则

## 1. 文件定位

本文件用于正式区分“什么算已知”。

它要解决的是：

- 不把 GPT 数据源里的内容误写成 Codex 已知
- 不把当前分支状态误写成主读取分支正式已知
- 不把聊天背景误写成仓库正式事实

## 2. 第一层：GPT 已知

属于 `GPT 已知` 的内容包括：

- 聊天里说过但还没写入仓库的内容
- GPT Project 数据源里的内容
- 外部资料
- 还没进入当前仓库正文的总结

硬规则：

- GPT 数据源里有，不等于 Codex 已知
- 聊天里说过，不等于 Codex 已知

## 3. 第二层：Codex 条件已知

属于 `Codex 条件已知` 的内容包括：

- 已进入 `codex_source/*`
- 已进入 `codex_log/latest.md`
- 已进入 bridge
- 已在当前执行单里被显式带入

但还没进入当前分支 `project_source` 正文或还没同步回主读取分支时，只能算这一层。

## 4. 第三层：Codex 正式已知

`Codex 正式已知` 继续拆成两个子状态：

### 4.1 当前分支正式已知

满足以下条件：

- 已进入当前分支的 `project_source` 正文
- 已进入当前分支的 Codex 读取口径
- 但还未同步回 `codex/user-readable-map`

### 4.2 主读取分支正式已知

满足以下条件：

- 已满足“当前分支正式已知”
- 且已同步回 `codex/user-readable-map`

## 5. 当前最容易混淆的三种情况

### 情况 A

- GPT 数据源里有
- 仓库没写

=> 只能算 `GPT 已知`

### 情况 B

- 当前分支里写了
- `codex/user-readable-map` 还没更新

=> 只能算 `当前分支正式已知`

### 情况 C

- 当前分支里写了
- 并且已同步回 `codex/user-readable-map`

=> 才能算 `主读取分支正式已知`

## 6. 一句话规则

不得把 `GPT 已知` 写成 `Codex 条件已知`，不得把 `Codex 条件已知` 写成 `当前分支正式已知`，更不得把“当前分支已写入”偷换成“主读取分支正式已知”。
