# 主读取分支与分支回流规则

## 1. 文件定位

本文件用于写清：

- 当前仓库默认主读取分支是什么
- 任务分支与主读取分支的关系是什么
- 什么情况下才算“仓库当前正式状态”
- 每轮收尾必须回报哪些同步锚点

它不是：

- Git 教程
- PR 模板
- 单次任务日志

## 2. 当前默认主读取分支

当前仓库默认主读取分支固定为：

- `codex/user-readable-map`

默认含义：

- 新会话若要接当前仓库“最新正式状态”，优先读取 `codex/user-readable-map`
- `codex_log/latest.md` 必须以该分支上的版本为准
- 其他任务分支即使已经 push，也不能自动替代主读取分支的接手口径

## 3. 任务分支与正式状态的关系

允许存在：

- 任务分支
- 验证分支
- 临时修复分支

但必须明确：

- 任务分支里的实现，只代表“该分支已完成”
- 只有当结果已同步回 `codex/user-readable-map` 后，才算“当前仓库正式状态”
- 若未完成回流，就必须诚实写成：
  - 已在任务分支完成
  - 尚未同步回主读取分支

## 4. 默认回流动作

当任务分支已经形成可判断小闭环时，默认回流动作至少包括：

1. 更新任务分支上的 `codex_log/latest.md`
2. 命中写日志条件时，补完整执行日志
3. 将结果同步到 `codex/user-readable-map`
   - 可用 `merge`
   - 可用 `cherry-pick`
   - 若无法直接回流，则改用 `PR`
4. 把 `codex/user-readable-map` push 到 GitHub

补充边界：

- 若走 `PR` 路径，必须给出 PR 编号或链接
- 若可安全直接回流到 `codex/user-readable-map`，优先不要把“已完成实现”长期留在孤立任务分支

## 5. 每轮收尾必须回报的 4 个同步锚点

每轮仓库型任务收尾，必须明确回报：

1. 当前工作分支
2. 最新提交 SHA
3. 是否已 push 到 GitHub
4. 是否已同步回主读取分支 `codex/user-readable-map`

若第 4 项答案是否定的，必须继续写清：

- 当前结果还停在哪个任务分支
- 新会话如果要看本轮结果，应读哪个分支 / 哪个 PR

## 6. `codex_log/latest.md` 的使用规则

`codex_log/latest.md` 必须遵守：

- 它是主读取分支的接手入口，不是任意任务分支的私有便签
- 若任务分支已形成新事实但未回流，不能把主读取分支上的 `latest` 写成已经同步
- 一旦结果回流到 `codex/user-readable-map`，主读取分支上的 `latest` 必须立刻更新到新事实

## 7. 冲突处理

若出现以下任一情况，必须先如实汇报：

- 任务分支与主读取分支发生冲突
- 无法安全 `merge` / `cherry-pick`
- 无 push 权限
- 无 PR 权限
- 不确定当前主读取分支是否仍是 `codex/user-readable-map`

不得把“任务分支已完成”偷换成“主读取分支已完成”。
