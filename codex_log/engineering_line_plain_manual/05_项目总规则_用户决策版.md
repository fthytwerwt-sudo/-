# 项目总规则｜用户决策版

## 0. 一句话总原则

用户只负责核心决策，不负责内部流程排障。

凡是会改变项目目标、当前阶段、外部调用、权限成本、审美方向、文案语义、状态推进和长期默认规则的事，必须返回用户确认。

凡是格式补齐、字段解释、只读审计、报告补全、路径索引、状态保守化和低风险验证，默认由 ChatGPT（总控判断层）和 Codex（写入执行器）处理。

## 1. 项目目标：最终要达成什么

最终目标不是“把外部项目装进来”，也不是“画一张流程图”。最终目标是形成一条可长期维护的工程线，让《视频工厂》能持续把用户目标、仓库事实、工具能力、执行结果和复盘记录接在一起。

这条工程线要能做到：

- 用户提出任务。
- 系统判断目标。
- 读取仓库事实。
- 必要时检索资料。
- 调用允许的工具。
- 生成结果。
- 失败时回到正确层级。
- 必要时返回用户确认。
- 写入 report（报告）、trace（追踪）和 log（日志）。
- 不把未完成状态写成完成状态。

如果某个新模块不能帮助这条工程线更稳定地判断、执行、验收和回退，它就不能直接进入主线。

## 2. 当前阶段：现在只做到哪一步

当前已经确认：

- adapter（适配器）候选已经合并进 main（主分支）。
- 当前已有 engineering_line_audit（工程线审计）报告和 plain_language_manual（人话版手册）。
- runtime（运行时）没有启用。
- service（服务）没有启动。
- RAG（检索增强生成）没有成为默认真实运行时。
- Tool Registry（工具注册表）没有正式闭环。
- Evaluator（评估器）和 Failure Route（失败路由）仍未完全统一。
- Checkpoint / Trace / Log（检查点 / 追踪 / 日志）仍未完整闭环。

因此，现在不能写成 production_readiness（生产可用状态）成立，不能写成 content_validation（内容验证）通过，不能写成 send_ready（可发送状态）开启，也不能写成工程融合已经完成。

当前适合做的是：把目标、阶段、状态、事实源、工具权限、失败回退和人工确认边界先说清楚，再让后续任务按一片一片的方式落地。

## 3. 用户拍板边界：哪些必须问用户

### 3.1 改变项目目标

以下情况必须问用户：

- 是否改变项目最终目标。
- 是否从《视频工厂》变成通用工程平台。
- 是否从辅助执行变成自动运行服务。
- 是否改变商业化、内容化、产品化方向。

判断口径：只要会改变“这个项目到底服务什么结果”，就不是 Codex（写入执行器）可以自行决定的事。

### 3.2 改变当前阶段

以下情况必须问用户：

- 是否把 candidate（候选）升级为 formal（正式）。
- 是否把 probe_only（仅探测）升级为 implemented（已实现）。
- 是否把 documented_only（仅文档）升级为 active（已启用）。
- 是否把审计完成写成工程完成。

判断口径：阶段升级必须有证据、验收和回退；没有用户确认，AI（人工智能）只能保守记录当前阶段。

### 3.3 启用外部能力 / 权限 / 成本

以下情况必须问用户：

- 启用 runtime（运行时）。
- 启动 service（服务）。
- 调用 external API（外部接口）。
- 调用 TTS（语音合成）。
- 调用 DashVector（阿里向量数据库）。
- 运行 Chroma ingestion（Chroma 入库）。
- 上传素材、声音、私密文件。
- 产生费用、额度消耗、联网调用。

判断口径：只要涉及外部权限、外部费用、隐私材料或真实服务调用，默认返回用户确认。

### 3.4 改变内容表达或审美

以下情况必须问用户：

- 改 locked copy（锁定文案）语义。
- 改账号表达方向。
- 改卡片视觉方向。
- 改视频比例。
- 改声音身份。
- 改真人 / 录屏 / PPT（演示文稿） / 卡片的承载比例。
- 使用降级方案替代原目标。

判断口径：Codex（写入执行器）可以调整格式、断句、排版和证据组织，但不能替用户改变表达立场、审美方向或内容语义。

### 3.5 改变事实源优先级

以下情况必须问用户：

- RAG（检索增强生成）结果和仓库文件冲突且无法自动裁决。
- 外部资料和 main（主分支）当前事实冲突。
- 旧 log（日志）和 latest（最新日志）冲突且影响执行方向。
- GPT Project（GPT 项目静态包）和 GitHub（远端仓库）当前事实冲突。

判断口径：仓库当前事实优先，但当冲突会改变执行方向时，必须把冲突点交回用户或 ChatGPT（总控判断层）裁决。

### 3.6 改变长期默认规则

以下情况必须问用户：

- 把某个新机制设为长期默认。
- 降权旧机制。
- 删除旧入口。
- 替换项目主流程。
- 改变未来 Codex（写入执行器）默认执行方式。

判断口径：临时补丁可以由 AI（人工智能）整理，长期默认规则必须由用户确认。

### 3.7 推进发布 / 内容 / 发送状态

以下情况必须问用户：

- 推进 content_validation（内容验证）。
- 推进 send_ready（可发送状态）。
- 推进 publish_candidate（可发布候选片）。
- 推进 production_readiness（生产可用状态）。

判断口径：技术链路跑通不等于内容过线，文件存在不等于可以发送，候选结果不等于生产可用。

## 4. AI 默认处理边界：哪些不用问用户

### 4.1 格式与可读性

默认由 ChatGPT（总控判断层）和 Codex（写入执行器）处理：

- 英文字段补中文备注。
- Markdown（标记文档）排版修正。
- 标题层级整理。
- 表格字段补解释。
- 技术词补中文含义。
- 报告目录补齐。

这些动作不改变项目目标、状态、权限或事实源，只提升可读性。

### 4.2 只读审计

默认处理：

- 读取仓库文件。
- 检查文件是否存在。
- 检查报告是否缺字段。
- 检查状态是否偷换。
- 检查是否有旧口径残留。
- 检查是否有未授权外部调用。

只读审计不等于执行完成。它只能告诉用户“现在证据是什么、缺口在哪里”。

### 4.3 低风险补全

默认处理：

- 补 done_when（完成标准）。
- 补 blocked_if（阻断条件）。
- 补 status_boundary（状态边界）。
- 补 files_read（已读取文件）。
- 补 validation_result（验证结果）。
- 补 latest（最新日志）摘要。
- 补 README（目录说明）索引。

这些补全只能让任务更清楚，不能把候选、探测、文档状态升级为正式状态。

### 4.4 状态保守化

默认处理：

- 把夸大的 completed（完成）改成 blocked（阻断）或 partial（部分完成）。
- 把 production_readiness（生产可用状态）的夸大写法改成 not_claimed（未声称）。
- 把 runtime（运行时）就绪的夸大写法改成 probe_only（仅探测）。
- 把内容验证通过的夸大写法拆成 technical_validation（技术验证）和 content_validation（内容验证）分别判断。

状态保守化是安全动作，不需要用户先确认；但如果要反向推进状态，就必须问用户。

### 4.5 执行层常规检查

默认处理：

- git diff check（Git 差异检查）。
- secret scan（密钥扫描）。
- 文件路径检查。
- JSON / YAML / Markdown parse（JSON / YAML / Markdown 结构解析）。
- 报告存在检查。
- 禁止状态推进扫描。

这些检查属于验收动作。检查失败时，Codex（写入执行器）应先修复或阻断，不能把失败结果写成完成。

## 5. 高风险问题：哪些必须停下来

| stopline（停止线） | 为什么高风险 | 应该返回给谁 | 用户未确认时怎么处理 | 是否允许降级 |
| --- | --- | --- | --- | --- |
| 需要用户授权的外部调用 | 可能产生费用、联网、隐私或外部副作用 | 用户 | 记录为 blocked（阻断），不调用 | 只能提出降级方案，不能执行 |
| 需要 secret / API key（密钥 / 接口密钥） | 可能泄露凭证或越权访问 | 用户 | 不读取、不写入、不要求用户明文贴密钥 | 不允许自动降级成真实调用 |
| 需要上传素材、声音或私密文件 | 涉及隐私、肖像、声音和版权 | 用户 | 停在说明和清单层 | 只能做本地只读清单 |
| 需要开 runtime（运行时） | 会从文档 / 探测进入真实执行链 | 用户 | 保持 runtime_enabled（运行时启用）为 false（未启用） | 不允许自动启用 |
| 需要启 service（服务） | 可能打开端口、接受请求或改变边界 | 用户 | 保持 service_started（服务启动）为 false（未启动） | 不允许自动启用 |
| 需要改 locked copy（锁定文案）语义 | 会改变内容表达和用户判断 | 用户 / ChatGPT（总控判断层） | 输出 copy_change_request（文案修改请求） | 未确认不允许降级改写 |
| 需要改变审美方向 | 会改变品牌感、卡片方向和用户偏好 | 用户 | 给出对比点，不直接改方向 | 只能建议，不直接替换 |
| 需要用降级方案 | 降级会损失原目标价值 | 用户 | 写清损失和阻断原因 | 需用户明确确认 |
| 需要把候选模块转正式 | 会改变项目默认入口 | 用户 / ChatGPT（总控判断层） | 保持 candidate（候选）或 probe_only（仅探测） | 不允许静默升级 |
| 事实源冲突无法裁决 | 可能把旧事实写成当前事实 | ChatGPT（总控判断层） / 用户 | 记录冲突，不继续执行依赖结论 | 不允许用摘要替代原文 |
| 素材不能证明文案 | 会造成口播和画面不一致 | ChatGPT（总控判断层） / 用户 | 返回文案层或素材层 | 不允许用弱相关素材硬撑 |
| 卡片会遮挡核心证据 | 会影响用户理解和内容可信度 | ChatGPT（总控判断层） / 用户 | 调整位置；调不了则 blocked（阻断） | 不允许遮挡后继续交付 |
| RAG（检索增强生成）结果没有 source_path（来源路径）或 readback（回读） | 无法证明资料来源 | ChatGPT（总控判断层） | 不把结果写成事实 | 不允许把向量摘要当事实 |
| 工具可能写仓库或外呼，但权限不清 | 可能越过 active_write_executor（当前激活写入执行器） | 用户 / ChatGPT（总控判断层） | 停在 tool permission review（工具权限复审） | 不允许真实调用 |
| Codex（写入执行器）无法区分本轮文件和 unrelated dirty files（无关脏文件） | 可能误提交用户改动 | 用户 | 停止提交，先列出冲突文件 | 不允许全量暂存 |
| secret scan（密钥扫描）失败 | 可能把凭证推到远端 | 用户 | 停止 commit（提交）和 push（推送） | 不允许绕过扫描 |
| push（推送）失败 | 远端状态未同步，不能声称完成 | 用户 / Git sync（Git 同步）层 | 保留本地提交状态并回报失败 | 不允许写成已远端同步 |

## 6. 模块标准：每个模块输入 / 输出 / 验收

### 6.1 项目目标

它负责什么：
决定这件事最终服务什么结果。

输入：
- 用户目标、当前项目身份、长期运营方向。

输出：
- project_goal（项目目标）和不做事项。

验收：
- 用户能看懂“为什么做这件事”和“什么不算完成”。

用户必须拍板：
- 改变商业化、内容化、产品化或自动化方向。

AI 默认处理：
- 把模糊目标拆成可验证的目标、阶段和边界。

Codex 禁止擅自处理：
- 把工具接入目标改成项目总目标。

失败回退：
- 回到目标层，让用户或 ChatGPT（总控判断层）重新定目标。

### 6.2 当前阶段

它负责什么：
说明现在只做到哪一步，防止把候选、探测、文档写成正式。

输入：
- latest（最新日志）、报告、commit（提交）、probe（探测）结果。

输出：
- current_stage（当前阶段）和 status_boundary（状态边界）。

验收：
- runtime（运行时）、service（服务）、RAG（检索增强生成）、media（媒体）等状态没有被误推进。

用户必须拍板：
- 阶段升级、正式入口启用、生产可用状态声明。

AI 默认处理：
- 把夸大的状态改成保守状态。

Codex 禁止擅自处理：
- 把 probe_only（仅探测）写成 formal（正式）。

失败回退：
- 回到事实源裁决层核对证据。

### 6.3 状态地图

它负责什么：
给每个模块贴清楚状态标签。

输入：
- 文件、报告、schema（结构）、fixture（样例）、probe（探测）、运行记录。

输出：
- state_map（状态地图），例如 missing（缺失）、documented_only（仅文档）、candidate（候选）、probe_only（仅探测）、formal（正式）。

验收：
- 每个状态都有对应 evidence（证据）。

用户必须拍板：
- 把关键模块升级为 formal（正式）。

AI 默认处理：
- 标记缺口、冲突和证据来源。

Codex 禁止擅自处理：
- 用文件存在替代验收证据。

失败回退：
- 回到 report（报告）层补证据。

### 6.4 事实源裁决

它负责什么：
决定哪个来源能作为当前事实。

输入：
- GitHub（远端仓库）文件、main（主分支）内容、latest（最新日志）、review_loop（复盘闭环）、GPT Project（GPT 项目静态包）、RAG（检索增强生成）结果。

输出：
- fact_source_arbitration（事实源裁决）和冲突清单。

验收：
- 当前仓库原文件优先，旧口径和摘要不得覆盖当前事实。

用户必须拍板：
- 无法自动裁决且会改变执行方向的冲突。

AI 默认处理：
- 回读原文件、列冲突、保守阻断。

Codex 禁止擅自处理：
- 用 RAG（检索增强生成）摘要替代 source_readback（原文回读）。

失败回退：
- 回到 source_readback（原文回读）层。

### 6.5 Workflow（工作流）

它负责什么：
决定任务进入哪条工作流。

输入：
- route_decision（路由判断）、任务类型、用户目标、必读文件。

输出：
- workflow_route_decision（工作流归位判断）。

验收：
- 只进入已允许的 workflow（工作流），未归位能力不得硬塞。

用户必须拍板：
- 新增正式 workflow（工作流）或改变旧 workflow（工作流）边界。

AI 默认处理：
- 把任务归入已有 workflow（工作流）或登记为候选。

Codex 禁止擅自处理：
- 因为上游有 agent（智能体）就新增项目 workflow（工作流）。

失败回退：
- 回到 mechanism_repair_flow（机制修补流）。

### 6.6 State / Node / Edge（状态 / 节点 / 边）

它负责什么：
把工作流拆成可执行的状态、节点和流转条件。

输入：
- workflow（工作流）、阶段标签、进入条件、退出条件。

输出：
- state（状态）、node（节点）、edge（边）定义。

验收：
- 每个 node（节点）有输入、输出、阻断条件和日志。

用户必须拍板：
- 改变节点代表的业务决策或阶段升级。

AI 默认处理：
- 补齐节点字段、流转条件和阻断规则。

Codex 禁止擅自处理：
- 让 graph（图）替代 Project State Action Router（项目状态动作总控器）。

失败回退：
- 回到状态地图或 workflow（工作流）层。

### 6.7 RAG（检索增强生成）

它负责什么：
辅助找资料，不负责直接拍板项目事实。

输入：
- query（检索问题）、retrieval_manifest（检索清单）、source_path（来源路径）、readback（回读）。

输出：
- 带来源的检索结果和 gap_status（缺口状态）。

验收：
- 每条可用结果必须有 source_path（来源路径）、chunk_id（切片编号）和 readback（回读）。

用户必须拍板：
- 真实调用 DashVector（阿里向量数据库）或 Chroma（本地向量库）以及替换检索主路线。

AI 默认处理：
- 用本地文件回读验证检索结论。

Codex 禁止擅自处理：
- 把向量结果直接写成项目事实。

失败回退：
- 回到事实源裁决层或 source_readback（原文回读）层。

### 6.8 Tool Registry（工具注册表）

它负责什么：
登记哪些工具能用、能做什么、不能做什么。

输入：
- 工具名称、权限、输入输出、费用、外呼和写入能力。

输出：
- tool_permission（工具权限）和 allowed_actions（允许动作）。

验收：
- 工具不能绕过 active_write_executor（当前激活写入执行器）。

用户必须拍板：
- 启用会外呼、写仓库、产生成本或读取隐私文件的工具。

AI 默认处理：
- 标记工具为 disabled_by_default（默认禁用）或 probe_only（仅探测）。

Codex 禁止擅自处理：
- 未授权调用外部工具。

失败回退：
- 回到 tool permission review（工具权限复审）层。

### 6.9 Retriever（检索器）

它负责什么：
按规则取回资料。

输入：
- query（检索问题）、provider（检索来源）、过滤条件、权限边界。

输出：
- retrieval_result（检索结果）和 manifest（清单）。

验收：
- 检索结果必须能回指原文。

用户必须拍板：
- 改变主检索路线或启用真实外部检索服务。

AI 默认处理：
- 检查结果是否缺 source_path（来源路径）和 readback（回读）。

Codex 禁止擅自处理：
- 用检索器输出替代仓库事实。

失败回退：
- 回到 RAG（检索增强生成）或事实源裁决层。

### 6.10 Vector Store（向量库）

它负责什么：
存储和检索向量化资料。

输入：
- 文档、metadata（元数据）、chunk（切片）、embedding（向量）。

输出：
- 可检索的向量索引和来源元数据。

验收：
- DashVector（阿里向量数据库）仍是主路线；Chroma（本地向量库）只能做 sandbox（沙盒）参考，除非用户另行确认。

用户必须拍板：
- 真实写入、真实调用、替换主向量库。

AI 默认处理：
- 检查 metadata（元数据）是否保留 source_readback（原文回读）能力。

Codex 禁止擅自处理：
- 运行 Chroma ingestion（Chroma 入库）或真实调用 DashVector（阿里向量数据库）。

失败回退：
- 回到清洗和检索边界层。

### 6.11 Evaluator（评估器）

它负责什么：
判断结果是否达到验收标准。

输入：
- 输出文件、报告、状态、blocked case（阻断样例）、用户目标。

输出：
- evaluation_result（评估结果）和 failed_reason（失败原因）。

验收：
- 技术成功、内容成功、状态成功必须分开判断。

用户必须拍板：
- 把评估结论升级为内容通过、可发送或生产可用。

AI 默认处理：
- 检查缺字段、状态偷换和完成真实性。

Codex 禁止擅自处理：
- 把技术验证通过写成内容验证通过。

失败回退：
- 回到失败路由层。

### 6.12 Failure Route（失败路由）

它负责什么：
决定失败后回到哪一层修。

输入：
- failed_reason（失败原因）、blocked_if（阻断条件）、当前模块。

输出：
- fallback_layer（回退层）和 next_safe_step（下一步安全动作）。

验收：
- 每个失败都有明确归位，不让用户排查内部流程。

用户必须拍板：
- 失败后是否接受降级方案或改变目标。

AI 默认处理：
- 把失败归到目标层、事实源层、工具层、内容层或 Git sync（Git 同步）层。

Codex 禁止擅自处理：
- 失败原因不明却继续推进。

失败回退：
- 回到 Failure Route（失败路由）自身并补分类。

### 6.13 Human-in-the-loop（人工介入）

它负责什么：
定义什么时候必须人来确认。

输入：
- 高风险动作、冲突、审美判断、内容语义、状态推进。

输出：
- human_review_request（人工复审请求）。

验收：
- 用户只看关键决策，不被要求排查内部细节。

用户必须拍板：
- 外部调用、权限成本、降级方案、内容语义、状态推进。

AI 默认处理：
- 整理问题、列选项、保守阻断。

Codex 禁止擅自处理：
- 用“我认为可以”替代用户确认。

失败回退：
- 回到用户确认或 ChatGPT（总控判断层）判断。

### 6.14 Guardrails（护栏）

它负责什么：
防止越权、误写、误提交和状态偷换。

输入：
- 禁止文件、禁止状态、权限边界、扫描结果。

输出：
- guardrail_result（护栏结果）。

验收：
- 触发护栏时必须 blocked（阻断）或修复，不能继续完成。

用户必须拍板：
- 是否放宽护栏。

AI 默认处理：
- 执行扫描、识别违规、阻断高风险动作。

Codex 禁止擅自处理：
- 绕过 secret scan（密钥扫描）或使用 git add .（全量暂存）。

失败回退：
- 回到 validation（验证）层。

### 6.15 卡片判断

它负责什么：
判断视频里哪些话需要卡片辅助，哪些不需要。

输入：
- 口播句子、真实证据、观众理解难点、画面可读性。

输出：
- card_decision（卡片判断）和 card_type（卡片类型）。

验收：
- 不是先问做几张卡，而是先判断这句话是不是关键判断、观众会不会漏掉、有没有真实证据、会不会挡住录屏。

用户必须拍板：
- 卡片视觉方向变化、审美方向变化、关键表达变化。

AI 默认处理：
- 判断 card_type（卡片类型）：data_result_card（数据成果卡）、judgment_card（判断卡）、summary_card（总结卡）、boundary_card（边界卡）、process_summary_card（流程总结卡）。

Codex 禁止擅自处理：
- 普通转场强行加卡；卡片遮挡真实证据后继续交付；把容易误解成成功的内容做成成果卡。

失败回退：
- 如果只是普通转场，不加卡；如果容易被误解成已成功，优先 boundary_card（边界卡）；如果会遮挡真实证据，必须 blocked（阻断）或换位置。

### 6.16 文案修改

它负责什么：
保护 locked copy（锁定文案）语义，同时允许执行层做必要格式处理。

输入：
- locked_topic（锁定选题）、locked_title（锁定标题）、locked_final_script（锁定终稿）、素材证据。

输出：
- copy_change_request（文案修改请求）或格式化后的执行文案。

验收：
- 标点、换行、字幕分句可以调整，语义、标题、开头和核心判断不能擅自改。

用户必须拍板：
- 改语义、改标题、改表达方向、接受降级文案。

AI 默认处理：
- 识别文案与素材不匹配，提出修改请求。

Codex 禁止擅自处理：
- 自己重写用户锁定文案。

失败回退：
- 回到文案层或素材层。

### 6.17 素材证据

它负责什么：
判断画面、截图、录屏是否能证明口播。

输入：
- 素材路径、时间码、截图、录屏、source_segment_inventory（素材片段清单）。

输出：
- material_evidence_contract（素材证据契约）。

验收：
- 每个关键判断都有可见证据或明确缺口。

用户必须拍板：
- 素材缺失时是否补录、替换或接受降级。

AI 默认处理：
- 标记 evidence_strength（证据强度）和 missing_material（缺素材）。

Codex 禁止擅自处理：
- 用弱相关素材冒充直接证据。

失败回退：
- 回到素材层或文案层。

### 6.18 Report / Trace / Log（报告 / 追踪 / 日志）

它负责什么：
留下可回读、可复审、可追踪的记录。

输入：
- files_read（已读取文件）、validation_result（验证结果）、status_boundary（状态边界）、git_sync_status（Git 同步状态）。

输出：
- report（报告）、trace（追踪）、log（日志）。

验收：
- 用户能看出做了什么、没做什么、验证了什么、下一步是什么。

用户必须拍板：
- 把报告结论升级为项目状态结论。

AI 默认处理：
- 补齐字段、索引和 latest（最新日志）摘要。

Codex 禁止擅自处理：
- 用 report（报告）替代真实执行。

失败回退：
- 回到报告补全层。

### 6.19 Runtime（运行时）

它负责什么：
让候选链路进入可调用的执行环境。

输入：
- graph（图）、service boundary（服务边界）、tool permission（工具权限）、写入边界。

输出：
- runtime_probe（运行时探测）或 runtime_status（运行时状态）。

验收：
- 未经授权只能停在 probe_only（仅探测），且不能直接写 repo（仓库）。

用户必须拍板：
- 启用运行时、启动服务、允许外呼、允许成本、允许长期运行。

AI 默认处理：
- 做 no-service（无服务）或 no-render（无渲染）探测报告。

Codex 禁止擅自处理：
- 启动正式 runtime（运行时）或让 service（服务）直接写仓库。

失败回退：
- 回到权限、服务边界或用户授权层。

### 6.20 Git sync（Git 同步）

它负责什么：
确认本地结果已经安全提交并同步远端。

输入：
- git diff（Git 差异）、staged files（已暂存文件）、commit（提交）、push（推送）、remote head（远端头指针）。

输出：
- git_sync_status（Git 同步状态）。

验收：
- 只暂存本轮允许文件，push（推送）后远端回读匹配本地 HEAD（当前提交）。

用户必须拍板：
- 是否处理 unrelated dirty files（无关脏文件）或重写历史。

AI 默认处理：
- path-limited stage（路径限定暂存）、commit（提交）、push（推送）、remote readback（远端回读）。

Codex 禁止擅自处理：
- 使用 git add .（全量暂存）或误提交无关文件。

失败回退：
- 回到 Git sync（Git 同步）层并报告失败原因。

## 7. 失败回退：出错后回到哪里修

| 问题 | 应该回到哪一层 | 谁处理 | 是否需要用户确认 | 禁止怎么处理 |
| --- | --- | --- | --- | --- |
| 目标不清 | 项目目标层 | 用户 / ChatGPT（总控判断层） | 是 | Codex（写入执行器）直接猜目标 |
| 状态冲突 | 事实源裁决层 | ChatGPT（总控判断层） / Codex（写入执行器） | 冲突影响方向时需要 | 用旧日志覆盖 current（当前）事实 |
| 文案不支撑素材 | 文案层 | ChatGPT（总控判断层） | 改语义时需要 | Codex（写入执行器）自行改锁定文案 |
| 素材不支撑口播 | 素材层 | Codex（写入执行器）先审计，用户决定补录或降级 | 需要 | 弱相关素材冒充证据 |
| 卡片挡证据 | 卡片判断层 | Codex（写入执行器） | 改视觉方向时需要 | 遮挡后继续交付 |
| RAG（检索增强生成）无来源 | 检索层 | Codex（写入执行器） | 通常不需要，除非改变事实源 | 把摘要写成事实 |

| 评估不过 | Evaluator（评估器）层 | ChatGPT（总控判断层） / Codex（写入执行器） | 若改变目标或降级则需要 | 技术通过冒充内容通过 |

## 8. 项目融合规则：新模块怎么接入旧主线

新模块不能直接变成正式入口。所有新模块必须先经历：

```text
missing（缺失）
-> documented_only（仅文档）
-> candidate（候选）
-> probe_only（仅探测）
-> formal（正式）
```

每一步都必须有 evidence（证据）：

- missing（缺失）：仓库里还没有对应文件、契约、样例或执行记录。
- documented_only（仅文档）：只有文档说明，没有可验证结构。
- candidate（候选）：有文件、代码、schema（结构）或 fixture（样例），但未接正式入口。
- probe_only（仅探测）：有探测通过，但不代表真实运行、真实服务或真实媒体链路。
- formal（正式）：有入口、有验收、有 report（报告）、有 failure route（失败回退）、有 status_boundary（状态边界）。

融合时必须守住这些规则：

- adapter（适配器）不能直接替代主线。
- RAG（检索增强生成）不能直接替代仓库事实。
- runtime（运行时）不能直接写 repo（仓库）。
- Tool（工具）不能越权调用。
- Evaluator（评估器）不能只停在文档。
- Report（报告）不能替代真实执行。
- Probe（探测）不能替代 production_readiness（生产可用状态）。

用户需要看的不是“有没有模块”，而是“这个模块现在处于哪一档、证据是什么、会不会改变项目目标或状态”。

## 9. Codex 执行格式：怎么下发、怎么回报

以后 Codex（写入执行器）每次回报必须回答：

1. 本轮修的是哪一层。
2. 本轮是否影响项目目标。
3. 本轮是否需要用户确认。
4. 本轮是否推进状态。
5. 本轮哪些是 AI（人工智能）默认处理。
6. 本轮哪些是用户必须拍板。
7. 本轮哪些被 blocked（阻断）。
8. 本轮失败后回哪一层。
9. 本轮改了哪些文件。
10. 本轮没改哪些高风险文件。
11. 本轮验证做了什么。
12. 下一步建议修哪一层。

必须禁止以下回报方式：

- 只写 completed（完成）。
- 只写文件路径。
- 只写技术通过。
- 把报告生成写成工程完成。
- 把 probe（探测）通过写成正式可用。
- 把 technical_validation（技术验证）写成 content_validation（内容验证）。
- 把本地文件写成已 push（推送）。

推荐回报结构：

```yaml
task_layer（本轮层级）: 说明修的是目标、状态、事实源、工具、内容、验证还是 Git sync（Git 同步）
user_decision_needed（是否需要用户拍板）: true / false（是 / 否）
status_promoted（是否推进状态）: false（未推进）
ai_default_actions（AI 默认处理）: 列出格式、审计、补字段、验证等低风险动作
blocked_items（阻断项）: 列出停下来的原因
files_changed（修改文件）: 只列本轮白名单文件
validation_result（验证结果）: 列出实际跑过的检查
next_safe_step（下一步安全动作）: 给出下一层建议
```

这份文档本身只是一份决策规则，不是 runtime（运行时）启用证明，不是 RAG（检索增强生成）正式接入证明，不是内容过线证明，也不是可发送证明。

## 10. 当前 8 个关键决策｜已确认版

本节是当前执行锚点。后续所有 engineering_fusion（工程融合）、schema（结构契约）、fixture（测试样例）、probe（探测脚本）、adapter（适配器）和 runtime hardening（运行时加固）任务，都必须先读取本节。

如果本节和旧文档、旧报告或旧 handoff（交接记录）冲突，以本节作为当前用户确认口径；旧材料只保留为 historical_reference（历史参考）。

### 10.1 项目最终目标

`已确认`: 《视频工厂》优先，先不升级成通用 AI（人工智能）工程平台，但工程结构要兼容未来复用。

这意味着：

- 新模块必须先服务《视频工厂》当前主线。
- 可以预留可复用结构，但不能把当前项目目标偷换成通用平台建设。
- 任何“把《视频工厂》升级成通用 AI 工程平台”的动作，必须另行返回用户确认。

### 10.2 当前阶段

`已确认`: 当前只允许做规则、状态地图、验收契约、接口层和 no-service / no-render（无服务 / 无渲染）验证。

当前不允许：

- 直接启 runtime（运行时）。
- 启动 service（服务）。
- 调用 external API（外部接口）。
- 调用 TTS（语音合成）。
- 真实调用 DashVector（阿里向量数据库）或 Chroma（本地向量库）。
- 生成 media（媒体）。

### 10.3 adapter（适配器）身份

`已确认`: adapter（适配器）是 execution framework candidate（执行框架候选），不替代原《视频工厂》主线。

这意味着：

- adapter（适配器）可以承载候选链路、契约验证、无服务探测和后续执行框架接线。
- adapter（适配器）不能替代 Project State Action Router（项目状态动作总控器）。
- adapter（适配器）不能因为本地 probe（探测）通过就变成 formal（正式）入口。

### 10.4 RAG（检索增强生成）策略

`已确认`: RAG（检索增强生成）变成默认选项。

这里的“默认选项”只表示：

- 默认进入 RAG（检索增强生成）判断链。
- 默认准备 retrieval_manifest（检索清单）。
- 默认要求 source_path（来源路径）、chunk_id（分块编号）和 readback（原文回读）。

这里的“默认选项”不表示：

- 默认真实调用 DashVector（阿里向量数据库）。
- 默认运行 Chroma ingestion（Chroma 入库）。
- 默认让向量结果覆盖 GitHub（远端仓库）当前事实。

RAG（检索增强生成）只能辅助找资料；仓库原文件事实优先。

### 10.5 runtime（运行时）边界

`已确认`: State / RAG / Tool / Evaluator / Failure / Trace（状态 / 检索增强生成 / 工具 / 评估器 / 失败 / 追踪）都接清楚后，才允许进入 runtime hardening（运行时加固）。

这意味着：

- 没有状态地图，不进 runtime hardening（运行时加固）。
- 没有 RAG（检索增强生成）默认判断边界，不进 runtime hardening（运行时加固）。
- 没有 Tool Registry（工具注册表）权限边界，不进 runtime hardening（运行时加固）。
- 没有 Evaluator / Failure Route / Guardrails（评估器 / 失败路由 / 护栏），不进 runtime hardening（运行时加固）。
- 没有 Report / Trace / Log（报告 / 追踪 / 日志）校验，不进 runtime hardening（运行时加固）。

### 10.6 卡片决策边界

`已确认`: 用户只确认视觉方向和重大审美变化；AI（人工智能）默认判断卡片类型、位置、数量，但不得挡证据或新增事实。

这意味着：

- card_type（卡片类型）默认由 AI（人工智能）按内容和证据判断。
- card_position（卡片位置）默认由 AI（人工智能）避开证据区域。
- card_count（卡片数量）默认由 AI（人工智能）按必要性控制。
- 只要改变视觉方向、重大审美方向或新增事实，必须返回用户确认。

### 10.7 文案权限边界

`已确认`: 融合完成前，Codex（写入执行器）只能改格式、断句、标点和 TTS（语音合成）停顿，不能改语义、标题和核心判断。

融合完成后，是否放宽 Codex（写入执行器）的文案权限，需要另行讨论。

这意味着：

- Codex（写入执行器）可以做执行层可读性处理。
- Codex（写入执行器）不能替用户重写 locked copy（锁定文案）。
- Codex（写入执行器）发现文案和素材不匹配时，应输出 copy_change_request（文案修改请求）或 blocked（阻断）。

### 10.8 失败返回规则

`已确认`: 低风险问题自动回退修；影响目标、权限、成本、审美、语义、状态升级、外部调用的，必须返回用户确认。

低风险自动处理包括：

- 格式补齐。
- 字段补中文。
- 路径索引补全。
- 报告字段补全。
- 状态保守化。
- 本地 no-service / no-render（无服务 / 无渲染）验证。

必须返回用户确认包括：

- 改项目目标。
- 改当前阶段。
- 启用 runtime（运行时）或 service（服务）。
- 外部 API（外部接口）、TTS（语音合成）、DashVector（阿里向量数据库）、Chroma（本地向量库）真实调用。
- 改审美方向。
- 改文案语义、标题或核心判断。
- 推进 content_validation（内容验证）、send_ready（可发送状态）或 production_readiness（生产可用状态）。
