# 02｜常见场景决策手册

## 场景 1：口径冲突了怎么办

用户看到的现象：
两个文件说法不一样，比如一个说 adapter（适配器）已经进 main（主分支），另一个还停在 branch-local（分支本地）旧状态。

Codex 容易犯的错：
直接挑一个自己喜欢的说法继续执行，或者把旧 handoff（交接件）当成当前事实。

正确处理顺序：
1. 先标记 `conflict（冲突）`。
2. 比较事实源优先级。
3. GitHub main（主分支当前事实）优先于旧日志、旧 handoff（交接件）和旧报告。
4. 旧口径只保留为历史，不覆盖当前事实。
5. 冲突未裁决前，不继续执行功能。

允许做：
- 写冲突登记表。
- 写事实源优先级。
- 写 next_safe_step（下一步安全动作）。

禁止做：
- 禁止把旧口径直接写成当前事实。
- 禁止为了继续执行而忽略冲突。
- 禁止把 `candidate（候选）` 写成 `formal（正式）`。

做到哪算通过：
- 冲突点、当前采用事实、历史保留口径、阻断条件都写清。

必须阻断的情况：
- 无法判断哪个文件是当前事实源。
- 需要改 `GPT数据源（项目事实目录）` 才能裁决，但本轮未授权。

给用户看的例子：
如果旧报告说“还在 adapter 分支”，但 latest（最新日志）和 main（主分支）合并报告说“候选已合并 main（主分支）”，那当前事实按 main（主分支）合并报告走；旧报告只说明历史过程。

## 场景 2：卡片要不要加

用户看到的现象：
一段口播里有判断、有数据、有流程，用户不确定要不要加卡片。

Codex 容易犯的错：
先问“做几张卡”，或者凭感觉加一堆卡，结果挡住真实录屏证据。

正确处理顺序：
1. 先判断这句话是不是关键判断。
2. 判断观众会不会漏掉。
3. 判断有没有真实证据。
4. 判断会不会挡住录屏。
5. 判断卡片类型：`data_result_card（数据成果卡）`、`judgment_card（判断卡）`、`summary_card（总结卡）`、`boundary_card（边界卡）`、`process_summary_card（流程总结卡）`。
6. 普通转场不加卡。
7. 容易被误解成已成功时，优先用 `boundary_card（边界卡）`。
8. 会遮挡真实证据时，必须 `blocked（阻断）` 或换位置。

允许做：
- 先写 `card_placement_decision（卡片位置决策）`。
- 给每张卡写 needed（需要）或 not_needed_reason（不需要理由）。

禁止做：
- 禁止用卡片替代真实证据。
- 禁止遮挡商品卡、表格、聊天框、关键按钮、关键数据。
- 禁止改 locked copy（锁定文案）语义。

做到哪算通过：
- 每张卡为什么出现、放哪里、证明什么、会不会遮挡，都写清。

必须阻断的情况：
- 卡片挡住核心证据。
- 卡片文字需要改 locked copy（锁定文案）语义。
- 素材没有证据，只想用卡片硬证明。

给用户看的例子：
口播说“这只是探测通过，还不是正式接入”，这适合 `boundary_card（边界卡）`。如果画面正在展示真实后台数据，就不能把卡片盖在数据上。

## 场景 3：RAG（检索增强生成）检索结果和仓库文件冲突怎么办

用户看到的现象：
RAG（检索增强生成）找出来一段摘要，但仓库原文件说法不同。

Codex 容易犯的错：
把向量结果直接写成项目事实，或者只看 `page_content（页面正文）` 不看来源路径。

正确处理顺序：
1. RAG（检索增强生成）只能辅助找资料。
2. 仓库原文件事实优先。
3. 检索结果必须有 `source_path（来源路径）`。
4. 检索结果必须做 `readback（原文回读）`。
5. 没有回读或回读冲突时，标记 `conflict（冲突）` 或 `blocked（阻断）`。

允许做：
- 用 RAG（检索增强生成）找候选资料。
- 用 `retrieval_manifest（检索清单）` 记录来源。
- 用 `source_readback（原文回读）` 核事实。

禁止做：
- 禁止把向量摘要直接写成项目事实。
- 禁止让 Chroma（本地向量库）替代 DashVector（阿里向量数据库）主路线。
- 禁止没有 `source_path（来源路径）` 就推进执行。

做到哪算通过：
- 检索清单、原文路径、当前分支回读、冲突判断都齐全。

必须阻断的情况：
- 只有正文，没有来源路径。
- 只有摘要，读不到原文件。
- 摘要和原文件冲突。

给用户看的例子：
RAG（检索增强生成）说“runtime（运行时）已启用”，但 latest（最新日志）写 `runtime_enabled（运行时启用）: false（未启用）`，就必须按 latest（最新日志）和原文件走。

## 场景 4：Codex 想改文案怎么办

用户看到的现象：
Codex（写入执行器）觉得原文太长、画面支撑不了，想顺手改稿。

Codex 容易犯的错：
直接改标题、改开头、改核心判断，最后视频和用户确认的文案不是一回事。

正确处理顺序：
1. 先检查 `locked_copy（锁定文案）` 是否存在。
2. Codex（写入执行器）只能改标点、换行、字幕分句、TTS（语音合成）停顿这类不改变语义的内容。
3. 如果语义、标题、核心判断、开头句需要改，输出 `copy_change_request（文案修改请求）`。
4. ChatGPT（总控脑）或用户确认后才能改语义。

允许做：
- 调整字幕断句。
- 调整画面映射。
- 提出文案修改请求。

禁止做：
- 禁止擅自改 locked copy（锁定文案）。
- 禁止把视觉标题卡标题改成另一个意思。
- 禁止为了素材方便而改掉核心判断。

做到哪算通过：
- 保留原文语义，或有明确 `copy_change_request（文案修改请求）` 和用户 / ChatGPT（总控脑）确认。

必须阻断的情况：
- 不改文案就无法执行，但又没有确认。
- 素材支撑不了原文核心判断。

给用户看的例子：
原文说“这不是正式接入”，Codex（写入执行器）不能改成“已经接入成功”。如果想改成更顺口，必须先请求确认。

## 场景 5：素材不能证明口播怎么办

用户看到的现象：
口播说 A，但录屏里只能看到一个相近场景，不能直接证明 A。

Codex 容易犯的错：
用主题相近素材硬剪，或者让观众自己猜画面和口播的关系。

正确处理顺序：
1. 不能硬剪。
2. 不能用主题相近素材冒充证据。
3. 判断能不能退回改文案。
4. 判断能不能补素材。
5. 判断能不能改成 `boundary_card（边界卡）`。
6. 都不行就 `blocked（阻断）`。

允许做：
- 写素材缺口。
- 写补素材建议。
- 输出文案修改请求。
- 改成边界说明卡。

禁止做：
- 禁止用背景素材冒充直接证据。
- 禁止把 `background_context_only（背景上下文）` 当 `direct_evidence（直接证据）`。
- 禁止在证据缺失时生成视频完成结论。

做到哪算通过：
- 每个 line_group（句组）都有能支撑口播的素材、卡片或明确阻断。

必须阻断的情况：
- 数据句没有数据来源。
- 判断句没有证据也没有判断卡。
- 动作句没有动作画面。

给用户看的例子：
口播说“这个工具把流程缩短了一半”，画面必须有对比证据或数据成果卡。只有工具首页录屏不够。

## 场景 6：技术跑通但内容不行怎么办

用户看到的现象：
脚本能跑、文件能生成、probe（探测）通过，但视频不好看、口播不顺、证据不对。

Codex 容易犯的错：
把 `technical_validation（技术验证）` 写成 `content_validation（内容验证）`。

正确处理顺序：
1. 技术通过只能说明链路能跑。
2. 内容、人感、画面、声音、证据、字幕、卡片都要单独复审。
3. 如果内容不行，回质量复审或素材 / 文案层，不写完成。

允许做：
- 写 technical_validation（技术验证）结果。
- 写 remaining_blockers（剩余阻断）。
- 写 next_safe_step（下一步安全动作）。

禁止做：
- 禁止把技术通过写成内容通过。
- 禁止把文件存在写成任务完成。
- 禁止把内部诊断产物写成用户交付物。

做到哪算通过：
- 技术状态、内容状态、人工复审状态分开写。

必须阻断的情况：
- 用户未确认内容和可发送状态。
- 只有技术预览或静态报告。

给用户看的例子：
no-render probe（无渲染探测）通过，只说明规则能检查；不说明真实视频剪出来好看。

## 场景 7：什么时候必须人工确认

用户看到的现象：
某一步涉及声音、审美、文案语义、降级方案或发布状态。

Codex 容易犯的错：
用自己的判断替用户拍板，直接推进 `send_ready（可发送状态）`。

正确处理顺序：
1. 声音最终选择必须人工确认。
2. 审美判断必须人工确认。
3. 文案语义改动必须人工确认。
4. 降级方案必须人工确认。
5. 发布前 `send_ready（可发送状态）` 必须人工确认。
6. 任何改变目标或状态的操作都必须人工确认。

允许做：
- 生成候选方案。
- 生成对比说明。
- 标记 `pending_user_review（等待用户复审）`。

禁止做：
- 禁止自动通过人审。
- 禁止默认接受降级方案。
- 禁止替用户确认发布。

做到哪算通过：
- 人审项、确认人、确认结论、未确认时的阻断状态写清。

必须阻断的情况：
- 需要用户审美判断但缺反馈。
- 需要用户确认声音但缺试听。
- 需要用户授权降级但未授权。

给用户看的例子：
Codex（写入执行器）可以说“这张卡片我认为可读”，但不能说“用户已确认审美通过”，除非用户真的确认。

## 场景 8：什么时候允许进入 runtime（运行时）

用户看到的现象：
前面已有 adapter（适配器）候选、probe（探测）、service boundary（服务边界），想知道能不能直接开 runtime（运行时）。

Codex 容易犯的错：
看到 main（主分支）已有候选代码，就写成 runtime（运行时）可启用。

正确处理顺序：
1. 先完成 `engineering_state_map（工程状态地图）`。
2. 再完成 `State / Node / Edge（状态 / 节点 / 边）`。
3. 再完成 `RAG / Tool / Retriever（检索增强生成 / 工具 / 检索器）`。
4. 再完成 `Evaluator / Failure Route（评估器 / 失败路由）`。
5. 再完成 `Trace / Report / Log（追踪 / 报告 / 日志）`。
6. 需要安装依赖、端口、secret（密钥）、外部 API（外部接口）、TTS（语音合成）、DashVector（阿里向量数据库）、Chroma（本地向量库）、media（媒体）时，必须另开授权任务。
7. runtime（运行时）不能直接写 repo（仓库）。

允许做：
- 做隔离 no-write runtime probe（不写仓库运行时探测）。
- 做 handoff（交接）验证。
- 写阻断报告。

禁止做：
- 禁止未授权安装依赖。
- 禁止打开端口。
- 禁止 runtime（运行时）直接写仓库。
- 禁止调用外部 API（外部接口）或生成媒体。

做到哪算通过：
- 前置 5 个任务切片都有通过证据，runtime（运行时）只在隔离、不写仓库、不外呼的边界内探测。

必须阻断的情况：
- 需要 secret（密钥）。
- 需要真实外部 API（外部接口）。
- 需要真实 DashVector（阿里向量数据库）或 Chroma（本地向量库）。
- runtime（运行时）想写仓库、commit（提交）或 push（推送）。

给用户看的例子：
如果只是本地 in-process probe（进程内探测）通过，只能说“候选链能被调用”，不能说“runtime（运行时）已经正式上线”。
