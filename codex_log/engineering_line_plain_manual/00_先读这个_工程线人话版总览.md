# 00｜先读这个：工程线人话版总览

## 这份手册解决什么问题

上一轮 Codex（写入执行器）生成了工程线审计报告，但那份报告更像工程师看的地图：里面有 `adapter（适配器）`、`runtime（运行时）`、`RAG（检索增强生成）`、`Tool Registry（工具注册表）`、`Evaluator（评估器）`、`Trace（追踪）` 这些词。

这份手册把它翻译成人话：现在项目到底是什么状态，哪些东西只是候选，哪些东西只是探测过，哪些地方缺口最大，用户后续要怎么下发需求，Codex（写入执行器）才不会乱改。

## 现在项目处在哪

当前结论可以先记住一句：

> `adapter（适配器）` 候选已经合并进 `main（主分支）`，但工程线还没有正式跑起来。

更具体一点：

- `adapter（适配器）` 候选代码已经进入 `main（主分支）`。
- `runtime（运行时）` 没有正式启用。
- `service（服务）` 没有正式启动。
- `RAG（检索增强生成）` 没有接入正式运行链。
- `Tool（工具）` 没有进入正式工具执行链。
- `DashVector（阿里向量数据库）` 本轮没有真实调用。
- `Chroma ingestion（Chroma 入库）` 本轮没有运行。
- `TTS（语音合成）` 本轮没有调用。
- `media（媒体）` 本轮没有生成。
- `content_validation（内容验证）` 没有推进。
- `send_ready（可发送状态）` 没有开启。
- `production_readiness（生产可用状态）` 没有声称成立。

所以，当前只是“工程线审计完成 + 后续修复路线图清楚”，不是“工程线改造完成”。

## 为什么不能直接修 RAG（检索增强生成）或 runtime（运行时）

因为现在最大的问题不是某个单点工具没接上，而是状态地图还没统一。

如果不先区分下面这些状态，后面很容易继续把“探测过”误写成“完成”：

- `formal（正式）`：已经进入正式项目入口，可以当当前事实使用。
- `candidate（候选）`：看起来可用，但还不能当正式入口。
- `probe_only（仅探测）`：只证明某个本地探测能跑，不代表正式可用。
- `documented_only（仅文档）`：只是写了方案，还没有代码或验证。
- `missing（缺失）`：这一层还没有实现。
- `conflict（冲突）`：新旧口径或文件结论互相打架，必须先裁决。

如果现在直接修 `RAG（检索增强生成）` 或 `runtime（运行时）`，Codex（写入执行器）可能会把 `probe_only（仅探测）` 当成 `formal（正式）`，把报告里写过的方案当成已经执行完成。这样后面越接越乱。

## 下一步先修什么

正确顺序是：

1. 先修 `engineering_state_map（工程状态地图）`。
2. 再修 `acceptance_contract（验收契约）`。
3. 然后才考虑 `State / Node / Edge（状态 / 节点 / 边）`。
4. 再往后才是 `RAG（检索增强生成） / Retriever（检索器） / Tool Registry（工具注册表）`。
5. 最后才轮到隔离的 `runtime（运行时）` 加固。

也就是说：

- 不是先修 `RAG（检索增强生成）`。
- 不是先修 `runtime（运行时）`。
- 不是先启动 `service（服务）`。
- 不是先跑真实媒体生成。

第一步必须让每个模块都能被标成 `formal（正式）`、`candidate（候选）`、`probe_only（仅探测）`、`documented_only（仅文档）`、`missing（缺失）` 或 `conflict（冲突）`，并且每个状态都要有证据路径和阻断条件。

## 用户应该怎么看这套手册

如果你看不懂工程审计报告，先看这套手册。

建议流程是：

1. 先看本文件，确认“现在项目还没正式启用工程运行线”。
2. 再看 `01_工程线人话版施工图.md`，理解未来工程线怎么接。
3. 遇到具体问题，看 `02_常见场景决策手册.md`。
4. 你想改项目时，用 `03_以后修改下发模板.md` 里的模板先描述需求。
5. 你先和 ChatGPT（总控脑）聊清楚。
6. ChatGPT（总控脑）确认方案后，再让 Codex（写入执行器）按 `Task Slice（任务切片）` 执行。

## 一句话结论

当前不要急着让 Codex（写入执行器）“把 RAG（检索增强生成）接上”或“把 runtime（运行时）跑起来”。先让 Codex（写入执行器）补 `engineering_state_map（工程状态地图）` 和 `acceptance_contract（验收契约）`，否则后面会继续混淆“候选、探测、文档、正式”这几层。
