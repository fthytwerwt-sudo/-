# 01｜工程线人话版施工图

## 先看整条线

```text
用户提出一个需求
-> 先判断任务类型
-> 判断当前模块状态
-> 决定走哪条工程线
-> 如果需要资料，就走 RAG（检索增强生成） / Retriever（检索器）
-> 如果需要工具，就走 Tool Registry（工具注册表）
-> 如果需要卡片，就走 card_placement_decision（卡片位置决策）
-> 如果失败，就走 Failure Route（失败路由）
-> 如果需要人判断，就走 Human-in-the-loop（人工介入）
-> 最后写 Report（报告） / Trace（追踪） / Log（日志）
```

这条线的重点不是“把所有工具都打开”，而是每一步先回答：当前状态是什么，允许做什么，失败后回哪一层，谁有最终判断权。

## Goal / Milestone / Acceptance（目标 / 里程碑 / 验收）

它是干什么的：
把“我要做什么”拆成目标、阶段和完成标准，防止 Codex（写入执行器）只完成局部文件就说完成。

现在状态：
`documented_only（仅文档） / partial（部分成立）`。审计报告里已经写了路线图，但还没有统一的机器可检查验收契约。

现在不能误解成什么：
不能误解成 `formal（正式）` 完成，也不能把“路线图写了”当成“工程改造已完成”。

以后怎么修：
1. 建 `engineering_state_map（工程状态地图）`。
2. 建 `acceptance_contract（验收契约）`。
3. 每个 `Task Slice（任务切片）` 都写清 `done_when（做到哪算完成）` 和 `blocked（阻断）` 条件。

## Workflow（工作流）

它是干什么的：
判断用户这轮需求应该进哪条工作流，例如文案、素材证据、审美剪辑、质量复审、数据复盘或机制修补。

现在状态：
正式 6 类 `Workflow（工作流）` 是 `formal（正式）`；adapter（适配器）里的 6 个候选 workflow 是 `candidate（候选）`。

现在不能误解成什么：
不能把 adapter（适配器）候选 workflow 当成已经替代正式 workflow，也不能新增一个 workflow 就直接写进正式入口。

以后怎么修：
1. 把正式 workflow 和 adapter（适配器）候选 workflow 做一张状态对齐表。
2. 给每个 workflow 写 evidence path（证据路径）。
3. 缺状态或缺证据时必须 `blocked（阻断）`。

## State / Node / Edge（状态 / 节点 / 边）

它是干什么的：
把一个任务拆成可走的图：`State（状态）` 保存当前信息，`Node（节点）` 做一步判断或验证，`Edge（边）` 决定下一步走哪。

现在状态：
`probe_only（仅探测）`。已有 fake graph（假图探测）和 no-service probe（无服务探测），但没有正式 `LangGraph（图式工作流框架）` 运行图。

现在不能误解成什么：
不能把 fake graph（假图探测）写成真实 `runtime（运行时）` 已启用。

以后怎么修：
1. 先定义 `State schema（状态结构）`。
2. 再定义 `Node / Edge contract（节点 / 边契约）`。
3. 仍然用 no-service probe（无服务探测）验证，不启动服务。

## RAG（检索增强生成） / Retriever（检索器） / Vector Store（向量库）

它是干什么的：
帮助找资料。`Retriever（检索器）` 负责查，`Vector Store（向量库）` 负责存索引，`RAG（检索增强生成）` 负责把检索结果带回任务。

现在状态：
`partial（部分成立） / documented_only（仅文档）`。有 DashVector（阿里向量数据库）最小 smoke（烟测）和 readback（回读）证据，但 adapter（适配器） workflow 还没有正式接 Retriever（检索器）。

现在不能误解成什么：
不能把向量检索结果当仓库事实。没有 `source_path（来源路径）` 和 `readback（原文回读）` 的结果不能直接写进项目结论。

以后怎么修：
1. 先做 `retrieval_manifest（检索清单）` 适配。
2. 强制 `source_readback（原文回读）`。
3. 后续再单独授权真实 DashVector（阿里向量数据库）或 Chroma（本地向量库）调用。

## Tool Registry（工具注册表）

它是干什么的：
列清楚哪些工具能用、什么时候能用、输入输出是什么、哪些动作绝对禁止。

现在状态：
`missing（缺失）`。上游有工具层概念，但本仓库 adapter（适配器）里还没有正式工具注册表。

现在不能误解成什么：
不能靠 prompt（提示词）临场说“用某个工具”，也不能让工具绕过 Codex（写入执行器）直接写仓库。

以后怎么修：
1. 先做只读 `Tool Registry schema（工具注册表结构）`。
2. 给每个工具写 allowed action（允许动作）和 forbidden action（禁止动作）。
3. 真实工具调用必须另开授权任务。

## Evaluator（评估器）

它是干什么的：
判断某一步是否真的通过，比如是否误把技术通过写成内容通过，是否把 RAG（检索增强生成）摘要当事实。

现在状态：
`partial（部分成立） / probe_only（仅探测）`。已有 completion truth guard（完成真实性护栏）和 blocked fixture（阻断样例），但还没有统一 Evaluator Registry（评估器注册表）。

现在不能误解成什么：
不能把“某个 probe（探测）通过”当成所有 workflow（工作流）都通过。

以后怎么修：
1. 做 `evaluator_result schema（评估结果结构）`。
2. 每条 workflow（工作流）绑定自己的评估器。
3. 失败时进入 Failure Route（失败路由）。

## Failure Route（失败路由）

它是干什么的：
失败时决定回哪一层：回目标层、文案层、素材层、RAG（检索增强生成）层、工具层、人工确认层，还是直接 `blocked（阻断）`。

现在状态：
`partial（部分成立）`。现在有很多 `blocked_if（阻断条件）`，但缺一张统一失败路由表。

现在不能误解成什么：
不能只写“失败了”，也不能让 Codex（写入执行器）自己猜下一步修哪里。

以后怎么修：
1. 给每种失败写 `failure_owner（失败归属）`。
2. 给每种失败写 `next_safe_step（下一步安全动作）`。
3. 缺证据路径时必须 `blocked（阻断）`。

## Human-in-the-loop（人工介入）

它是干什么的：
遇到审美、声音、语义、降级方案、发布前确认等问题时，必须让人判断。

现在状态：
`documented_only（仅文档） / fixture（样例）`。有人工复审结构，但还没有真实 runtime（运行时）里的中断和恢复机制。

现在不能误解成什么：
不能把“需要人工确认”自动跳过，更不能让 Codex（写入执行器）替用户确认 `send_ready（可发送状态）`。

以后怎么修：
1. 做 `human_review_handoff（人工复审交接件）`。
2. 每个人审点写清必须由谁确认。
3. 没有人审结论时保持 `pending_user_review（等待用户复审）` 或 `blocked（阻断）`。

## Guardrails（护栏）

它是干什么的：
防止越权：不写仓库、不误推进状态、不调用未授权外部接口、不把探测写成完成。

现在状态：
`partial（部分成立）`。已有 service no-write（服务不写仓库）、completion truth（完成真实性检查）和状态扫描，但还比较分散。

现在不能误解成什么：
不能因为某个护栏在一个脚本里跑过，就认为所有任务都自动安全。

以后怎么修：
1. 建 `guardrail_runner（护栏运行器）`。
2. 汇总 no-write（不写仓库）、status promotion scan（状态推进扫描）、secret scan（密钥扫描）。
3. 每个任务收尾都跑同一套检查。

## Checkpoint（检查点） / Trace（追踪） / Report（报告） / Log（日志）

它是干什么的：
让每一步能回看：走到哪里、读了什么、为什么阻断、报告在哪、Git 是否同步。

现在状态：
`schema_only（只有结构） / partial（部分成立）`。有 report（报告）和 latest log（最新日志），但没有统一 trace event（追踪事件）和 checkpoint store（检查点存储）。

现在不能误解成什么：
不能把 `memory（运行时记忆）` 当项目事实，仓库原文件和日志才是事实源。

以后怎么修：
1. 先做本地 `trace_event JSONL（追踪事件 JSONL）`。
2. 再做 report validator（报告校验器）。
3. 不接外部 telemetry（遥测服务），除非后续单独授权。

## Runtime（运行时）

它是干什么的：
让工程链条能真正按图执行，而不是每轮只靠人读报告和手动跑 probe（探测）。

现在状态：
`missing（缺失） / probe_only（仅探测）`。main（主分支）里有候选入口和 in-process probe（进程内探测），但没有正式 runtime（运行时）部署。

现在不能误解成什么：
不能把 in-process probe（进程内探测）写成 service（服务）已部署，也不能让 runtime（运行时）直接写 repo（仓库）。

以后怎么修：
1. 前 5 个 `Task Slice（任务切片）` 通过后，再做 isolated runtime hardening（隔离运行时加固）。
2. runtime（运行时）只能 route（路由）、validate（验证）、block（阻断）、handoff（交接）。
3. 真实写仓库仍然交给 `active_write_executor = codex（当前激活写入执行器为 Codex）`。
