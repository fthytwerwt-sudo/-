# 20260621 用户可读 RAG 决策工程线审查指南

## 1. 这次到底在收什么口

这次不是实现第二轮代码，也不是修完 DashVector 同步。它只是把 Round 1 设计报告补成用户可审查状态：设计报告不能再留 Git pending 占位，第二轮 validator 清单必须完整，用户需要一份不用读完整技术报告也能判断是否放行的审查指南。

## 2. 为什么现在同步慢

当前检测层已经能识别只有 23 个可索引文件变化，但执行层仍疑似拿完整 `chunk_manifest` 跑 embedding / upsert。最新分块清单约 5657 个 chunks，所以即使只改了少量文件，外部同步仍可能像全量任务一样慢。

核心判断：

```yaml
sync_slow_reason:
  changed_indexable_file_count: 23
  latest_chunk_manifest_chunk_count: 5657
  detection_layer: file_delta_detected
  execution_layer: still_full_manifest_embedding_upsert_suspected
  current_vector_sync_status: blocked
```

## 3. 为什么只修增量还不够

只修增量可以让同步变快，但不能解决旧口径污染 RAG 召回的问题。旧文档、历史计划、目标态草案如果没有权威层级、过期状态和冲突组裁决，仍可能被 Codex 当成当前事实。

所以第二轮必须同时处理两条线：

```yaml
required_pair:
  true_incremental_vector_sync: 解决同步慢和外部 API 调用过多
  rag_authority_overlay: 解决旧口径、历史日志、目标态计划污染当前判断
```

## 4. 用户审查时看哪六点

```yaml
review_focus:
  - 是否接受 active manifest allowlist + authority overlay post-filter 作为默认防污染策略
  - 是否接受第二轮先跑 dry-run/fake-client，再授权真实 DashVector delta sync
  - 是否接受旧 chunk 默认 tombstone/supersede，物理 delete 延后到 probe 通过后
  - 是否接受 hard_gate_before_weight，即硬闸门先于权重引擎
  - 是否接受高风险任务不能使用过期索引直接拍板
  - 是否接受每次关键决策输出 decision_audit_report，解释为什么选这个动作
```

## 6. 什么必须阻断

```yaml
blocked_if:
  - 第二轮想跳过 dry-run 直接真实 upsert
  - 第二轮想删除旧向量但没有 delete/metadata update probe 证据
  - 仍把 historical_log、target_state_plan、archive_only 当成 current_formal_fact
  - 使用过期索引处理高风险任务且没有原文回读
  - hard gate 失败后仍进入 weighted decision
  - 把 weighted decision engine 写成已经通过真实任务验证
  - 把向量同步 blocked 写成 RAG 最新
```

## 9. 用户需要拍板的三个点

```yaml
user_review_points:
  - 是否同意 active manifest allowlist + authority overlay post-filter 成为默认 RAG 供料过滤策略
  - 是否同意第二轮必须先 dry-run/fake-client，通过本地 validator 后再考虑真实 delta sync
```
