# DeepSeek 只读供料管线稳定性修复

## 1. 本轮定位

- `项目`: 《视频工厂》
- `目标`: 把 `DeepSeek readonly explorer` 从“最小 API 可用”推进到“能稳定给 Codex 提供资料包”
- `执行边界`: 不修改机制正文，不修改视频 / 声音 / 发布状态，不把 DeepSeek 写成生产执行通过

## 2. 脚本修复

- `已确认` `scripts/deepseek_readonly_explorer.py` 新增输入压缩能力
  - `MAX_CHARS_PER_CONTEXT_FILE = 6000`
  - `MAX_TOTAL_CONTEXT_CHARS = 18000`
- `已确认` 真实任务上下文改为自动抽取标题、开头和任务相关片段，不再默认整文件直塞
- `已确认` 新增 `3` 次受控重试：
  - `single_call_safe`
  - `compressed_retry`
  - `minimal_retry`
- `已确认` 每次尝试都会记录：
  - `attempt_index`
  - `failure_reason`
  - `prompt_size_chars`
  - `context_size_chars`
  - `finish_reason`
- `已确认` schema 校验已加强：
  - 顶层必须是 object
  - 四个顶层 key 必须存在
  - 顶层值不能是空字符串
  - 文件型任务里 `must_read_file_map` 不能空
  - `candidate_summary.summary` 不能为空
- `已确认` 新增 `local fallback`：
  - `deepseek_generation_status = failed`
  - `context_pack_validation = fallback_local_only`
  - `fallback_status = used`
  - fallback 只提供 Codex 最小资料，不等于 DeepSeek 结论

## 3. Smoke test 结果

结论：

DeepSeek 真实任务 JSON 输出仍不稳定，但 fallback 已能稳定给 Codex 产出一份可读资料包。

## 5. 供料价值判断

- `已确认` fallback 资料包包含：
  - `prefetch_context_pack`
  - `must_read_file_map`
  - `risk_and_conflict_report`
  - `candidate_summary`
- `已确认` fallback 能指出下一轮应优先读：
  - `codex_source/00_codex_readme.md`
  - `GPT数据源/10_OPC一人公司闭环与多AI协作机制.md`
  - `GPT数据源/06_当前主线锚点_API生成真人_用户录制素材_少量PPT_云端剪辑.md`
  - `GPT数据源/05_文案路由规则.md`
- `已确认` fallback 能提示旧 SOP 风险：
  - `fixed_material_anchor（固定素材锚点）` 单独阅读时可能被误解为固定 SOP
  - `locked reference` 的执行闸门可能被读成固定流程
  - `反 SOP 口径` 与 `历史强约束口径` 需要 Codex 下一轮做冲突辨析
